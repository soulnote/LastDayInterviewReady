
### Q1. What is serialization in Java?

**Simple me:**  
Serialization ka matlab hai **object ko byte stream me convert karna** taaki:
- Use network bhej sako (RMI, REST APIs, messaging)
- File me save kar sako (disk)
- Cache me store kar sako (Redis, etc.)

**Deserialization** ulta hai — byte stream se object wapas bana dena.

**How to do in Java:**
```java
class Student implements Serializable {
    int rollNo;
    String name;
}

// Serialize
ObjectOutputStream out = new ObjectOutputStream(new FileOutputStream("file.ser"));
out.writeObject(student);

// Deserialize
ObjectInputStream in = new ObjectInputStream(new FileInputStream("file.ser"));
Student s = (Student) in.readObject();
```

**SDE-2 level point:**  
Serialization me constructor nahi chalta. Object directly byte stream se recreate hota hai.  
Agar tum sensitive data (password, credit card) store kar rahe ho to serialization vulnerable ho sakta hai — isliye external libraries (JSON, Protobuf) better hote hain.

### Q2. What is serialVersionUID and its importance in Java?

**Definition:**  
`serialVersionUID` ek unique ID hai jo class ka version identify karta hai. Har `Serializable` class ke paas ek hota hai.

**Importance (SDE-2 level):**  
Jab tum object serialize karte ho, to `serialVersionUID` bhi byte stream me store ho jata hai.  
Deserialize karte time, JVM compare karta hai:
- Agar class ka UID aur byte stream ka UID match kare → success
- Agar match nahi kare → `InvalidClassException`

**Kyun important hai?**  
Maan lo tumne object serialize kiya, phir class me field add/remove/rename kar di. Agar UID define nahi kiya to Java compiler apne hisaab se generate karega (class ka hash). Woh badal jayega → deserialize fail hoga.

**Best practice:**  
Hamesha explicitly define karo:
```java
private static final long serialVersionUID = 1L;
```

**Interview me bolna:** *"serialVersionUID version control karta hai. Agar class evolve ho rahi hai aur backward compatibility maintain karni hai, to explicit UID define karo."*

### Q3. What is purpose of Transient Keyword in Java?

**Simple me:**  
`transient` keyword se aap JVM ko bolte ho — *"Is field ko serialize mat karna"*

**Example:**
```java
class User implements Serializable {
    String username;
    transient String password;  // password serialize nahi hoga
}
```

**Use cases:**
1. **Sensitive data** — password, PAN, credit card
2. **Derived fields** — jo calculate ho sakte hain (e.g., age from birthdate)
3. **Large temporary data** — cache, logs jo serialize nahi karna chahte
4. **Platform-specific data** — Thread, Socket, FileDescriptor (ye serializable nahi hote)

**SDE-2 level point:**  
Deserialize karte time transient fields default value lete hain (null, 0, false). Agar tumhe value restore karni hai, to `readObject()` method override karke custom logic likh sakte ho.

### Q4. Can we serialize static variables in Java?

**No — static variables serialize nahi hote.**

**Reason:**  
Serialization **object ke instance** ko save karta hai, class ke nahi. Static variables class level pe hote hain, object level pe nahi.

**Example:**
```java
class Employee implements Serializable {
    int id;
    static String companyName;  // Ye serialize nahi hoga
}
```

**Agar static variable serialize karna hai to kya kare?**  
Directly nahi ho sakta. Alternatives:
- `readObject()`/`writeObject()` override karke manually handle karo
- Static variable ko transient non-static bana ke rakho, phir manually manage karo

**SDE-2 level nuance:**  
Deserialize karte time static variable apni **current class value** leta hai, byte stream se nahi. Ye common mistake hoti hai.

### Q5. What happens if we don't define serialVersionUID?

**Java automatically generate karta hai** based on class structure — fields, methods, modifiers, etc.

**Problem (SDE-2 level):**  
Automatic UID **sensitive hota hai** — class me koi bhi minor change (ek field add karna, method rename karna) se UID badal jata hai.

**Consequences:**
1. **Backward compatibility break hoti hai** — purana serialize data new class version se deserialize nahi ho paega
2. **InvalidClassException** aayega
3. **Runtime pe pata chalta hai** — compile time nahi

**Real-world scenario:**  
Ek application running hai, usne objects serialize karke file me store kar di. Tumne class me ek getter method add kardi (minor change). Automatic UID badal gaya → ab purani file read nahi ho paegi → production me error.

**Best practice:**  
Hamesha explicit UID define karo, aur jab class evolve karo to UID update karne ka decision deliberately lo (backward compatibility chahiye to UID same rakho, nahi to badlao).

**Interview me bolna:**  
*"Agar UID define nahi kiya to Java runtime class structure ka hash generate karta hai. Yeh minor changes se bhi change ho jata hai, jo serialization backward compatibility tod deta hai. Isliye production systems me explicit UID define karte hain."*

### Bonus (SDE-2 level advanced points):

**Serialization vs Externalizable:**
- `Serializable` — automatic, but performance slower
- `Externalizable` — full control, `writeExternal()`/`readExternal()` override, faster but more code

**Inheritance and serialization:**
- Parent class `Serializable` hai → child automatically serializable
- Parent `Serializable` nahi hai → parent ke fields serialize nahi honge (default values aayengi)

**Custom serialization:**
```java
private void writeObject(ObjectOutputStream oos) throws IOException {
    oos.defaultWriteObject();  // default fields
    // custom logic
}

private void readObject(ObjectInputStream ois) throws IOException, ClassNotFoundException {
    ois.defaultReadObject();
    // custom reconstruction logic
}
```
