# **Maven - Complete Guide**

## **Q1. How to Add Libraries at Runtime Using Maven Plugins?**

### 🎯 **Understanding the Problem**

```yaml
Question: "How to add libraries at runtime using Maven plugins?"

Interpretation: Multiple scenarios could be asked:

1. Download dependencies dynamically during application execution
2. Load JARs from remote repository at runtime (not build time)
3. Use Maven plugins to manage dependencies in custom ways
4. Classpath manipulation at runtime
```

### 📌 **Scenario 1: Download Dependencies at Runtime (Not Build Time)**


```java
// Runtime Dependency Downloader
@Component
public class RuntimeDependencyLoader {
    
    @Autowired
    private RepositorySystem repositorySystem;
    
    @Autowired
    private RepositorySystemSession session;
    
    public void downloadAndLoadDependency(String groupId, String artifactId, String version) {
        // 1. Create artifact request
        Artifact artifact = new DefaultArtifact(groupId, artifactId, "jar", version);
        ArtifactRequest request = new ArtifactRequest();
        request.setArtifact(artifact);
        request.setRepositories(getRepositories());
        
        // 2. Download artifact
        ArtifactResult result = repositorySystem.resolveArtifact(session, request);
        File jarFile = result.getArtifact().getFile();
        
        // 3. Load into classloader at runtime
        loadJarIntoClasspath(jarFile);
        
        System.out.println("Dependency loaded: " + artifactId + ":" + version);
    }
    
    private void loadJarIntoClasspath(File jarFile) throws Exception {
        URLClassLoader classLoader = (URLClassLoader) ClassLoader.getSystemClassLoader();
        Method method = URLClassLoader.class.getDeclaredMethod("addURL", URL.class);
        method.setAccessible(true);
        method.invoke(classLoader, jarFile.toURI().toURL());
    }
}
```

### 📌 **Scenario 2: Using Maven Dependency Plugin to Copy JARs**

```xml
<!-- pom.xml - Copy dependencies to specific folder at build time -->
<build>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-dependency-plugin</artifactId>
            <version>3.6.0</version>
            <executions>
                <execution>
                    <id>copy-dependencies</id>
                    <phase>package</phase>
                    <goals>
                        <goal>copy-dependencies</goal>
                    </goals>
                    <configuration>
                        <outputDirectory>${project.build.directory}/libs</outputDirectory>
                        <overWriteReleases>false</overWriteReleases>
                        <overWriteSnapshots>false</overWriteSnapshots>
                        <overWriteIfNewer>true</overWriteIfNewer>
                    </configuration>
                </execution>
            </executions>
        </plugin>
    </plugins>
</build>
```

```java
// Runtime loading from pre-copied JARs
@Component
public class PluginClassLoader {
    
    @PostConstruct
    public void loadPlugins() throws Exception {
        File libDir = new File("libs");
        if (libDir.exists()) {
            for (File jar : libDir.listFiles((dir, name) -> name.endsWith(".jar"))) {
                loadJar(jar);
                System.out.println("Loaded plugin: " + jar.getName());
            }
        }
    }
    
    private void loadJar(File jar) throws Exception {
        URLClassLoader child = new URLClassLoader(
            new URL[]{jar.toURI().toURL()},
            this.getClass().getClassLoader()
        );
        
        // Use the loaded classes
        Class<?> clazz = child.loadClass("com.example.PluginClass");
        // ... use reflection to call methods
    }
}
```

### 📌 **Scenario 4: Custom ClassLoader for Dynamic Loading**

```java
// Complete runtime dependency management system
@Component
public class DynamicDependencyManager {
    
    private final Map<String, ClassLoader> moduleClassLoaders = new ConcurrentHashMap<>();
    private final String mavenLocalRepo = System.getProperty("user.home") + "/.m2/repository";
    
    public ClassLoader loadModule(String groupId, String artifactId, String version) 
            throws Exception {
        
        String key = groupId + ":" + artifactId + ":" + version;
        
        if (moduleClassLoaders.containsKey(key)) {
            return moduleClassLoaders.get(key);
        }
        
        // Find JAR file in local Maven repository
        String jarPath = mavenLocalRepo + "/" + 
            groupId.replace('.', '/') + "/" + 
            artifactId + "/" + 
            version + "/" + 
            artifactId + "-" + version + ".jar";
        
        File jarFile = new File(jarPath);
        
        if (!jarFile.exists()) {
            // Download if not present
            downloadDependency(groupId, artifactId, version);
        }
        
        // Create isolated classloader
        URLClassLoader classLoader = new URLClassLoader(
            new URL[]{jarFile.toURI().toURL()},
            Thread.currentThread().getContextClassLoader()
        );
        
        moduleClassLoaders.put(key, classLoader);
        return classLoader;
    }
    
    private void downloadDependency(String groupId, String artifactId, String version) {
        // Use Maven resolver to download
        // Or invoke Maven CLI: mvn dependency:get ...
        executeMavenCommand("dependency:get", 
            "-Dartifact=" + groupId + ":" + artifactId + ":" + version);
    }
    
    private void executeMavenCommand(String... args) {
        try {
            ProcessBuilder pb = new ProcessBuilder("mvn", args);
            pb.inheritIO();
            Process process = pb.start();
            process.waitFor();
        } catch (Exception e) {
            throw new RuntimeException("Failed to download dependency", e);
        }
    }
}

// Usage example
@Service
public class PluginExecutor {
    
    @Autowired
    private DynamicDependencyManager dependencyManager;
    
    public void executePlugin(String pluginName) throws Exception {
        // Load plugin JAR at runtime
        ClassLoader pluginLoader = dependencyManager.loadModule(
            "com.mycompany.plugins", 
            pluginName, 
            "1.0.0"
        );
        
        // Load and instantiate plugin class
        Class<?> pluginClass = pluginLoader.loadClass("com.mycompany.Plugin");
        Object plugin = pluginClass.getDeclaredConstructor().newInstance();
        
        // Execute plugin method
        Method executeMethod = pluginClass.getMethod("execute");
        executeMethod.invoke(plugin);
    }
}
```

### 📌 **Scenario 5: Using Maven Invoker Programmatically**

```java
// Programmatically run Maven to download dependencies
@Component
public class MavenInvoker {
    
    public void downloadDependencies(String pomPath) {
        Invoker invoker = new DefaultInvoker();
        invoker.setMavenHome(new File(System.getenv("MAVEN_HOME")));
        
        InvocationRequest request = new DefaultInvocationRequest();
        request.setPomFile(new File(pomPath));
        request.setGoals(Collections.singletonList("dependency:copy-dependencies"));
        
        InvocationResult result = invoker.execute(request);
        
        if (result.getExitCode() != 0) {
            throw new RuntimeException("Maven execution failed");
        }
    }
}
```

## **Q2. Different Types of Dependencies in Maven**

### 📊 **Complete Dependency Scope Reference**

```xml
<!-- Maven has 6 dependency scopes -->
<dependency>
    <groupId>org.springframework</groupId>
    <artifactId>spring-core</artifactId>
    <version>6.0.0</version>
    <scope>compile</scope>  <!-- Default scope -->
</dependency>
```

### 🎯 **Scope Comparison Table**

| Scope | Compile Classpath | Test Classpath | Runtime Classpath | Packaged | Example |
|-------|------------------|----------------|-------------------|----------|---------|
| **compile** (default) | ✅ | ✅ | ✅ | ✅ | Spring Core, Commons Lang |
| **provided** | ✅ | ✅ | ❌ | ❌ | Servlet API, Lombok |
| **runtime** | ❌ | ✅ | ✅ | ✅ | JDBC Driver |
| **test** | ❌ | ✅ | ❌ | ❌ | JUnit, Mockito |
| **system** | ✅ | ✅ | ❌ | ❌ | System JAR (discouraged) |
| **import** | Special - only for pom type dependencies | | | | BOM imports |

### 📝 **Detailed Explanation with Examples**

```xml
<!-- 1. COMPILE Scope (Default) - Available everywhere -->
<dependency>
    <groupId>org.apache.commons</groupId>
    <artifactId>commons-lang3</artifactId>
    <version>3.14.0</version>
    <scope>compile</scope>  <!-- Can omit as this is default -->
</dependency>
<!-- When to use: Core business logic dependencies -->
<!-- Available in: main code, test code, packaged JAR -->


<!-- 2. PROVIDED Scope - Expected to be provided by runtime environment -->
<dependency>
    <groupId>jakarta.servlet</groupId>
    <artifactId>jakarta.servlet-api</artifactId>
    <version>6.0.0</version>
    <scope>provided</scope>
</dependency>
<!-- When to use: 
     - Servlet API (provided by Tomcat/Jetty)
     - Lombok (processed at compile time)
     - Java EE APIs
-->
<!-- Not packaged in final JAR/WAR -->


<!-- 3. RUNTIME Scope - Needed at runtime but not for compilation -->
<dependency>
    <groupId>org.postgresql</groupId>
    <artifactId>postgresql</artifactId>
    <version>42.7.1</version>
    <scope>runtime</scope>
</dependency>
<!-- When to use:
     - JDBC drivers
     - Logging implementations
     - Dynamic dependencies loaded via reflection
-->
<!-- Available in: test and runtime, NOT during compilation -->


<!-- 4. TEST Scope - Only for testing -->
<dependency>
    <groupId>org.junit.jupiter</groupId>
    <artifactId>junit-jupiter</artifactId>
    <version>5.10.1</version>
    <scope>test</scope>
</dependency>
<dependency>
    <groupId>org.mockito</groupId>
    <artifactId>mockito-core</artifactId>
    <version>5.8.0</version>
    <scope>test</scope>
</dependency>
<!-- When to use:
     - Unit testing frameworks
     - Mocking libraries
     - Test utilities
-->
<!-- NOT available in main code or packaged JAR -->


<!-- 5. SYSTEM Scope - Discouraged! Use repository instead -->
<dependency>
    <groupId>com.custom</groupId>
    <artifactId>custom-lib</artifactId>
    <version>1.0</version>
    <scope>system</scope>
    <systemPath>${project.basedir}/lib/custom-lib.jar</systemPath>
</dependency>
<!-- ⚠️ DISCOURAGED: Breaks reproducibility, not recommended -->


<!-- 6. IMPORT Scope - For BOM (Bill of Materials) -->
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-dependencies</artifactId>
            <version>3.2.0</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
<!-- When to use:
     - Importing Spring Boot BOM
     - Managing versions across multiple modules
-->
```

### 💻 **Practical Examples**


### 🎯 **When to Use Each Scope - Decision Guide**

```yaml
Decision Tree for Scope Selection:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Start: Is this needed at compile time?
  │
  ├─ YES → Is it provided by runtime environment?
  │         ├─ YES → scope="provided" (Servlet API, Lombok)
  │         └─ NO  → scope="compile" (Default)
  │
  └─ NO → Is it only for testing?
           ├─ YES → scope="test" (JUnit, Mockito)
           └─ NO  → scope="runtime" (JDBC Driver)

Special Cases:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
• Multi-module projects → Use <dependencyManagement>
• BOM imports → scope="import" with type="pom"
• System libraries → AVOID scope="system"
```

### 💡 **Common Mistakes & Solutions**

```xml
<!-- ❌ MISTAKE 1: Putting test dependencies in compile scope -->
<dependency>
    <groupId>junit</groupId>
    <artifactId>junit</artifactId>
    <version>4.13.2</version>
    <!-- Missing scope="test" → Will be packaged! -->
</dependency>
<!-- ✅ FIX: Add test scope -->

<!-- ❌ MISTAKE 2: Including provided dependencies in WAR -->
<dependency>
    <groupId>jakarta.servlet</groupId>
    <artifactId>jakarta.servlet-api</artifactId>
    <scope>compile</scope>  <!-- Wrong! -->
</dependency>
<!-- ✅ FIX: Use provided scope -->

<!-- ❌ MISTAKE 3: Using system scope -->
<dependency>
    <scope>system</scope>
    <systemPath>${basedir}/lib/custom.jar</systemPath>
</dependency>
<!-- ✅ FIX: Install to local repo or use repository -->
```

### 📋 **Quick Reference Card**

```yaml
Dependency Scopes Quick Guide:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

compile (default):
  - Use for: Business logic, utilities
  - Packaged: YES
  - Example: Spring Core, Apache Commons

provided:
  - Use for: Container/runtime provided
  - Packaged: NO
  - Example: Servlet API, Lombok

runtime:
  - Use for: Runtime only, not compile
  - Packaged: YES
  - Example: JDBC Drivers

test:
  - Use for: Testing only
  - Packaged: NO
  - Example: JUnit, Mockito

import:
  - Use for: BOM imports
  - Packaged: N/A
  - Example: Spring Boot Dependencies

system:
  - Use for: AVOID! Use repository instead
  - Packaged: NO
  - Example: N/A (deprecated)
```

### 🚀 **Best Practices Summary**

```yaml
1. Always specify scope explicitly (don't rely on defaults)
2. Use provided for container-specific dependencies
3. Use test scope for all testing dependencies
4. Avoid system scope completely
5. Use dependencyManagement for version control
6. Regularly audit dependencies (mvn dependency:analyze)
7. Exclude unnecessary transitive dependencies
8. Keep dependencies up to date
```

**Remember:** Wrong scope = Bloated JARs, ClassNotFoundExceptions, or runtime failures! 🎯
