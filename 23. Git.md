## **Q1. How to Merge Branches in Git & Types of Merges**

### 📌 **Basic Merge Command**

```bash
# Step 1: Switch to target branch (where you want to merge INTO)
git checkout main

# Step 2: Update main branch with latest changes
git pull origin main

# Step 3: Merge feature branch into main
git merge feature-branch

# Step 4: Push merged changes
git push origin main
```

### 🎯 **Types of Git Merges**

```yaml
1. Fast-Forward Merge (Linear)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
When: No new commits on target branch since feature branch created

Before:
main:     A---B---C
                \
feature:         D---E

After merge (fast-forward):
main:     A---B---C---D---E

Command: git merge feature (default when possible)

2. Three-Way Merge (Non-Fast-Forward)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
When: Both branches have new commits

Before:
main:     A---B---C---F---G
                \
feature:         D---E

After merge (creates merge commit):
main:     A---B---C---F---G---M
                \           /
                 D---------E

Command: git merge feature (creates merge commit)

3. Squash Merge
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
When: Want to combine all feature commits into one

Before:
feature:  D---E---F (3 commits)

After squash:
main:     A---B---C---S (1 commit with all changes)

Command: git merge --squash feature

4. Octopus Merge
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
When: Merging multiple branches at once

Command: git merge branch1 branch2 branch3
```

### 💻 **Practical Examples**

```bash
# ========== FAST-FORWARD MERGE ==========
# Create feature branch
git checkout -b feature/login
# Make changes, commit
echo "login code" > login.html
git add .
git commit -m "Add login feature"

# Switch to main (no new commits on main)
git checkout main

# Fast-forward merge happens automatically
git merge feature/login
# Output: Updating a1b2c3d..e4f5g6h
#         Fast-forward
#          login.html | 1 +
#          1 file changed, 1 insertion(+)

# ========== THREE-WAY MERGE (with merge commit) ==========
git checkout -b feature/payment
echo "payment code" > payment.html
git add .
git commit -m "Add payment feature"

# Meanwhile, someone else commits to main
git checkout main
echo "README update" > README.md
git add .
git commit -m "Update README"

# Now merge (creates merge commit)
git merge feature/payment
# Output: Merge made by the 'recursive' strategy.
# Creates merge commit with message: "Merge branch 'feature/payment'"

# ========== FORCE NO-FAST-FORWARD (always create merge commit) ==========
git merge --no-ff feature/payment

# ========== SQUASH MERGE ==========
git checkout main
git merge --squash feature/login
git commit -m "Add login feature (squashed)"
```

## **Q2. Difference Between Git Pull and Git Fetch**

### 📊 **Comparison Table**

| Aspect | Git Fetch | Git Pull |
|--------|-----------|----------|
| **What it does** | Downloads changes only | Downloads + merges changes |
| **Working directory** | ❌ Does NOT change | ✅ Changes working directory |
| **Safety** | Safe (no automatic merge) | Can cause conflicts |
| **Use case** | See what changed first | Directly update local branch |
| **Command** | `git fetch origin` | `git pull origin main` |
| **Equivalent to** | Just download | Fetch + Merge |

### 💻 **Visual Explanation**

```bash
# ========== GIT FETCH ==========
# Before fetch:
Local main:     A---B---C
Remote main:    A---B---C---D---E

git fetch origin

# After fetch (downloaded but not merged):
Local main:     A---B---C
Remote tracking: A---B---C---D---E (origin/main)

# You can now see what changed:
git log origin/main --oneline
git diff main origin/main

# Then decide when to merge:
git merge origin/main  # or git rebase


# ========== GIT PULL ==========
# Before pull:
Local main:     A---B---C
Remote main:    A---B---C---D---E

git pull origin main

# After pull (downloaded AND merged):
Local main:     A---B---C---D---E

# If there are conflicts, they happen immediately
```

### 💡 **Best Practices**

```bash
# ✅ RECOMMENDED: Fetch first, review, then merge
git fetch origin
git log origin/main..HEAD  # See what's different
git diff origin/main       # See actual changes
git merge origin/main      # Merge when ready

# ❌ NOT RECOMMENDED: Pull without review (can cause unexpected conflicts)
git pull origin main

# ✅ SAFE PULL with rebase
git pull --rebase origin main  # Fetch + rebase instead of merge

# ✅ Check what pull will do before executing
git fetch origin
git log HEAD..origin/main --oneline  # See incoming commits
```

## **Q3. Difference Between Git Merge and Git Rebase**

### 📊 **Comparison Table**

| Aspect | Git Merge | Git Rebase |
|--------|-----------|------------|
| **History** | Preserves complete history | Rewrites history (linear) |
| **Merge commits** | Creates merge commit | No merge commits |
| **Branch structure** | Shows branching clearly | Linear history (looks like single branch) |
| **Conflict resolution** | Once (at merge time) | Multiple times (per commit) |
| **Safety on shared branches** | ✅ Safe | ❌ Dangerous (never rebase shared branches) |
| **Use case** | Public/shared branches | Private/feature branches |

### 💻 **Visual Example**

```bash
# Starting point:
# Both main and feature have commits

# ========== MERGE ==========
Before merge:
main:     A---B---C---F
              \
feature:       D---E

git checkout main
git merge feature

After merge:
main:     A---B---C---F---M
              \         /
               D-------E
# Result: Branching history preserved, merge commit 'M'


# ========== REBASE ==========
Before rebase:
main:     A---B---C---F
              \
feature:       D---E

git checkout feature
git rebase main

After rebase:
main:     A---B---C---F
                       \
feature:                D'---E'  (new commits!)

# Now fast-forward merge is possible
git checkout main
git merge feature

Final linear history:
main:     A---B---C---F---D'---E'
```

### 💻 **Practical Examples**

```bash
# ========== MERGE EXAMPLE ==========
# Create feature branch
git checkout -b feature/awesome
echo "feature work" > feature.txt
git add .
git commit -m "Add feature"

# Switch to main, add commits
git checkout main
echo "main work" > main.txt
git add .
git commit -m "Update main"

# Merge feature
git merge feature/awesome
# Creates merge commit: "Merge branch 'feature/awesome'"

# View history (shows branching)
git log --graph --oneline
# *   a1b2c3d Merge branch 'feature/awesome'
# |\  
# | * d4e5f6g Add feature
# * | b7c8d9e Update main
# |/  
# * e9f0g1h Initial commit


# ========== REBASE EXAMPLE ==========
# Same starting point
git checkout feature/awesome
git rebase main
# Output: First, rewinding head to replay your work on top of it...
# Applying: Add feature

# Now fast-forward merge
git checkout main
git merge feature/awesome
# Output: Fast-forward

# View history (linear!)
git log --graph --oneline
# * d4e5f6g Add feature
# * b7c8d9e Update main
# * e9f0g1h Initial commit
```

### ⚠️ **Golden Rule of Rebase**

```bash
# ✅ CORRECT: Rebase private/local branches ONLY
git checkout feature/my-work
git rebase main  # Safe - only you have this branch

# ❌ NEVER DO THIS: Rebase shared/public branches
git checkout main
git rebase feature/other  # DANGER! Others have this branch

# ❌ NEVER rebase a branch that others are working on
git push --force  # This will break everyone's work!

# ✅ SAFE alternative: Merge instead
git checkout main
git merge feature/other  # Safe for shared branches
```

## **Q4. How to Resolve Git Merge Conflict**

### 📋 **Step-by-Step Conflict Resolution**

```bash
# Step 1: Attempt merge (conflict occurs)
git merge feature-branch
# Output: Auto-merging app.js
# CONFLICT (content): Merge conflict in app.js
# Automatic merge failed; fix conflicts and then commit the result.

# Step 2: Check conflicted files
git status
# Output: both modified:   app.js

# Step 3: Open conflicted file
cat app.js
```

### 🔍 **Understanding Conflict Markers**

```javascript
// app.js - Conflicted file
<<<<<<< HEAD (Current branch - main)
function greet() {
    return "Hello from main branch!";
}
=======
function greet() {
    return "Hello from feature branch!";
}
>>>>>>> feature-branch (Incoming branch)

// After resolution (choose one or combine)
function greet() {
    return "Hello from both branches!";
}
```

### 🛠️ **Conflict Resolution Methods**

```bash
# ========== METHOD 1: Manual Resolution ==========
# Edit the file, remove conflict markers, keep desired code
# Then mark as resolved
git add app.js
git commit -m "Resolved merge conflict in app.js"

# ========== METHOD 2: Use One Side Only ==========
# Keep our version (HEAD/main branch)
git checkout --ours app.js
git add app.js

# Keep their version (feature branch)
git checkout --theirs app.js
git add app.js

# ========== METHOD 3: Using Merge Tool ==========
# Open visual merge tool
git mergetool

# Common tools: VSCode, IntelliJ, Meld, P4Merge
# Configure mergetool:
git config --global merge.tool vscode
git config --global mergetool.vscode.cmd "code --wait $MERGED"

# ========== METHOD 4: Abort merge ==========
# If too complicated, abort and try different strategy
git merge --abort
```

### 💻 **Complete Conflict Resolution Example**

```bash
# Scenario: Two branches modified same file

# Step 1: Start merge
git checkout main
git merge feature/login
# CONFLICT in login.html

# Step 2: View conflict
cat login.html
<<<<<<< HEAD
<div>Welcome User! (main)</div>
=======
<div>Welcome User! (feature)</div>
>>>>>>> feature/login

# Step 3: Edit file to resolve
vim login.html
# Keep: <div>Welcome User!</div>

# Step 4: Mark as resolved
git add login.html

# Step 5: Complete merge
git commit -m "Merge branch 'feature/login'"

# Step 6: Verify resolution
git log --oneline --graph
```

### 🚀 **Advanced Conflict Resolution**

```bash
# ========== Resolving Multiple Files ==========
# List all conflicted files
git diff --name-only --diff-filter=U

# Resolve all using ours/theirs
git checkout --ours .  # Keep all our changes
git checkout --theirs .  # Keep all their changes

# ========== Using Diff3 Style (Better visibility) ==========
git config --global merge.conflictstyle diff3
# Now conflicts show:
<<<<<<< HEAD
// Our changes
||||||| base
// Original code (very helpful!)
=======
// Their changes
>>>>>>>

# ========== Resolving Binary Files ==========
# For images, PDFs, etc. - choose one version
git checkout --ours logo.png
git add logo.png
git commit -m "Used our version of logo"

# ========== Preventing Future Conflicts ==========
# More frequent merges
# Smaller, focused commits
# Clear communication within team
```

## **Q5. How to Uncommit Changes Pushed to GitHub**

### 🔄 **Methods to Uncommit/Undo Pushed Changes**

```yaml
Method 1: git revert (SAFEST - Recommended)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
- Creates new commit that undoes changes
- History preserved
- Safe for shared branches
- No force push needed

Method 2: git reset (DANGEROUS - For private branches only)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
- Removes commits entirely
- Rewrites history
- Requires force push
- DANGEROUS if others have pulled

Method 3: git rebase -i (Interactive rebase)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
- Squash, edit, or drop commits
- Rewrites history
- Requires force push
```

### 💻 **Method 1: Git Revert (Safe for Shared Branches)**

```bash
# ========== Revert Last Commit ==========
git revert HEAD
# Opens editor for commit message
# Creates new commit: "Revert 'Previous commit message'"

git push origin main
# No force needed! Safe for team


# ========== Revert Specific Commit ==========
git log --oneline
# a1b2c3d Fix login bug
# d4e5f6g Add payment feature
# g7h8i9j Initial commit

# Revert the payment feature commit
git revert d4e5f6g
# Creates new commit that undoes d4e5f6g

git push origin main


# ========== Revert Multiple Commits ==========
# Revert last 3 commits
git revert HEAD~3..HEAD

# Revert range but keep them as separate revert commits
git revert -n HEAD~3..HEAD  # -n = no commit
git commit -m "Revert commits 1-3"
git push origin main


# ========== Revert with Specific Message ==========
git revert HEAD -m "Reverting due to critical bug"
git push origin main
```

### 💻 **Method 2: Git Reset (Dangerous - Private Branches Only)**

```bash
# ⚠️ WARNING: Only use on branches no one else is using!

# ========== Soft Reset (Keep changes in staging) ==========
git reset --soft HEAD~1
# Commit removed, changes are staged (ready to commit again)
git status  # Shows changes staged

# Fix something, then recommit
git commit -m "Fixed commit message"


# ========== Mixed Reset (Keep changes unstaged) ==========
git reset HEAD~1
# Commit removed, changes are unstaged (modified but not added)
git status  # Shows changes not staged for commit


# ========== Hard Reset (Discard changes completely) ⚠️ ==========
git reset --hard HEAD~1
# Commit AND changes are GONE! Cannot recover!
git status  # Working directory is clean


# ========== Force Push After Reset ==========
# After any reset on a pushed branch, you must force push
git push --force origin feature-branch

# Safer force push (fails if someone else pushed)
git push --force-with-lease origin feature-branch


# ========== Reset to Specific Commit ==========
git reset --hard a1b2c3d  # Go back to commit a1b2c3d
git push --force origin main
```

### 💻 **Method 3: Interactive Rebase**

```bash
# ========== Interactive Rebase to Remove/Edit Commits ==========
# Rebase last 3 commits
git rebase -i HEAD~3

# Opens editor with:
pick a1b2c3d Commit message 1
pick d4e5f6g Commit message 2
pick g7h8i9j Commit message 3

# Change 'pick' to:
# drop - Remove commit completely
# reword - Change commit message
# edit - Stop to amend commit
# squash - Combine with previous commit
# fixup - Combine, discard message

# Example: Remove middle commit
pick a1b2c3d Commit message 1
drop d4e5f6g Commit message 2  # This commit will be removed
pick g7h8i9j Commit message 3

# Save and exit
# Force push after rebase
git push --force-with-lease origin main
```

### 🆘 **Emergency Recovery**

```bash
# ========== If You Force Pushed Wrong Branch ==========
# Check reflog (your local history)
git reflog
# a1b2c3d HEAD@{0}: commit: Add feature
# d4e5f6g HEAD@{1}: commit: Fix bug
# g7h8i9j HEAD@{2}: commit: Initial commit

# Recover lost commit
git checkout d4e5f6g
git branch recover-branch

# Or reset back to that commit
git reset --hard d4e5f6g


# ========== Undo Git Revert ==========
# If you reverted by mistake, revert the revert!
git log --oneline
# x1y2z3w Revert "Add payment feature"  (the revert commit)
# d4e5f6g Add payment feature  (original commit)

# Revert the revert
git revert x1y2z3w
# This brings back the original change!
```

## **📚 Frequently Asked Git Questions**

### **Q6: What's the difference between git and GitHub?**

```yaml
Git:
  - Version control system (local)
  - Runs on your computer
  - Tracks changes, manages history
  - Command line tool

GitHub:
  - Cloud hosting service for Git repos
  - Adds collaboration features
  - PRs, Issues, Actions, Wiki
  - Remote repository hosting

Analogy:
  Git = Engine
  GitHub = Car that uses the engine
```

### **Q7: What is a Git stash?**

```bash
# Save uncommitted changes temporarily
git stash save "WIP: login feature"

# List stashes
git stash list
# stash@{0}: On main: WIP: login feature

# Apply latest stash (keep in stash)
git stash apply

# Apply and remove from stash
git stash pop

# Apply specific stash
git stash apply stash@{1}

# Create branch from stash
git stash branch new-feature

# Clear all stashes
git stash clear
```

### **Q8: What is cherry-pick in Git?**

```bash
# Apply specific commit from another branch
git checkout main
git cherry-pick a1b2c3d  # Apply commit a1b2c3d to main

# Cherry-pick multiple commits
git cherry-pick a1b2c3d d4e5f6g

# Cherry-pick range
git cherry-pick a1b2c3d..d4e5f6g

# Cherry-pick without committing
git cherry-pick -n a1b2c3d  # Stage changes only
```

### **Q9: How to undo git add before commit?**

```bash
# Unstage specific file
git reset HEAD app.js

# Unstage all files
git reset

# Unstage and keep changes
git restore --staged app.js

# Unstage all and discard changes (careful!)
git reset --hard HEAD
```

### **Q10: How to change last commit message?**

```bash
# Change last commit message
git commit --amend -m "New commit message"

# Add forgotten files to last commit
git add forgotten-file.js
git commit --amend --no-edit  # Keep same message

# Change older commit message (interactive rebase)
git rebase -i HEAD~3
# Change 'pick' to 'reword' for the commit
# Save, edit message, save
```

### **Q11: How to see commit history?**

```bash
# Standard log
git log

# One line per commit
git log --oneline

# Graph view
git log --graph --oneline --decorate

# Last N commits
git log -5

# Search by author
git log --author="Raj"

# Search by message
git log --grep="bug fix"

# Show changes
git log -p
```

### **Q12: What's the difference between origin/main and main?**

```yaml
main:
  - Local branch on your computer
  - You work on this

origin/main:
  - Remote tracking branch
  - Represents state of main on GitHub
  - Updated by git fetch/pull

# See difference
git log origin/main..main  # Commits you have that remote doesn't
git log main..origin/main  # Commits remote has that you don't
```

### **Q13: How to delete a branch?**

```bash
# Delete local branch
git branch -d feature/old  # Safe (merged)
git branch -D feature/old  # Force (even if not merged)

# Delete remote branch
git push origin --delete feature/old

# Delete remote tracking reference
git remote prune origin  # Remove stale remote branches
```

### **Q14: How to recover deleted branch?**

```bash
# Find deleted branch commit hash
git reflog
# a1b2c3d HEAD@{10}: checkout: moving from feature to main

# Recreate branch
git checkout -b feature/recovered a1b2c3d
```

### **Q15: What's a .gitignore file?**

```bash
# .gitignore - Specifies intentionally untracked files

# Examples:
*.class          # Ignore compiled Java files
/node_modules/   # Ignore node dependencies
.env             # Ignore environment variables
*.log            # Ignore log files
.DS_Store        # Ignore macOS system files

# Check why file is ignored
git check-ignore -v app.log

# Force add ignored file
git add -f config.env
```
