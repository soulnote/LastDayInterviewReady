
### Q1. What are Java Servlets?

**Simple me:**  
Servlet ek **Java class** hai jo server-side pe run hoti hai aur client (browser, mobile app, etc.) ke requests ko handle karti hai.

**Real-world analogy:**  
Restaurant me waiter — client order deta hai, waiter kitchen (server) tak order pahunchata hai aur response wapas laata hai.

**Technical definition:**  
Servlet ek **server-side component** hai jo:
- HTTP requests receive karta hai (GET, POST, PUT, DELETE)
- Business logic execute karta hai (database call, calculation)
- Dynamic response generate karta hai (HTML, JSON, XML)
- Client ko response bhejta hai

**Example:**  
```java
@WebServlet("/login")
public class LoginServlet extends HttpServlet {
    protected void doPost(HttpServletRequest req, HttpServletResponse resp) 
            throws ServletException, IOException {
        String username = req.getParameter("username");
        // Database validation logic
        resp.getWriter().write("Welcome " + username);
    }
}
```

**SDE-2 level point:**  
Servlets **thread-safe nahi hote** by default. Ek instance multiple threads ke beech share hota hai — isliye instance variables avoid karo, local variables use karo. Ya phir synchronization handle karo.

### Q2. Explain Servlet Life Cycle

Servlet container (Tomcat, Jetty, JBoss) servlet ke life cycle ko manage karta hai.

**5 stages (SDE-2 level depth me):**

```
[Unloaded] → [Loaded] → [Instantiated] → [Initialized] → [Ready to Serve] → [Destroyed]
```

**Step-by-step:**

| Stage | Method Called | Kab hota hai | Ek baar / Multiple baar |
|-------|--------------|--------------|-------------------------|
| 1. Loading | - | Container servlet class load karta hai (first request ya server start pe) | Ek baar |
| 2. Instantiation | Constructor | Class load hone ke baad object create hota hai | Ek baar |
| 3. Initialization | `init(ServletConfig config)` | Object create hone ke baad. Resources open karo (DB connection, file) | Ek baar |
| 4. Request Handling | `service(HttpServletRequest, HttpServletResponse)` | Har client request pe. Multiple threads call kar sakte hain | Multiple baar |
| 5. Destruction | `destroy()` | Server shutdown ya servlet reload pe. Resources release karo | Ek baar |

**Important nuances (SDE-2 level):**

```java
public class MyServlet extends HttpServlet {
    
    // Ek baar chalega - connection pool initialize karo
    @Override
    public void init(ServletConfig config) {
        // Thread-safe initialization
    }
    
    // Har request pe chalega - NOT thread-safe!
    @Override
    protected void service(HttpServletRequest req, HttpServletResponse resp) {
        // Use LOCAL variables only
        String data = req.getParameter("data");
        // Avoid instance variables
    }
    
    // Ek baar chalega - cleanup
    @Override
    public void destroy() {
        // Close DB connections, release resources
    }
}
```

**Interview me bolna:**  
*"Servlet life cycle container controlled hai. init() ek baar, service() multiple times (multithreaded), destroy() ek baar. Service method thread-safe nahi hai — isliye hum instance variables avoid karte hain."*

### Q3. Differences between GenericServlet vs HttpServlet

| Feature | GenericServlet | HttpServlet |
|---------|---------------|-------------|
| **Protocol** | Protocol-independent | HTTP protocol specific |
| **Abstract class** | Haan (implements Servlet, ServletConfig) | Haan (extends GenericServlet) |
| **Main method** | `service(ServletRequest req, ServletResponse resp)` | `service(HttpServletRequest req, HttpServletResponse resp)` + `doGet()`, `doPost()`, etc. |
| **Use case** | Custom protocols (FTP, SMTP, etc.) | Web applications (REST, web pages) |
| **Methods to override** | Sirf `service()` | `doGet()`, `doPost()`, `doPut()`, `doDelete()` |
| **Request/Response** | Generic `ServletRequest`/`ServletResponse` | HTTP specific (headers, cookies, session, methods) |
| **Practical usage** | Rare — almost never used | 99.9% use cases me yahi |

**Example comparison:**

```java
// GenericServlet - rare use
public class MyGenericServlet extends GenericServlet {
    public void service(ServletRequest req, ServletResponse res) {
        // Protocol independent logic
    }
}

// HttpServlet - common use
@WebServlet("/api/user")
public class UserServlet extends HttpServlet {
    protected void doGet(HttpServletRequest req, HttpServletResponse resp) {
        // GET request handle
    }
    protected void doPost(HttpServletRequest req, HttpServletResponse resp) {
        // POST request handle
    }
}
```

**SDE-2 level point:**  
GenericServlet abstract hai, usme sirf `service()` override karna padta hai. HttpServlet usko extend karta hai aur HTTP methods ke liye convenience methods provide karta hai (`doGet`, `doPost`, etc.). Aaj kal REST APIs me mostly `@WebServlet` annotation ke saath HttpServlet use hota hai.

### Q4. What is Servlet Chaining?

**Definition:**  
Servlet chaining ek technique hai jahan **multiple servlets ek pipeline me chain hote hain**. Ek servlet ka output doosre servlet ka input hota hai.

**How it works:**

```
Request → Servlet A → Servlet B → Servlet C → Response
           ↓            ↓            ↓
       Process 1    Process 2    Process 3
```

**Ways to achieve chaining:**

**1. Using `RequestDispatcher` (most common):**
```java
public class FirstServlet extends HttpServlet {
    protected void doGet(HttpServletRequest req, HttpServletResponse resp) {
        // Some processing
        req.setAttribute("data", "processed data");
        
        // Chain to next servlet
        RequestDispatcher rd = req.getRequestDispatcher("/secondServlet");
        rd.forward(req, resp);  // forward - response not committed yet
    }
}
```

**2. Using `include()` (not true chain but similar):**
```java
RequestDispatcher rd = req.getRequestDispatcher("/footerServlet");
rd.include(req, resp);  // multiple servlets contribute to single response
```

**3. Using servlet filters (modern approach):**
Filters chaining ka better version hai — more flexible, reusable.

**Use cases (SDE-2 level):**

| Scenario | Kaise use kare |
|----------|---------------|
| **Authentication → Authorization → Business Logic** | Chain 1: Check login, Chain 2: Check role, Chain 3: Process request |
| **Logging → Compression → Encryption** | Chain 1: Log request, Chain 2: Compress data, Chain 3: Encrypt response |
| **Input validation → Data enrichment → DB save** | Chain 1: Validate, Chain 2: Add metadata, Chain 3: Save |

**Advantages:**
- **Separation of concerns** — each servlet ek kaam kare
- **Reusability** — same servlet multiple chains me use kar sakte ho
- **Modularity** — code maintainable

**Disadvantages:**
- Performance overhead (multiple servlet calls)
- Debugging mushkil hota hai
- Request/response object ek hi thread me pass hota hai

**Modern alternative:**  
Aaj kal servlet chaining ki jagah **Spring MVC Interceptors** ya **Servlet Filters** use karte hain. Woh zyada powerful hai.

**Interview me bolna:**  
*"Servlet chaining ek pipeline pattern hai jahan multiple servlets sequentially execute hote hain. RequestDispatcher.forward() se achieve hota hai. Modern applications me filters ya Spring interceptors better alternatives hain kyunki woh more flexible aur reusable hote hain."*

### Bonus (SDE-2 level advanced points):

**Servlet vs JSP vs Filter:**
- **Servlet** — Business logic + response generation (HTML/JSON)
- **JSP** — Presentation layer (HTML with Java code)
- **Filter** — Pre-processing / Post-processing (logging, auth, compression)

**Async Servlets (Servlet 3.0+):**
```java
@WebServlet(value = "/async", asyncSupported = true)
public class AsyncServlet extends HttpServlet {
    protected void doGet(HttpServletRequest req, HttpServletResponse resp) {
        AsyncContext ctx = req.startAsync();
        // Long running task in separate thread
        ctx.start(() -> {
            // Process
            ctx.complete();
        });
    }
}
```
Isse thread utilization better hota hai — long running tasks thread ko block nahi karte.

**Common interview trap:**  
*"Servlet singleton hai kya?"*  
Answer: Haan — ek instance, multiple threads. Isliye instance variables thread-safe nahi hote.
