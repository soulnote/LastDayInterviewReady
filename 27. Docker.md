## What is Docker? 

**Simple Intuition:**
Docker = Lightweight virtual machine but better. Jaise ek suitcase mein app + dependencies (Node.js, Python, libraries) sab pack kardo. Kisi bhi machine le jao, wahi suitcase kholo → app chalega.

**Technical Definition:**
Docker is a containerization platform that packages applications and their dependencies into isolated units called containers.

**Docker vs VM vs Kubernetes:**

| | Docker Container | Virtual Machine | Kubernetes |
|---|----------------|----------------|------------|
| Size | MBs | GBs | Orchestrator |
| Startup | Seconds | Minutes | Manages containers |
| Isolation | Process level | Hardware level | Container management |
| OS | Shares host kernel | Full OS per VM | Works with Docker |

```
┌─────────────────────────────────────────────────────────────┐
│                    DOCKER ARCHITECTURE                       │
├─────────────────────────────────────────────────────────────┤
│  Docker Client (docker CLI)                                 │
│       ↓                                                     │
│  Docker Daemon (dockerd)                                    │
│       ↓                                                     │
│  containerd (container runtime)                            │
│       ↓                                                     │
│  runc (low-level container runner)                         │
│       ↓                                                     │
│  Container (isolated process)                              │
└─────────────────────────────────────────────────────────────┘
```

## Q1. List common Docker mistakes and how to avoid them in production?

### Top 10 Docker Production Mistakes

| # | Mistake | Why Bad | How to Avoid |
|---|---------|---------|--------------|
| 1 | **Using `latest` tag** | "Latest" kya hai pata nahi. Dev → prod same image? No guarantee | Use specific tags: `v1.2.3` or `git-sha` |
| 2 | **Running as root** | Container root = host root (security nightmare) | `USER appuser` in Dockerfile |
| 3 | **Large image size** | Slow pulls, more storage, security surface | Multi-stage builds, Alpine base |
| 4 | **No .dockerignore** | Secret files, node_modules, .git copied in image | Create `.dockerignore` file |
| 5 | **Hardcoded secrets** | `ENV PASSWORD=123` → anyone can inspect | Use Docker secrets, env vars from outside |
| 6 | **Unoptimized layers** | Each RUN creates layer. Rebuilds slow | Chain commands: `RUN apt-get update && apt-get install` |
| 7 | **No health checks** | Container dead but still receiving traffic | Add `HEALTHCHECK` to Dockerfile |
| 8 | **Exposing unnecessary ports** | More attack surface | Expose only needed ports |
| 9 | **No log rotation** | Container logs fill disk | Use `--log-opt max-size=10m` |
| 10 | **Ignoring resource limits** | One container eats all CPU/memory | `--memory="512m" --cpus="1.0"` |

### Detailed Examples

**Mistake 1: Using `latest`**
```dockerfile
# ❌ BAD
FROM node:latest

# ✅ GOOD
FROM node:18.17.0-alpine
```

**Mistake 2: Running as root**
```dockerfile
# ❌ BAD
FROM node:18
COPY . /app
CMD ["node", "app.js"]

# ✅ GOOD
FROM node:18-alpine
RUN addgroup -g 1001 -S nodejs && \
    adduser -S nodejs -u 1001
USER nodejs
COPY --chown=nodejs:nodejs . /app
CMD ["node", "app.js"]
```

**Mistake 3: Large image size**
```dockerfile
# ❌ BAD: 1.2GB image
FROM ubuntu:latest
RUN apt-get update
RUN apt-get install -y python3
RUN apt-get install -y build-essential
COPY . .

# ✅ GOOD: 120MB image (multi-stage)
FROM python:3.9-alpine as builder
COPY requirements.txt .
RUN pip install --user -r requirements.txt

FROM python:3.9-alpine
COPY --from=builder /root/.local /root/.local
COPY . .
CMD ["python", "app.py"]
```

**Mistake 4: .dockerignore**
```bash
# .dockerignore file
node_modules/
.git/
.env
*.log
Dockerfile
README.md
.gitignore
```

**Mistake 5: Hardcoded secrets**
```bash
# ❌ BAD
docker run -e DB_PASSWORD=secret123 myapp

# ✅ GOOD
echo "secret123" | docker secret create db_password -
docker service create --secret db_password myapp
# Or use env file
docker run --env-file .env.production myapp
```

**Mistake 9: Log rotation**
```bash
# ❌ BAD
docker run myapp

# ✅ GOOD
docker run \
  --log-driver json-file \
  --log-opt max-size=10m \
  --log-opt max-file=3 \
  myapp
```

**Mistake 10: Resource limits**
```bash
# ❌ BAD
docker run myapp

# ✅ GOOD
docker run \
  --memory="512m" \
  --memory-swap="1g" \
  --cpus="1.0" \
  --cpu-shares=512 \
  myapp
```

### Production Docker Run Command (Complete)

```bash
docker run -d \
  --name myapp \
  --restart unless-stopped \
  --memory="512m" \
  --cpus="1.0" \
  --log-opt max-size=10m \
  --log-opt max-file=3 \
  --read-only \
  --tmpfs /tmp:rw,noexec,nosuid,size=64m \
  --security-opt=no-new-privileges:true \
  --cap-drop=ALL \
  --cap-add=NET_BIND_SERVICE \
  -p 8080:8080 \
  myapp:1.2.3
```

## Q2. A critical security vulnerability has been discovered in one of your base images. How would you fix this?

### Step-by-Step Incident Response

#### Step 1: Identify Affected Images

```bash
# Find all images using vulnerable base
docker images --format "table {{.Repository}}\t{{.Tag}}\t{{.ID}}" | grep -E "ubuntu:20.04|node:14"

# Check running containers
docker ps --format "table {{.Names}}\t{{.Image}}"

# Scan for vulnerabilities (use Trivy, Grype, Snyk)
docker scan myapp:latest  # Docker Scout
trivy image myapp:latest
grype myapp:latest
```

#### Step 2: Assess Impact

```yaml
# Vulnerability assessment matrix
| Severity | Action | Timeline |
|----------|--------|----------|
| Critical | Immediate rebuild | < 4 hours |
| High | Rebuild within 24h | 1 day |
| Medium | Next release cycle | 1 week |
| Low | Monitor | Next sprint |
```

#### Step 3: Fix the Base Image

**Option A: Update base image**
```dockerfile
# ❌ Vulnerable
FROM ubuntu:20.04
RUN apt-get update && apt-get install -y openssl=1.1.1f

# ✅ Fixed
FROM ubuntu:22.04  # or ubuntu:20.04 with patched openssl
RUN apt-get update && apt-get install -y openssl=1.1.1f-1ubuntu2.19
```

**Option B: Minimal distroless image**
```dockerfile
# Even more secure
FROM gcr.io/distroless/base-debian11
COPY --from=builder /app /app
# No shell, no package manager, minimal attack surface
```

**Option C: Patch in multi-stage**
```dockerfile
FROM ubuntu:20.04 as builder
RUN apt-get update && \
    apt-get install -y wget && \
    wget https://security.ubuntu.com/ubuntu/pool/main/o/openssl/openssl_1.1.1f-1ubuntu2.19_amd64.deb

FROM ubuntu:20.04
COPY --from=builder /openssl_1.1.1f-1ubuntu2.19_amd64.deb .
RUN dpkg -i openssl*.deb && rm openssl*.deb
```

#### Step 4: Rebuild and Test

```bash
# Rebuild with new base
docker build -t myapp:1.2.4-security-fix .

# Scan again
docker scan myapp:1.2.4-security-fix

# Test functionality
docker run -d --name test myapp:1.2.4-security-fix
docker exec test curl -v https://api.example.com  # Test SSL
```

#### Step 5: Rolling Update

```bash
# Kubernetes rolling update
kubectl set image deployment/myapp myapp=myapp:1.2.4-security-fix
kubectl rollout status deployment/myapp

# Docker Swarm
docker service update --image myapp:1.2.4-security-fix myapp

# Standalone Docker
docker stop myapp-old
docker rm myapp-old
docker run -d --name myapp myapp:1.2.4-security-fix
```

#### Step 6: Prevention for Future

```dockerfile
# Use specific digest instead of tag
FROM node:18.17.0-alpine@sha256:abcdef123...

# Regular scanning in CI/CD
# .github/workflows/docker-scan.yml
name: Security Scan
on: [push]
jobs:
  scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Build image
        run: docker build -t myapp .
      - name: Scan with Trivy
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: myapp
          severity: CRITICAL,HIGH
```

### Incident Response Checklist

```bash
# 1. DETECT
docker scan myapp:latest | grep CRITICAL

# 2. ASSESS
docker ps --filter "status=running" --format "{{.Image}}" | sort | uniq -c

# 3. FIX
# Update base image in Dockerfile

# 4. REBUILD
docker build -t myapp:patched .
docker scan myapp:patched  # Verify fix

# 5. DEPLOY
kubectl set image deployment/myapp myapp=myapp:patched

# 6. VERIFY
kubectl rollout status deployment/myapp
kubectl logs -f deployment/myapp
```

## Q3. What is the difference between a docker image and a running docker container?

### Concept Explanation

**Simple Intuition:**
- **Docker Image** = Recipe/Blueprint (cake ka recipe)
- **Docker Container** = Actual cake (baked and ready to eat)

**Technical:**
- **Image** = Immutable, read-only template with layers
- **Container** = Runnable instance of an image with writable layer

### Architecture Diagram

```
┌─────────────────────────────────────────────────────────────┐
│                     DOCKER IMAGE                            │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  Layer 4: App Code (read-only)                      │   │
│  ├─────────────────────────────────────────────────────┤   │
│  │  Layer 3: Dependencies (npm install)                │   │
│  ├─────────────────────────────────────────────────────┤   │
│  │  Layer 2: Base OS (Ubuntu 22.04)                    │   │
│  ├─────────────────────────────────────────────────────┤   │
│  │  Layer 1: Kernel (shared with host)                 │   │
│  └─────────────────────────────────────────────────────┘   │
│                    IMAGE = READ-ONLY                        │
└─────────────────────────────────────────────────────────────┘
                            │
                            │ docker run
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                   DOCKER CONTAINER                          │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  Writable Layer (Container Layer)                   │   │
│  │  - Log files created                                 │   │
│  │  - Temp files written                                │   │
│  │  - Config changes                                    │   │
│  ├─────────────────────────────────────────────────────┤   │
│  │  Layer 4: App Code (read-only from image)           │   │
│  │  Layer 3: Dependencies (read-only from image)       │   │
│  │  Layer 2: Base OS (read-only from image)            │   │
│  │  Layer 1: Kernel (shared with host)                 │   │
│  └─────────────────────────────────────────────────────┘   │
│         CONTAINER = IMAGE + WRITABLE LAYER + PROCESS       │
└─────────────────────────────────────────────────────────────┘
```

### Detailed Comparison Table

| Aspect | Docker Image | Docker Container |
|--------|-------------|------------------|
| **State** | Immutable (cannot change) | Mutable (can change) |
| **Persistence** | Stored on disk | Running in memory |
| **Lifecycle** | Build once, use many times | Start, stop, delete |
| **Can you modify?** | No (build new version) | Yes (but lost after restart) |
| **Has process?** | No | Yes (has PID) |
| **Has network?** | No | Yes (has IP, ports) |
| **Storage** | Image registry (Docker Hub) | Container storage (ephemeral) |
| **Size** | Smaller (compressed layers) | Larger (writable layer added) |
| **Sharing** | Shared among containers | Isolated per container |

### Commands Demonstration

```bash
# List images (blueprints)
docker images
# REPOSITORY   TAG       IMAGE ID       SIZE
# nginx        latest    abc123def      142MB
# myapp        1.0       456xyz789      500MB

# List containers (running instances)
docker ps
# CONTAINER ID   IMAGE         STATUS         NAMES
# container123   nginx:latest  Up 2 hours     web-server

# Create container from image (but don't run)
docker create --name mycontainer nginx:latest

# Run container (create + start)
docker run -d --name running-container nginx:latest

# See difference: image is static, container is dynamic
docker inspect nginx:latest  # Image metadata
docker inspect running-container  # Container metadata (has IP, PID, mounts)

# See writable layer in action
docker exec running-container touch /tmp/test.txt  # Works (writable layer)
docker commit running-container myapp:with-change  # Creates new image from container

# Container dies, image remains
docker stop running-container
docker rm running-container
docker images  # nginx:latest still there
```

### Copy-on-Write (CoW) Mechanism

```bash
# When container modifies a file from image:
1. Container tries to write to /etc/nginx/nginx.conf
2. Docker copies file from image layer to writable layer
3. Container modifies the copy in writable layer
4. Original image layer unchanged

# Check storage
docker system df
# TYPE            TOTAL     ACTIVE    SIZE      RECLAIMABLE
# Images          5         2         1.2GB     800MB
# Containers      3         2         150MB     50MB
```

### When to Use What

| Use Case | Image | Container |
|----------|-------|-----------|
| Share with team | ✅ (push to registry) | ❌ (can't share running state) |
| Run in production | ❌ (need to run) | ✅ |
| Version control | ✅ (tag versions) | ❌ |
| Development | ❌ (need to run) | ✅ |
| CI/CD artifact | ✅ (build and push) | ❌ |

## Q4. What is a multi-stage Docker build and how does it help optimize container images?

### Concept Explanation

**Simple Intuition:**
Ek hi Dockerfile mein do phases:
1. **Build phase** (kitchen) - Install compilers, build tools, compile code (large image)
2. **Run phase** (dining table) - Only compiled binary + minimal runtime (small image)

**Technical:** Multi-stage builds allow copying artifacts from one stage to another, discarding intermediate layers.

### Problem: Without Multi-stage

```dockerfile
# ❌ BAD: Single stage = huge image
FROM golang:1.19
WORKDIR /app
COPY . .
RUN go build -o myapp .  # Requires Go compiler
# Go compiler still in final image! (800MB)
CMD ["./myapp"]
```

**Result:** 800MB image with Go compiler that runtime doesn't need.

### Solution: With Multi-stage

```dockerfile
# ✅ GOOD: Multi-stage build
# Stage 1: Builder (has compilers)
FROM golang:1.19 AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -o myapp .

# Stage 2: Runtime (only binary)
FROM alpine:latest
RUN apk --no-cache add ca-certificates
WORKDIR /root/
COPY --from=builder /app/myapp .
CMD ["./myapp"]
```

**Result:** 12MB image (98% smaller!)

### Architecture Diagram

```
┌─────────────────────────────────────────────────────────────────┐
│                    MULTI-STAGE BUILD FLOW                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  STAGE 1: BUILDER (golang:1.19 ~800MB)                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  - Go compiler                                            │   │
│  │  - Build tools (git, make)                               │   │
│  │  - Source code                                            │   │
│  │  - Dependencies                                           │   │
│  │  - Compiles binary (myapp)                               │   │
│  └─────────────────────────────────────────────────────────┘   │
│                           │                                      │
│                           │ COPY --from=builder                 │
│                           ▼                                      │
│  STAGE 2: RUNTIME (alpine:latest ~5MB)                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  - Minimal OS (Alpine Linux)                            │   │
│  │  - Only compiled binary (myapp)                         │   │
│  │  - No compilers, no source code                         │   │
│  │  - No build dependencies                                 │   │
│  └─────────────────────────────────────────────────────────┘   │
│                           │                                      │
│                           ▼                                      │
│                    FINAL IMAGE: ~12MB                          │
│                    (98% size reduction)                        │
└─────────────────────────────────────────────────────────────────┘
```

### Real-World Examples

**Example 1: Node.js Application**

```dockerfile
# ❌ Without multi-stage: 950MB
FROM node:18
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build
CMD ["npm", "start"]

# ✅ With multi-stage: 180MB
# Stage 1: Build
FROM node:18 AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build
RUN npm prune --production

# Stage 2: Run
FROM node:18-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./
USER node
CMD ["node", "dist/main.js"]
```

**Example 2: Python Application**

```dockerfile
# Multi-stage Python
FROM python:3.11 AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --user --no-cache-dir -r requirements.txt

FROM python:3.11-slim
WORKDIR /app
COPY --from=builder /root/.local /root/.local
COPY . .
ENV PATH=/root/.local/bin:$PATH
CMD ["python", "app.py"]
```

**Example 3: Java Spring Boot**

```dockerfile
# Multi-stage Java
FROM maven:3.8-openjdk-11 AS builder
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline
COPY src ./src
RUN mvn package -DskipTests

FROM openjdk:11-jre-slim
WORKDIR /app
COPY --from=builder /app/target/*.jar app.jar
EXPOSE 8080
CMD ["java", "-jar", "app.jar"]
```

### Advanced Techniques

**1. Named stages for different environments**

```dockerfile
# Development stage
FROM node:18 AS development
CMD ["npm", "run", "dev"]

# Build stage
FROM node:18 AS builder
# ... build steps ...

# Production stage
FROM node:18-alpine AS production
COPY --from=builder /app/dist ./dist
CMD ["node", "dist/main.js"]

# Test stage
FROM builder AS test
RUN npm run test
```

**2. Multiple artifacts from one build**

```dockerfile
FROM golang:1.19 AS builder
WORKDIR /app
COPY . .
RUN go build -o api ./cmd/api
RUN go build -o worker ./cmd/worker

FROM alpine:latest
COPY --from=builder /app/api /usr/local/bin/
COPY --from=builder /app/worker /usr/local/bin/
CMD ["api"]  # Default, override for worker
```

**3. Using external image as stage**

```dockerfile
# Copy from any image, not just previous stages
FROM scratch
COPY --from=alpine:latest /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/
COPY --from=builder /app/binary /binary
```

### Build Command

```bash
# Build specific stage
docker build --target builder -t myapp:builder .
docker build --target production -t myapp:latest .

# Build with cache optimization
docker build --build-arg BUILDKIT_PROGRESS=plain -t myapp .

# Use BuildKit for better multi-stage
export DOCKER_BUILDKIT=1
docker build -t myapp:optimized .
```

### Best Practices Summary

```dockerfile
# Ultimate multi-stage template
FROM base AS deps
  # Install dependencies only

FROM deps AS build  
  # Build application

FROM scratch AS final
  # Distroless or scratch for maximum security
  COPY --from=build /app/binary /binary
  USER 1000:1000
  ENTRYPOINT ["/binary"]
```

| Benefit | Impact |
|---------|--------|

| **Better security** | No build tools in runtime |
| **Faster builds** | Layer caching per stage |
| **Clearer Dockerfiles** | Separate build vs runtime concerns |
| **Lower bandwidth** | Registry storage costs down |
