## 1.1 API Design Principles

### The 7 Golden Principles

| Principle | Explanation (Hinglish) | Example |
|-----------|----------------------|---------|
| **1. Consistency** | Har endpoint same pattern follow kare | `/users/123` and `/orders/456` same style |
| **2. Simplicity** | Kam se kam complexity, jyada se jyada clarity | Avoid nested resources > 3 levels |
| **3. Predictability** | Client anumaan laga sake ki kya hoga | DELETE se hamesha delete hota hai |
| **4. Statelessness** | Har request complete information rakhe | Token har baar bhejna |
| **5. Backward Compatibility** | Old clients kaam karte rahein | Never remove fields, only add |
| **6. Performance** | Fast responses, minimal data | Pagination, field selection |
| **7. Security First** | Default secure ho | HTTPS mandatory, rate limiting |

```http
# 4. Request Body (Complex data)

  "shippingAddress": {...}
}
```

### B. Response Design Patterns

```json

  "meta": {                  // Metadata

// 2. Paginated Response
{
  "data": [...],
  "pagination": {
    "currentPage": 2,

// 3. Collection Response with Includes

    "name": "John",

      { "id": "456", "total": 100 },
      { "id": "789", "total": 200 }
    ],
    "profile": {
      "avatar": "https://...",
      "bio": "..."
    }
  }
}

// 4. Field Selection (Sparse Fields)

{
  "id": "123",
  "name": "John",
  "email": "john@example.com"
  // No other fields
}
```

### C. Idempotency Design

```python
# Idempotent POST using idempotency key
POST /payments
Idempotency-Key: client-generated-uuid-12345

{
  "amount": 100,
  "source": "card_123"
}

# Server implementation (pseudocode)
def create_payment(request):
    idempotency_key = request.headers['Idempotency-Key']
    
    # Check if already processed
    existing = redis.get(f"idempotent:{idempotency_key}")
    if existing:
        return existing  # Return same response
    
    # Process payment
    payment = process_payment(request.body)
    
    # Store response with idempotency key
    redis.setex(f"idempotent:{idempotency_key}", 86400, payment)
    
    return payment
```

### B. Error Categories & Status Codes

| Category | Status Codes | When to Use |
|----------|--------------|-------------|
| **Validation** | 400, 422 | Request body invalid, missing fields |
| **Authentication** | 401 | No/invalid API key, expired token |
| **Authorization** | 403 | Valid auth but insufficient permissions |
| **Not Found** | 404 | Resource doesn't exist |
| **Conflict** | 409 | Duplicate resource, state conflict |
| **Rate Limit** | 429 | Too many requests |
| **Server Error** | 500, 502, 503 | Internal errors, dependencies down |

## 1.4 Versioning Strategies

### Strategy Comparison

| Strategy | Example | Pros | Cons | Adoption |
|----------|---------|------|------|----------|
| **URI Path** | `/v1/users` | Simple, visible, cacheable | URI pollution | ⭐⭐⭐⭐⭐ Most common |
| **Custom Header** | `API-Version: 1` | Clean URIs, RESTful | Not visible, complex caching | ⭐⭐⭐ |
| **Content Negotiation** | `Accept: app/vnd.api.v1+json` | Pure REST | Complex, hard to debug | ⭐⭐ |
| **Query Parameter** | `/users?version=1` | Easy | Not RESTful, caching issues | ⭐ |

### Best Practice: URI Path Versioning

```http
# Version 1 (initial)
GET /v1/users/123
Response: { "name": "John", "email": "john@example.com" }

# Version 2 (breaking change - email becomes userEmail)
GET /v2/users/123
Response: { "name": "John", "userEmail": "john@example.com" }

# Support multiple versions simultaneously
/v1/* -> Service A
/v2/* -> Service B
```

### Backward Compatibility Rules

```
✅ Backward Compatible Changes (Safe):
   - Adding new optional fields
   - Adding new endpoints
   - Adding new HTTP methods
   - Expanding enum values
   - Making error messages more specific

❌ Breaking Changes (Need version bump):
   - Removing fields
   - Renaming fields
   - Changing field types (string → int)
   - Changing enum values
   - Adding required fields
   - Changing response structure
   - Changing error codes meaning
```

## 2.1 API Gateway

### What is API Gateway?

**Definition:**  
API Gateway ek **reverse proxy** hai jo client requests ko accept karta hai, routing karta hai, aur cross-cutting concerns handle karta hai.

**Hinglish:**  
API Gateway ek **receptionist** ki tarah hai. Saare requests pehle yahan aati hain, ye check karta hai ki request valid hai ya nahi, phir appropriate backend service ko forward karta hai.

### API Gateway Responsibilities

```
API Gateway Features
    │
    ├── Request Routing
    ├── Authentication & Authorization
    ├── Rate Limiting & Throttling
    ├── Request/Response Transformation
    ├── Caching
    ├── Logging & Monitoring
    ├── Circuit Breaking
    ├── Retry & Timeout
    ├── Load Balancing
    └── API Composition (Aggregation)
```

### Popular API Gateways

| Gateway | Language | Best For | Features |
|---------|----------|----------|----------|
| **Kong** | Lua/Go | Microservices | Plugins, gRPC, service mesh |
| **NGINX** | C | High performance | Low latency, simple |
| **Envoy** | C++ | Service mesh | Advanced L7 routing |
| **Traefik** | Go | Kubernetes | Auto-discovery, Let's Encrypt |
| **AWS API Gateway** | Managed | Serverless | Lambda, CloudWatch |
| **Zuul** | Java | Netflix stack | Dynamic routing |

### Architecture with API Gateway

```

    │
    ↓

    ├── Auth Service (validate JWT)
```

## 2.2 Load Balancing

### Load Balancing Algorithms

| Algorithm | How it works | Use Case |
|-----------|--------------|----------|
| **Round Robin** | Sequential distribution | Equal capacity servers |
| **Least Connections** | Send to fewest active connections | Variable request durations |
| **IP Hash** | Same client → same server | Session affinity (sticky sessions) |
| **Weighted Round Robin** | Based on server capacity | Unequal server specs |
| **Least Response Time** | Fastest server gets more | Performance sensitive |

## 2.3 Rate Limiting & Throttling

### Rate Limiting Strategies

| Strategy | Description | Example |
|----------|-------------|---------|
| **Fixed Window** | Reset at fixed intervals | 100 requests per minute, resets at :00 |
| **Sliding Window** | Rolling window | 100 requests in last 60 seconds |
| **Token Bucket** | Tokens refilled at steady rate | 10 tokens/sec, max 100 |
| **Leaky Bucket** | Queue + constant rate | 10 req/sec, buffer 20 |
| **Concurrent** | Parallel request limit | Max 10 concurrent requests |

### Implementation Example (Redis-based)

```python
class RateLimiter:
    def __init__(self, redis_client):
        self.redis = redis_client
    
    def check_rate_limit(self, api_key, limit=100, window=60):
        """
        Sliding window rate limiter
        limit: max requests per window
        window: time window in seconds
        """
        key = f"ratelimit:{api_key}"
        current_time = time.time()
        window_start = current_time - window
        
        # Remove old entries
        self.redis.zremrangebyscore(key, 0, window_start)
        
        # Count current requests
        count = self.redis.zcard(key)
        
        if count >= limit:
            # Get oldest request time for Retry-After header
            oldest = self.redis.zrange(key, 0, 0, withscores=True)[0][1]
            retry_after = int(window - (current_time - oldest))
            return False, retry_after
        
        # Add current request
        self.redis.zadd(key, {str(current_time): current_time})
        self.redis.expire(key, window)
        
        remaining = limit - (count + 1)
        return True, remaining
```

## 2.4 Caching Strategies

### Cache Hierarchy

```
Browser Cache (1ms)
        ↓ miss
CDN Cache (10ms)
        ↓ miss
API Gateway Cache (20ms)
        ↓ miss
Redis/Memcached (1-5ms)
        ↓ miss
Database (10-100ms)
```

### Caching Patterns

| Pattern | Description | When to Use |
|---------|-------------|-------------|
| **Cache-Aside** | App checks cache first, then DB | Read-heavy workloads |
| **Write-Through** | Write to cache + DB simultaneously | Write-heavy, consistency needed |
| **Write-Behind** | Write to cache, async to DB | High write throughput |
| **Refresh-Ahead** | Auto-refresh cache before expiry | Predictable access patterns |

### Cache Invalidation Strategies

```python
# Cache-Aside Pattern
def get_user(user_id):
    # 1. Check cache
    cache_key = f"user:{user_id}"
    user = redis.get(cache_key)
    
    if user:
        return user
    
    # 2. Cache miss - get from DB
    user = db.query("SELECT * FROM users WHERE id = ?", user_id)
    
    # 3. Store in cache
    redis.setex(cache_key, 3600, user)
    
    return user

def update_user(user_id, data):
    # 1. Update database
    db.update("UPDATE users SET ... WHERE id = ?", user_id, data)
    
    # 2. Invalidate cache
    redis.delete(f"user:{user_id}")
    
    # OR update cache (write-through)
    redis.setex(f"user:{user_id}", 3600, updated_user)
```

### Cache Headers Best Practices

```http
# Public, cacheable resource (product catalog)
Cache-Control: public, max-age=3600
ETag: "abc123"
Age: 120

# Private, user-specific resource
Cache-Control: private, max-age=300
Vary: Authorization, Accept-Encoding

# Never cache (sensitive data)
Cache-Control: no-store, private
Pragma: no-cache

# Always revalidate (dynamic content)
Cache-Control: no-cache, must-revalidate
ETag: "def456"
```

```python
# Request counter
http_requests_total = Counter(
    'http_requests_total',
    'Total HTTP requests',
    ['method', 'endpoint', 'status']
)

# Response time histogram
http_request_duration_seconds = Histogram(
    'http_request_duration_seconds',
    'HTTP request duration',
    ['method', 'endpoint'],
    buckets=[0.1, 0.5, 1, 2, 5]
)

# Active requests gauge
active_requests = Gauge(
    'active_requests',
    'Number of active requests'
)

# Track in middleware
@app.middleware
async def metrics_middleware(request, call_next):
    active_requests.inc()
    start = time.time()
    
    response = await call_next(request)
    
    duration = time.time() - start
    http_request_duration_seconds.labels(
        method=request.method,
        endpoint=request.url.path
    ).observe(duration)
    
    http_requests_total.labels(
        method=request.method,
        endpoint=request.url.path,
        status=response.status_code
    ).inc()
    
    active_requests.dec()
    return response
```

## 3.1 Authentication (Who you are?)

### Authentication Methods Comparison

| Method | How it works | Security | Use Case |
|--------|--------------|----------|----------|
| **API Key** | Static key in header | Low (can be leaked) | Internal services, rate limiting |
| **HTTP Basic Auth** | Base64(username:password) | Low (unless HTTPS) | Legacy systems |
| **JWT (JSON Web Token)** | Signed token | Medium (stateless) | Stateless auth, microservices |
| **OAuth 2.0** | Tokens with scopes | High (delegated access) | Third-party access, SSO |
| **mTLS** | Client certificate | Very High | Service-to-service (internal) |

### API Key Authentication

```http
# API Key in header (preferred)
GET /users/123
X-API-Key: sk_live_abc123def456

# API Key in query (not recommended - logs mein chala jayega)
GET /users/123?api_key=sk_live_abc123def456

# Implementation
def validate_api_key(request):
    api_key = request.headers.get('X-API-Key')
    
    if not api_key:
        return None
    
    # Check in Redis/Database
    org_id = redis.get(f"api_key:{api_key}")
    
    if not org_id:
        return None
    
    return org_id  # Return organization ID for rate limiting
```

### JWT (JSON Web Token) — Most Common

```python
import jwt
from datetime import datetime, timedelta

# JWT Structure: Header.Payload.Signature
# eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VySWQiOiIxMjMiLCJyb2xlIjoiYWRtaW4iLCJleHAiOjE3MDUzMTUyMDB9.abc123...

# Generate JWT
def generate_jwt(user_id, role):
    payload = {
        'userId': user_id,
        'role': role,
        'permissions': ['read', 'write', 'delete'],
        'iat': datetime.utcnow(),  # Issued at
        'exp': datetime.utcnow() + timedelta(hours=1),  # Expiry
        'jti': str(uuid.uuid4())  # JWT ID (for blacklist)
    }
    
    token = jwt.encode(
        payload,
        os.environ['JWT_SECRET'],
        algorithm='HS256'
    )
    
    return token

# Validate JWT
def validate_jwt(token):
    try:
        payload = jwt.decode(
            token,
            os.environ['JWT_SECRET'],
            algorithms=['HS256']
        )
        
        # Check if token is blacklisted
        if redis.exists(f"blacklist:{payload['jti']}"):
            return None
        
        return payload
    except jwt.ExpiredSignatureError:
        return None
    except jwt.InvalidTokenError:
        return None
```

### JWT Best Practices

| Practice | Why? | Implementation |
|----------|------|----------------|
| **Short expiry** | Reduce risk of stolen tokens | 15-60 minutes |
| **Refresh tokens** | Better user experience | Long-lived refresh token, short-lived access token |
| **Blacklist on logout** | Immediate revocation | Redis store of revoked JTI |
| **Use HTTPS** | Prevent token interception | TLS everywhere |
| **Store in httpOnly cookie** | XSS protection | Cookie instead of localStorage |

## 3.2 Authorization (What you can do?)

### Authorization Models

| Model | How it works | Complexity | Use Case |
|-------|--------------|------------|----------|
| **RBAC (Role-Based)** | User has role, role has permissions | Medium | Most common |
| **ABAC (Attribute-Based)** | Dynamic rules based on attributes | High | Complex access control |
| **ACL (Access Control List)** | Resource has list of users | Low | Simple file systems |
| **Policy-Based** | Centralized policies (OPA) | High | Multi-cloud, microservices |

### RBAC Implementation

```python
# Role-Permission mapping
ROLES = {
    'admin': ['read', 'write', 'delete', 'manage_users'],
    'user': ['read', 'write'],
    'viewer': ['read'],
    'guest': []
}

# User-Role mapping (in database)
users = {
    'user_123': {'role': 'admin', 'department': 'engineering'},
    'user_456': {'role': 'user', 'department': 'sales'},
    'user_789': {'role': 'viewer', 'department': 'hr'}
}

# Authorization middleware
def authorize(required_permission):
    def decorator(func):
        @wraps(func)
        def wrapper(user_id, *args, **kwargs):
            user_role = get_user_role(user_id)
            user_permissions = ROLES.get(user_role, [])
            
            if required_permission not in user_permissions:
                raise HTTPException(
                    status_code=403,
                    detail=f"User {user_id} doesn't have {required_permission} permission"
                )
            
            return func(user_id, *args, **kwargs)
        return wrapper
    return decorator

# Usage
@app.delete('/users/{user_id}')
@authorize('delete')
def delete_user(current_user_id, user_id_to_delete):
    # Only users with 'delete' permission can delete
    return {"status": "deleted"}

# Resource-level authorization (owner check)
@app.patch('/orders/{order_id}')
def update_order(current_user_id, order_id):
    order = db.get_order(order_id)
    
    # Check if user owns this order
    if order.user_id != current_user_id:
        # Admin can override
        if not has_permission(current_user_id, 'admin'):
            raise HTTPException(403, "You can only update your own orders")
    
    return update_order_in_db(order_id)
```

## 3.3 API Keys vs JWT vs OAuth2

### Detailed Comparison

| Feature | API Key | JWT | OAuth2 |
|---------|---------|-----|--------|
| **State** | Stateless | Stateless | Usually stateful (refresh token) |
| **Revocation** | Hard (need to regenerate) | Hard (needs blacklist) | Easy (revoke refresh token) |
| **Expiry** | Long-lived (months/years) | Short-lived (15-60 min) | Short-lived access + long refresh |
| **Scope/Permission** | Fixed (per key) | Inside token (custom claims) | Granular scopes |
| **Rotation** | Manual | Automatic (via refresh) | Automatic |
| **Use case** | Internal services, rate limiting | Microservices, SPA | Third-party apps, SSO |
| **Security** | Low (if leaked, game over) | Medium (short expiry helps) | High (consent, scopes) |

### Decision Tree

```
Is it for a third-party application?
    ├── Yes → OAuth 2.0
    └── No ↓

Is it for a single-page application (React/Vue)?
    ├── Yes → JWT (with refresh tokens, httpOnly cookie)
    └── No ↓

Is it for internal service-to-service?
    ├── Yes → mTLS or API Key
    └── No ↓

Is it for a mobile app?
    ├── Yes → OAuth 2.0 (PKCE)
    └── No → API Key or JWT
```

## 3.4 Common API Attacks & Mitigation

### Attack Matrix

| Attack | What it is | Mitigation |
|--------|-----------|------------|
| **Injection (SQL, NoSQL, Command)** | Attacker injects malicious code | Parameterized queries, input validation |
| **XSS (Cross-Site Scripting)** | Malicious script in response | Output encoding, CSP header |
| **CSRF (Cross-Site Request Forgery)** | Attacker tricks user's browser | CSRF tokens, SameSite cookies |
| **Mass Assignment** | Attacker sends extra fields | Whitelist allowed fields |
| **IDOR (Insecure Direct Object Reference)** | Access unauthorized resources | Authorization checks on every request |
| **Rate Limit Bypass** | Too many requests | Distributed rate limiting |
| **Man-in-the-Middle** | Intercept traffic | HTTPS only, HSTS |
| **Replay Attack** | Replay valid request | Nonce, timestamp in request |
| **Parameter Pollution** | Duplicate parameters | Validate and reject duplicates |
| **JWT Algorithm Confusion** | Change algorithm to 'none' | Validate algorithm, use strong secrets |

### SQL Injection Prevention

```python
# ❌ VULNERABLE - String concatenation
user_id = request.args.get('user_id')
query = f"SELECT * FROM users WHERE id = {user_id}"
db.execute(query)  # user_id = "1 OR 1=1" → returns ALL users

# ✅ SAFE - Parameterized queries
user_id = request.args.get('user_id')
query = "SELECT * FROM users WHERE id = %s"
db.execute(query, (user_id,))  # Parameterized = safe

# ✅ SAFE - ORM
user = User.query.filter_by(id=user_id).first()
```

### IDOR Prevention

```python
# ❌ VULNERABLE - No ownership check
@app.get('/api/invoices/{invoice_id}')
def get_invoice(invoice_id):
    return db.get_invoice(invoice_id)  # Any user can see any invoice

# ✅ SAFE - Ownership check
@app.get('/api/invoices/{invoice_id}')
def get_invoice(current_user_id, invoice_id):
    invoice = db.get_invoice(invoice_id)
    
    # Check ownership
    if invoice.user_id != current_user_id:
        # Admin override with audit log
        if not is_admin(current_user_id):
            audit_log.warning(f"IDOR attempt: User {current_user_id} tried to access invoice {invoice_id}")
            raise HTTPException(403, "Access denied")
    
    return invoice
```

### CSRF Prevention

```python
# CSRF Token pattern
@app.get('/api/csrf-token')
def get_csrf_token(request):
    token = secrets.token_urlsafe(32)
    request.session['csrf_token'] = token
    return {'csrfToken': token}

@app.post('/api/transfer-money')
def transfer_money(request):
    # Validate CSRF token
    request_token = request.headers.get('X-CSRF-Token')
    session_token = request.session.get('csrf_token')
    
    if not request_token or request_token != session_token:
        raise HTTPException(403, "Invalid CSRF token")
    
    # Process transfer
    return transfer(request.body)

# Better: Use SameSite cookies
Set-Cookie: sessionId=abc123; SameSite=Lax; HttpOnly; Secure
```
