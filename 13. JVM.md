
## Q1. What is Metaspace in Java 8?

**Definition:** Metaspace JVM ka **new memory region** hai jo **class metadata** store karta hai. Java 8 me **PermGen (Permanent Generation)** ki jagah aaya.

**PermGen vs Metaspace (Major difference):**

| Aspect | PermGen (Java 7 & earlier) | Metaspace (Java 8+) |
|--------|---------------------------|---------------------|
| **Location** | Heap memory ka part | Native memory (off-heap) |
| **Default size** | 82 MB (limited) | Unlimited (auto-growing) |
| **Max size** | `-XX:MaxPermSize` | `-XX:MaxMetaspaceSize` (default unlimited) |
| **OOM risk** | High (famous PermGen OOM) | Low (unless memory leak) |
| **Garbage Collection** | Full GC ke saath | Independent, can shrink |
| **String pool** | Yes (Java 7 moved to heap) | No (strings heap me) |

**Metaspace me kya store hota hai:**
```java
public class Employee {
    // Ye sab Metaspace me jayenge:
    private static String companyName;     // Static field metadata
    private static final int VERSION = 1;  // Static final constant
    
    public void work() {}  // Method bytecode
    public void sleep() {} // Method bytecode
    
    // Class name, superclass, interfaces
    // Runtime constant pool
    // Method table (vtable, itable)
    // JIT compiled code
}
```

**Why Metaspace? (SDE-2 level answer):**
1. **PermGen fixed size hoti thi** → web applications redeploy karte time OOM aata tha
2. **Native memory** → OS ke saath share hoti hai, automatically grow/shrink ho sakti hai
3. **Better performance** → GC alag se manage karta hai

**Monitor Metaspace:**
```bash
# Check Metaspace usage
jstat -gc <pid> | awk '{print $8 " " $9}'
# $8 = Metaspace capacity, $9 = Metaspace used

# JVM flags
-XX:MetaspaceSize=128m          # Initial size
-XX:MaxMetaspaceSize=256m       # Max size (prevent unlimited growth)
-XX:MinMetaspaceFreeRatio=40    # Min free percentage
-XX:MaxMetaspaceFreeRatio=70    # Max free percentage
```

## Q2. Is It Possible to Resurrect an Object That became Eligible for Garbage Collection?

**Answer:** **Yes, but only once and through `finalize()` (deprecated).**

**Detailed explanation with code:**

```java
public class ResurrectionDemo {
    private static ResurrectionDemo zombieRef = null;
    private String name;
    
    public ResurrectionDemo(String name) {
        this.name = name;
    }
    
    @Override
    protected void finalize() throws Throwable {
        super.finalize();
        System.out.println(name + " is being finalized... resurrecting!");
        zombieRef = this;  // Resurrection happens here!
    }
    
    public static void main(String[] args) throws Exception {
        // Create object
        ResurrectionDemo obj = new ResurrectionDemo("Zombie");
        
        // Remove strong reference
        obj = null;
        
        // Request GC
        System.gc();
        Thread.sleep(1000);
        
        // Object is alive again!
        if (zombieRef != null) {
            System.out.println(zombieRef.name + " is alive again!");
            
            // Try to kill it again
            zombieRef = null;
            System.gc();
            Thread.sleep(1000);
            
            // Second time - finalize won't be called again
            if (zombieRef == null) {
                System.out.println("Finally dead!");
            }
        }
    }
}
```

**Output:**
```
Zombie is being finalized... resurrecting!
Zombie is alive again!
Finally dead!
```

**SDE-2 level critical points:**

| Aspect | Detail |
|--------|--------|
| **How many times?** | Only once per object |
| **Finalize thread** | Low priority daemon thread |
| **Delay** | No guarantee when finalize() runs |
| **Performance** | 100x slower than normal GC |
| **Modern alternative** | `Cleaner` (Java 9+) or `PhantomReference` |
| **Status** | Deprecated (Java 9), Removed (future versions) |

**Why resurrection is bad:**
- Creates **memory leaks** (objects never die)
- **Performance nightmare** (GC delays)
- **Unpredictable behavior** (timing issues)
- **Security risks** (sensitive data remains alive)

**Interview me bolna:** 
*"Technically possible via finalize() but it's deprecated and considered an anti-pattern. Modern Java uses Cleaner or PhantomReference for cleanup without resurrection."*

## Q4. What Are Strong, Weak, Soft, and Phantom References?

**SDE-2 level deep dive with real examples:**

### 1. Strong Reference (Default)
```java
Object obj = new Object();  // Strong reference
// JVM will NEVER collect this object until reference is null
obj = null;  // Now eligible for GC
```

### 2. Soft Reference (Memory-sensitive cache)
```java
// Cache that survives until memory pressure
SoftReference<byte[]> cacheRef = new SoftReference<>(new byte[100_000_000]);

// When memory is low (before OOM), this will be collected
byte[] data = cacheRef.get();
if (data == null) {
    // Reload from disk/database
    data = loadFromDisk();
    cacheRef = new SoftReference<>(data);
}
```

**Real use case - Image caching:**
```java
public class ImageCache {
    private Map<String, SoftReference<BufferedImage>> cache = new HashMap<>();
    
    public BufferedImage getImage(String path) {
        SoftReference<BufferedImage> ref = cache.get(path);
        BufferedImage img = (ref != null) ? ref.get() : null;
        
        if (img == null) {
            img = loadImageFromDisk(path);  // Expensive operation
            cache.put(path, new SoftReference<>(img));
        }
        return img;
    }
}
```

### 3. Weak Reference (Metadata, WeakHashMap)
```java
// GC collects on next cycle regardless of memory
WeakReference<Object> weakRef = new WeakReference<>(new Object());

// Real-world: WeakHashMap
WeakHashMap<Key, Value> metadata = new WeakHashMap<>();
Key key = new Key("session123");
metadata.put(key, new Value("user data"));
key = null;  // Strong reference removed
// Next GC → entry automatically removed from WeakHashMap!
```

**Use case - ThreadLocal cleanup:**
```java
class ThreadLocalLeak {
    // ThreadLocal uses WeakReference for keys
    private static ThreadLocal<byte[]> threadLocal = new ThreadLocal<>();
    
    void goodPattern() {
        threadLocal.set(new byte[1024]);
        try {
            // Use threadLocal
        } finally {
            threadLocal.remove();  // Critical! Prevents leak
        }
    }
}
```

### 4. Phantom Reference (Cleanup after GC)
```java
// PhantomReference.get() ALWAYS returns null
// Used with ReferenceQueue to detect when object is GC'd

public class ResourceCleaner {
    private static final ReferenceQueue<Object> QUEUE = new ReferenceQueue<>();
    
    static class CleanupPhantomReference extends PhantomReference<Object> {
        private final Runnable cleanupAction;
        
        CleanupPhantomReference(Object referent, Runnable action) {
            super(referent, QUEUE);
            this.cleanupAction = action;
        }
        
        void cleanup() {
            cleanupAction.run();
        }
    }
    
    public static void main(String[] args) {
        Object resource = new Object();
        CleanupPhantomReference phantomRef = 
            new CleanupPhantomReference(resource, () -> System.out.println("Cleaning up"));
        
        resource = null;  // Remove strong reference
        
        // Cleaner thread
        new Thread(() -> {
            try {
                CleanupPhantomReference ref = (CleanupPhantomReference) QUEUE.remove();
                ref.cleanup();  // Runs AFTER object is GC'd
            } catch (InterruptedException e) {}
        }).start();
    }
}
```

**Comparison table for interview:**

| Aspect | Strong | Soft | Weak | Phantom |
|--------|--------|------|------|---------|
| **GC collection time** | Never directly | When memory low | Next GC cycle | After finalization |
| **get() returns** | Object | Object or null | Object or null | Always null |
| **ReferenceQueue needed?** | No | Optional | Optional | **Yes** (practically) |
| **Can resurrect?** | Yes | No (if collected) | No | No |
| **Use case** | Normal objects | Cache | Metadata, WeakHashMap | Cleanup, resource release |
| **Memory leak risk** | High if not cleared | Medium | Low | None |

## Q6. Explain difference between Minor GC and Major GC in Java?

**Heap Structure reminder:**
```
┌─────────────────────────────────────────────────────┐
│                        HEAP                          │
├──────────────┬──────────────────┬───────────────────┤
│  Young Gen   │      Old Gen     │    (Metaspace)    │
├──┬────┬──────┤                  │   (Native Memory) │
│E │ S0 │ S1   │                  │                   │
└──┴────┴──────┴──────────────────┴───────────────────┘
 E = Eden, S0/S1 = Survivor Spaces
```

### Detailed Comparison:

| Aspect | Minor GC | Major GC (Full GC) |
|--------|----------|-------------------|
| **Target region** | Young Generation (Eden + Survivor) | Old Generation + Young Gen |
| **Frequency** | Very frequent (seconds to minutes) | Infrequent (hours to days) |
| **Duration** | Very fast (10-50 ms) | Slow (100ms to several seconds) |
| **Pause time** | Short pause (usually < 100ms) | Long pause (STW - Stop The World) |
| **Trigger** | Eden fills up | Old Gen fills up / System.gc() |
| **Objects collected** | Short-lived, unreferenced | Long-lived, unreferenced |
| **Impact** | Low | High (can affect user experience) |
| **GC Algorithm** | Copying (stop-copy) | Mark-sweep-compact or concurrent |
| **Log indicator** | `[GC` | `[Full GC` |

### Visual Example:

```java
// Minor GC logs
[GC (Allocation Failure) 512M->128M(1024M), 0.025 secs]
//          ↑              ↑       ↑          ↑
//       Trigger       Before  After    Duration

// Major/Full GC logs
[Full GC (Ergonomics) 1024M->256M(1024M), 0.850 secs]
//    ↑                    ↑       ↑          ↑
//  Stop-World         Before   After    Much slower!
```

### How objects move through generations:

```java
// Step-by-step object lifecycle
Object obj = new Object();  // 1. Created in Eden

// After Minor GC #1
// If obj still referenced → moves to Survivor0 (S0)
// Age = 1

// After Minor GC #2
// Still alive → moves to Survivor1 (S1)
// Age = 2

// After Minor GC #N (usually age threshold = 15)
// Age >= 15 → PROMOTED to Old Generation

// When Old Gen fills → Major GC happens
```

### SDE-2 level important points:

**1. Stop-The-World (STW):**
- Minor GC: Short STW (all application threads pause)
- Major GC: Long STW (more noticeable)

**2. GC Logs analysis:**
```bash
# Enable GC logging (Java 9+)
-XX:+PrintGCDetails -Xlog:gc*:file=gc.log

# Java 8
-XX:+PrintGCDetails -XX:+PrintGCTimeStamps -Xloggc:gc.log
```

**3. Performance optimization:**
```java
// Minimize Major GC by:
// 1. Tuning young gen size
// 2. Reducing object promotion rate
// 3. Using appropriate GC algorithm
// 4. Avoiding System.gc() calls
```

**4. Common misconception:**
> "Full GC = Major GC" — Not exactly. Major GC cleans Old Gen, Full GC cleans entire heap (Young + Old + Metaspace)

## Q7. Explain how Garbage Collection works in Java?

**SDE-2 level comprehensive explanation:**

### Phase 1: Marking (Find live objects)

```
Before Marking:          After Marking:
┌───┬───┬───┬───┐       ┌───┬───┬───┬───┐
│ A │ B │ C │ D │       │ A │ B │ C │ D │
│ ✓ │ ✓ │   │   │       │ M │ M │   │   │  M = Marked (live)
└───┴───┴───┴───┘       └───┴───┴───┴───┘
```

### Phase 2: Sweeping (Remove unmarked)

```
After Sweep:
┌───┬───┬───┬───┐
│ A │ B │   │   │  (Dead objects removed)
└───┴───┴───┴───┘
```

### Phase 3: Compacting (Optional - remove fragmentation)

```
After Compaction:
┌───┬───┬───┬───┬───┐
│ A │ B │   │   │   │  (Free space consolidated)
└───┴───┴───┴───┴───┘
```

### Generational GC Process (Detailed):

```java
// Simplified algorithm
public class GarbageCollector {
    
    // 1. Minor GC (Young Generation)
    void minorGC() {
        // Stop all application threads (STW)
        stopTheWorld();
        
        // Mark live objects in Eden + Survivor
        Set<Object> liveObjects = markLiveObjects(youngGen);
        
        // Copy live objects to another survivor space
        copyToSurvivor(liveObjects);
        
        // Increment age of survived objects
        for (Object obj : liveObjects) {
            obj.age++;
            if (obj.age >= TENURING_THRESHOLD) {
                promoteToOldGen(obj);  // Move to Old Gen
            }
        }
        
        // Clear Eden and current survivor
        clearYoungGen();
        
        // Resume application
        resumeApplication();
    }
    
    // 2. Major GC (Old Generation)
    void majorGC() {
        stopTheWorld();
        
        // Mark all live objects (Young + Old)
        Set<Object> liveObjects = markLiveObjects(heap);
        
        // Sweep dead objects
        sweepDeadObjects(liveObjects);
        
        // Compact to remove fragmentation
        compact();
        
        resumeApplication();
    }
}
```

### Different GC Algorithms:

| GC Name | Algorithm | Best For | Pause Time |
|---------|-----------|----------|------------|
| **Serial GC** | Single-threaded mark-sweep-compact | Small apps, single CPU | Long |
| **Parallel GC** | Multi-threaded mark-sweep-compact | High throughput, batch jobs | Medium |
| **CMS (deprecated)** | Concurrent mark-sweep | Low latency (but fragmented) | Short |
| **G1GC** | Region-based, incremental | Balanced (throughput + latency) | Short (< 200ms) |
| **ZGC** | Colored pointers, load barriers | Very low latency (< 10ms) | Sub-millisecond |
| **Shenandoah** | Brookes pointers | Very low latency | Sub-millisecond |

```
Application Threads: ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

**1. Object Age Calculation:**

// Each Minor GC = age + 1
// When age >= threshold → promoted to Old Gen
```

**2. Dynamic Tenuring:**

## Q8. Can you explain the different sections of the heap memory in Java?

**Complete heap structure (SDE-2 level detail):**

```
                              HEAP MEMORY
┌────────────────────────────────────────────────────────────────┐
│                         YOUNG GENERATION                        │
│  ┌─────────────────┐  ┌─────────────┐  ┌─────────────┐         │
│  │      EDEN       │  │ SURVIVOR 0  │  │ SURVIVOR 1  │         │
│  │    (80-90%)     │  │   (5-10%)   │  │   (5-10%)   │         │
│  └─────────────────┘  └─────────────┘  └─────────────┘         │
├─────────────────────────────────────────────────────────────────┤
│                    OLD GENERATION (TENURED)                      │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │         Long-lived objects (age >= threshold)            │  │
│  │         Usually 60-80% of total heap                     │  │
│  └──────────────────────────────────────────────────────────┘  │
├─────────────────────────────────────────────────────────────────┤
│                    METASPACE (Off-Heap)                          │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │    Class metadata, method bytecode, constant pool        │  │
│  │    Native memory (not part of heap)                      │  │
│  └──────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

### Detailed Breakdown:

| Region | Purpose | Size (typical) | GC Collection | Contents |
|--------|---------|----------------|---------------|----------|
| **Eden** | New objects created here | 80-90% of Young Gen | Minor GC | Short-lived objects |
| **Survivor 0 (S0)** | Objects surviving 1st GC | 5-10% of Young Gen | Minor GC | Intermediate age objects |
| **Survivor 1 (S1)** | Objects surviving 2nd GC | 5-10% of Young Gen | Minor GC | Intermediate age objects |
| **Old Gen** | Long-lived objects | 60-80% of total heap | Major/Full GC | Tenured objects |
| **Metaspace** | Class metadata | Native (default unlimited) | Special GC | Class definitions |

### Object flow through heap sections:

```java
// Step-by-step journey
public class HeapJourney {
    
    // STEP 1: Object created
    public void createObject() {
        MyObject obj = new MyObject();  
        // ↓ Goes to EDEN
    }
    
    // STEP 2: Minor GC #1 (if obj still referenced)
    // ↓ Moves to SURVIVOR 0 (age = 1)
    
    // STEP 3: Minor GC #2 (still referenced)
    // ↓ Moves to SURVIVOR 1 (age = 2)
    
    // STEP 4: Minor GC #3 (still referenced)
    // ↓ Moves to SURVIVOR 0 (age = 3)
    // ... continues until age >= 15
    
    // STEP 5: Promotion
    // ↓ Moves to OLD GENERATION (age = 15+)
    
    // STEP 6: Object becomes unreferenced
    obj = null;
    // ↓ Will be collected in next MAJOR GC
}
```

### Size ratios and tuning:

```bash
# JVM flags for heap sizing

# Total heap size
-Xms2g -Xmx2g                    # Initial/Max heap = 2GB

# Young/Old ratio
-XX:NewRatio=2                   # Old:Young = 2:1 (Old = 2/3, Young = 1/3)
-XX:NewRatio=3                   # Old:Young = 3:1 (Old = 3/4, Young = 1/4)

# Survivor space ratio
-XX:SurvivorRatio=8              # Eden:S0 = 8:1 (Eden = 8/10, S0 = 1/10, S1 = 1/10)

# Explicit sizing
-XX:NewSize=512m                 # Young gen initial size
-XX:MaxNewSize=512m              # Young gen max size
-XX:SurvivorSpaceSize=64m        # Each survivor space size
```

### Example calculation:

```
Total Heap = 2GB (2048 MB)
NewRatio = 2

Young Gen = Total / (NewRatio + 1) = 2048 / 3 = 682 MB
Old Gen = 2048 - 682 = 1366 MB

SurvivorRatio = 8
Eden = Young * (SurvivorRatio / (SurvivorRatio + 2)) = 682 * (8/10) = 545 MB
S0 = Young * (1/10) = 68 MB
S1 = Young * (1/10) = 68 MB
```

### Special regions (Advanced):

**1. Large Object Space (Humongous Objects in G1):**
```java
// Objects > 50% of region size go directly to Old Gen
byte[] hugeArray = new byte[100 * 1024 * 1024];  // 100MB
// ↓ Bypasses Young Gen completely
```

**2. TLAB (Thread Local Allocation Buffer):**
```
Each thread gets small Eden region
Thread 1: ┌──────┐
Thread 2: ┌──────┐
Thread 3: ┌──────┘
No synchronization needed!
```

**3. PLAB (Promotion Local Allocation Buffer):**
```
During GC, threads use local buffers for promotion
Reduces contention during GC
```

|---------|----------------|
