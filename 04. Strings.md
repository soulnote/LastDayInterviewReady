
## **Q1. Why String is immutable in Java?**

**Answer:**  
String immutable hai **security, performance, aur caching** ke liye. Java designers ne intentionally immutable banaya.

**Main Reasons:**

**1. String Constant Pool (SCP) - Caching:**
```java
String s1 = "Hello";
String s2 = "Hello";  // Same object from pool
// If String was mutable, changing s1 would affect s2
```

**2. Security (Most Important):**
```java
// Database connection example
void connectToDatabase(String url, String username, String password) {
    // If String was mutable, hacker could change password reference
    // after security check but before connection
}

// Class loading - package names are Strings
// If mutable, could change package name at runtime - security disaster
```

**3. Thread Safety:**
```java
// Immutable = automatically thread-safe
// Multiple threads can share same String without synchronization
String shared = "config";
// No matter how many threads access, value never changes
```

**4. HashCode Caching:**
```java
String str = "Hello";
str.hashCode();  // Calculated once, cached
// HashMap performance improves because hash code doesn't change
```

**5. Class Loading:**
```java
// JVM uses Strings for class names
// If mutable, could change class name after loading
Class.forName("com.myapp.User");  // Safe because String immutable
```

**What if String was mutable?**
```java
// Hypothetical mutable String
MutableString s1 = "Hello";
MutableString s2 = s1;  // Same reference
s1.setCharAt(0, 'J');   // s1 = "Jello"
// s2 also becomes "Jello" - unexpected behavior!
```

**Interview tip:** "String immutability ke 5 pillars: **Security**, **String Pool**, **Thread Safety**, **HashCode Caching**, **Class Loading**. Yeh sab milke Java ko robust banate hain."

## **Q2. String Constant Pool (SCP) in Java**

**Answer:**  
SCP (String Constant Pool) **Heap memory** ka ek special area hai jahan String literals store hote hain. Iska purpose **memory optimization** aur **reusability** hai.

**Location over Java versions:**
- **Java 6 and below:** PermGen (Method Area)
- **Java 7+:** Heap (moved to avoid PermGen OutOfMemoryError)
- **Java 8+:** Heap (PermGen removed, MetaSpace introduced but SCP still in Heap)

**How SCP works:**
```java
String s1 = "Hello";  // Created in SCP
String s2 = "Hello";  // Reuses same object from SCP
String s3 = "Hello";  // Same object again

System.out.println(s1 == s2);  // true (same reference)
```

**Memory diagram:**
```
Heap Memory:
┌─────────────────────────────────────┐
│  Eden │ Survivor │ Old Gen │  SCP   │
│       │         │         │        │
│       │         │         │ "Hello"│ ← s1, s2, s3 point here
│       │         │         │ "World"│
│       │         │         │ "Java" │
└─────────────────────────────────────┘
```

**What goes to SCP?**
```java
// 1. String literals - YES
String s1 = "Hello";           // Goes to SCP

// 2. intern() called - YES
String s2 = new String("Hi").intern();  // Goes to SCP

// 3. new String() - NO (goes to heap)
String s3 = new String("Hello");  // Creates separate object in heap

// 4. Concatenation of literals - YES (compile-time constant)
String s4 = "Hel" + "lo";  // "Hello" - compile time, goes to SCP

// 5. Concatenation with variable - NO (runtime)
String s5 = "Hel";
String s6 = s5 + "lo";  // Creates new object in heap
```

**SCP Size and Limits:**
```java
// Can't directly set SCP size, but can set Heap size
// -Xmx2g  // Total heap 2GB, SCP part of it

// To see SCP usage (jcmd)
jcmd <pid> VM.native_memory
```

**Interview tip:** "SCP is a **string pool/cache** in heap (Java 7+). Literals go here automatically. `new String()` creates separate heap object. Use `intern()` to manually add to pool. SCP prevents duplicate strings and saves memory."

## **Q3. Is it possible to compare Strings using the == operator?**

**Answer:**  
**Yes, possible but WRONG for content comparison.** `==` compares **reference** (memory address), not content.

**Demonstration:**
```java
String s1 = "Hello";
String s2 = "Hello";
String s3 = new String("Hello");

// Reference comparison
System.out.println(s1 == s2);  // true (same object from SCP)
System.out.println(s1 == s3);  // false (different objects)

// Content comparison - CORRECT way
System.out.println(s1.equals(s3));  // true
```

**When == works (but shouldn't rely on):**
```java
// Case 1: Same literal - works but don't rely
String a = "Java";
String b = "Java";
System.out.println(a == b);  // true (SCP reuse)

// Case 2: compile-time constants
final String c = "Hello";
final String d = "Hello";
String e = c + d;  // Compile-time constant?
// Be careful - not always predictable

// Case 3: intern() forces pool
String f = new String("Hi").intern();
String g = "Hi";
System.out.println(f == g);  // true (both in SCP now)
```

**Why never use == for content:**
```java
// User input example
String userInput = scanner.nextLine();  // User types "admin"
String hardcoded = "admin";

// This is DANGEROUS
if (userInput == hardcoded) {  // Almost always false
    grantAccess();  // Never executes!
}

// CORRECT way
if (userInput.equals(hardcoded)) {
    grantAccess();  // Works correctly
}
```

**Interview tip:** "`==` compares references, not content. Always use `.equals()` for String content comparison. The only time `==` works is when both strings are from SCP, but never rely on this - it's a bad practice."

## **Q4. What does String Intern Method do?**

**Answer:**  
`intern()` method **String Constant Pool mein object daalta hai** aur us pool se reference return karta hai. Agar string already pool mein hai, to existing reference return hota hai.

**How intern() works:**
```java
String s1 = new String("Hello");  // Heap object
String s2 = s1.intern();          // Puts in SCP, returns pool reference
String s3 = "Hello";              // From SCP

System.out.println(s1 == s2);  // false (heap vs pool)
System.out.println(s2 == s3);  // true (same pool object)
```

**Before and after intern():**
```java
// Without intern
String heap1 = new String("Java");
String heap2 = new String("Java");
String literal = "Java";

System.out.println(heap1 == heap2);   // false (different heap objects)
System.out.println(heap1 == literal); // false (heap vs pool)

// With intern
String interned1 = heap1.intern();
String interned2 = heap2.intern();

System.out.println(interned1 == interned2); // true (same pool object)
System.out.println(interned1 == literal);   // true (same pool object)
```

**Performance impact:**
```java
// ❌ BAD - Without intern (creates many objects)
for (int i = 0; i < 1_000_000; i++) {
    String s = new String("Data");  // 1 million heap objects!
    process(s);
}
// Memory: 1 million separate objects

// ✅ GOOD - With intern (reuses from pool)
for (int i = 0; i < 1_000_000; i++) {
    String s = new String("Data").intern();  // Single pool object
    process(s);
}
// Memory: 1 object in pool + references
```

**When to use intern():**
```java
// Good use - many duplicate strings
String[] thousandsOfRecords = loadData();
Set<String> uniqueValues = new HashSet<>();
for (String record : thousandsOfRecords) {
    uniqueValues.add(record.intern());  // Memory efficient
}

// Bad use - strings are mostly unique
String uniqueId = generateUniqueId().intern();  // Wastes memory
```

**Java 7+ change:**
```java
// Java 6: intern() put in PermGen (limited space)
// Java 7+: intern() put in Heap (more space, better GC)
```

**Interview tip:** "`intern()` manually adds string to SCP and returns pool reference. Useful for memory optimization when many duplicate strings. But don't overuse - only when duplicates are high. Java 7+ moved intern pool to heap, so better than Java 6."

## **Q5. StringBuffer vs StringBuilder**

**Answer:**

| Feature | StringBuffer | StringBuilder |
|---------|--------------|---------------|
| **Thread Safety** | ✅ Yes (synchronized) | ❌ No |
| **Performance** | Slower (due to synchronization) | Faster |
| **Introduced in** | Java 1.0 | Java 1.5 |
| **Use case** | Multi-threaded environment | Single-threaded |

**Performance difference:**
```java
// StringBuffer - thread-safe but slower
StringBuffer sb = new StringBuffer();
for (int i = 0; i < 100000; i++) {
    sb.append("a");  // Each append is synchronized
}
// Time: ~15-20ms

// StringBuilder - faster
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 100000; i++) {
    sb.append("a");  // No synchronization overhead
}
// Time: ~5-10ms
```

**Thread safety demonstration:**
```java
// StringBuffer - safe for multiple threads
StringBuffer buffer = new StringBuffer();
Runnable task = () -> {
    for (int i = 0; i < 1000; i++) {
        buffer.append("a");  // Synchronized internally
    }
};
// Multiple threads can safely use same buffer

// StringBuilder - UNSAFE for multiple threads
StringBuilder builder = new StringBuilder();
Runnable badTask = () -> {
    for (int i = 0; i < 1000; i++) {
        builder.append("a");  // No synchronization
    }
};
// May cause data corruption in multi-threaded environment
```

**When to use which:**
```java
// ✅ Use StringBuilder - Local variable (single thread)
public String buildName(String first, String last) {
    StringBuilder sb = new StringBuilder();
    sb.append(first).append(" ").append(last);
    return sb.toString();
}

// ✅ Use StringBuffer - Shared across threads
public class Logger {
    private StringBuffer logBuffer = new StringBuffer();  // Shared
    
    public synchronized void log(String message) {
        logBuffer.append(message).append("\n");
    }
}

// ✅ Alternative - Use StringBuilder with external sync
private StringBuilder sb = new StringBuilder();
public void append(String s) {
    synchronized(this) {
        sb.append(s);  // Manual sync
    }
}
```

**Interview tip:** "StringBuilder is faster but not thread-safe. StringBuffer is thread-safe but slower. In **99% of cases** (local variables), use StringBuilder. Use StringBuffer only when string buffer is shared across threads."

## **Q6. What is use of StringTokenizer?**

**Answer:**  
StringTokenizer **string ko tokens (smaller parts) mein todta hai** based on delimiter. Though legacy class (since Java 1.0), still used for simple parsing.

**Basic usage:**
```java
String data = "apple,banana,orange,mango";
StringTokenizer st = new StringTokenizer(data, ",");

while (st.hasMoreTokens()) {
    String token = st.nextToken();
    System.out.println(token);
}
// Output:
// apple
// banana
// orange
// mango
```

**Different constructors:**
```java
// 1. Default delimiter (whitespace: space, tab, newline)
StringTokenizer st1 = new StringTokenizer("Hello World Java");
while(st1.hasMoreTokens()) {
    System.out.println(st1.nextToken());
}
// Output: Hello, World, Java

// 2. Custom delimiter
StringTokenizer st2 = new StringTokenizer("a|b|c|d", "|");

// 3. Return delimiters as tokens
StringTokenizer st3 = new StringTokenizer("a|b|c", "|", true);
// Returns: a, |, b, |, c

// 4. Count tokens
int count = st.countTokens();
```

**Real-world use cases:**
```java
// 1. CSV parsing (simple)
String csv = "101,Rahul,Delhi,25";
StringTokenizer st = new StringTokenizer(csv, ",");
int id = Integer.parseInt(st.nextToken());
String name = st.nextToken();
String city = st.nextToken();
int age = Integer.parseInt(st.nextToken());

// 2. Command parsing
String command = "COPY source.txt destination.txt";
StringTokenizer st = new StringTokenizer(command);
String operation = st.nextToken();  // "COPY"
String source = st.nextToken();     // "source.txt"
String dest = st.nextToken();       // "destination.txt"

// 3. Configuration parsing
String config = "host=localhost;port=8080;timeout=30";
StringTokenizer st = new StringTokenizer(config, ";");
while(st.hasMoreTokens()) {
    String pair = st.nextToken();
    StringTokenizer pairSt = new StringTokenizer(pair, "=");
    String key = pairSt.nextToken();
    String value = pairSt.nextToken();
    // Process key-value
}
```

**Modern alternatives (preferred):**
```java
// 1. String.split() - Most common
String[] fruits = "apple,banana,orange".split(",");

// 2. Scanner
Scanner sc = new Scanner("apple banana orange");
while(sc.hasNext()) {
    System.out.println(sc.next());
}

// 3. Pattern (regex)
Pattern pattern = Pattern.compile(",");
String[] result = pattern.split("a,b,c");
```

**Comparison:**
| Feature | StringTokenizer | String.split() |
|---------|----------------|----------------|
| **Performance** | Slightly faster | Good |
| **Regex support** | No | Yes |
| **Delimiter as token** | Yes | No |
| **Recommended** | Legacy | ✅ Modern |

**Interview tip:** "StringTokenizer legacy class hai (since JDK 1.0). Simple parsing ke liye kaam aata hai. But modern code mein `String.split()` ya `Scanner` prefer karo. Use StringTokenizer only when you need to return delimiters as tokens."

## **Q7. Difference between String a = "ravi" and String a2 = new String("ravi")**

**Answer:**

| Feature | `String a = "ravi"` | `String a2 = new String("ravi")` |
|---------|---------------------|----------------------------------|
| **Memory location** | String Constant Pool (SCP) | Heap (regular object area) |
| **Object creation** | 0 or 1 object (reuses from pool) | Always creates new object |
| **Performance** | Faster | Slower (extra object) |
| **Memory usage** | Less (shared) | More (new object each time) |

**Detailed example:**
```java
// Literal way
String s1 = "ravi";  // Created in SCP
String s2 = "ravi";  // Reuses s1 object from SCP

System.out.println(s1 == s2);  // true (same reference)

// new String() way
String s3 = new String("ravi");  // New heap object
String s4 = new String("ravi");  // Another new heap object

System.out.println(s3 == s4);  // false (different objects)
System.out.println(s1 == s3);  // false (SCP vs Heap)
System.out.println(s1.equals(s3));  // true (same content)
```

**Memory diagram:**
```
Heap Memory:
┌──────────────────────────────────────┐
│                                      │
│  String Constant Pool (SCP)          │
│  ┌─────────────────────────────┐    │
│  │         "ravi"              │ ←── s1, s2 point here
│  └─────────────────────────────┘    │
│                                      │
│  Regular Heap Area                   │
│  ┌─────────────────────────────┐    │
│  │  String Object "ravi"       │ ←── s3 points here
│  └─────────────────────────────┘    │
│  ┌─────────────────────────────┐    │
│  │  String Object "ravi"       │ ←── s4 points here
│  └─────────────────────────────┘    │
└──────────────────────────────────────┘
```

**How many objects created?**
```java
// Case 1: String s = "ravi";
// If "ravi" already in pool: 0 new objects
// If not in pool: 1 object in pool

// Case 2: String s = new String("ravi");
// Always: 1 or 2 objects
// - 1 object in heap (always)
// - If "ravi" not in pool: 1 object in pool also (total 2)
```

**Performance impact:**
```java
// ❌ BAD - Creating unnecessary objects
for (int i = 0; i < 100000; i++) {
    String s = new String("Hello");  // 100,000 heap objects!
}

// ✅ GOOD - Using literal
for (int i = 0; i < 100000; i++) {
    String s = "Hello";  // Single SCP object reused
}
```

**Interview tip:** "Literal (`= "ravi"`) uses SCP - memory efficient and faster. `new String("ravi")` always creates new heap object - use only when you need a distinct object. In 99% cases, use literal. `new String()` is almost never needed."

## **Q8. Difference between String, StringBuffer, StringBuilder**

**Answer:**

| Feature | String | StringBuffer | StringBuilder |
|---------|--------|--------------|---------------|
| **Mutability** | Immutable | Mutable | Mutable |
| **Thread Safety** | Safe (immutable) | ✅ Synchronized | ❌ Not safe |
| **Performance** | Slow for modifications | Moderate | Fastest |
| **Memory** | Creates new objects on change | Modifies existing | Modifies existing |
| **Use case** | Fixed/rarely changed | Multi-threaded modifications | Single-threaded modifications |

**Performance comparison:**
```java
// String - worst for modification
String str = "Hello";
for (int i = 0; i < 10000; i++) {
    str += "!";  // Creates new String object each time!
}
// Creates 10,000 temporary String objects
// Time: ~50-100ms

// StringBuffer - better
StringBuffer sb = new StringBuffer("Hello");
for (int i = 0; i < 10000; i++) {
    sb.append("!");  // Modifies same object
}
// Time: ~5-10ms

// StringBuilder - best
StringBuilder sb = new StringBuilder("Hello");
for (int i = 0; i < 10000; i++) {
    sb.append("!");  // Modifies same object, no sync
}
// Time: ~2-5ms
```

**When to use which:**
```java
// 1. String - When string doesn't change
String message = "Welcome to Java";
String errorCode = "ERR_404";
String name = "John";  // Even if name changes rarely, String is fine

// 2. StringBuilder - When building strings in single thread
public String buildUserProfile(User user) {
    StringBuilder sb = new StringBuilder();
    sb.append("Name: ").append(user.getName())
      .append(", Age: ").append(user.getAge())
      .append(", City: ").append(user.getCity());
    return sb.toString();
}

// 3. StringBuffer - When multiple threads modify same buffer
public class SharedLogger {
    private StringBuffer logBuffer = new StringBuffer();
    
    public void log(String message) {
        synchronized(this) {
            logBuffer.append(message).append("\n");
        }
    }
}
```

**Method availability:**
```java
// All three have similar methods, but:
String str = "Hello";
str.charAt(0);  // Available
// str.append("!");  // ❌ NOT available

StringBuilder sb = new StringBuilder("Hello");
sb.append("!");     // ✅ Available
sb.insert(0, "Hi"); // ✅ Available
sb.reverse();       // ✅ Available
sb.delete(0, 2);    // ✅ Available
```

**Interview tip:** "String for fixed text, StringBuilder for single-threaded modifications, StringBuffer for multi-threaded modifications. StringBuilder is 10-15% faster than StringBuffer. Never use String for repeated concatenation in loops."

## **Q9. Millions of temporary StringBuilder objects causing GC pauses**

**Answer:**  
(Yeh pehle Q2 mein answer kar chuka hoon, let me give you **concise version** focusing on StringBuilder specifically)

**Problem:** Millions of short-lived StringBuilder objects → Young GC frequently triggers → STW pauses.

**Root cause:**
```java
// ❌ Bad pattern
for (int i = 0; i < 1_000_000; i++) {
    StringBuilder sb = new StringBuilder();  // New object each iteration
    sb.append("Data ").append(i);
    String result = sb.toString();
    // sb eligible for GC immediately
}
```

**Solutions:**

**1. Reuse with ThreadLocal (Best):**
```java
public class StringBuilderPool {
    private static final ThreadLocal<StringBuilder> pool = 
        ThreadLocal.withInitial(() -> new StringBuilder(1024));
    
    public static StringBuilder get() {
        StringBuilder sb = pool.get();
        sb.setLength(0);  // Clear without new allocation
        return sb;
    }
}

// Usage
for (int i = 0; i < 1_000_000; i++) {
    StringBuilder sb = StringBuilderPool.get();
    sb.append("Data ").append(i);
    String result = sb.toString();
    // No GC pressure - same object reused
}
```

**2. Pre-allocate capacity:**
```java
// Estimate max size needed
StringBuilder sb = new StringBuilder(1024);  // Prevents resizing
```

**3. Return to pool pattern:**
```java
public class ReusableStringBuilder {
    private static final Queue<StringBuilder> pool = 
        new ConcurrentLinkedQueue<>();
    
    public static StringBuilder borrow() {
        StringBuilder sb = pool.poll();
        if (sb == null) {
            sb = new StringBuilder(1024);
        }
        sb.setLength(0);
        return sb;
    }
    
    public static void return_(StringBuilder sb) {
        if (sb != null && sb.capacity() < 4096) {
            pool.offer(sb);  // Only cache reasonably sized
        }
    }
}
```

**4. JVM tuning (if memory can be increased slightly):**
```bash
# Increase young generation to reduce GC frequency
-XX:NewSize=512m -XX:MaxNewSize=512m
# Use G1GC for better pause time control
-XX:+UseG1GC
-XX:MaxGCPauseMillis=50
```

**Interview tip:** "Main fix - **Reuse StringBuilder objects** with ThreadLocal or object pool. Set capacity appropriately. Never create StringBuilder inside loops. For extreme cases, use **String concatenation** with Java 9+ optimizations (invokedynamic)."

## **Q10. In what scenario StringBuilder should be preferred over String class?**

**Answer:**  
StringBuilder prefer karo jab **multiple modifications** karne honge string par.

**Scenarios:**

**1. Loops with concatenation (Most important):**
```java
// ❌ BAD - String in loop
String result = "";
for (int i = 0; i < 1000; i++) {
    result += i + ",";  // Creates 1000 intermediate String objects
}

// ✅ GOOD - StringBuilder in loop
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 1000; i++) {
    sb.append(i).append(",");  // Single object, modified
}
String result = sb.toString();
```

**2. Building large strings dynamically:**
```java
// ❌ BAD - Multiple String objects
String html = "<html>";
html += "<head><title>Page</title></head>";
html += "<body>";
for (User user : users) {
    html += "<div>" + user.getName() + "</div>";
}
html += "</body></html>";

// ✅ GOOD - StringBuilder
StringBuilder sb = new StringBuilder(1024);  // Pre-allocate
sb.append("<html>");
sb.append("<head><title>Page</title></head>");
sb.append("<body>");
for (User user : users) {
    sb.append("<div>").append(user.getName()).append("</div>");
}
sb.append("</body></html>");
```

**3. JSON/XML generation:**
```java
public String toJson(List<Product> products) {
    StringBuilder sb = new StringBuilder();
    sb.append("[");
    for (int i = 0; i < products.size(); i++) {
        if (i > 0) sb.append(",");
        sb.append("{")
          .append("\"id\":").append(products.get(i).getId())
          .append(",\"name\":\"").append(products.get(i).getName())
          .append("\"}");
    }
    sb.append("]");
    return sb.toString();
}
```

**4. Complex string manipulation:**
```java
// StringBuilder provides useful methods
StringBuilder sb = new StringBuilder("Hello World");
sb.insert(6, "Beautiful ");  // "Hello Beautiful World"
sb.delete(5, 6);              // "Hell Beautiful World"
sb.reverse();                 // "dlroW lufituaeB lleH"
sb.replace(0, 4, "Hi");       // "Hi lufituaeB lleH"
```

**When to use String instead:**
```java
// ✅ Use String for:
// 1. Simple concatenation (compiler optimizes)
String msg = "Hello " + name + "! Welcome to " + city;
// Compiler converts to StringBuilder internally

// 2. String literals
String constant = "FIXED_VALUE";

// 3. Method parameters (immutability is good)
public void process(String data) { }  // String is fine

// 4. Keys in HashMap (immutability ensures hashcode stability)
map.put(key, value);  // String as key is perfect
```

**Performance numbers:**
```java
// 10,000 concatenations
String:      ~50-100ms (creates 10,000 objects)
StringBuilder: ~2-5ms  (0 intermediate objects)
// 100x faster for large loops!
```

**Interview tip:** "Use StringBuilder when you have **multiple modifications** - loops, dynamic building, frequent appends. Use String for **fixed content** and **HashMap keys**. For simple concatenation of few strings, compiler optimizes String anyway."

## **Q11. Why char array is preferred to store password than strings?**

**Answer:**  
**Security reason** - char array can be **explicitly cleared** (overwritten), String cannot.

**Core problem with String:**
```java
String password = "secret123";
// Once created, String is immutable
// Can't erase password from memory
// Password stays in heap/SCP until GC
// GC is unpredictable - password could stay for minutes/hours
```

**With char array:**
```java
char[] password = {'s', 'e', 'c', 'r', 'e', 't', '1', '2', '3'};
// Use password...
Arrays.fill(password, ' ');  // IMMEDIATELY overwrite!
// Password cleared from memory right after use
```

**Security vulnerability demonstration:**
```java
// BAD - Using String for password
public void login(String password) {
    // Password stored in String
    // Memory dump could reveal password even after method ends
    // String stays in memory until GC
}

// GOOD - Using char array
public void login(char[] password) {
    try {
        // Authenticate
        authenticate(password);
    } finally {
        Arrays.fill(password, ' ');  // Clear immediately
    }
}
```

**Memory dump attack:**
```java
// Attacker can take heap dump
// jmap -dump:live,format=b,file=heap.bin <pid>

// Searching heap dump reveals:
// - String passwords: Plain text visible
// - char[] passwords: Can be overwritten before dump
```

**Why String is dangerous:**
```java
String pwd = "secret";
// 1. String goes to String Constant Pool (might never be GC'd)
// 2. Even if not in pool, GC timing unpredictable
// 3. String interning makes it permanent
// 4. Substring creates new String referencing same char[] (Java 6 issue)
```

**Best practice code:**
```java
public class SecureAuthenticator {
    
    public boolean authenticate(char[] password) {
        try {
            // Verify password
            boolean isValid = verify(password);
            return isValid;
        } finally {
            // ALWAYS clear password from memory
            clearArray(password);
        }
    }
    
    private void clearArray(char[] array) {
        if (array != null) {
            Arrays.fill(array, '\0');  // Overwrite with null char
        }
    }
}

// Usage
char[] pwd = passwordField.getPassword();  // Swing returns char[]
authenticator.authenticate(pwd);
Arrays.fill(pwd, '\0');  // Clear even after method
```

**Additional security measures:**
```java
// 1. Use try-finally for guaranteed cleanup
try {
    // use password
} finally {
    Arrays.fill(password, '\0');
}

// 2. Don't log passwords
logger.info("User logged in");  // Don't log password

// 3. Don't store passwords in collections
// List<String> passwords = new ArrayList<>();  // NO!

// 4. Use SecureString in some frameworks
```

**Interview tip:** "String immutable hai - memory se erase nahi ho sakta. char array ko immediately overwrite kar sakte ho. Memory dump attacks se bachata hai. Financial/security applications mein **always use char[] for passwords**. Also, never log passwords, never store in collections."

## **Q12. Where does memory get allocated for a String when we instantiate it with the new operator?**

**Answer:**  
`new String("...")` se **Heap memory** (regular object area) mein allocation hota hai, **String Constant Pool mein nahi** (unless intern() called).

**Memory allocation:**
```java
String s = new String("Hello");
```

**What happens:**
1. **Heap memory** mein new String object created
2. **String Constant Pool** mein already "Hello" check hota hai
3. If "Hello" not in pool → one object created in pool also
4. New heap object **references** the char[] from pool (in most JVM implementations)

**Memory diagram:**
```
Heap Memory:
┌─────────────────────────────────────────────┐
│                                             │
│  String Constant Pool (SCP)                 │
│  ┌──────────────────────────────────┐      │
│  │      char[] ['H','e','l','l','o']│      │
│  └──────────────────────────────────┘      │
│           ↑                                 │
│           │ (references)                    │
│  Regular Heap Area                          │
│  ┌──────────────────────────────────┐      │
│  │  String object                    │      │
│  │  - hash: 0 (cached later)        │      │
│  │  - value: ref to SCP char[] ─────┘      │
│  └──────────────────────────────────┘      │
└─────────────────────────────────────────────┘
```

**How many objects created?**
```java
// Case 1: String literal not in pool
String s = new String("NewString");
// Creates 2 objects:
// 1. In SCP: "NewString" (if not exists)
// 2. In Heap: new String object

// Case 2: String literal already in pool
String existing = "Hello";  // Creates in SCP
String s = new String("Hello");
// Creates 1 new object:
// 1. In Heap: new String object (reuses SCP char[])
```

**Verify with code:**
```java
String s1 = "Hello";
String s2 = new String("Hello");
String s3 = new String("Hello");

System.out.println(s1 == s2);  // false (SCP vs Heap)
System.out.println(s2 == s3);  // false (different heap objects)
System.out.println(s1 == s1.intern());  // true (same pool)

// Check memory address (conceptually)
System.identityHashCode(s1);  // Different from s2
System.identityHashCode(s2);  // Different from s3
```

**Java version differences:**
```java
// Java 6 and before
// - SCP in PermGen (limited memory)
// - new String() always creates separate char[] copy

// Java 7+
// - SCP moved to Heap
// - Optimization: new String() may share char[] with pool
// - Still separate String object in heap
```

**Memory impact:**
```java
// ❌ Memory waste
for (int i = 0; i < 100000; i++) {
    String s = new String("Hello");  // 100,000 heap objects!
}

// ✅ Memory efficient
for (int i = 0; i < 100000; i++) {
    String s = "Hello";  // Single SCP object reused
}
```

**Interview tip:** "`new String()` allocates in **heap** (regular object area), not in SCP. It creates a new String object each time. The underlying char[] may be shared with SCP for optimization. Avoid `new String()` unless you specifically need a distinct object. Use literal for normal cases."
