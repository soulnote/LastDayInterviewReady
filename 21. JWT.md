

## Q1. How JWT token works in Java?

**Answer:**  
JWT (JSON Web Token) is a **compact, URL-safe token** that carries claims between parties. It consists of three Base64Url-encoded parts separated by dots: **Header.Payload.Signature**.

**JWT Structure:**

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4iLCJpYXQiOjE1MTYyMzkwMjJ9.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

Part 1 (Header):     eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
Part 2 (Payload):    eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4iLCJpYXQiOjE1MTYyMzkwMjJ9
Part 3 (Signature):  SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
```

```java
@Component
public class JwtTokenProvider {

    @Value("${jwt.secret}")
    private String jwtSecret;
    
    @Value("${jwt.expiration}")
    private int jwtExpirationMs;
    
    // 1. Generate JWT Token
    public String generateToken(Authentication authentication) {
        UserDetails userDetails = (UserDetails) authentication.getPrincipal();
        
        return Jwts.builder()
            .setSubject(userDetails.getUsername())
            .setIssuedAt(new Date())
            .setExpiration(new Date(System.currentTimeMillis() + jwtExpirationMs))
            .signWith(getSigningKey(), SignatureAlgorithm.HS512)
            .compact();
    }
    
    // 2. Extract username from token
    public String getUsernameFromToken(String token) {
        return Jwts.parserBuilder()
            .setSigningKey(getSigningKey())
            .build()
            .parseClaimsJws(token)
            .getBody()
            .getSubject();
    }
    
    // 3. Validate JWT token
    public boolean validateToken(String token) {
        try {
            Jwts.parserBuilder()
                .setSigningKey(getSigningKey())
                .build()
                .parseClaimsJws(token);
            return true;
        } catch (JwtException | IllegalArgumentException e) {
            return false;
        }
    }
    
    // 4. Get signing key
    private Key getSigningKey() {
        byte[] keyBytes = Decoders.BASE64.decode(jwtSecret);
        return Keys.hmacShaKeyFor(keyBytes);
    }
}
```

**JWT Authentication Filter:**

```java
@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {
    
    @Autowired
    private JwtTokenProvider tokenProvider;
    
    @Autowired
    private UserDetailsService userDetailsService;
    
    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain filterChain) throws ServletException, IOException {
        
        String jwt = parseJwt(request);
        
        if (jwt != null && tokenProvider.validateToken(jwt)) {
            String username = tokenProvider.getUsernameFromToken(jwt);
            UserDetails userDetails = userDetailsService.loadUserByUsername(username);
            
            UsernamePasswordAuthenticationToken authentication = 
                new UsernamePasswordAuthenticationToken(userDetails, null, userDetails.getAuthorities());
            
            SecurityContextHolder.getContext().setAuthentication(authentication);
        }
        
        filterChain.doFilter(request, response);
    }
    
    private String parseJwt(HttpServletRequest request) {
        String headerAuth = request.getHeader("Authorization");
        if (headerAuth != null && headerAuth.startsWith("Bearer ")) {
            return headerAuth.substring(7);
        }
        return null;
    }
}
```

## Q2. What are differences between Authentication and Authorization?

**Answer:**

| Aspect | Authentication | Authorization |
|--------|----------------|---------------|
| **Purpose** | Verify identity | Verify permissions |
| **Question** | "Who are you?" | "What can you do?" |
| **Timing** | First (login) | After authentication |
| **Data** | Username, password, biometrics | Roles, permissions, scopes |
| **HTTP Status** | 401 Unauthorized | 403 Forbidden |
| **Storage** | Credentials | Permissions/Roles |
| **Example** | "I am John" | "John can READ, not WRITE" |

**Code Example:**

```java
@RestController
public class DocumentController {
    
    // AUTHENTICATION - User must be logged in
    @GetMapping("/documents")
    public List<Document> getDocuments() {
        Authentication auth = SecurityContextHolder.getContext().getAuthentication();
        if (auth == null || !auth.isAuthenticated()) {
            throw new ResponseStatusException(HttpStatus.UNAUTHORIZED);
        }
        return documentService.findAll();
    }
    
    // AUTHORIZATION - User must have ADMIN role
    @DeleteMapping("/documents/{id}")
    @PreAuthorize("hasRole('ADMIN')")
    public void deleteDocument(@PathVariable Long id) {
        documentService.delete(id);
    }
    
    // AUTHORIZATION - User must own the document
    @PutMapping("/documents/{id}")
    @PreAuthorize("#id == authentication.principal.userId or hasRole('ADMIN')")
    public Document updateDocument(@PathVariable Long id, @RequestBody Document doc) {
        return documentService.update(id, doc);
    }
}
```

**Interview Tip:** "Authentication = identity verification (401). Authorization = permission verification (403). Authentication happens first, then authorization. Spring Security handles both."

## Q3. How JWT authentication is stateless?

**Answer:**  
JWT is stateless because the **server does NOT store any session information**. All necessary user data is contained within the JWT token itself.

**Stateless vs Stateful Comparison:**

```
STATEFUL (Session-based):
┌────────┐     ┌────────┐     ┌─────────────────┐
│ Client │────▶│ Server │────▶│ Session Store   │
│        │◀────│        │     │ (Redis/Memory)  │
└────────┘     └────────┘     └─────────────────┘
- Server stores session ID
- Needs shared session store for scaling

STATELESS (JWT-based):
┌────────┐     ┌────────┐
│ Client │────▶│ Server │
│        │◀────│        │
└────────┘     └────────┘
- JWT contains all user data
- No server-side storage
- Scales horizontally easily
```

**Why JWT is Stateless:**

```java
@Component
public class StatelessJwtService {
    
    // 1. Login - Only time database is touched
    public String login(String username, String password) {
        User user = userRepository.findByUsername(username);  // DB call
        if (passwordEncoder.matches(password, user.getPassword())) {
            // Generate JWT with ALL user data inside
            return Jwts.builder()
                .setSubject(user.getUsername())
                .claim("userId", user.getId())
                .claim("roles", user.getRoles())
                .claim("permissions", user.getPermissions())
                .setExpiration(new Date(System.currentTimeMillis() + 3600000))
                .signWith(SECRET_KEY)
                .compact();
        }
        throw new BadCredentialsException("Invalid credentials");
    }
    
    // 2. Subsequent requests - NO DATABASE CALL!
    public Authentication validateToken(String token) {
        // Validate signature and extract claims - NO DB!
        Claims claims = Jwts.parserBuilder()
            .setSigningKey(SECRET_KEY)
            .build()
            .parseClaimsJws(token)
            .getBody();
        
        // User data comes from JWT, not database
        String username = claims.getSubject();
        List<String> roles = claims.get("roles", List.class);
        
        List<GrantedAuthority> authorities = roles.stream()
            .map(role -> new SimpleGrantedAuthority("ROLE_" + role))
            .collect(Collectors.toList());
        
        return new UsernamePasswordAuthenticationToken(username, null, authorities);
    }
}
```

**Stateless Configuration:**

```java

    }
}
```

**Interview Tip:** "JWT is stateless because server doesn't store session. All user data (roles, permissions) in token. Validate cryptographically without DB. Scales horizontally easily. Downside: hard to revoke individual tokens."

## Q4. What are differences between stateless and stateful authentication?

**Answer:**

| Feature | Stateless (JWT) | Stateful (Session) |
|---------|-----------------|-------------------|
| **Server storage** | None | Session store (Redis/DB) |
| **Client storage** | JWT token | Session cookie (Session ID) |
| **Scalability** | Excellent (any server) | Needs shared session store |
| **Logout** | Complex (token blacklist) | Simple (invalidate session) |
| **Token size** | Large (contains claims) | Small (just Session ID) |
| **Performance** | Fast (no lookup) | Slower (session lookup) |
| **CSRF protection** | Not needed | Required |
| **Revocation** | Hard (until expiry) | Easy (immediate) |
| **Use case** | REST APIs, microservices | Traditional web apps |

**Code Examples:**

```java
// STATEFUL (Session-based)
@RestController
public class StatefulAuthController {
    
    @PostMapping("/login")
    public ResponseEntity<?> login(HttpSession session, @RequestBody LoginRequest request) {
        User user = authenticate(request);
        session.setAttribute("user", user);  // Store in session
        session.setAttribute("roles", user.getRoles());
        return ResponseEntity.ok().build();
    }
    
    @GetMapping("/profile")
    public User getProfile(HttpSession session) {
        User user = (User) session.getAttribute("user");  // Read from session
        if (user == null) throw new RuntimeException("Not logged in");
        return user;
    }
    
    @PostMapping("/logout")
    public void logout(HttpSession session) {
        session.invalidate();  // Simple logout
    }
}

// STATELESS (JWT-based)
@RestController
public class StatelessAuthController {
    
    @PostMapping("/login")
    public ResponseEntity<JwtResponse> login(@RequestBody LoginRequest request) {
        User user = authenticate(request);
        String token = jwtProvider.generateToken(user);  // No session!
        return ResponseEntity.ok(new JwtResponse(token));
    }
    
    @GetMapping("/profile")
    public User getProfile(@RequestHeader("Authorization") String token) {
        String username = jwtProvider.getUsernameFromToken(token);  // From JWT
        return userService.findByUsername(username);
    }
    
    @PostMapping("/logout")
    public void logout(@RequestHeader("Authorization") String token) {
        // Complex - need token blacklist or just let it expire
        tokenBlacklist.add(token);
    }
}
```

**When to Use Which:**

```java
// ✅ Use STATEFUL (Session) for:
// - Traditional web apps (Thymeleaf, JSP)
// - Need immediate logout/revocation
// - Small to medium scale
// - Users trust browser

// ✅ Use STATELESS (JWT) for:
// - REST APIs / Microservices
// - Mobile app backends
// - High scalability requirements
// - Multiple services/instances
```

**Interview Tip:** "Stateful = server stores session (needs shared store). Stateless = JWT contains all data (no server storage). Stateful easier logout, Stateless better scaling. Stateful needs CSRF, Stateless doesn't."

## Q5. Which algorithm implementation you have used to generate a JWT token?

**Answer:**  
I have used **HS256 (HMAC-SHA256)** for symmetric signing and **RS256 (RSA-SHA256)** for asymmetric signing.

**HS256 Implementation (Symmetric):**

```java
@Component
public class HmacJwtService {
    
    @Value("${jwt.secret}")
    private String secret;  // Base64 encoded secret (min 256 bits)
    
    private Key getSigningKey() {
        byte[] keyBytes = Decoders.BASE64.decode(secret);
        return Keys.hmacShaKeyFor(keyBytes);
    }
    
    public String generateToken(String username) {
        return Jwts.builder()
            .setSubject(username)
            .setIssuedAt(new Date())
            .setExpiration(new Date(System.currentTimeMillis() + 3600000))
            .signWith(getSigningKey(), SignatureAlgorithm.HS256)
            .compact();
    }
}
```

**RS256 Implementation (Asymmetric):**

```java
@Component
public class RsaJwtService {
    
    @Value("${jwt.private-key}")
    private Resource privateKeyResource;
    
    @Value("${jwt.public-key}")
    private Resource publicKeyResource;
    
    private PrivateKey privateKey;
    private PublicKey publicKey;
    
    @PostConstruct
    public void init() throws IOException {
        String privateKeyContent = new String(privateKeyResource.getInputStream().readAllBytes());
        String publicKeyContent = new String(publicKeyResource.getInputStream().readAllBytes());
        
        privateKey = loadPrivateKey(privateKeyContent);
        publicKey = loadPublicKey(publicKeyContent);
    }
    
    public String generateToken(String username) {
        return Jwts.builder()
            .setSubject(username)
            .setIssuedAt(new Date())
            .setExpiration(new Date(System.currentTimeMillis() + 3600000))
            .signWith(privateKey, SignatureAlgorithm.RS256)
            .compact();
    }
    
    public boolean validateToken(String token) {
        try {
            Jwts.parserBuilder()
                .setSigningKey(publicKey)
                .build()
                .parseClaimsJws(token);
            return true;
        } catch (JwtException e) {
            return false;
        }
    }
}
```

**Generate RSA Keys:**

```bash
# Generate private key
openssl genrsa -out private.pem 2048

# Extract public key
openssl rsa -in private.pem -pubout -out public.pem

# Convert to PKCS8 for Java
openssl pkcs8 -topk8 -inform PEM -outform PEM -in private.pem -out private-pkcs8.pem -nocrypt
```

**Interview Tip:** "HS256 (symmetric) - single secret key, fast, for single service. RS256 (asymmetric) - private/public key pair, for microservices. HS256 needs 256-bit secret. RS256 needs 2048-bit RSA key."

## Q6. Types of Signing Algorithms while creating JWT Token?

**Answer:**  

| Algorithm | Type | Key Size | Speed | Use Case |
|-----------|------|----------|-------|----------|
| **HS256** | Symmetric (HMAC-SHA256) | 256-bit | Fastest | Single service |
| **HS384** | Symmetric (HMAC-SHA384) | 384-bit | Fast | Higher security |
| **HS512** | Symmetric (HMAC-SHA512) | 512-bit | Fast | Highest security |
| **RS256** | Asymmetric (RSA-SHA256) | 2048-bit | Moderate | Microservices |
| **RS384** | Asymmetric (RSA-SHA384) | 2048-bit | Moderate | Higher security |
| **RS512** | Asymmetric (RSA-SHA512) | 2048-bit | Moderate | Highest security |
| **ES256** | Asymmetric (ECDSA-P256) | 256-bit | Fast | Mobile/IoT |
| **ES384** | Asymmetric (ECDSA-P384) | 384-bit | Fast | Higher security |
| **ES512** | Asymmetric (ECDSA-P521) | 521-bit | Moderate | Highest security |
| **PS256** | Asymmetric (RSA-PSS) | 2048-bit | Slower | FIPS compliance |

**Algorithm Selection Guide:**

```java
@Service
public class AlgorithmSelector {
    
    public SignatureAlgorithm selectAlgorithm(Environment env) {
        // Single service, internal use
        if (isSingleService()) {
            return SignatureAlgorithm.HS256;  // Fastest, simplest
        }
        
        // Microservices with public key distribution
        if (needsPublicKeyDistribution()) {
            return SignatureAlgorithm.RS256;  // Industry standard
        }
        
        // Mobile/IoT devices (limited battery/compute)
        if (isMobileOrIoT()) {
            return SignatureAlgorithm.ES256;  // Smaller keys, faster
        }
        
        // Financial/Government compliance
        if (needsHighSecurity()) {
            return SignatureAlgorithm.PS256;  // FIPS 140-2 compliant
        }
        
        return SignatureAlgorithm.RS256;
    }
}
```

**Interview Tip:** "HS256 (symmetric, fast) - single service. RS256 (asymmetric) - microservices. ES256 (elliptic curve) - mobile/IoT. PS256 (RSA-PSS) - highest security. Never use 'none' algorithm."

## Q7. How do you handle JWT tokens in a React app?

**Answer:**  
Store JWT in **memory** (state) or **httpOnly cookie** (most secure), include in Authorization header.

**Option 1: HttpOnly Cookie (Most Secure)**

```javascript
// Server sets cookie with httpOnly flag
@PostMapping("/login")
public ResponseEntity<?> login(@RequestBody LoginRequest request) {
    String token = jwtProvider.generateToken(user);
    
    ResponseCookie cookie = ResponseCookie.from("jwt_token", token)
        .httpOnly(true)   // Not accessible via JavaScript
        .secure(true)     // HTTPS only
        .sameSite("Strict")
        .path("/")
        .maxAge(Duration.ofHours(1))
        .build();
    
    return ResponseEntity.ok()
        .header(HttpHeaders.SET_COOKIE, cookie.toString())
        .body(new LoginResponse("Login successful"));
}

// React - no token handling needed, browser sends cookie automatically
fetch('/api/user/profile', {
    method: 'GET',
    credentials: 'include'  // Important: send cookies
});
```

**Option 2: Memory Storage (SPA)**

```javascript
// auth.service.js
class AuthService {
    constructor() {
        this.token = null;
        this.user = null;
    }
    
    async login(username, password) {
        const response = await fetch('/api/auth/login', {
            method: 'POST',
            headers: { 'Content-Type': 'application/json' },
            body: JSON.stringify({ username, password })
        });
        
        const data = await response.json();
        this.token = data.accessToken;
        this.user = data.user;
        
        // Store in memory only (not localStorage)
        return data;
    }
    
    getToken() {
        return this.token;
    }
    
    logout() {
        this.token = null;
        this.user = null;
    }
}

// Axios interceptor to add token
axios.interceptors.request.use(config => {
    const token = authService.getToken();
    if (token) {
        config.headers['Authorization'] = `Bearer ${token}`;
    }
    return config;
});
```

**Token Refresh Logic:**

```javascript
// Axios response interceptor for token refresh
let isRefreshing = false;
let failedQueue = [];

axios.interceptors.response.use(
    response => response,
    async error => {
        const originalRequest = error.config;
        
        if (error.response?.status === 401 && !originalRequest._retry) {
            if (isRefreshing) {
                return new Promise((resolve, reject) => {
                    failedQueue.push({ resolve, reject });
                }).then(() => {
                    return axios(originalRequest);
                }).catch(err => Promise.reject(err));
            }
            
            originalRequest._retry = true;
            isRefreshing = true;
            
            try {
                const response = await axios.post('/api/auth/refresh', {}, {
                    withCredentials: true
                });
                const newToken = response.data.accessToken;
                authService.setToken(newToken);
                
                failedQueue.forEach(({ resolve }) => resolve());
                return axios(originalRequest);
            } catch (refreshError) {
                failedQueue.forEach(({ reject }) => reject(refreshError));
                authService.logout();
                window.location.href = '/login';
                return Promise.reject(refreshError);
            } finally {
                isRefreshing = false;
                failedQueue = [];
            }
        }
        
        return Promise.reject(error);
    }
);
```

**Interview Tip:** "Store JWT in **httpOnly cookie** (most secure, XSS protected) or **memory** (SPA). Never store in localStorage (XSS vulnerable). Use axios interceptors for automatic token refresh. Implement logout to clear token."

## Q8. What if a person gets valid token from other and tries to hit API from Postman?

**Answer:**  
Token theft allows attacker to **impersonate the legitimate user** until token expires. Multiple mitigation strategies needed.

**Mitigation Strategies:**

**1. Short Token Expiration (TTL):**

```java
@Component
public class ShortLivedJwtService {
    
    // Access token: 15 minutes only
    public String generateAccessToken(String username) {
        return Jwts.builder()
            .setSubject(username)
            .setExpiration(new Date(System.currentTimeMillis() + 900000)) // 15 min
            .signWith(SECRET_KEY)
            .compact();
    }
    
    // Refresh token: 7 days, stored in Redis
    public String generateRefreshToken(String username) {
        String refreshToken = UUID.randomUUID().toString();
        redisTemplate.opsForValue().set(
            "refresh:" + refreshToken,
            username,
            7, TimeUnit.DAYS
        );
        return refreshToken;
    }
}
```

**2. Token Fingerprinting (Bind to Device):**

```java
@Component
public class FingerprintJwtService {
    
    public String generateToken(String username, HttpServletRequest request) {
        String fingerprint = generateFingerprint(request);
        
        return Jwts.builder()
            .setSubject(username)
            .claim("fingerprint", fingerprint)
            .claim("userAgent", request.getHeader("User-Agent"))
            .claim("ipHash", hashIp(request.getRemoteAddr()))
            .setExpiration(new Date(System.currentTimeMillis() + 900000))
            .signWith(SECRET_KEY)
            .compact();
    }
    
    public boolean validateToken(String token, HttpServletRequest request) {
        Claims claims = getClaims(token);
        String storedFingerprint = claims.get("fingerprint", String.class);
        String currentFingerprint = generateFingerprint(request);
        
        if (!storedFingerprint.equals(currentFingerprint)) {
            // Token stolen! Revoke all user tokens
            revokeAllUserTokens(claims.getSubject());
            return false;
        }
        return true;
    }
    
    private String generateFingerprint(HttpServletRequest request) {
        String userAgent = request.getHeader("User-Agent");
        String acceptLanguage = request.getHeader("Accept-Language");
        return DigestUtils.sha256Hex(userAgent + acceptLanguage + SECRET_SALT);
    }
}
```

**3. Token Revocation (Blacklist):**

```java
@Service
public class TokenRevocationService {
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    public void revokeToken(String token) {
        long ttl = getRemainingTTL(token);
        redisTemplate.opsForValue().set(
            "blacklist:" + token,
            "revoked",
            ttl, TimeUnit.MILLISECONDS
        );
    }
    
    public boolean isRevoked(String token) {
        return Boolean.TRUE.equals(redisTemplate.hasKey("blacklist:" + token));
    }
}

// JWT Filter with revocation check
@Component
public class RevocationAwareJwtFilter extends OncePerRequestFilter {
    
    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain filterChain) throws ServletException, IOException {
        
        String token = extractToken(request);
        
        if (token != null && tokenRevocationService.isRevoked(token)) {
            response.setStatus(HttpServletResponse.SC_UNAUTHORIZED);
            response.getWriter().write("Token has been revoked");
            return;
        }
        
        filterChain.doFilter(request, response);
    }
}
```

**4. Token Rotation (Each request gets new token):**

```java
@Component
public class TokenRotationService {
    
    public String rotateToken(String oldToken) {
        if (isTokenRevoked(oldToken)) {
            throw new SecurityException("Token revoked - possible theft detected");
        }
        
        // Revoke old token
        revokeToken(oldToken);
        
        // Generate new token with same user data
        String username = getUsernameFromToken(oldToken);
        return generateToken(username);
    }
}
```

**5. Anomaly Detection:**

```java
@Component
public class AnomalyDetector {
    
    public void detectAnomaly(String token, HttpServletRequest request) {
        String username = getUsernameFromToken(token);
        String currentIp = request.getRemoteAddr();
        String previousIp = redisTemplate.opsForValue().get("user:lastIp:" + username);
        
        if (previousIp != null && !previousIp.equals(currentIp)) {
            // IP changed - possible token theft
            logSecurityEvent("IP change detected for user: " + username);
            
            // Require additional verification or revoke token
            revokeToken(token);
            sendAlertEmail(username, "New IP detected: " + currentIp);
        }
        
        // Update last used IP
        redisTemplate.opsForValue().set("user:lastIp:" + username, currentIp, 1, TimeUnit.HOURS);
    }
}
```

**Defense in Depth Summary:**

| Layer | Mitigation | Effectiveness |
|-------|------------|---------------|
| **Transport** | HTTPS only | Prevents network sniffing |
| **Storage** | HttpOnly cookie | Prevents XSS theft |
| **Lifetime** | Short TTL (15 min) | Limits damage window |
| **Fingerprint** | Bind to device | Prevents replay from different device |
| **Revocation** | Blacklist | Immediate invalidation |
| **Rotation** | New token per request | Old token becomes invalid |
| **Anomaly** | IP/UA monitoring | Detects suspicious activity |

**Interview Tip:** "Token theft is serious. Mitigations: short TTL (15 min), token fingerprinting (bind to device), revocation blacklist, token rotation, anomaly detection. Defense in depth - multiple layers. Never rely on single protection."

```java
@Service
public class JwtSecurityService {
    
    // 1. Token Generation with proper claims
    public String generateToken(User user) {
        Map<String, Object> claims = new HashMap<>();
        claims.put("userId", user.getId());
        claims.put("roles", user.getRoles());
        claims.put("permissions", user.getPermissions());
        claims.put("tenantId", user.getTenantId());
        
        return Jwts.builder()
            .setClaims(claims)
            .setSubject(user.getUsername())
            .setIssuedAt(new Date())
            .setExpiration(new Date(System.currentTimeMillis() + ACCESS_TOKEN_TTL))
            .signWith(getPrivateKey(), SignatureAlgorithm.RS256)
            .compact();
    }
    
    // 2. Token Validation with multiple checks
    public boolean validateToken(String token, HttpServletRequest request) {
        try {
            Claims claims = getClaims(token);
            
            // Check expiration
            if (claims.getExpiration().before(new Date())) {
                log.warn("Token expired");
                return false;
            }
            
            // Check fingerprint
            if (!validateFingerprint(claims, request)) {
                log.warn("Fingerprint mismatch - possible token theft");
                revokeToken(token);
                return false;
            }
            
            // Check revocation
            if (isTokenRevoked(token)) {
                log.warn("Token revoked");
                return false;
            }
            
            return true;
        } catch (JwtException e) {
            log.error("Token validation failed", e);
            return false;
        }
    }
    
    // 3. Token Revocation
    public void revokeToken(String token) {
        long ttl = getRemainingTTL(token);
        redisTemplate.opsForValue().set(
            "blacklist:" + token,
            "revoked",
            ttl,
            TimeUnit.MILLISECONDS
        );
    }
    
    // 4. Refresh Token Rotation
    public TokenPair refreshTokens(String refreshToken) {
        if (!validateRefreshToken(refreshToken)) {
            throw new SecurityException("Invalid refresh token");
        }
        
        // Revoke old refresh token
        revokeRefreshToken(refreshToken);
        
        // Generate new token pair
        User user = getUserFromRefreshToken(refreshToken);
        String newAccessToken = generateToken(user);
        String newRefreshToken = generateRefreshToken(user);
        
        return new TokenPair(newAccessToken, newRefreshToken);
    }
}
```

```java
            null,

        );
```

## Q16. A user reports being logged out frequently. What could be the issue?

**Answer:**  
Common causes: **short token expiration**, **token invalidation**, **clock skew**, **multiple devices**, or **token storage issues**.

**Common Issues and Solutions:**

**1. Short Token Expiration (Most Common):**

```java
// Issue: Access token expires too quickly (e.g., 5 minutes)
@Component
public class TokenExpirationService {
    
    // ❌ Too short - user gets logged out frequently
    private final int TOO_SHORT = 5 * 60 * 1000;  // 5 minutes
    
    // ✅ Reasonable - 15-60 minutes
    private final int REASONABLE = 60 * 60 * 1000;  // 60 minutes
    
    // ✅ With sliding expiration
    public String generateTokenWithSlidingExpiration(String username) {
        return Jwts.builder()
            .setSubject(username)
            .setIssuedAt(new Date())
            .setExpiration(new Date(System.currentTimeMillis() + REASONABLE))
            .claim("lastActivity", System.currentTimeMillis())
            .signWith(SECRET_KEY)
            .compact();
    }
    
    // Validate with sliding window
    public boolean validateWithSlidingWindow(String token) {
        Claims claims = getClaims(token);
        long lastActivity = claims.get("lastActivity", Long.class);
        long now = System.currentTimeMillis();
        
        // Refresh if token is old but still valid
        if (now - lastActivity > 15 * 60 * 1000) {  // 15 min inactivity
            return false;  // Force refresh
        }
        
        // Update last activity
        claims.put("lastActivity", now);
        return true;
    }
}
```

**2. Clock Skew Between Servers:**

```java
                return true;
            }
            return false;
        }
    }
}
```

## Q17. How can you revoke a compromised JWT?

**Answer:**  
JWT is stateless, so revocation requires **additional state** like a blacklist or version tracking.

**Method 1: Token Blacklist (Redis)**

```java
@Service
public class JwtRevocationService {
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    // Revoke token immediately
    public void revokeToken(String token) {
        long ttl = getRemainingTTL(token);
        redisTemplate.opsForValue().set(
            "blacklist:" + token,
            "revoked",
            ttl, TimeUnit.MILLISECONDS
        );
        log.info("Token revoked: {}", token);
    }
    
    // Check if token is revoked
    public boolean isRevoked(String token) {
        return Boolean.TRUE.equals(redisTemplate.hasKey("blacklist:" + token));
    }
    
    // Revoke all tokens for a user (e.g., password change)
    public void revokeAllUserTokens(String username) {
        long version = redisTemplate.opsForValue().increment("user:token-version:" + username);
        log.info("User {} token version incremented to {}", username, version);
    }
}
```

```java
@Service

    public String generateToken(User user) {

        return Jwts.builder()
            .setSubject(user.getUsername())

            .setExpiration(new Date(System.currentTimeMillis() + 3600000))

            .compact();

    public boolean validateTokenVersion(String token) {
        Claims claims = getClaims(token);
        String username = claims.getSubject();
        long tokenVersion = claims.get("version", Long.class);
        long currentVersion = getCurrentVersion(username);
        
        return tokenVersion == currentVersion;
    }
    
    // Revoke all tokens by incrementing version
    public void revokeAllUserTokens(String username) {
        long newVersion = redisTemplate.opsForValue().increment("version:" + username);
        log.info("User {} tokens revoked, new version: {}", username, newVersion);
    }
    
    private long getCurrentVersion(String username) {
        String version = redisTemplate.opsForValue().get("version:" + username);
        return version != null ? Long.parseLong(version) : 1;
```

## Q18. How does it differ from a session-based cookie?

**Answer:**

| Aspect | JWT (Stateless) | Session Cookie (Stateful) |
|--------|-----------------|--------------------------|
| **Storage** | Token stored client-side | Session ID stored in cookie |
| **Server State** | No session storage | Session storage (Redis/DB) |
| **Validation** | Cryptographic signature | Session lookup |
| **Scalability** | Any node can validate | Needs shared session store |
| **Logout** | Token must expire or be blacklisted | Session invalidation |
| **Size** | Larger (contains claims) | Small (just ID) |
| **CSRF** | Not vulnerable | Vulnerable |
| **XSS** | Vulnerable if not in HttpOnly | Protected by HttpOnly |

## Q19. How do you handle JWT expiration in long-lived sessions?

**Answer:**  
Use **refresh token rotation** and **silent authentication** to maintain long-lived sessions without storing access tokens long-term.

**Refresh Token Pattern:**

```java
@Component
public class RefreshTokenService {
    
    private static final int ACCESS_TOKEN_TTL = 15 * 60 * 1000;  // 15 minutes
    private static final int REFRESH_TOKEN_TTL = 7 * 24 * 60 * 60 * 1000;  // 7 days
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    // Generate token pair
    public TokenPair generateTokenPair(User user) {
        String accessToken = generateAccessToken(user);
        String refreshToken = generateRefreshToken(user);
        
        // Store refresh token in Redis (revocable)
        redisTemplate.opsForValue().set(
            "refresh:" + refreshToken,
            user.getUsername(),
            REFRESH_TOKEN_TTL,
            TimeUnit.MILLISECONDS
        );
        
        return new TokenPair(accessToken, refreshToken);
    }
    
    // Refresh access token
    public TokenPair refreshTokens(String refreshToken) {
        // Validate refresh token
        String username = redisTemplate.opsForValue().get("refresh:" + refreshToken);
        if (username == null) {
            throw new SecurityException("Invalid or expired refresh token");
        }
        
        // OPTIONAL: Rotate refresh token (more secure)
        redisTemplate.delete("refresh:" + refreshToken);
        
        User user = userService.findByUsername(username);
        return generateTokenPair(user);  // New access + new refresh
    }
    
    private String generateAccessToken(User user) {
        return Jwts.builder()
            .setSubject(user.getUsername())
            .setExpiration(new Date(System.currentTimeMillis() + ACCESS_TOKEN_TTL))
            .signWith(SECRET_KEY)
            .compact();
    }
}
```

**Frontend Silent Refresh:**

```javascript
// axios.interceptor.js
let isRefreshing = false;
let refreshSubscribers = [];

function onRefreshed(token) {
    refreshSubscribers.forEach(callback => callback(token));
    refreshSubscribers = [];
}

axios.interceptors.response.use(
    response => response,
    async error => {
        const originalRequest = error.config;
        
        if (error.response?.status === 401 && !originalRequest._retry) {
            if (isRefreshing) {
                // Queue request while refresh is in progress
                return new Promise(resolve => {
                    refreshSubscribers.push(token => {
                        originalRequest.headers['Authorization'] = `Bearer ${token}`;
                        resolve(axios(originalRequest));
                    });
                });
            }
            
            originalRequest._retry = true;
            isRefreshing = true;
            
            try {
                const response = await axios.post('/api/auth/refresh', {}, {
                    withCredentials: true
                });
                const newToken = response.data.accessToken;
                
                // Update stored token
                localStorage.setItem('accessToken', newToken);
                onRefreshed(newToken);
                
                // Retry original request
                originalRequest.headers['Authorization'] = `Bearer ${newToken}`;
                return axios(originalRequest);
            } catch (refreshError) {
                // Refresh failed - redirect to login
                window.location.href = '/login';
                return Promise.reject(refreshError);
            } finally {
                isRefreshing = false;
            }
        }
        
        return Promise.reject(error);
    }
);
```

```java
            claims.setExpiration(new Date(now + 60 * 60 * 1000));
            claims.setLastActivity(now);
            
            return Jwts.builder()
                .setClaims(claims)
                .signWith(SECRET_KEY)
                .compact();
        }
        
        return token;  // No extension
    }
}
```

## Q20. How does the structure of a JWT support its functionality?

**Answer:**  
JWT's **3-part structure** (Header.Payload.Signature) enables **self-contained, verifiable, and secure** token-based authentication.

**JWT Structure:**

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4iLCJpYXQiOjE1MTYyMzkwMjJ9.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

Part 1 (Header):     eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
Part 2 (Payload):    eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4iLCJpYXQiOjE1MTYyMzkwMjJ9
Part 3 (Signature):  SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
```

**Part 1: Header (Algorithm & Type)**

```json
// Decoded Header
{
  "alg": "HS256",      // Signing algorithm
  "typ": "JWT"         // Token type
}
```

**Purpose:** Tells the server which algorithm was used to sign the token.

**Part 2: Payload (Claims/Data)**

```json
// Decoded Payload
{
  "sub": "1234567890",        // Subject (user ID)
  "name": "John Doe",         // User name
  "iat": 1516239022,          // Issued at
  "exp": 1516242622,          // Expiration
  "roles": ["USER", "ADMIN"], // Custom claims
  "tenantId": "acme"          // Multi-tenancy
}
```

**Purpose:** Contains all user information - no database lookup needed.

**Part 3: Signature (Verification)**

```
Signature = HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secret
)
```

**Purpose:** Ensures token hasn't been tampered with.

**How Structure Supports Functionality:**

```java
public class JwtStructureDemo {
    
    // 1. SELF-CONTAINED (No DB lookup)
    // Payload contains all user data
    public Authentication authenticate(String token) {
        Claims claims = parseJwt(token);
        
        // No database call! Everything from token
        String username = claims.getSubject();
        List<String> roles = claims.get("roles", List.class);
        String tenantId = claims.get("tenantId", String.class);
        
        return createAuthentication(username, roles, tenantId);
    }
    
    // 2. VERIFIABLE (Signature ensures integrity)
    public boolean verifyToken(String token) {
        try {
            // If signature valid, token hasn't been tampered with
            Jwts.parserBuilder()
                .setSigningKey(SECRET_KEY)
                .build()
                .parseClaimsJws(token);
            return true;
        } catch (JwtException e) {
            return false;  // Token tampered or invalid
        }
    }
    
    // 3. EXPIRABLE (exp claim for time-based validity)
    public boolean isExpired(String token) {
        Claims claims = getClaims(token);
        Date expiration = claims.getExpiration();
        return expiration.before(new Date());
    }
    
    // 4. EXTENSIBLE (Custom claims)
    public String addCustomClaims(String username, Map<String, Object> customData) {
        return Jwts.builder()
            .setSubject(username)
            .addClaims(customData)  // Add any custom data
            .signWith(SECRET_KEY)
            .compact();
    }
}
```

**Structure Benefits:**

| Component | Purpose | Benefit |
|-----------|---------|---------|
| **Header** | Algorithm declaration | Server knows how to verify |
| **Payload** | User data + claims | Self-contained, no DB lookup |
| **Signature** | Cryptographic verification | Tamper-proof, integrity |

**Interview Tip:** "JWT structure: **Header** (algorithm), **Payload** (claims/user data), **Signature** (verification). Enables: self-contained (no DB), verifiable (tamper-proof), expirable (time-based), extensible (custom claims)."

```java
            .httpOnly(true)
            .secure(true)
            .sameSite("Strict")
            .path("/")
            .maxAge(15 * 60)
            .build();

        return ResponseEntity.ok().build();
    }

            .secure(true)
            .sameSite("Strict")
            .path("/")
            .maxAge(0)  // Delete cookie
```

```javascript
    if (!isLoggedIn()) {
        return <Navigate to="/login" replace />;
    }
    
    return children;
};

// Usage in App
function App() {
    return (
        <BrowserRouter>
            <Routes>
                <Route path="/login" element={<Login />} />
                <Route path="/dashboard" element={
                    <ProtectedRoute>
                        <Dashboard />
                    </ProtectedRoute>
                } />
            </Routes>
        </BrowserRouter>
    );
}
```

**Interview Tip:** "JWT in web app: login generates token, store in httpOnly cookie (secure, XSS protected), send with requests automatically, implement silent refresh, clear cookie on logout. Never store in localStorage (XSS vulnerable)."

## Q22. Potential risks of using JWTs, and how can you mitigate them?

**Answer:**

| Risk | Description | Mitigation |
|------|-------------|------------|
| **Token Theft** | Attacker steals token | Short TTL, HttpOnly cookie, fingerprinting |
| **No Revocation** | Can't invalidate easily | Blacklist, short TTL, refresh tokens |
| **XSS Attacks** | Script steals token | HttpOnly cookie, CSP headers |
| **CSRF Attacks** | Cross-site requests | SameSite cookie, CSRF tokens |
| **Token Replay** | Reusing old tokens | JTI (JWT ID) with cache |
| **Algorithm Confusion** | Attacker changes algorithm | Validate algorithm, use strong keys |
| **Information Leakage** | Sensitive data in payload | Don't store sensitive data, encrypt |
| **Token Bloat** | Large tokens | Minimize claims, use references |

**Comprehensive Mitigation:**

```java
@Configuration
public class JwtSecurityMitigation {
    
    // 1. Short TTL + Refresh Tokens
    @Bean
    public JwtTokenProvider tokenProvider() {
        return new JwtTokenProvider(
            15 * 60 * 1000,  // Access token: 15 min
            7 * 24 * 60 * 60 * 1000  // Refresh token: 7 days
        );
    }
    
    // 2. Token Fingerprinting
    public String generateTokenWithFingerprint(User user, HttpServletRequest request) {
        String fingerprint = createFingerprint(request);
        
        return Jwts.builder()
            .setSubject(user.getUsername())
            .claim("fingerprint", fingerprint)
            .claim("userAgent", request.getHeader("User-Agent"))
            .signWith(PRIVATE_KEY, SignatureAlgorithm.RS256)  // Strong algorithm
            .compact();
    }
    
    // 3. JTI (JWT ID) for Replay Prevention
    public String generateTokenWithJti(User user) {
        String jti = UUID.randomUUID().toString();
        
        // Store JTI in cache with short TTL
        redisTemplate.opsForValue().set("jti:" + jti, "used", 15, TimeUnit.MINUTES);
        
        return Jwts.builder()
            .setId(jti)
            .setSubject(user.getUsername())
            .signWith(PRIVATE_KEY)
            .compact();
    }
    
    // 4. Validate Algorithm (Prevent Algorithm Confusion)
    public boolean validateToken(String token) {
        try {
            Jwts.parserBuilder()
                .setSigningKey(PUBLIC_KEY)
                .require("alg", "RS256")  // Enforce expected algorithm
                .build()
                .parseClaimsJws(token);
            return true;
        } catch (JwtException e) {
            return false;
        }
    }
    
    // 5. Don't store sensitive data in JWT
    // ❌ Bad
    claims.put("password", user.getPassword());
    claims.put("ssn", user.getSsn());
    
    // ✅ Good - only non-sensitive identifiers
    claims.put("userId", user.getId());
    claims.put("roles", user.getRoles());
}
```

**Security Headers (Mitigate XSS/CSRF):**

```java
@Configuration
public class SecurityHeadersConfig {
    
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .headers(headers -> headers
                .contentSecurityPolicy("default-src 'self'; script-src 'self'")
                .httpStrictTransportSecurity(hsts -> hsts.maxAgeInSeconds(31536000))
                .xssProtection(xss -> xss.block(true))
            )
            .csrf(csrf -> csrf
                .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse())
            );
        return http.build();
    }
}
```

**Interview Tip:** "JWT risks: theft (short TTL + HttpOnly), revocation (blacklist), XSS (HttpOnly cookie), CSRF (SameSite cookie), replay (JTI), algorithm confusion (validate alg). Defense in depth: multiple layers of protection."

## Q23. Prevent CSRF attacks when using JWTs in cookies?

**Answer:**  
When JWT is stored in cookie, implement **CSRF protection** using **SameSite cookie attribute** or **CSRF tokens**.

**Method 1: SameSite Cookie Attribute (Recommended)**

```java
@Configuration
public class CsrfProtectionConfig {
    
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable())  // Not needed with SameSite=Strict
            .headers(headers -> headers
                .addHeaderWriter((request, response) -> {
                    // Set SameSite cookie attribute
                    response.setHeader("Set-Cookie", 
                        "jwt_token=value; HttpOnly; Secure; SameSite=Strict");
                })
            );
        return http.build();
    }
    
    // Configure cookie with SameSite
    public ResponseCookie createSecureCookie(String token) {
        return ResponseCookie.from("jwt_token", token)
            .httpOnly(true)
            .secure(true)
            .sameSite("Strict")  // Prevents CSRF
            .path("/")
            .maxAge(900)
            .build();
    }
}
```

**Method 2: CSRF Token (For SameSite=Lax/None)**

```java
@Configuration
public class CsrfTokenConfig {
    
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf
                .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse())
                .csrfTokenRequestHandler(new CsrfTokenRequestAttributeHandler())
            )
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**").permitAll()
                .anyRequest().authenticated()
            );
        
        return http.build();
    }
}
```

**Frontend - Include CSRF Token:**

```javascript
// Get CSRF token from cookie
function getCsrfToken() {
    const cookies = document.cookie.split(';');
    for (let cookie of cookies) {
        const [name, value] = cookie.trim().split('=');
        if (name === 'XSRF-TOKEN') {
            return value;
        }
    }
    return null;
}

        'Content-Type': 'application/json'
    },

    const csrfToken = getCsrfToken();
    if (csrfToken) {
        config.headers['X-XSRF-TOKEN'] = csrfToken;
    }
    return config;
});
```

**Method 3: Double-Submit Cookie (JWT-specific)**

```java
@Service
public class DoubleSubmitCsrfProtection {
    
    // Generate CSRF token and store in separate cookie
    public void setCsrfToken(HttpServletResponse response) {
        String csrfToken = UUID.randomUUID().toString();
        
        ResponseCookie csrfCookie = ResponseCookie.from("csrf_token", csrfToken)
            .httpOnly(false)  // Accessible by JS
            .secure(true)
            .sameSite("Strict")
            .path("/")
            .build();
        
        response.addHeader(HttpHeaders.SET_COOKIE, csrfCookie.toString());
        
        // Also store in JWT claim
        // Client must send both cookies and JWT claim matches
    }
    
    // Validate CSRF token
    public boolean validateCsrf(String csrfFromCookie, String csrfFromHeader) {
        return csrfFromCookie != null && csrfFromCookie.equals(csrfFromHeader);
    }
}
```

**Best Practice Recommendation:**

```java
@Configuration
public class BestPracticeCsrfConfig {
    
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            // 1. Use SameSite=Strict (prevents most CSRF)
            // 2. For cross-origin requests, use CSRF tokens
            // 3. For APIs, prefer JWT in Authorization header (no CSRF)
            
            .csrf(csrf -> {
                if (isApiRequest()) {
                    csrf.disable();  // Stateless API
                } else {
                    csrf.csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse());
                }
            });
        
        return http.build();
    }
}
```

**Interview Tip:** "Prevent CSRF with JWT cookies: **SameSite=Strict** (best, prevents cross-site requests), **CSRF tokens** (for cross-origin), **Double-submit cookie** (JWT-specific). For pure APIs, prefer Authorization header (no CSRF vulnerability)."

## Q24. Why You Usually Don't Store JWT Access Tokens in a Database?

**Answer:**  
Storing JWT in database defeats the **stateless** purpose of JWT and adds unnecessary overhead.

**Reasons:**

| Reason | Explanation |
|--------|-------------|
| **Stateless Purpose** | JWT is designed to avoid server storage |
| **Performance** | Database lookup on every request kills performance |
| **Scalability** | Creates bottleneck, requires shared database |
| **Complexity** | Adds unnecessary infrastructure |
| **Token Nature** | Tokens are meant to be self-validating |

**Why NOT to store:**

```java
// ❌ Bad - Storing JWT in database
@Repository
public class BadJwtStorage {
    
    public void saveToken(String token, String username) {
        // Database write on every login
        jdbcTemplate.update("INSERT INTO jwt_tokens (token, username) VALUES (?, ?)", token, username);
    }
    
    public boolean validateToken(String token) {
        // Database read on EVERY request! Kills performance
        int count = jdbcTemplate.queryForObject(
            "SELECT COUNT(*) FROM jwt_tokens WHERE token = ? AND revoked = false", 
            Integer.class, token);
        return count > 0;
    }
}

// ✅ Good - Stateless validation
@Component
public class GoodJwtValidation {
    
    public boolean validateToken(String token) {
        // No database! Just cryptographic validation
        try {
            Jwts.parserBuilder()
                .setSigningKey(SECRET_KEY)
                .build()
                .parseClaimsJws(token);
            return true;
        } catch (JwtException e) {
            return false;
        }
    }
}
```

**When you MIGHT store tokens:**

```java
// Legitimate use cases for token storage:
// 1. Blacklist (revoked tokens)
@Service
public class TokenBlacklist {
    // Store ONLY revoked tokens, not valid ones
    public void revokeToken(String token) {
        redisTemplate.opsForValue().set("blacklist:" + token, "revoked", ttl);
    }
}

// 2. Refresh tokens (long-lived, need revocation)
@Service
public class RefreshTokenStorage {
    // Store refresh tokens (not access tokens)
    public void storeRefreshToken(String refreshToken, String username) {
        redisTemplate.opsForValue().set("refresh:" + refreshToken, username, 7, TimeUnit.DAYS);
    }
}

// 3. Audit/Observability (async, not for validation)
@Service
public class TokenAuditService {
    // Log token issuance for audit (not for validation)
    @Async
    public void logTokenIssuance(String username, String jti) {
        auditRepository.save(new TokenAudit(username, jti, LocalDateTime.now()));
    }
}
```

**Interview Tip:** "Don't store access tokens in DB - defeats stateless purpose. JWT validation is cryptographic (fast, no DB). Only store: blacklisted tokens (revocation), refresh tokens (need revocation), audit logs (async). Performance: DB lookup = 5-10ms, crypto validation = <1ms."

## Q25. Why do I need to store the refresh token in the database?

**Answer:**  
Refresh tokens need storage because they are **long-lived and need revocation capability**.

**Reasons to Store Refresh Tokens:**

| Reason | Explanation |
|--------|-------------|
| **Revocation** | Need to invalidate on logout/password change |
| **Theft Detection** | Detect token replay (one-time use) |
| **Device Management** | Track active sessions per user |
| **Security** | Limit number of active devices |
| **Audit** | Track token usage history |

**Complete Refresh Token Implementation:**

```java
@Entity
public class RefreshToken {
    @Id
    private String id;  // The refresh token value
    private String username;
    private String deviceId;
    private String deviceName;
    private LocalDateTime issuedAt;
    private LocalDateTime expiresAt;
    private LocalDateTime revokedAt;
    private String revokedReason;
}

@Service
public class RefreshTokenService {
    
    @Autowired
    private RefreshTokenRepository refreshTokenRepository;
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    // 1. Create refresh token with storage
    public String createRefreshToken(User user, String deviceId, String deviceName) {
        String refreshToken = UUID.randomUUID().toString();
        
        RefreshToken token = new RefreshToken();
        token.setId(refreshToken);
        token.setUsername(user.getUsername());
        token.setDeviceId(deviceId);
        token.setDeviceName(deviceName);
        token.setIssuedAt(LocalDateTime.now());
        token.setExpiresAt(LocalDateTime.now().plusDays(7));
        
        refreshTokenRepository.save(token);
        
        // Also store in Redis for faster lookup
        redisTemplate.opsForValue().set(
            "refresh:" + refreshToken,
            user.getUsername(),
            7, TimeUnit.DAYS
        );
        
        return refreshToken;
    }
    
    // 2. Validate refresh token (needs DB/Redis lookup)
    public String validateRefreshToken(String refreshToken) {
        // Check Redis first (fast path)
        String username = redisTemplate.opsForValue().get("refresh:" + refreshToken);
        if (username == null) {
            // Check database (fallback)
            Optional<RefreshToken> token = refreshTokenRepository.findById(refreshToken);
            if (token.isEmpty() || token.get().getRevokedAt() != null) {
                throw new SecurityException("Invalid refresh token");
            }
            if (token.get().getExpiresAt().isBefore(LocalDateTime.now())) {
                throw new SecurityException("Refresh token expired");
            }
            return token.get().getUsername();
        }
        return username;
    }
    
    // 3. Revoke refresh token (logout)
    public void revokeRefreshToken(String refreshToken) {
        // Remove from Redis
        redisTemplate.delete("refresh:" + refreshToken);
        
        // Update database
        refreshTokenRepository.findById(refreshToken).ifPresent(token -> {
            token.setRevokedAt(LocalDateTime.now());
            token.setRevokedReason("User logout");
            refreshTokenRepository.save(token);
        });
    }
    
    // 4. Revoke all tokens for user (password change)
    public void revokeAllUserTokens(String username) {
        // Revoke in database
        refreshTokenRepository.revokeAllByUsername(username);
        
        // Revoke in Redis (pattern delete)
        Set<String> keys = redisTemplate.keys("refresh:*");
        for (String key : keys) {
            String value = redisTemplate.opsForValue().get(key);
            if (username.equals(value)) {
                redisTemplate.delete(key);
            }
        }
    }
    
    // 5. Rotate refresh token (more secure)
    public TokenPair rotateRefreshToken(String oldRefreshToken, String deviceId) {
        String username = validateRefreshToken(oldRefreshToken);
        
        // Revoke old token
        revokeRefreshToken(oldRefreshToken);
        
        // Create new token pair
        User user = userService.findByUsername(username);
        String newAccessToken = generateAccessToken(user);
        String newRefreshToken = createRefreshToken(user, deviceId, "Rotated");
        
        return new TokenPair(newAccessToken, newRefreshToken);
    }
    
    // 6. Get active devices for user
    public List<DeviceInfo> getActiveDevices(String username) {
        return refreshTokenRepository.findByUsernameAndRevokedAtIsNull(username)
            .stream()
            .map(token -> new DeviceInfo(token.getDeviceId(), token.getDeviceName(), token.getIssuedAt()))
            .collect(Collectors.toList());
    }
    
    // 7. Limit number of active devices
    private static final int MAX_DEVICES = 5;
    
    public void enforceDeviceLimit(String username) {
        long activeCount = refreshTokenRepository.countByUsernameAndRevokedAtIsNull(username);
        if (activeCount > MAX_DEVICES) {
            // Revoke oldest token
            RefreshToken oldest = refreshTokenRepository.findOldestByUsername(username);
            oldest.setRevokedAt(LocalDateTime.now());
            oldest.setRevokedReason("Device limit exceeded");
            refreshTokenRepository.save(oldest);
        }
    }
}
```

**Refresh Token Flow:**

```
┌─────────────┐                                    ┌─────────────┐
│   Client    │                                    │   Server    │
└──────┬──────┘                                    └──────┬──────┘
       │                                                  │
       │  1. Login (username/password)                    │
       │─────────────────────────────────────────────────>│
       │                                                  │
       │                                                  │ 2. Generate:
       │                                                  │    - Access Token (15 min)
       │                                                  │    - Refresh Token (store in DB)
       │                                                  │
       │  3. { accessToken, refreshToken }               │
       │<─────────────────────────────────────────────────│
       │                                                  │
       │  4. Use accessToken for API calls               │
       │─────────────────────────────────────────────────>│
       │                                                  │
       │  5. Access token expires (401)                  │

       │  8. New accessToken + new refreshToken          │

       │                                                  │

       │─────────────────────────────────────────────────>│
```

**Interview Tip:** "Refresh tokens need storage because they're long-lived and revocable. Store in DB/Redis with: username, device info, expiry, revocation status. Benefits: logout, device management, theft detection, token rotation. Access tokens remain stateless (crypto validation only)."
