# 🏃 SCRUM - Complete Agile Framework Guide

## What is SCRUM? (Core Concept)

### Simple Intuition

**Real-world analogy:** 
Scrum = Team ka kitchen recipe. Ek dinner party (project) host karni hai. Poori raat continuous cooking ke jagah, **2-hour sprints** mein kaam karte ho. Har 2 hours baad team check karti hai - "Kya ban gaya? Kya problem hai?" Har sprint ke end mein ek dish ready hoti hai.

**Technical Definition:**
Scrum is a lightweight Agile framework for managing complex product development. It uses fixed-length iterations (sprints), defined roles (Product Owner, Scrum Master, Team), and ceremonies to deliver incremental value.

### Scrum Framework at a Glance

```
┌─────────────────────────────────────────────────────────────────────┐
│                         SCRUM FRAMEWORK                              │
```

## Q1. What does a Scrum Master do during the day?

### Daily Activities Breakdown

**Role:** Scrum Master = Team's coach, facilitator, and impediment remover. Not a project manager. Not a boss. A servant leader.

### Typical Day Timeline

```
9:00 AM - Check team's progress board (Jira/Trello)
9:30 AM - Facilitate Daily Standup (15 min)
    - What did you do yesterday?
    - What will you do today?
    - Any blockers?
10:00 AM - Remove impediments (talk to other teams, unblock resources)
11:00 AM - Coach team member on story splitting
12:00 PM - Lunch
1:00 PM - Update burndown chart, track sprint progress
2:00 PM - Facilitate backlog refinement session
3:00 PM - Shield team from external interruptions
4:00 PM - Work with Product Owner on priority
5:00 PM - Prepare for tomorrow's sprint planning
```

### Detailed Responsibilities

| Activity | What SM Does | Time Spent |
|----------|--------------|------------|
| **Facilitate Daily Standup** | Ensures timebox (15 min), identifies blockers | 30 min |
| **Remove Impediments** | Unblock team (CI broken? Need access? Other team dependency?) | 2-3 hours |
| **Coach Team** | Teach Scrum practices, pair programming, TDD | 1-2 hours |
| **Shield Team** | Stop random requests, say "No" to stakeholders | 1 hour |
| **Facilitate Ceremonies** | Sprint Planning, Review, Retrospective | 2-4 hours/week |
| **Track Metrics** | Velocity, burndown, cycle time | 30 min |
| **Work with PO** | Backlog health, definition of ready, priorities | 1 hour |
| **Continuous Improvement** | Implement retrospective action items | 1 hour |

### Real SM Day (Example)

```yaml
8:45 AM: Check Slack, emails (no urgent fires)
9:00 AM: Review Jira board, update blocked tickets
9:30 AM: Daily Standup (team of 7)
  - Dev1: "API integration done, need QA environment"
  - Dev2: "Stuck on auth, need AWS IAM access"
  - SM action: Get IAM access by EOD
10:00 AM: Meet with DevOps to unblock auth issue
10:30 AM: Update burndown chart
11:00 AM: 1-on-1 coaching with junior dev on story estimation
11:30 AM: Shield team from product manager's "urgent" request
12:30 PM: Lunch
1:30 PM: Backlog refinement session (45 min)
2:30 PM: Prepare Sprint Review demo
3:30 PM: Remove blocker: CI pipeline broken (escalate to DevOps)
4:30 PM: Update stakeholders on sprint progress
5:00 PM: Plan tomorrow, update impediment log
```

## Q2. How did you as a Scrum Master tackle daily challenges?

### Top 10 Daily Challenges & Solutions

| # | Challenge | Solution | Real Example |
|---|-----------|----------|--------------|
| 1 | **Standup going too long (30+ min)** | Timebox, use "parking lot" for deep dives | "Let's discuss offline after standup" |
| 2 | **Team member silent/unengaged** | Round-robin, ask direct questions | "Sarah, what's your update today?" |
| 3 | **Same blocker reported daily** | Escalate, create action item with owner | "I'll escalate to infra team today" |
| 4 | **Product Owner adding mid-sprint work** | Reinforce sprint commitment, defer to next sprint | "Let's add to backlog, plan next sprint" |
| 5 | **Team missing Definition of Done** | Create checklist, peer review | "Does this have tests? Documentation?" |
| 6 | **External interruptions (meetings, requests)** | Block calendar time, create "focus hours" | "Team has 10-12 no-meeting block" |
| 7 | **Stories too big (can't finish in sprint)** | Teach story splitting techniques | "Split into: DB schema, API, UI" |
| 8 | **Low velocity/overcommitment** | Use historical velocity, capacity planning | "Last sprint 30 points, plan 28-32" |
| 9 | **Conflict between team members** | Facilitate discussion, focus on process not person | "Let's focus on code, not developer" |
| 10 | **Stakeholders unhappy with progress** | Transparent metrics, manage expectations | "Here's burndown, velocity trend" |

### Detailed Challenge Responses

**Challenge 1: Standup going too long**

```bash
# Problem: Team discussing technical solutions in standup
Dev1: "So I was thinking about using Redis cache, and then we could..."
Dev2: "But what about Redis vs Memcached?"

# SM Intervention:
"Good discussion! Let's park this. Join the architecture huddle at 11 AM. 
What's everyone else's update? 30 seconds each."

# Result: Standup back to 15 min
```

**Challenge 2: Team consistently misses Sprint Goal**

```bash
# Problem: 3 sprints in a row, only 60% completion

# SM Actions:
1. Review Definition of Ready (stories not ready before sprint)
2. Check velocity vs commitment (overcommitting)
3. Identify hidden work (bugs, tech debt, meetings)

# Solution implemented:
- Stories require: clear acceptance criteria, design approved
- Reduce sprint commitment by 20%
- Add 20% buffer for unplanned work

# Result: 90% completion next sprint
```

**Challenge 3: Product Owner keeps changing priorities**

```bash
# Problem: Mid-sprint, PO wants to add high-priority feature

# SM Response:
"I understand this is important. But we committed to this sprint goal.
Two options:
1. We swap with an equal-sized story (PO chooses what to remove)
2. We add to backlog, plan next sprint

Which works for you?"

# PO chooses option 2, team stays focused
```

**Challenge 4: Team not improving despite retrospectives**

```bash
# Problem: Same issues every retro (testing, handoffs)

# SM Action:
- Make action items SMART (Specific, Measurable, Achievable, Relevant, Time-bound)
- Assign owners
- Track in next sprint's "improvement tasks"
- Measure before/after

# Example:
From: "Improve testing"
To: "Add 80% unit test coverage for user-service by sprint end"
Owner: Dev lead
Measured: Coverage report
```

## Q3. How do you handle a situation where your team consistently misses Sprint Goals?

### Root Cause Analysis Framework

```
┌─────────────────────────────────────────────────────────────┐
│         Why Sprint Goals Missed? (5 Whys Analysis)          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Symptom: Team misses 3 sprints in a row                    │
│                                                              │
│  Why #1: Stories not completed                              │
│      ↓                                                       │
│  Why #2: Stories too large, unexpected dependencies         │
│      ↓                                                       │
│  Why #3: No dependency analysis before sprint               │
│      ↓                                                       │
│  Why #4: Rushed sprint planning (1 hour for 2-week sprint)  │
│      ↓                                                       │
│  Why #5: Leadership pushes for faster planning              │
│                                                              │
│  Root Cause: Inadequate sprint planning                     │
│  Solution: 4-hour planning, dependency mapping              │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### Step-by-Step Resolution Plan

#### Step 1: Immediate Actions (Same Sprint)

```yaml
1. Stop the bleeding:
   - Cancel non-critical stories
   - Reduce scope to achievable goal
   - Communicate transparently to stakeholders

2. Protect team morale:
   - No blame game
   - Focus on learning, not failure
   - Celebrate what WAS accomplished
```

#### Step 2: Diagnostic (Retrospective)

```markdown
# Retrospective Questions:

1. **Estimation Accuracy:**
   - Are stories properly estimated?
   - What's our velocity trend (last 6 sprints)?
   
2. **Definition of Ready:**
   - Do stories meet "ready" before sprint?
   - Are acceptance criteria clear?
   
3. **External Factors:**
   - How much unplanned work arrived?
   - Dependencies on other teams?
   
4. **Team Capacity:**
   - Holidays, sick days, training?
   - Context switching (meetings, support)?
   
5. **Technical Issues:**
   - CI/CD failures?
   - Environment instability?
```

#### Step 3: Implement Fixes

**Fix A: Capacity-Based Planning**

```bash
# Before: Plan 40 points (optimistic)
# After: Historical velocity = 30 points average
# Plan: 25-30 points (conservative)

Calculation:
Team availability = 7 devs × 6 hours/day (remove meetings) × 10 days = 420 hours
Subtract: 20% buffer = 336 hours
Story points = 336/10 (hours per point) = 33 points
Plan: 30 points
```

**Fix B: Improve Definition of Ready**

```yaml
Definition of Ready (DoR) Checklist:
  [ ] Story estimated (points)
  [ ] Acceptance criteria written
  [ ] Design approved
  [ ] Dependencies identified
  [ ] No external blockers
  [ ] Team understands the work
  [ ] Tests planned
```

**Fix C: Split Large Stories**

```bash
# 21-point story (too big)
"Build payment gateway integration"

# Split into 5 stories:
1. Setup payment provider SDK (3 points)
2. Implement charge API (5 points)
3. Implement refund API (3 points)
4. Add webhook handling (5 points)
5. Error handling + retry logic (5 points)
```

**Fix D: Dependency Management**

```yaml
Sprint Planning Additions:
  - Dependency board (visual)
  - Pre-sprint sync with dependent teams
  - Spike stories for unknown dependencies
```

#### Step 4: Recovery Sprint

```yaml
Recovery Sprint Strategy:
  - Reduce capacity by 30% (focus on quality)
  - Add technical debt stories
  - No new features, only complete in-progress work
  - Daily check-ins on progress
  - Goal: Build momentum, restore confidence
```

#### Step 5: Long-term Prevention

```yaml
Sprint Metrics to Track:
  - Planned vs Actual points
  - Sprint Goal success rate (target >80%)
  - Unplanned work percentage (target <10%)
  - Story size distribution (small <5 points)
  
Process Improvements:
  - Sprint planning: 1 hour per week of sprint
  - Backlog refinement: Weekly, 1 hour
  - Technical spikes for unknowns
  - Cross-team dependency board
```

### Communication Template (to Stakeholders)

```markdown
Subject: Sprint Goal Missed - Action Plan

Team missed Sprint 23 goal (60% completion)

Root Causes Identified:
1. Underestimated API dependency (3 days delay)
2. 30% unplanned production bugs
3. Team capacity reduced (1 member sick)

Action Plan:
✅ Dependency mapping added to planning
✅ Bug buffer increased (20% of capacity)
✅ Cross-training initiated
✅ Reduced sprint commitment by 20% for next sprint

Expected improvement: 85% completion next sprint

Questions? Join our retrospective tomorrow at 2 PM.
```

### The 5 Core Ceremonies

```

│                    SCRUM CEREMONIES TIMELINE                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Sprint Start (Day 1)                                           │
│       │                                                         │
│       ├──► Sprint Planning (2-4 hours)                         │
│       │        - Define sprint goal                            │
│       │        - Select backlog items                          │
│       │        - Create tasks                                  │
│       │                                                         │
│       ▼                                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    SPRINT (2-4 weeks)                    │   │
│  │                                                          │   │
│  │  Daily Standup (15 min each day)                        │   │
│  │       - What did you do yesterday?                      │   │
│  │       - What will you do today?                         │   │
│  │       - Any blockers?                                   │   │
│  │                                                          │   │
│  │  Backlog Refinement (weekly, 1 hour)                    │   │
│  │       - Estimate stories                                │   │
│  │       - Split large stories                             │   │
│  │       - Define acceptance criteria                      │   │
│  │                                                          │   │
│  └─────────────────────────────────────────────────────────┘   │
│       │                                                         │
│       ▼                                                         │
│  Sprint End (Last Day)                                          │
│       │                                                         │
│       ├──► Sprint Review (1-2 hours)                           │
│       │        - Demo completed work                          │
│       │        - Get stakeholder feedback                     │
│       │        - Update backlog                               │
│       │                                                         │
│       └──► Sprint Retrospective (1-2 hours)                    │
│                - What went well?                               │
│                - What went wrong?                              │
│                - What to improve?                              │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### Detailed Ceremony Breakdown

| Ceremony | Duration | Who | Purpose | Output |
|----------|----------|-----|---------|--------|
| **Sprint Planning** | 2-4 hrs (2-week sprint) | SM, PO, Team | Define sprint goal, select stories | Sprint Backlog, Tasks |
| **Daily Standup** | 15 min | SM, Team | Sync progress, identify blockers | Impediment list |
| **Backlog Refinement** | 1 hr/week | SM, PO, Team | Groom backlog, estimate | Refined stories |
| **Sprint Review** | 1-2 hrs | SM, PO, Team, Stakeholders | Demo work, get feedback | Updated backlog |
| **Sprint Retrospective** | 1-2 hrs | SM, Team | Inspect process, improve | Action items |

### Ceremony-Specific Details

#### 1. Sprint Planning (Day 1, Morning)

```yaml
Agenda:
  9:00-9:30: Product Owner presents top priorities
  9:30-10:30: Team discusses capacity, selects stories
  10:30-11:30: Break down stories into tasks
  11:30-12:00: Team commits to sprint goal

Questions answered:
  - What can we deliver this sprint? (scope)
  - How will we do the work? (tasks)
  - What's our sprint goal? (one sentence)

Output:
  - Sprint Goal: "Enable user login via Google OAuth"
  - Sprint Backlog: 5 stories, 32 points
```

#### 2. Daily Standup (Every day, 9:30 AM)

```yaml
Format (3 questions, 1 min each):
  Dev1: "Yesterday: Completed login API. Today: OAuth integration. Blocked: None"
  Dev2: "Yesterday: Fixed 3 bugs. Today: Profile page. Blocked: Waiting for design"
  Dev3: "Yesterday: Code review. Today: Unit tests. Blocked: Test environment down"

SM Role:
  - Keep time (15 min hard stop)
  - Note blockers (assign action items)
  - No technical deep dives
  - No status reporting to management
```

#### 3. Backlog Refinement (Weekly, 2 PM)

```yaml
Activities:
  - Review upcoming stories (next 2-3 sprints)
  - Estimate using Planning Poker
  - Split stories > 13 points
  - Add acceptance criteria
  - Identify dependencies

Example:
  Story: "As a user, I want to reset my password"
  
  Refinement output:
    - Acceptance criteria: Email link, token expires in 1hr, new password strength check
    - Tasks: Email service, token generator, password validator
    - Estimate: 8 points
    - Dependency: Email service API
```

#### 4. Sprint Review (Last day, 10 AM)

```yaml
Format (Demo-focused):
  10:00-10:15: Product Owner sets context
  10:15-11:00: Team demos working software (not slides!)
  11:00-11:30: Stakeholder feedback
  11:30-12:00: Update backlog based on feedback

Rules:
  - NO slides, only working software
  - Stakeholders can ask questions
  - Feedback captured but NOT acted on in current sprint
  - Celebrate achievements

Example Demo:
  "Here's the Google OAuth login. Click 'Login with Google' → 
   Redirects to Google → Authorize → Back to app with user profile.
   Works in Chrome, Firefox. Edge case handled: user declines access."
```

#### 5. Sprint Retrospective (Last day, 2 PM)

```yaml
Format (Inspect & Adapt):
  Start-Stop-Continue method:
  
  Start doing:
    - Automated deployment checks
    - Pair programming on complex stories
    
  Stop doing:
    - Long standups
    - Mid-sprint scope changes
    
  Continue doing:
    - Code reviews
    - Documentation

Action Items (SMART):
  - "Implement CI pipeline check by end of next sprint" (Owner: DevOps lead)
  - "Reduce standup to 15 min with timer" (Owner: SM)
  - "Add unit test coverage to 70%" (Owner: Dev team)
```

## Q5. What is PI Planning?

### Concept Explanation

**PI Planning** = Program Increment Planning (SAFe framework)

**Simple Intuition:**
Regular Scrum = 2-week sprint. PI Planning = 12-week (3 months) mega-planning. Poori train (multiple teams) ek saath baithti hai aur plan karti hai ki agle 3 months mein kya deliver karna hai.

### PI Planning Overview

```
┌─────────────────────────────────────────────────────────────────┐
│                    PI PLANNING (2 DAYS)                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  DAY 1 (Morning) - Context & Vision                            │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 9:00 - Business context (VP Product)                    │   │
│  │ 10:00 - Product vision, roadmap                         │   │
│  │ 11:00 - Architecture vision                             │   │
│  └─────────────────────────────────────────────────────────┘   │
│                           │                                      │
│  DAY 1 (Afternoon) - Team Breakouts                            │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 1:00 - Teams identify dependencies                      │   │
│  │ 2:00 - Draft plans (features per team)                  │   │
│  │ 4:00 - Identify risks, impediments                      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                           │                                      │
│  DAY 2 (Morning) - Plan Review                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 9:00 - Each team presents plan (5 min each)             │   │
│  │ 11:00 - Cross-team dependency resolution                │   │
│  └─────────────────────────────────────────────────────────┘   │
│                           │                                      │
│  DAY 2 (Afternoon) - Finalize                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 1:00 - Final plan adjustments                           │   │
│  │ 2:00 - Management review & approval                     │   │
│  │ 3:00 - Confidence vote (fist of five)                   │   │
│  │ 4:00 - PI Objectives committed                          │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### PI Planning Structure

```yaml
Duration: 2 days (full-time, all teams)

Participants:
  - All development teams (5-10 teams, 50-100 people)
  - Product Management
  - System Architects
  - Business Owners
  - Scrum Masters
  - Release Train Engineer (RTE)

Outputs:
  - PI Objectives (team and program level)
  - ART (Agile Release Train) backlog
  - Dependency matrix
  - Risk register
  - Confidence vote results

PI Length: 8-12 weeks (4-6 sprints)
```

### Sample PI Plan

```yaml
PI Goal: "Enable payment processing for 5 new countries"

Teams:
  Team A (Payment API):
    - Sprint 1: Integrate Stripe API
    - Sprint 2: Add fraud detection
    - Sprint 3: Test 5 countries
    - Sprint 4: Performance tuning
    
  Team B (Frontend):
    - Sprint 1: Checkout page redesign
    - Sprint 2: Country selector
    - Sprint 3: Localization
    - Sprint 4: Analytics
    
  Team C (Compliance):
    - Sprint 1: Legal review 5 countries
    - Sprint 2: Tax calculation
    - Sprint 3: Reporting
    - Sprint 4: Audit log

Dependencies:
  - Team A → Team B: API contract by end of Sprint 1
  - Team C → Team A: Tax rules by Sprint 2

Risks:
  - High: Currency conversion API unstable (Mitigation: backup provider)
  - Medium: Legal approval delays (Mitigation: pre-work)

Confidence Vote: 4.2/5 (85% confident)
```

## Q6. What are 12 Principles of Agile?

### The 12 Principles (Agile Manifesto)

```
┌─────────────────────────────────────────────────────────────────┐
│                   12 AGILE PRINCIPLES                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. Customer Satisfaction through early & continuous delivery  │
│  2. Welcome changing requirements (even late)                   │
│  3. Deliver working software frequently (weeks, not months)     │
│  4. Business & developers work together daily                   │
│  5. Build projects around motivated individuals                 │
│  6. Face-to-face conversation (most efficient)                  │
│  7. Working software = primary measure of progress              │
│  8. Sustainable development (constant pace indefinitely)        │
│  9. Continuous attention to technical excellence                │
│  10. Simplicity (maximize work not done)                        │
│  11. Self-organizing teams (best architectures emerge)          │
│  12. Regular reflection & adjustment (improve continuously)     │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### Detailed Breakdown with Examples

| # | Principle | Real-World Example | Anti-Pattern |
|---|-----------|-------------------|---------------|
| 1 | **Early & continuous delivery** | Deploy to prod every day, not every month | Annual release planning |
| 2 | **Welcome changing requirements** | "Customer wants new feature mid-sprint? Add to backlog, plan next sprint" | "No changes after requirements signed" |
| 3 | **Working software frequently** | Demo every 2 weeks, ship every sprint | "We'll show you when it's done (6 months)" |
| 4 | **Business & developers daily** | Product Owner in same room, daily standup | PO in separate building, emails only |
| 5 | **Motivated individuals** | "Here's the goal, you figure out how" | "Follow this Gantt chart exactly" |
| 6 | **Face-to-face conversation** | Whiteboard session, pair programming | 20 emails back-and-forth |
| 7 | **Working software = progress** | "Working payment API = 50% done" | "500 pages of documentation = 50% done" |
| 8 | **Sustainable development** | 40-hour weeks, no death marches | Crunch mode before release |
| 9 | **Technical excellence** | Refactoring, unit tests, CI/CD | "We'll fix it after release" |
| 10 | **Simplicity** | YAGNI (You Aren't Gonna Need It) | "Let's build for future scalability" |
| 11 | **Self-organizing teams** | Team decides how to implement | Manager assigns tasks |
| 12 | **Regular reflection** | Sprint retrospective every 2 weeks | No process improvement ever |

### Agile vs Waterfall Comparison

| Aspect | Agile | Waterfall |
|--------|-------|-----------|
| Requirements | Evolving | Fixed up front |
| Delivery | Incremental (2-4 weeks) | Single (months later) |
| Customer feedback | Continuous | At end only |
| Change response | Welcomed | Rejected/expensive |
| Documentation | Just enough | Comprehensive |
| Team structure | Cross-functional | Siloed |
| Success measure | Working software | Following plan |

## Q7. What is Scrum?

### Complete Scrum Definition

**Official (Scrum Guide):**
> Scrum is a lightweight framework that helps people, teams, and organizations generate value through adaptive solutions for complex problems.

**Simple:**
Scrum = Small team (3-9 people) works in short cycles (sprints) to deliver working software incrementally. Has 3 roles, 5 events, 3 artifacts.

```yaml
Roles (3):
  - Product Owner: What to build? (priorities, backlog)
  - Scrum Master: How to build? (process, coaching)
  - Development Team: Build it (3-9 people, cross-functional)

Events (5):
  - Sprint Planning (start)
  - Daily Standup (every day)
  - Sprint Review (end)
  - Sprint Retrospective (end)
  - Sprint (container, 2-4 weeks)

Artifacts (3):
  - Product Backlog: All desired features (ordered by value)
  - Sprint Backlog: Features for current sprint + plan
  - Increment: Working software at sprint end

Values (5):
  - Commitment, Courage, Focus, Openness, Respect
```

### Scrum vs Other Frameworks

| Framework | Best For | Scale |
|-----------|----------|-------|
| **Scrum** | Single team (3-9 people) | Small |
| **Scrum of Scrums** | Multiple Scrum teams | Medium |
| **SAFe** | Enterprise (100+ teams) | Large |
| **LeSS** | Scaling Scrum (2-8 teams) | Medium |
| **Kanban** | Continuous flow (no sprints) | Small |

### Quick Interview Answers

**Q: What's the difference between Scrum Master and Project Manager?**

| Scrum Master | Project Manager |
|--------------|-----------------|
| Servant leader | Command & control |
| Removes impediments | Assigns tasks |
| Facilitates ceremonies | Creates Gantt charts |
| No authority over team | Has authority |
| Focus on process | Focus on timeline/budget |

**Q: When to use Scrum?**
- Complex problems (not predictable)
- Requirements change frequently
- Need fast feedback
- Team can be cross-functional

**Q: When NOT to use Scrum?**
- Simple, predictable work (use Kanban)
- Fixed price/contract (use Waterfall)
- Team can't be dedicated
- Regulatory requires upfront documentation
