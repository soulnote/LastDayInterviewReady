
## **Q1. What is Idempotency in REST API?**

### 📌 **Simple Definition**

**Idempotency = Same request multiple times karne par bhi result same aata hai jaise ek baar kiya ho**

**Real-life analogy:**
- **Idempotent:** Lift ka button ek baar press karo ya 10 baar, lift aayegi (same result)
- **Non-idempotent:** "Cart mein product add karo" - 1 baar press = 1 product, 10 baar press = 10 products

### 🎯 **HTTP Methods Idempotency Status**

| HTTP Method | Idempotent? | Why? |
|-------------|-------------|------|
| **GET** | ✅ Yes | Data sirf read hota hai, kuch change nahi hota |
| **PUT** | ✅ Yes | Same data se update karoge toh same state |
| **DELETE** | ✅ Yes | Pehle delete kiya, dobara delete karoge toh same (resource already gone) |
| **HEAD** | ✅ Yes | GET jaisa but without body |
| **OPTIONS** | ✅ Yes | Information retrieve karta hai |
| **POST** | ❌ No | Har baar naya resource create hota hai |
| **PATCH** | ❌ No (partially) | Partial update multiple times different result de sakta hai |

### 💻 **Code Examples**

```java
// ========== IDEMPOTENT EXAMPLE (PUT) ==========
// Request 1: Update user with ID 123
PUT /api/users/123
{
    "name": "Raj",
    "email": "raj@example.com"
}
// Response: 200 OK, User updated

// Request 2: Same request again
PUT /api/users/123
{
    "name": "Raj",
    "email": "raj@example.com"
}
// Response: 200 OK, User already has same data
// Result: SAME as first request ✅


// ========== NON-IDEMPOTENT EXAMPLE (POST) ==========
// Request 1: Create order
POST /api/orders
{
    "product": "iPhone",
    "quantity": 1
}
// Response: 201 Created, Order ID: 1001

// Request 2: Same request again
POST /api/orders
{
    "product": "iPhone",
    "quantity": 1
}
// Response: 201 Created, Order ID: 1002
// Result: DIFFERENT! New order created ❌


// ========== IDEMPOTENT POST using Idempotency-Key ==========
// Request 1
POST /api/payments
Idempotency-Key: uuid-1234-5678
{
    "amount": 100
}
// Response: 200 OK, Payment ID: 5001

// Request 2 (same key - retry due to network issue)
POST /api/payments
Idempotency-Key: uuid-1234-5678  // Same key!
{
    "amount": 100
}
// Response: 200 OK, Payment ID: 5001 (same as before)
// No duplicate payment created ✅
```

### 🔧 **Implementing Idempotency Key**

```java
// Spring Boot implementation
@Component
public class IdempotencyFilter implements Filter {
    
    private final Map<String, Object> idempotencyStore = new ConcurrentHashMap<>();
    
    @Override
    public void doFilter(ServletRequest request, ServletResponse response, 
                         FilterChain chain) throws IOException, ServletException {
        
        HttpServletRequest httpRequest = (HttpServletRequest) request;
        String idempotencyKey = httpRequest.getHeader("Idempotency-Key");
        
        // Only for POST/PUT/PATCH
        if (idempotencyKey != null && !"GET".equals(httpRequest.getMethod())) {
            
            // Check if already processed
            if (idempotencyStore.containsKey(idempotencyKey)) {
                // Return cached response
                ((HttpServletResponse) response).setStatus(200);
                response.getWriter().write("Already processed");
                return;
            }
            
            // Store in cache for future requests (24 hours)
            idempotencyStore.put(idempotencyKey, "PROCESSED");
            scheduleRemoval(idempotencyKey, 24, TimeUnit.HOURS);
        }
        
        chain.doFilter(request, response);
    }
}

// Client side implementation
public class IdempotentPaymentClient {
    public void makePayment(double amount) {
        String idempotencyKey = UUID.randomUUID().toString();
        
        HttpRequest request = HttpRequest.newBuilder()
            .uri(URI.create("https://api.example.com/payments"))
            .header("Idempotency-Key", idempotencyKey)
            .header("Content-Type", "application/json")
            .method("POST", HttpRequest.BodyPublishers.ofString(
                "{\"amount\": " + amount + "}"))
            .build();
        
        // Retry with same key if fails
        for (int i = 0; i < 3; i++) {
            try {
                HttpResponse<String> response = client.send(request, 
                    HttpResponse.BodyHandlers.ofString());
                if (response.statusCode() == 200) {
                    break; // Success
                }
            } catch (Exception e) {
                // Retry with same idempotency key
                continue;
            }
        }
    }
}
```

## **Q2. What are the Best Practices for Designing RESTful APIs?**

### 📋 **Complete Best Practices Checklist**

```yaml
1. Naming Conventions:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ Use nouns, not verbs:
   /users (not /getUsers)
   /products (not /fetchProducts)

✅ Use plural nouns:
   /users (not /user)
   /orders (not /order)

✅ Use lowercase with hyphens:
   /user-orders (not /userOrders or /user_orders)

✅ Use sub-resources for relationships:
   /users/123/orders (not /orders?userId=123)


2. HTTP Methods:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
GET    → /users          → List all users
GET    → /users/123      → Get specific user
POST   → /users          → Create new user
PUT    → /users/123      → Update entire user
PATCH  → /users/123      → Partial update
DELETE → /users/123      → Delete user


3. Status Codes:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
2xx Success:
  200 OK          → GET, PUT, DELETE success
  201 Created     → POST success (return location header)
  204 No Content  → DELETE success (no response body)

3xx Redirection:
  301 Moved Permanently  → Resource moved
  304 Not Modified       → Cache hit

4xx Client Errors:
  400 Bad Request    → Invalid request
  401 Unauthorized   → Authentication required
  403 Forbidden      → No permission
  404 Not Found      → Resource doesn't exist
  409 Conflict       → Duplicate resource
  422 Unprocessable  → Validation failed
  429 Too Many Requests → Rate limit exceeded

5xx Server Errors:
  500 Internal Server Error → Generic error
  502 Bad Gateway          → Upstream error
  503 Service Unavailable  → Server down


4. Versioning:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Option 1: URL Path (Most common)
  /v1/users
  /v2/users

Option 2: Custom Header
  Accept: application/vnd.myapi.v1+json

Option 3: Query Parameter
  /users?version=1


5. Filtering, Sorting, Pagination:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
GET /users?status=active&age=25        # Filtering
GET /users?sort=name:asc,age:desc      # Sorting
GET /users?page=2&limit=20             # Pagination
GET /users?fields=id,name,email        # Field selection

Response headers:
  X-Total-Count: 1000
  X-Page: 2
  X-Per-Page: 20
  Link: <https://api.com/users?page=3>; rel="next"


6. Security:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ Always use HTTPS
✅ Implement rate limiting
✅ Validate all inputs
✅ Use JWT or OAuth 2.0
✅ Sanitize outputs (prevent XSS)
✅ Log security events


7. Documentation:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ Use OpenAPI (Swagger) specification
✅ Provide examples for each endpoint
✅ Document error codes
✅ Include authentication details
✅ Keep documentation up to date
```

### 💻 **Complete REST API Example**

```java
@RestController
@RequestMapping("/api/v1/users")
public class UserController {
    
    @Autowired
    private UserService userService;
    
    // GET /users - List with pagination
    @GetMapping
    public ResponseEntity<Page<UserResponse>> getUsers(
            @RequestParam(defaultValue = "0") int page,
            @RequestParam(defaultValue = "20") int size,
            @RequestParam(required = false) String status,
            @RequestParam(defaultValue = "id,asc") String[] sort) {
        
        Page<UserResponse> users = userService.getUsers(page, size, status, sort);
        
        return ResponseEntity.ok()
                .header("X-Total-Count", String.valueOf(users.getTotalElements()))
                .header("X-Page", String.valueOf(page))
                .header("X-Per-Page", String.valueOf(size))
                .body(users);
    }
    
    // GET /users/{id} - Get single user
    @GetMapping("/{id}")
    public ResponseEntity<UserResponse> getUser(@PathVariable Long id) {
        UserResponse user = userService.getUser(id);
        return ResponseEntity.ok(user);
    }
    
    // POST /users - Create user
    @PostMapping
    public ResponseEntity<UserResponse> createUser(
            @Valid @RequestBody CreateUserRequest request) {
        
        UserResponse created = userService.createUser(request);
        
        return ResponseEntity
                .created(URI.create("/api/v1/users/" + created.getId()))
                .body(created);
    }
    
    // PUT /users/{id} - Full update
    @PutMapping("/{id}")
    public ResponseEntity<UserResponse> updateUser(
            @PathVariable Long id,
            @Valid @RequestBody UpdateUserRequest request) {
        
        UserResponse updated = userService.updateUser(id, request);
        return ResponseEntity.ok(updated);
    }
    
    // PATCH /users/{id} - Partial update
    @PatchMapping("/{id}")
    public ResponseEntity<UserResponse> patchUser(
            @PathVariable Long id,
            @RequestBody Map<String, Object> updates) {
        
        UserResponse patched = userService.patchUser(id, updates);
        return ResponseEntity.ok(patched);
    }
    
    // DELETE /users/{id} - Delete user
    @DeleteMapping("/{id}")
    public ResponseEntity<Void> deleteUser(@PathVariable Long id) {
        userService.deleteUser(id);
        return ResponseEntity.noContent().build();
    }
    
    // Exception Handler
    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(ResourceNotFoundException ex) {
        return ResponseEntity
                .status(HttpStatus.NOT_FOUND)
                .body(new ErrorResponse("NOT_FOUND", ex.getMessage()));
    }
}
```

## **Q3. What is the Difference Between SOAP and REST?**

### 📊 **Detailed Comparison Table**

| Aspect | SOAP | REST |
|--------|------|------|
| **Full Form** | Simple Object Access Protocol | Representational State Transfer |
| **Protocol** | Protocol (strict rules) | Architectural style (guidelines) |
| **Transport** | HTTP, SMTP, TCP, UDP | HTTP/HTTPS only |
| **Message Format** | XML only | JSON, XML, HTML, Plain Text |
| **State** | Stateless or Stateful | Stateless |
| **Security** | WS-Security (built-in) | HTTPS + OAuth/JWT |
| **Performance** | Slower (XML parsing) | Faster (JSON lightweight) |
| **Caching** | Not supported | Built-in HTTP caching |
| **Error Handling** | Built-in fault structure | HTTP status codes |
| **WSDL** | Yes (WSDL contract) | No (OpenAPI optional) |
| **Use Cases** | Enterprise, Banking, Telecom | Web APIs, Mobile, Public APIs |

```java
// ========== 1. JWT Authentication ==========
@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {
    
    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                   HttpServletResponse response,
                                   FilterChain chain) throws IOException, ServletException {
        
        String token = extractToken(request);
        
        if (token != null && validateToken(token)) {
            // Extract user info from token
            String username = extractUsername(token);
            List<String> roles = extractRoles(token);
            
            // Set authentication in context
            UsernamePasswordAuthenticationToken auth = 
                new UsernamePasswordAuthenticationToken(username, null, 
                    roles.stream().map(SimpleGrantedAuthority::new).collect(Collectors.toList()));
            
            SecurityContextHolder.getContext().setAuthentication(auth);
        }
        
        chain.doFilter(request, response);
    }
    
    private String extractToken(HttpServletRequest request) {
        String bearerToken = request.getHeader("Authorization");
        if (bearerToken != null && bearerToken.startsWith("Bearer ")) {
            return bearerToken.substring(7);
        }
        return null;
    }
}

// ========== 2. Rate Limiting ==========
@Component
public class RateLimitingFilter implements Filter {
    
    private final Map<String, AtomicInteger> requestCounts = new ConcurrentHashMap<>();
    private final Map<String, Long> resetTimes = new ConcurrentHashMap<>();
    
    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
                        FilterChain chain) throws IOException, ServletException {
        
        HttpServletRequest httpRequest = (HttpServletRequest) request;
        String clientId = getClientId(httpRequest);
        
        // 100 requests per minute
        int limit = 100;
        long windowMs = 60000;
        
        long currentTime = System.currentTimeMillis();
        Long resetTime = resetTimes.get(clientId);
        
        if (resetTime == null || currentTime > resetTime) {
            // New window
            resetTimes.put(clientId, currentTime + windowMs);
            requestCounts.put(clientId, new AtomicInteger(1));
        } else {
            int count = requestCounts.computeIfAbsent(clientId, 
                k -> new AtomicInteger(0)).incrementAndGet();
            
            if (count > limit) {
                ((HttpServletResponse) response).setStatus(429);
                response.getWriter().write("Rate limit exceeded");
                return;
            }
        }
        
        chain.doFilter(request, response);
    }
}

// ========== 3. Input Validation ==========
@RestControllerAdvice
public class ValidationHandler {
    
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<Map<String, String>> handleValidationExceptions(
            MethodArgumentNotValidException ex) {
        
        Map<String, String> errors = new HashMap<>();
        ex.getBindingResult().getAllErrors().forEach((error) -> {
            String fieldName = ((FieldError) error).getField();
            String errorMessage = error.getDefaultMessage();
            errors.put(fieldName, errorMessage);
        });
        
        return ResponseEntity.badRequest().body(errors);
    }
}

// DTO with validation
public class CreateUserRequest {

    @Size(min = 2, max = 50, message = "Name must be between 2 and 50 characters")
    @Pattern(regexp = "^[a-zA-Z\\s]+$", message = "Name can only contain letters")
    private String name;
    
    @NotBlank(message = "Email is required")
    @Email(message = "Invalid email format")
    private String email;
    
    @NotBlank(message = "Password is required")
    @Pattern(regexp = "^(?=.*[0-9])(?=.*[a-z])(?=.*[A-Z])(?=.*[@#$%^&+=])(?=\\S+$).{8,}$",
             message = "Password must contain at least 8 characters, one digit, one lowercase, one uppercase, one special character")
    private String password;
}

// Use parameterized queries (JPA/Hibernate)
@Repository
public class UserRepository {
    
    @PersistenceContext
    private EntityManager entityManager;
    
    // Safe - uses parameterized query
    public User findByEmail(String email) {
        return entityManager.createQuery(
            "SELECT u FROM User u WHERE u.email = :email", User.class)
            .setParameter("email", email)
            .getSingleResult();
    }
    
    // NEVER DO THIS - vulnerable to SQL injection
    public User findByEmailUnsafe(String email) {
        // ❌ NEVER use string concatenation
        String query = "SELECT * FROM users WHERE email = '" + email + "'";
        // This allows: email = "' OR '1'='1"
        return executeNativeQuery(query);
```

## **Q5. Difference Between PUT and POST**

| Aspect | POST | PUT |
|--------|------|-----|
| **Purpose** | Create new resource | Update existing resource (or create at known ID) |
| **Idempotency** | ❌ Non-idempotent | ✅ Idempotent |
| **URL** | Collection URL (`/users`) | Specific URL (`/users/123`) |
| **Response** | 201 Created (with location header) | 200 OK or 204 No Content |
| **Resource ID** | Server generates | Client provides |

### 💻 **Examples**

```java
// ========== POST - Create (Server generates ID) ==========
// Request
POST /api/users
{
    "name": "Raj",
    "email": "raj@example.com"
}

// Response
HTTP/1.1 201 Created
Location: /api/users/123
{
    "id": 123,
    "name": "Raj",
    "email": "raj@example.com"
}

// ========== PUT - Update/Create at known ID ==========
// Request
PUT /api/users/123
{
    "name": "Raj Kumar",
    "email": "raj.kumar@example.com"
}

// Response
HTTP/1.1 200 OK
{
    "id": 123,
    "name": "Raj Kumar",
    "email": "raj.kumar@example.com"
}

// ========== PUT for Create (if client knows ID) ==========
// Request

{
    "name": "New User",
    "email": "new@example.com"
}

// Response (if 456 didn't exist)
HTTP/1.1 201 Created
Location: /api/users/456
```

## **Q6. What is HATEOAS?**

### 📌 **Definition**

**HATEOAS (Hypermedia as the Engine of Application State)** = API responses mein links bhejna jo client ko bataye aage kya actions possible hain

**Real-life analogy:**
- **Without HATEOAS:** Restaurant menu mein sirf dish names (client ko pata hona chahiye kaise order karna hai)
- **With HATEOAS:** Menu mein har dish ke saath "Add to cart", "See reviews", "Order now" links bhi hain

### 💻 **Example**

```json
// ========== WITHOUT HATEOAS ==========
GET /api/orders/123
{
    "orderId": 123,
    "status": "PENDING",
    "total": 99.99
}
// Client needs to KNOW: /api/orders/123/pay, /api/orders/123/cancel


// ========== WITH HATEOAS ==========
GET /api/orders/123
{
    "orderId": 123,
    "status": "PENDING",
    "total": 99.99,
    "_links": {
        "self": { "href": "/api/orders/123" },
        "pay": { 
            "href": "/api/orders/123/payments",
            "method": "POST"
        },
        "cancel": {
            "href": "/api/orders/123",
```

## **Q7. Common Security Vulnerabilities in REST API**

### 📋 **OWASP Top 10 for APIs**

| # | Vulnerability | Description | Prevention |
|---|---------------|-------------|------------|
| 1 | **Broken Object Level Authorization** | User can access other user's data | Check permissions for every request |
| 2 | **Broken Authentication** | Weak token generation/validation | Use strong JWT, implement proper expiry |
| 3 | **Excessive Data Exposure** | API returns more data than needed | Filter sensitive fields |
| 4 | **Lack of Rate Limiting** | No limits on request frequency | Implement rate limiting |
| 5 | **Broken Function Level Authorization** | User can access admin endpoints | Role-based access control |
| 6 | **Mass Assignment** | User can update fields they shouldn't | Use DTOs, whitelist allowed fields |
| 7 | **Security Misconfiguration** | Default passwords, verbose errors | Secure defaults, custom error pages |
| 8 | **Injection** | SQL, NoSQL, Command injection | Parameterized queries, input validation |
| 9 | **Improper Assets Management** | Exposed API documentation, old versions | Proper versioning, hide internal endpoints |
| 10 | **Insufficient Logging** | No audit trail of attacks | Log all auth attempts, monitor anomalies |

### 💻 **Examples & Fixes**

```java
// ========== 1. Broken Object Level Authorization ==========
// ❌ VULNERABLE
@GetMapping("/users/{id}")
public User getUser(@PathVariable Long id) {
    // No check if user owns this ID!
    return userRepository.findById(id);
}

// ✅ FIXED
@GetMapping("/users/{id}")
public User getUser(@PathVariable Long id, @AuthenticationPrincipal User currentUser) {
    if (!currentUser.getId().equals(id) && !currentUser.isAdmin()) {
        throw new AccessDeniedException("Cannot access other user's data");
    }
    return userRepository.findById(id);
}

// ========== 2. Mass Assignment Protection ==========
// ❌ VULNERABLE
@PostMapping("/users")
public User createUser(@RequestBody User user) {
    // User can set isAdmin=true in request body!
    return userRepository.save(user);
}

// ✅ FIXED - Use DTO
public class CreateUserDTO {
    private String name;
    private String email;
    private String password;
    // No isAdmin field!
}

@PostMapping("/users")
public User createUser(@Valid @RequestBody CreateUserDTO dto) {
    User user = new User();
    user.setName(dto.getName());
    user.setEmail(dto.getEmail());
    user.setPassword(encodePassword(dto.getPassword()));
    user.setIsAdmin(false); // Server sets this
    return userRepository.save(user);
}

// ========== 3. Excessive Data Exposure ==========
// ❌ VULNERABLE - Sends everything including password hash, internal IDs
@GetMapping("/users/me")
public User getCurrentUser() {
    return userRepository.findById(currentUserId);
}

// ✅ FIXED - Use projection/DTO
public class UserProfileDTO {
    private Long id;
    private String name;
    private String email;
    private String avatarUrl;
    // No password, no internal fields
}

// ========== 4. SQL Injection Prevention ==========
// ❌ VULNERABLE
@GetMapping("/search")
public List<User> searchUsers(@RequestParam String name) {
    String query = "SELECT * FROM users WHERE name LIKE '%" + name + "%'";
    return jdbcTemplate.query(query, userRowMapper);
}

// ✅ FIXED
@GetMapping("/search")
public List<User> searchUsers(@RequestParam String name) {
    return userRepository.findByNameContainingIgnoreCase(name);
}
```

## **Q8. How Does Versioning Work in REST APIs?**

### 📊 **Versioning Strategies**

```yaml
1. URI Path Versioning (Most Common):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
GET /v1/users
GET /v2/users

Pros: Easy to implement, cache-friendly, visible
Cons: Breaks REST principle (URI should represent resource, not version)

2. Custom Header Versioning:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
GET /users
Accept-version: v1

GET /users
Accept-version: v2

Pros: URI stays clean, good for internal APIs
Cons: Harder to test, not cache-friendly

3. Content Negotiation (Accept Header):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
GET /users
Accept: application/vnd.myapi.v1+json

GET /users
Accept: application/vnd.myapi.v2+json

Pros: RESTful, follows HTTP standards
Cons: Complex, not user-friendly

4. Query Parameter Versioning:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
GET /users?version=1
GET /users?version=2

Pros: Simple
Cons: Can be cached incorrectly, not RESTful
```

### 💻 **Implementation Examples**

## **Q9. POST is Non-Idempotent - How to Convert to Idempotent?**

### 🔧 **Solution: Idempotency Key Pattern**

```java
// ========== Complete Idempotency Implementation ==========

// 1. Create Idempotency Store
@Component
public class IdempotencyService {
    
    private final RedisTemplate<String, IdempotencyRecord> redisTemplate;
    
    // Store for 24 hours
    private static final Duration TTL = Duration.ofHours(24);
    
    public boolean isProcessed(String idempotencyKey) {
        return redisTemplate.hasKey(idempotencyKey);
    }
    
    public void storeResult(String idempotencyKey, Object result) {
        IdempotencyRecord record = new IdempotencyRecord();
        record.setResult(result);
        record.setProcessedAt(Instant.now());
        redisTemplate.opsForValue().set(idempotencyKey, record, TTL);
    }
    
    public Object getCachedResult(String idempotencyKey) {
        IdempotencyRecord record = redisTemplate.opsForValue().get(idempotencyKey);
        return record != null ? record.getResult() : null;
    }
}

// 2. Idempotent POST endpoint
@RestController
public class PaymentController {
    
    @Autowired
    private IdempotencyService idempotencyService;
    
    @PostMapping("/api/payments")
    public ResponseEntity<?> createPayment(
            @RequestHeader(value = "Idempotency-Key", required = true) String idempotencyKey,
            @RequestBody PaymentRequest request) {
        
        // Check if already processed
        if (idempotencyService.isProcessed(idempotencyKey)) {
            Object cachedResult = idempotencyService.getCachedResult(idempotencyKey);
            return ResponseEntity.ok(cachedResult);
        }
        
        // Process the payment (ONLY ONCE)
        PaymentResult result = paymentService.processPayment(request);
        
        // Store result for future retries
        idempotencyService.storeResult(idempotencyKey, result);
        
        return ResponseEntity.ok(result);
    }
}

// 3. Database-level uniqueness constraint
@Entity
@Table(uniqueConstraints = @UniqueConstraint(columnNames = "idempotencyKey"))
public class PaymentTransaction {
    
    @Id
    private Long id;
    
    @Column(nullable = false, unique = true)
    private String idempotencyKey;
    
    private String paymentId;
    private BigDecimal amount;
    private String status;
}

// 4. Client-side implementation
public class IdempotentPaymentClient {
    
    public PaymentResult makePayment(PaymentRequest request) {
        String idempotencyKey = UUID.randomUUID().toString();
        
        // Retry up to 3 times with same key
        for (int attempt = 1; attempt <= 3; attempt++) {
            try {
                PaymentResult result = sendRequest(request, idempotencyKey);
                return result;
            } catch (NetworkException e) {
                // Network issue - retry with same key
                continue;
            } catch (TimeoutException e) {
                // Timeout - retry with same key
                continue;
            }
        }
        throw new PaymentException("Failed after 3 retries");
    }
}

// 5. Alternative: Use database unique constraint for natural keys
@PostMapping("/api/orders")
public ResponseEntity<Order> createOrder(@RequestBody OrderRequest request) {
    // Use business key for idempotency
    String naturalKey = request.getCustomerId() + "_" + request.getProductId() + "_" + request.getDate();
    
    try {
        Order order = orderRepository.findByNaturalKey(naturalKey);
        if (order != null) {
            return ResponseEntity.ok(order);
        }
        
        order = orderService.createOrder(request);
        order.setNaturalKey(naturalKey);
        return ResponseEntity.ok(order);
        
    } catch (DataIntegrityViolationException e) {
        // Concurrent request - fetch and return existing
        return ResponseEntity.ok(orderRepository.findByNaturalKey(naturalKey));
    }
}
```

## **Q10. What Happens If You Send JSON Body in GET Request?**

### 📌 **Short Answer**

**Technically possible but NOT recommended.** Most servers and proxies may ignore or reject the body.

### 🔍 **Detailed Explanation**

```yaml
HTTP Specification Says:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
RFC 7231: GET requests MAY have a body, but:
- Semantically, GET means "retrieve", not "send data"
- Servers are not required to read or parse GET body
- Many servers/proxies will ignore or reject it

What Actually Happens:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. Browser/Client: Can send (most HTTP clients allow)
2. Proxies/CDNs: May strip the body
3. Servers:
   - Tomcat: Rejects by default (400 Bad Request)
   - Nginx: Ignores the body
   - Spring Boot: Ignores (no @RequestBody binding)
   - Express.js: Body is undefined
4. Caching: GET requests are cached by URL only (body ignored)
```

### 💻 **Testing Different Behaviors**

```java

HttpRequest request = HttpRequest.newBuilder()
    .uri(URI.create("https://api.example.com/search"))
    .header("Content-Type", "application/json")
    .method("GET", HttpRequest.BodyPublishers.ofString("{\"name\":\"Raj\"}"))
    .build();

// Result: Request is sent with body, but server may ignore

// 2. Using curl
curl -X GET https://api.example.com/search \
  -H "Content-Type: application/json" \
  -d '{"name":"Raj"}'

// Result: Warning message: "Warning: You can only send one data item with -d at a time"
// Most servers will ignore the body

// 3. Using Postman
// POSTMAN allows JSON body for GET but shows warning

// ========== SERVER BEHAVIORS ==========

// Spring Boot - Body is IGNORED
@GetMapping("/search")
public String search(@RequestBody(required = false) SearchRequest request) {
    // request will be null! Body is ignored
    return "Search results";
}

// Express.js - Body is NOT available
app.get('/search', (req, res) => {
    console.log(req.body); // undefined (unless custom middleware)
    res.send('Search results');
});

// To actually read body in GET (NOT RECOMMENDED)
// Need custom parser (most frameworks don't support)
```

### ✅ **Best Practice**

```java

// ✅ DO THIS - Use query parameters for simple data

// ✅ OR Use POST for complex search criteria

    "filters": {

    },

}
```
