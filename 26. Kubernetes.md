# ☸ What is Kubernetes?
### Simple Intuition (Real-world Analogy)

**Mummy ke rasoi ka example:**
- **Without Kubernetes:** Tum khud bartan lao, chulha jalao, sabzi kaat ke pakao, timer lagao, plate mein serve karo. Sab kuch manually.
- **With Kubernetes:** Ek head chef hai jo bolega — "5 logon ka khana banana hai" → automatically bartan nikalega, gas on karega, timing maintain karega, plate mein serve karega. Agar koi bartan tuta → turant replacement laega.

**Tech analogy:** Kubernetes = **Operating System for Containers**

Jaise Windows/Linux hardware manage karta hai (CPU, RAM, Disk), waise hi Kubernetes containers manage karta hai across multiple machines.

### Official Definition

> **Kubernetes (K8s)** = Open-source container orchestration platform that automates deployment, scaling, and management of containerized applications.

**"K8s" kyun?** K + 8 letters (ubernete) + s = K8s

### Internal Working (High Level)

```
Developer writes YAML → kubectl apply → API Server → Scheduler → Nodes run containers
                                                              ↓
                                                    Constantly reconciles:
                                                    "Is actual = desired?"
```

## 2. Core Components Breakdown

### What Kubernetes Solves?

| Problem Without K8s | Solution With K8s |
|---------------------|-------------------|
| Container manually start karna | Declarative YAML → Auto-schedule |
| Server crash → manual restart | Self-healing (auto-restart) |
| Traffic badhe to manually scale | Auto-scaling (HPA) |
| New version deploy → downtime | Rolling updates (0 downtime) |
| Multiple servers ke beech load balancing | Service + kube-proxy |
| Config hardcoded in image | ConfigMaps, Secrets |

### Key Capabilities

| Capability | Explanation |
|------------|-------------|
| **Service Discovery & Load Balancing** | Container ko IP milta hai, DNS name, traffic distribute |
| **Storage Orchestration** | Automatic storage mounting (local, cloud, NFS) |
| **Automated Rollouts & Rollbacks** | Deploy new version, auto-rollback if fails |
| **Self-healing** | Restart failed containers, reschedule dead nodes |
| **Secret & Config Management** | Sensitive data (passwords, tokens) without rebuilding image |
| **Batch Execution** | Run one-time jobs, scheduled jobs |

## 4. Step-by-Step Workflow (What happens when you deploy?)

```
User: kubectl run nginx --image=nginx

Step 1: kubectl → API Server (authentication, RBAC check)
Step 2: API Server → etcd (stores Pod spec)
Step 3: Scheduler (watches etcd for unscheduled Pods)
Step 4: Scheduler → selects best Node (resource, affinity, taints)
Step 5: Scheduler → updates Pod with nodeName
Step 6: Kubelet (on selected Node) → sees Pod assigned to it
Step 7: Kubelet → calls Container Runtime to pull image
Step 8: Container Runtime → runs container
Step 9: Kubelet → updates Pod status back to API Server
Step 10: API Server → etcd (status updated)

Total time: ~1-3 seconds
```

## 8. Failure Handling & Self-Healing

| Failure | Kubernetes Response |
|---------|---------------------|
| Container crashes | Kubelet restarts it |
| Node dies | Controller reschedules pods to healthy nodes |
| App unresponsive | Liveness probe → restart |
| Traffic spike | HPA adds replicas |
| Network partition | Pods reschedule, etcd handles split-brain |

## 9. Best Practices (Production Grade)

### Do's ✅
- Use Deployments, not bare Pods
- Set resource requests/limits
- Use namespaces for isolation
- Implement liveness + readiness probes
- Store config in ConfigMaps, secrets in Secrets
- Use RBAC for security
- Version control all YAMLs

### Don'ts ❌
- Don't run stateful apps without StatefulSet
- Don't use `latest` tag in production
- Don't run as root container
- Don't ignore pod anti-affinity
- Don't store secrets in images

## Q1. You want to deploy an application with multiple replicas. How do you scale your Kubernetes deployment?

### 1. Concept Explanation

**Simple Intuition:**  
Imagine a chai stall. 1 waiter = 1 replica. Traffic badh gaya → 3 waiters (replicas) laga do. Kubernetes `Deployment` aapka manager hai jo waiters ki count maintain karta hai.

**Internal Working:**  
Deployment → ReplicaSet controller → Pods. ReplicaSet ensures desired replica count. Scaling = updating ReplicaSet's `replicas` field.

### 2. Core Components Breakdown

| Component | Responsibility |
|-----------|----------------|
| Deployment | Desired state (image, replicas, update strategy) |
| ReplicaSet | Maintains exact number of Pods |
| Scheduler | Assigns new Pods to Nodes |
| Kubelet | Starts/stops containers |

### 3. Architecture Diagram

```
User → kubectl scale → API Server → etcd (state updated)
                                           ↓
                              Deployment Controller (reconciles)
                                           ↓
                                    ReplicaSet (replicas: 5)
                                           ↓
                              Scheduler (Pod → Node binding)
                                           ↓
                              Node1(Pod1) Node2(Pod2) Node3(Pod3)
```

### 4. Step-by-Step Workflow (Scaling from 3 → 5)

1. `kubectl scale deployment myapp --replicas=5`
2. API Server writes new state to etcd
3. Deployment controller detects difference (3 → 5)
4. Updates ReplicaSet spec
5. ReplicaSet controller sees 2 missing Pods
6. Creates 2 new Pod objects (Pending state)
7. Scheduler binds Pods to Nodes
8. Kubelets start containers

### 5. Commands & YAML

```bash
# Imperative (quick)
kubectl scale deployment myapp --replicas=5

# Declarative (gitops way)
kubectl edit deployment myapp   # change replicas: 5

# HPA (automatic)
kubectl autoscale deployment myapp --min=2 --max=10 --cpu-percent=50
```

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 5  # ← yahi line badalni hai
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: app
        image: nginx:latest
        resources:
          requests:
            cpu: "100m"
```

### 6. Real-World Use Case

**Zomato/Rapido** - During lunch hour, HPA auto-scales from 50 to 500 replicas based on CPU/memory or custom metrics (RPS).

### 7. Scaling & Performance

- **Horizontal** (more replicas) - HPA
- **Vertical** (bigger replicas) - VPA (alpha)
- **Cluster scaling** - Cluster Autoscaler (adds Nodes if insufficient)

### 8. Failure Handling

| Problem | Debug |
|---------|-------|
| New Pods pending | `kubectl describe pod` → Insufficient CPU |
| CrashLoopBackOff | Check image, health probes |
| Pending forever | `kubectl get events --all-namespaces` |

### 9. Best Practices

✅ Set `resources.requests` for HPA  
✅ Use HPA, not manual scaling in prod  
✅ Max replicas ≤ cluster capacity  
✅ Test scale-in (drain behavior)

❌ Scaling without resource limits  
❌ Ignoring PDB (PodDisruptionBudget) for critical apps

### 10. Interview Questions

**Q:** How is Deployment different from ReplicaSet?  
**A:** Deployment manages ReplicaSets (rollback, pause, versioned). ReplicaSet only maintains pod count.

**Q:** Scaling limit?  
**A:** Soft limit 5000 pods/node, API server can handle 5000 nodes, 150k pods.

### 11. Follow-up Questions

- "How does HPA work internally?" → Metrics Server → API aggregator → Controller adjusts replicas every 15s (default)
- "What is scale-from-zero?" → Knative, KEDA (event-driven)

### 12. Short Summary

```bash
kubectl scale deploy/myapp --replicas=N
# Internally: Deployment → ReplicaSet → Pods
# Auto: HPA + Metrics Server
# Blocked by: resource shortage, PDB
```

## Q2. You have a Kubernetes pod that is stuck in CrashLoopBackOff. How do you diagnose and fix it?

### 1. Concept Explanation

**Intuition:**  
CrashLoopBackOff = Container repeatedly start hota hai, crash hota hai. Kubernetes "back off" (wait) karta hai between retries (10s, 20s, 40s... 5min). Jaise bachcha rota hai to maa thoda wait karke deti hai doodh.

**Internal Working:**  
Kubelet checks container exit code (non-zero = crash). Back-off timer doubles exponentially. After 5min, reset.

### 2. Core Components

| Component | Role |
|-----------|------|
| Kubelet | Restarts container, manages backoff |
| Container Runtime (Docker/containerd) | Runs container, returns exit code |
| Pod | Container wrapper |

### 3. Architecture Diagram

```
Pod → Container starts → Exit code 1 → Crash
         ↓
Kubelet: "Oh, crashed"
         ↓
Backoff timer: wait 10s → restart → crash again → wait 20s → crash → wait 40s...
         ↓
Pod status: CrashLoopBackOff
```

### 4. Step-by-Step Diagnosis Workflow

```bash
# 1. See pod status
kubectl get pods
# NAME      READY   STATUS             RESTARTS      AGE
# mypod     0/1     CrashLoopBackOff   5 (2m ago)    10m

# 2. Describe pod (events + last state)
kubectl describe pod mypod

# 3. See logs (previous crashed container)
kubectl logs mypod --previous

# 4. If no logs, check container exit code
kubectl describe pod | grep "Exit Code"

# 5. Check events
kubectl get events --field-selector involvedObject.name=mypod
```

### 5. Common Causes & Fixes

| Cause | Fix |
|-------|-----|
| **Image not found** | Check image name, registry auth |
| **Entrypoint fails** | Fix command, check dependencies (DB down) |
| **Config wrong** | Check ConfigMap/Secret volume mounts |
| **Port conflict** | Change port |
| **OOMKilled** | Increase memory limits |
| **Liveness probe failing** | Increase failureThreshold, initialDelaySeconds |

### 6. Real-World Use Case

**Paytm** - Payment service pod crashlooping due to Redis connection timeout. Fixed by adding retry logic + health probe tuning.

### 7. Debugging Commands

```bash
# Interactive debug pod (same image)
kubectl run debug --image=myapp --rm -it -- /bin/sh

# Check if container can start manually
docker run -it myapp:latest /bin/sh

# Check configmap mount
kubectl exec -it mypod -- cat /etc/config/app.conf  # if running

# Port forward to test
kubectl port-forward pod/mypod 8080:8080
```

### 8. Failure Handling

**What happens internally:**
1. Container exits with non-zero
2. Kubelet increments restart counter
3. Backoff = min(300, 10 * 2^restarts) seconds
4. Status = CrashLoopBackOff
5. After 5min without crash, backoff resets

### 9. Best Practices

✅ Add `initialDelaySeconds` to probes  
✅ Use `terminationMessagePath` to capture crash output  
✅ Set `restartPolicy: Never` for jobs  
✅ Structured logging (stdout/stderr)

❌ Hardcoded secrets in image  
❌ Ignoring OOMKilled

### 10. Interview Questions

**Q:** CrashLoopBackOff vs Error vs Pending?  
**A:** Pending = not scheduled; Error = container started but exited; CrashLoopBackOff = repeated crashes with exponential backoff.

**Q:** Backoff max?  
**A:** 5 minutes (300 seconds), managed by kubelet.

### 11. Follow-up

- "How to get crash logs without `--previous`?" → Sidecar container logging, or `terminationMessagePolicy: FallbackToLogsOnError`
- "What if probe fails vs container crashes?" → Probe failure = restart (depending on policy); crash = always restart

### 12. Summary

```bash
# Diagnosis chain
kubectl logs <pod> --previous
kubectl describe pod <pod>
kubectl get events

# Common fixes: image, config, resources, probes
# Backoff: 10s,20s,40s,80s,160s,300s
```

## Q3. You need to expose a service in your Kubernetes cluster to the internet. How do you expose the service using a LoadBalancer?

### 1. Concept Explanation

**Intuition:**  
LoadBalancer type service = Cloud provider (AWS/GCP/Azure) automatically creates a cloud load balancer. Jaise mall ka guard → cloud LB hai, jo traffic ko cluster ke andar distribute karta hai.

**Internal Working:**  
Service controller (cloud-provider integration) detects `type: LoadBalancer` → calls cloud API to create LB → assigns external IP → updates service status.

### 2. Core Components

| Component | Role |
|-----------|------|
| Service (type LB) | Abstraction over Pods |
| Cloud Controller Manager (CCM) | Creates cloud LB |
| kube-proxy | Updates iptables/IPVS rules |
| Cloud LB | External traffic entry point |

### 3. Architecture Diagram

```
Internet
    ↓
[Cloud Load Balancer] ← created by CCM
    ↓ (external IP: 35.200.x.x)
    ↓
NodePort (30000-32767) on all Nodes
    ↓
kube-proxy (iptables/IPVS)
    ↓
Service (ClusterIP)
    ↓
Endpoints (Pod IPs)
    ↓
Pod1 (10.244.1.5)   Pod2 (10.244.2.8)
```

### 4. Step-by-Step Workflow

1. `kubectl expose deployment myapp --type=LoadBalancer --name=myapp-svc`
2. API Server creates Service object
3. Cloud Controller Manager watches for LoadBalancer type
4. CCM calls cloud API (AWS ELB, GCP LB)
5. Cloud LB gets public IP, health checks configured
6. CCM updates Service with `status.loadBalancer.ingress.ip`
7. Traffic flow: Internet → Cloud LB → NodePort → kube-proxy → Pod

### 5. Commands & YAML

```bash
# Quick expose
kubectl expose deployment myapp --type=LoadBalancer --port=80 --target-port=8080

# Check external IP (can take 2-3 min)
kubectl get svc myapp-svc
# NAME        TYPE           CLUSTER-IP     EXTERNAL-IP     PORT(S)
# myapp-svc   LoadBalancer   10.96.123.45   35.200.10.20    80:31234/TCP

# Get external IP
kubectl get svc myapp-svc -o jsonpath='{.status.loadBalancer.ingress[0].ip}'
```

```yaml
apiVersion: v1
kind: Service
metadata:
  name: myapp-svc
spec:
  type: LoadBalancer           # ← key line
  selector:
    app: myapp                 # Which pods to target
  ports:
  - protocol: TCP
    port: 80                   # Service port (external facing)
    targetPort: 8080           # Container port
    nodePort: 31234            # Optional, auto-assigned
  externalTrafficPolicy: Local # Preserves client IP
```

### 6. Real-World Use Case

**Swiggy** - API gateway exposed via AWS NLB (Network Load Balancer) for high throughput. HTTP/2 traffic.

### 7. Performance Considerations

| Cloud LB Type | Use Case |
|---------------|----------|
| Classic LB (CLB) | Basic HTTP |
| Application LB (ALB) | Layer 7, path-based routing |
| Network LB (NLB) | High throughput, TCP/UDP, static IP |

```yaml
# AWS NLB example
metadata:
  annotations:
    service.beta.kubernetes.io/aws-load-balancer-type: "nlb"
    service.beta.kubernetes.io/aws-load-balancer-cross-zone-load-balancing-enabled: "true"
```

### 8. Failure Handling

| Problem | Debug |
|---------|-------|
| External IP pending | Check CCM logs, cloud permissions |
| No route to host | Security group rules, NodePort reachable? |
| Health check failing | Set correct `spec.healthCheckNodePort` |

```bash
# Debug chain
kubectl describe svc myapp-svc
kubectl logs -n kube-system cloud-controller-manager-xxx
kubectl get endpoints myapp-svc  # Pods present?
```

### 9. Best Practices

✅ Use `externalTrafficPolicy: Local` to preserve client IP  
✅ Set `healthCheckNodePort` for custom health checks  
✅ Use annotations for cloud-specific features (SSL termination, etc.)  
✅ Consider Ingress for HTTP workloads (cheaper)

❌ LoadBalancer for every microservice (expensive → use Ingress)  
❌ Ignoring idle timeout settings

### 10. Interview Questions

**Q:** LoadBalancer vs NodePort vs ClusterIP?  
**A:** ClusterIP (internal only), NodePort (exposed on Node IP:port), LoadBalancer (cloud LB + NodePort + ClusterIP)

**Q:** How does kube-proxy fit in?  
**A:** Updates iptables rules on each node. Cloud LB sends traffic to NodePort, kube-proxy forwards to correct Pod.

### 11. Follow-up

- "What if you're on bare metal?" → MetalLB (L2/BGP)
- "How to get static IP?" → Cloud LB annotations for static IP allocation
- "LoadBalancer for UDP?" → Yes, but NLB recommended

### 12. Summary

```yaml
type: LoadBalancer
# Creates: Cloud LB → NodePort → Service → Pods
# Cloud provider specific (CCM handles)
# Check: kubectl get svc -w (wait for EXTERNAL-IP)
```

## Q4. You want to run a job in Kubernetes that executes once and then completes. How do you create a job?

### 1. Concept Explanation

**Intuition:**  
Cron job ki tarah but ek baar. Jaise DB migration script — run karo, khatam ho jao. Pods delete nahi hote, but Completed status mein chale jaate hain.

**Internal Working:**  
Job controller creates Pods until `completions` number succeed. After completion, Pods stay (for logs). Controller stops reconciling.

### 2. Core Components

| Component | Role |
|-----------|------|
| Job | API object defining task |
| Job Controller | Watches for Jobs, creates Pods |
| Pod | Runs the actual container |

### 3. Architecture Diagram

```
User → kubectl create -f job.yaml
              ↓
        API Server → etcd
              ↓
        Job Controller (watches)
              ↓
    Creates Pod with restartPolicy: Never/OnFailure
              ↓
        Pod runs container
              ↓
    Exit 0 → Pod status: Completed
    Exit 1 → Pod status: Error (retries)
              ↓
    Job controller updates status.succeeded/failed
```

### 4. Step-by-Step Workflow

1. Create Job YAML (completions=1, parallelism=1)
2. Job controller detects new Job
3. Creates Pod with `restartPolicy: Never`
4. Pod runs to completion (exit 0)
5. Job controller sets `status.succeeded=1`
6. Job marked as Complete, no new Pods created

### 5. Commands & YAML

```bash
# Simple job
kubectl create job my-job --image=busybox -- echo "Hello"

# Check status
kubectl get jobs
# NAME     COMPLETIONS   DURATION   AGE
# my-job   1/1           2s         10s

# Check pod
kubectl get pods
# NAME           READY   STATUS      RESTARTS   AGE
# my-job-xxx     0/1     Completed   0          15s

# Logs after completion
kubectl logs my-job-xxx

# Delete job (deletes pods too)

```

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: db-migration
spec:
  completions: 1          # Total successful pods needed
  parallelism: 1          # Run how many in parallel
  backoffLimit: 4         # Retries on failure
  ttlSecondsAfterFinished: 100  # Auto cleanup after 100s
  template:
    spec:
      restartPolicy: Never  # Never or OnFailure only
      containers:
      - name: migration
        image: myapp:latest
        command: ["node", "migrate.js"]
        env:
        - name: DB_URL
          valueFrom:
            secretKeyRef:
              name: db-secret
              key: url
```

### 6. Real-World Use Case

**Razorpay** - Daily reconciliation job. After batch completes, job stays for audit logs. TTL deletes after 7 days.

### 7. Advanced Patterns

```yaml
# Parallel processing (multiple items)
spec:
  completions: 100        # 100 total
  parallelism: 10         # 10 at a time
  
# Indexed jobs (each pod gets unique index)
spec:
  completions: 5
  parallelism: 5
  completionMode: Indexed
# Pods: myjob-0, myjob-1 (index via JOB_COMPLETION_INDEX env)
```

### 8. Failure Handling

| Scenario | Behavior |
|----------|----------|
| Pod fails (exit 1) | Retries up to `backoffLimit` |
| Pod deleted | Recreated |
| Node dies | Rescheduled |
| Timeout | No timeout by default (use `activeDeadlineSeconds`) |

```yaml
spec:
  activeDeadlineSeconds: 300  # 5 min timeout
  backoffLimit: 4
```

### 9. Best Practices

✅ Always set `restartPolicy: Never` or `OnFailure`  
✅ Use `ttlSecondsAfterFinished` for cleanup  
✅ Set resource limits (avoid unbounded memory)  
✅ Use `activeDeadlineSeconds` for timeouts  

❌ Using `restartPolicy: Always` (Jobs don't support it)  
❌ Not collecting logs before deletion

### 10. Interview Questions

**Q:** Job vs CronJob vs Deployment?  
**A:** Job (run to completion once), CronJob (scheduled Jobs), Deployment (long-running services)

**Q:** What happens if Job fails?  
**A:** BackoffLimit retries (default 6). After that, Job status = Failed.

### 11. Follow-up

- "How to run parallel jobs with queue?" → Work queues (RabbitMQ), indexed jobs, or job templates
- "Job vs Pod with restartPolicy: Never?" → Job gives retries, parallel completions, TTL

### 12. Summary

```yaml

kind: Job
spec:
  completions: 1
  template:
    spec:
      restartPolicy: Never
# Check: kubectl get jobs, kubectl logs job/my-job
# Cleanup: ttlSecondsAfterFinished or manual delete
```

## Q5. You need to restrict access to a Kubernetes service based on labels. How do you achieve this?

### 1. Concept Explanation

**Intuition:**  
NetworkPolicy = Firewall for Pods. Labels ke basis pe allow/deny traffic. Jaise club mein VIP entry: frontend label wale pods ko hi backend service access milta hai.

**Internal Working:**  
CNI (Calico/Cilium/Weave) implements NetworkPolicy via iptables/eBPF rules. API Server stores policy, CNI watches and programs data plane.

### 2. Core Components

| Component | Role |
|-----------|------|
| NetworkPolicy | K8s API object defining rules |
| CNI (Calico/Cilium) | Enforces rules (iptables/eBPF) |
| Pod Labels | Selectors for source/destination |

### 3. Architecture Diagram

```
[Pod A]  →  [Service]  →  [Pod B]
label: app=frontend          label: app=backend

Without NetworkPolicy: all traffic allowed

With NetworkPolicy:
    Ingress rule: allow only from pods with label app=frontend
    Egress rule: allow only to pods with label app=db

CNI adds iptables:
    -A KUBE-NWPLCY -m comment --comment "allow frontend" \
      -p tcp --dport 8080 -m set --match-set frontend-ips src -j ACCEPT
```

### 4. Step-by-Step Workflow

1. Create NetworkPolicy YAML (podSelector, policyTypes, ingress/egress rules)
2. kubectl apply → API Server stores policy
3. CNI controller watches NetworkPolicy events
4. CNI translates policy to data plane rules (iptables/eBPF)
5. Traffic is filtered at node level

### 5. Commands & YAML

```yaml
# Deny all ingress to backend pods except from frontend
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: backend-policy
spec:
  podSelector:
    matchLabels:
      app: backend           # Target these pods
  policyTypes:
  - Ingress                  # Apply to incoming traffic
  ingress:
  - from:
    - podSelector:           # Allow from pods with label
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080
```

```yaml
# Allow only from specific namespace
ingress:
- from:
  - namespaceSelector:
      matchLabels:
        name: production
    podSelector:
      matchLabels:
        app: api
```

```yaml
# Deny all traffic (default deny)
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress
  # No rules = deny all
```

```bash
# Check policies
kubectl get networkpolicies
kubectl describe networkpolicy backend-policy

# Test connectivity (from frontend pod)
kubectl exec -it frontend-pod -- curl backend-service:8080  # Should work
kubectl exec -it other-pod -- curl backend-service:8080     # Should fail
```

### 6. Real-World Use Case

**PhonePe** - Multi-tenant cluster. NetworkPolicy isolates payment service: only `role=payment-gateway` pods can access DB pods.

### 7. Advanced Scenarios

```yaml
# Egress filtering (pod to external)
egress:
- to:
  - ipBlock:
      cidr: 10.0.0.0/8
      except:
      - 10.0.1.0/24
  ports:
  - port: 443

# Combined selectors
from:
- namespaceSelector:
    matchLabels:
      tenant: acme
  podSelector:
    matchLabels:
      tier: frontend
```

### 8. Failure Handling & Debugging

| Problem | Debug |
|---------|-------|
| Policy not working | Check CNI (Calico: `calicoctl get networkpolicy`) |
| Connectivity blocked | `kubectl exec` with `tcpdump` or `nc -vz` |
| Wrong selector | `kubectl get pods --show-labels` |

```bash
# Debug commands
kubectl run test --image=busybox -it --rm -- /bin/sh
# Inside test pod:
wget -O- http://backend-service:8080

# Check iptables (on node)
iptables -L -n | grep -i "networkpolicy"
```

### 9. Best Practices

✅ Implement default deny (defense in depth)  
✅ Use namespaces + NetworkPolicy together  
✅ Log denied traffic (Calico/Cilium flow logs)  
✅ Test policies before prod (e.g., `kube-bench`)

❌ Relying only on NetworkPolicy (also use RBAC, mTLS)  
❌ Forgetting egress policies (pods can still call external APIs)

### 10. Interview Questions

**Q:** How does NetworkPolicy work without CNI?  
**A:** It doesn't. CNI must support it (Calico, Cilium, Weave, Antrea). Flannel (default) doesn't support.

**Q:** PodSelector vs NamespaceSelector?  
**A:** PodSelector = same namespace pods; NamespaceSelector = all pods in matched namespaces.

### 11. Follow-up

- "What about encryption between pods?" → Service Mesh (Istio, Linkerd) with mTLS
- "Can NetworkPolicy block node ports?" → No, node ports bypass NetworkPolicy. Use cloud security groups.

### 12. Summary

```yaml
# NetworkPolicy = Pod-level firewall
# Supported by: Calico, Cilium, Weave
# Default: allow all (if no policy)
# Use podSelector, namespaceSelector, ipBlock
# Test: kubectl exec test-pod -- curl target-service
```

## Q7. You have an application deployed on Kubernetes that is experiencing increased traffic. How would you scale the application to handle the increased load?

### Complete Answer (HPA + VPA + Cluster Autoscaler)

#### 1. Horizontal Pod Autoscaler (HPA) - Primary Solution

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 3
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80
  - type: Pods
    pods:
      metric:
        name: requests_per_second
      target:
        type: AverageValue
        averageValue: 500
```

```bash
# Create HPA
kubectl autoscale deployment myapp --cpu-percent=70 --min=3 --max=20

# Monitor
kubectl get hpa -w
kubectl describe hpa myapp-hpa
```

#### 2. Vertical Pod Autoscaler (VPA) - Companion

```yaml
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: myapp-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  updatePolicy:
    updateMode: "Auto"  # or "Off", "Initial", "Recreate"
  resourcePolicy:
    containerPolicies:
    - containerName: "*"
      minAllowed:
        cpu: "100m"
        memory: "256Mi"
      maxAllowed:
        cpu: "2"
        memory: "4Gi"
```

#### 3. Cluster Autoscaler (Node Level)

```bash
# AWS EKS
kubectl apply -f https://github.com/kubernetes/autoscaler/releases/download/cluster-autoscaler-1.27.0/cluster-autoscaler-autodiscover.yaml

# Annotations for cluster-autoscaler
spec:
  template:
    metadata:
      annotations:
        cluster-autoscaler.kubernetes.io/safe-to-evict: "true"
```

#### 4. Event-Driven Autoscaling (KEDA) - Advanced

```yaml
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: myapp-scaledobject
spec:
  scaleTargetRef:
    name: myapp
  triggers:
  - type: redis
    metadata:
      address: redis-master:6379
      listName: myqueue
      listLength: "5"
```

### Performance Optimization

```yaml
# Resource requests (critical for HPA)
resources:
  requests:
    cpu: "500m"
    memory: "1Gi"
  limits:
    cpu: "1"
    memory: "2Gi"
```

|---------|----------|

## Q9. While troubleshooting a networking issue, you noticed kube-proxy in the logs. What is the role of kube-proxy?

### Concept

**Intuition:**  
kube-proxy = Traffic cop on each node. Har node pe baitha hai, decide karta hai ki `my-service:80` pe aaya packet kis Pod ko jaana chahiye.

### Internal Working

```
Service: my-service (ClusterIP 10.96.0.1, port 80)
Endpoints: Pod1 (10.244.1.5:8080), Pod2 (10.244.2.8:8080)

kube-proxy modes:
1. iptables (default, Linux)
2. IPVS (high performance, round-robin)
3. userspace (deprecated)
```

### iptables Mode (Default)

```bash
# iptables rules on each node
iptables -t nat -L KUBE-SERVICES
# Chain KUBE-SVC-XXX (for my-service)
# Statistics: random 50% to Pod1, 50% to Pod2
```

### IPVS Mode (Better for large clusters)

```bash
# Enable IPVS
kubectl edit configmap -n kube-system kube-proxy
# mode: "ipvs"

ipvsadm -Ln
# TCP  10.96.0.1:80 rr
#   -> 10.244.1.5:8080   Masq    1      0          0         
#   -> 10.244.2.8:8080   Masq    1      0          0
```

### Debugging kube-proxy

```bash
# Check kube-proxy logs
kubectl logs -n kube-system kube-proxy-xxxxx

# Check iptables rules

```

### Common Issues

| Issue | Fix |
|-------|-----|
| Service not reachable | Check kube-proxy pod, iptables rules |
| Uneven load balancing | Switch to IPVS mode |
| High latency | Increase `conntrack` settings |

### Summary

```
kube-proxy = per-node network proxy
```

## Q10. What is Kubernetes Ingress?

### Concept

**Intuition:**  
LoadBalancer ek app ke liye, Ingress = ek hi LoadBalancer se multiple apps expose karo. Jaise society ka main gate (Ingress) → different buildings (Services) → flats (Pods).

### Architecture Diagram

```
Internet
    ↓
[LoadBalancer] (single, cloud LB)
    ↓
[Ingress Controller] (Nginx, Traefik, AWS ALB)
    ↓
Ingress Rules:
    /api/*    → api-service:8080
    /web/*    → web-service:80
    /static/* → static-service:9000
    ↓
Services → Pods
```

### Ingress YAML

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: myapp-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  tls:
  - hosts:
    - myapp.example.com
    secretName: tls-secret
  rules:
  - host: myapp.example.com
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 8080
      - path: /web
        pathType: Prefix
        backend:
          service:
            name: web-service
            port:
              number: 80
```

### Ingress Controllers

| Controller | Best for |
|------------|----------|
| Nginx Ingress | General purpose, most popular |
| Traefik | Dynamic config, automatic TLS |
| AWS ALB Ingress | AWS native, expensive |
| Contour | Envoy-based, high performance |
| Istio Gateway | Service mesh integration |

### Comparison: Ingress vs LoadBalancer

| Feature | Ingress | LoadBalancer |
|---------|---------|--------------|
| Cost | 1 LB for multiple services | 1 LB per service |
| Path-based routing | ✅ | ❌ |
| SSL termination | ✅ | ✅ (with extra cost) |
| Host-based routing | ✅ | ❌ |

### Summary

```yaml
# Ingress = Layer 7 routing (HTTP/HTTPS)
# Requires Ingress Controller (not built-in)
# Use for: microservices API gateway, path-based routing
# Check: kubectl get ingress, kubectl describe ingress
```

## Q11. What is the purpose of liveness and readiness probes in Kubernetes?

### Concept

**Intuition:**  
- **Liveness probe** = "Is app alive?" → Dead ho to restart karo
- **Readiness probe** = "Is app ready to serve traffic?" → Traffic do ya nahi

### Real-world analogy

```
Liveness: "Sans le raha hai?" (No → CPR/Crash)
Readiness: "Kaam kar sakta hai?" (No → Traffic mat bhej)
```

### YAML Example

```yaml
spec:
  containers:
  - name: myapp
    image: myapp:latest
    livenessProbe:
      httpGet:
        path: /healthz
        port: 8080
      initialDelaySeconds: 30   # Wait 30s before first probe
      periodSeconds: 10         # Check every 10s
      failureThreshold: 3       # Fail after 3 failures
      successThreshold: 1       # Recover after 1 success
      timeoutSeconds: 1         # Timeout after 1s
    
    readinessProbe:
      httpGet:
        path: /ready
        port: 8080
      initialDelaySeconds: 5
      periodSeconds: 5
      failureThreshold: 3
```

### Probe Types

| Type | Method | Use Case |
|------|--------|----------|
| httpGet | HTTP GET | Web apps, REST APIs |
| tcpSocket | TCP handshake | Databases, Redis |
| exec | Command runs | Custom checks |

```yaml
# exec probe
livenessProbe:
  exec:
    command:
    - cat
    - /tmp/healthy
  
# tcp probe
readinessProbe:
  tcpSocket:
    port: 3306
```

### Lifecycle Timeline

```
Pod Start → initialDelaySeconds → Ready? → Traffic
                    ↓
        Readiness probe: fail → Remove from Service endpoints
                    ↓
        Readiness probe: pass → Add back to endpoints
                    ↓
        Liveness probe: fail → Restart container
```

### Common Pitfalls

| Mistake | Consequence | Fix |
|---------|-------------|-----|
| No initialDelaySeconds | Probe fails before app starts | Set 30-60s |
| Same endpoint for both | Restart during startup | Separate /healthz (liveness) vs /ready (readiness) |
| Too aggressive | Throttling, crashloop | Increase periodSeconds, failureThreshold |

### Debugging

```bash
# Check probe status
kubectl describe pod mypod | grep -A 10 "Liveness\|Readiness"

# Check events
kubectl get events | grep -i probe

# Check if pod is ready
kubectl get pods -o wide
# READY column = ready/containers
```

### Summary Table

| | Liveness | Readiness |
|---|----------|-----------|
| **Purpose** | Keep app alive | Traffic routing |
| **On fail** | Restart container | Remove from Service |
| **Use when** | App deadlocked, crashed | App starting, overloaded |
| **Typical endpoint** | /healthz | /ready |
| **Default** | None (never check) | None |
