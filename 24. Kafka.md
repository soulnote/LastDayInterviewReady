
### **Q1. What is Apache Kafka?**
**Kafka ek distributed event streaming platform hai.**  
- Basically, ek **high-throughput, low-latency messaging system** jo real-time data pipeline me use hota hai.  
- Ye **publish-subscribe** model pe kaam karta hai: producers data bhejte hain topics me, consumers us data ko padhte hain.  
- Data **disk pe persist** hota hai (retention ke saath), isliye replay possible hai.  

> 🔹 Interview me bolna: *Kafka is like a central nervous system for microservices communication.*

---

### **Q2. List few use cases of Kafka**
- **Real-time analytics** (e.g., tracking user clicks)
- **Log aggregation** (collect logs from multiple services)
- **Stream processing** (with Kafka Streams or Flink)
- **Message queue replacement** (high throughput)
- **Event sourcing** (store state changes as events)
- **CDC (Change Data Capture)** – capture DB changes and stream

---

### **Q3. List few advantages of Kafka**
- **High throughput** – lakhs messages/sec
- **Low latency** – <10 ms (often)
- **Durability** – data disk pe rehta hai
- **Scalability** – partitions increase parallel processing
- **Fault tolerance** – replication ensures no data loss
- **Ordering guarantee** – within a partition

### **Q4. Difference between Queue and Topic**
| Feature | Queue | Topic |
|---------|-------|-------|
| Model | Point-to-point | Publish-subscribe |
| Consumer | Single consumer per message | Multiple consumer groups |
| Message deletion | After consumption | Retention-based |
| Use case | Load balancing | Broadcasting |

> 🔹 **Queue** – ek baar consume, delete.  
> 🔹 **Topic** – har consumer group apni copy padh sakta hai.

### **Q5. In which scenarios we should not use Kafka?**
- **Very low message volume** (Kafka overhead is high)
- **Small team without ops experience** (Kafka management complex)
- **Need of request-response (RPC)** – Kafka is async
- **Message priority or strict transactional across topics** (possible but complex)
- **If you need exactly-once without idempotency** – tricky but doable

### **Q6. In the Producer, when does QueueFullException occur?**
Jab producer ke internal buffer me memory full ho jaaye (due to `max.block.ms` or `buffer.memory` limit).  
> Reason: broker slow hai, network issue, ya acks=all but replicas down.

---

### **Q7. Does Kafka really guarantee the order of messages?**
**Haan, but only within a single partition.**  
- Same partition → order guarantee.  
- Different partitions → no cross-partition ordering.  
- If you need global order → single partition, but then parallelism suffers.

---

### **Q8. Can multiple producers send messages to one topic in Kafka? If same time, what happens?**
Yes, multiple producers can send to same topic.  
**Same time pe** → Kafka assigns sequential offsets. No data loss, no conflict.  
> Underlying mechanism: broker handles concurrent requests and appends to partition log in arrival order (small granularity).

---

### **Q9. How does partition work in Kafka?**
Topic ko logical chunks me todna hi **partition** hai.  
- Har partition ek ordered, immutable log hai.  
- Partition number se data store hota hai.  
- Multiple partitions = parallel reads/writes.

---

### **Q10. Explain how partitions, consumers, and consumer groups work**
- **Partitions** – unit of parallelism.
- **Consumer group** – group of consumers working together.
- **Ek partition ek consumer group me sirf ek consumer ko milega.**  
  Agar consumers zyada hain partitions se → idle consumers.  
  Agar partitions zyada hain → parallel processing.

> Example: 4 partitions, 2 consumers in same group → har consumer 2 partitions lega.

---

### **Q11. How does Kafka decide what partition a record should go to?**
Priority order:
1. If **key** provided → `hash(key) % partitions` (same key always same partition)
2. If **no key** but custom partitioner → as defined
3. Default **round-robin** (sticky partitioner in newer versions)

### **Q12. Write a simple SpringBoot Kafka producer and consumer**

**application.properties**
```properties
spring.kafka.bootstrap-servers=localhost:9092
spring.kafka.consumer.group-id=my-group
```

**Producer**
```java
@Service
public class Producer {
    @Autowired
    KafkaTemplate<String, String> kafkaTemplate;

    public void sendMsg(String msg) {
        kafkaTemplate.send("my-topic", msg);
    }
}
```

**Consumer**
```java
@Service
public class Consumer {
    @KafkaListener(topics = "my-topic", groupId = "my-group")
    public void listen(String msg) {
        System.out.println("Received: " + msg);
    }
}
```

### **Q13. Why should offset reset be used cautiously in production?**
Offset reset (`earliest` / `latest`) production me use karte ho to **data reprocessing ya data loss** ho sakta hai.  
- `earliest` → saara purana data dubara process hoga (may be huge)
- `latest` → naye messages se start, old data miss
> Best: explicitly manage offsets ya use committed offsets.

---

### **Q14. How do Kafka partitions and replicas work together for performance and fault tolerance?**
- **Partitions** → performance (parallelism)  
- **Replicas** → fault tolerance (leader-follower model)  
  - Leader handles all read/writes.  
  - Followers sync from leader.  
  - Leader fail ho → ISR me se naya leader elect.

> 🔹 **Performance** = partitions  
> 🔹 **Fault tolerance** = replicas + ISR

---

### **Q15. What is a Kafka consumer group, and why is it essential for scalability?**
**Consumer group** = set of consumers sharing same `group.id`.  
- Har partition ek consumer ko assign hota hai.  
- **Scalability** because:  
  - Add more consumers → more partitions consumed in parallel.  
  - Max parallelism = number of partitions.  
  - Rebalancing automatically hota hai.

---

### **Q16. How do kafka partitions and replicas work together to ensure performance and fault tolerance?**
Same as Q14 ka detailed version:  
- Partitions → parallel processing (performance)  
- Replicas → each partition ki copy (1 leader + N followers)  
- Leader partition handles traffic; followers sync.  
- Leader down → controller selects new leader from ISR (in-sync replicas)  
- Without replicas → broker down = data loss  
- Without partitions → single point of bottleneck

# **Scenerion based questions**
### **Q1. Kafka Production issue 1 - order is processed multiple times?**

**Symptom:** Same order/event consumer ne 2-3 baar process kar liya.

**Root causes:**
1. **Consumer processing ke baad offset commit nahi hua** (auto.commit false hai but manual commit miss ho gaya)
2. **Processing success hui but commit fail** (network ya broker issue)
3. **Rebalancing** – consumer died before commit, partition reassign ho gaya
4. **Idempotent producer enabled nahi** – producer retry kar raha tha

**Solution:**
- Enable **idempotent producer** (`enable.idempotence=true`)
- Use **exactly-once semantics** with transactions
- Consumer side pe **idempotent processing** (e.g., DB me unique constraint)

> 🔹 Interview me bolna: *"Multiple processing usually happens due to at-least-once delivery. We fixed by making consumer idempotent or moving to exactly-once with transactions."*

---

### **Q2. Kafka Production issue 2 - payment transactions are missing?**

**Symptom:** Payment event topic me aaya but consumer ne nahi dekha / DB me nahi hai.

**Root causes:**
1. **Offset commit too early** – consumer ne commit kar diya but processing fail (data loss)
2. **Consumer lag** – message abhi tak process nahi hua (not missing, just delayed)
3. **Producer acks=0** – producer ko fark nahi padta, broker gayab ho sakta hai
4. **Retention policy** – message expire ho gaya (log.retention.hours)
5. **Consumer subscribed to wrong partition / wrong group.id**

**Solution:**
- Use `acks=all` and `min.insync.replicas=2`
- Consumer me `enable.auto.commit=false`
- Monitoring lag via `kafka-consumer-groups --describe`
- Retention increase for critical topics

> 🔹 Real story: *"Humare paas ek case tha jaha acks=0 tha, broker restarted and we lost 3 seconds of payments. Ab acks=all hai."*

---

### **Q3. Will any messages be lost if one of Kafka Consumers crashes?**

**Depends on configuration:**

| Scenario | Data loss? |
|----------|-----------|
| `enable.auto.commit=true` + processing ke baad crash | ❌ Haan, loss possible (commit before processing) |
| `enable.auto.commit=false` + manual commit after success | ✅ No loss |
| Consumer crash, group rebalance | ✅ No loss (partitions reassign, last committed offset se start) |
| Consumer crash before commit + auto.commit=true | ❌ Loss |

**Best practice:** Disable auto commit, commit only after successful processing.

---

### **Q4. Can two consumers consume from same partition at the same time in kafka?**

**NO.** Ek partition, ek consumer group me **sirf ek consumer** ko assign hota hai.

- Same group me → impossible
- Different groups me → possible (har group ki apni offset)

> Reason: Kafka wants **ordered processing** per partition. Multiple consumers would break ordering.

---

### **Q5. What happens when number of consumers in kafka is greater than the number of partitions?**

**Extra consumers idle baithenge (do nothing).**

Example: 3 partitions, 5 consumers in same group
- 3 consumers active
- 2 consumers sleep (no partition assigned)

> 🔹 Interview me: *"Extra consumers waste hai. Max parallelism = number of partitions."*

---

### **Q6. Once Consumer B finishes its work on Partition B, does it wait idle until Consumer A finishes?**

**No.** Har consumer apni partition pe independent hai.

- Consumer A busy (partition A heavy)
- Consumer B free (partition B light)
- Consumer B **idle rahega**, A ki help nahi kar sakta

> Work stealing nahi hoti Kafka me. Solution: **more partitions** for better distribution.

---

### **Q7. Did you or devops had to reset kafka consumer offsets in dev or prod environment?**

**Yes, multiple times.**

**Situations:**
- **Buggy code deployed** – wrong data written, need to reprocess from earlier offset
- **Schema change** – old messages need reprocessing with new schema
- **Consumer lag too high** – skip old messages (seek to end)
- **Data corruption fix** – replay from beginning

**Commands used:**
```bash
# Reset to earliest
kafka-consumer-groups --bootstrap-server local:9092 --group my-group --topic my-topic --reset-offsets --to-earliest --execute

# Reset to specific offset
--to-offset 1000

# Shift by 10 messages
--shift-by -10
```

**Production caution:** Always take backup, reset during maintenance window, test on dev first.

### **Q8. Your microservice is consuming a Kafka topic, but suddenly lags behind by millions of messages. How do you diagnose?**

**Step-by-step diagnosis:**

**1. Check consumer lag**
```bash
kafka-consumer-groups --bootstrap-server broker:9092 --group my-group --describe
```
Lag column dekho (current offset - committed offset)

**2. Identify bottleneck**
- **CPU/Memory high?** → `top`, `htop`
- **Database slow?** → consumer downstream DB pe queries slow ho rahi
- **Network issue?** → `netstat`, `ping`, `traceroute`
- **GC pauses?** → enable GC logs

**3. Increase parallelism temporarily**
- Add more consumers (if partitions available)
- Or increase partitions (needs planning)

**4. Check broker side**
- Broker disk full? → retention not working
- Leader partition unbalanced?

**5. Fix actions**
- Scale consumers horizontally
- Optimize consumer processing (batch, async, parallel DB writes)
- Temp increase `fetch.max.bytes` and `max.poll.records`

**Real example:**
> *"Ek baar humara consumer PostgreSQL me insert kar raha tha without batch. 10 msg/sec pe fine tha, suddenly 10k msg/sec aaye → lag 2 million ho gaya. Humne batch inserts + async commit lagaya, lag zero ho gaya."*

### **Bonus: Quick memory trick for interview**

| Problem | Likely Cause | Fix |
|---------|-------------|-----|
| Duplicate processing | Offset commit before processing | Manual commit after success |
| Missing messages | acks=0 or commit too early | acks=all + idempotent |
| Consumer crash | Auto commit + crash | Disable auto commit |
| High lag | Slow consumer | More partitions + batch |
| Idle consumers | Consumers > partitions | Reduce consumers |
