## Java Collections Framework

The Java Collections Framework (JCF) is a unified architecture for storing and manipulating groups of objects. Here's a complete overview:

---

## Architecture Overview

```
java.util.Collection (interface)
├── List
│   ├── ArrayList
│   ├── LinkedList
│   └── Vector → Stack
├── Set
│   ├── HashSet
│   ├── LinkedHashSet
│   └── SortedSet → TreeSet
└── Queue
    ├── PriorityQueue
    ├── ArrayDeque
    └── Deque → LinkedList

java.util.Map (interface) — separate hierarchy
├── HashMap
├── LinkedHashMap
├── Hashtable
└── SortedMap → TreeMap
```

---

## 1. List — Ordered, allows duplicates

### ArrayList
- Backed by a dynamic array. Fast random access O(1), slow insert/delete in middle O(n).

```java
List<String> list = new ArrayList<>();
list.add("Apple");
list.add("Banana");
list.add(1, "Mango");       // insert at index
list.get(0);                // "Apple"
list.remove("Banana");
list.size();                // 2
Collections.sort(list);
```

### LinkedList
- Doubly linked list. Fast insert/delete O(1) at ends, slow random access O(n). Also implements `Deque`.

```java
LinkedList<String> ll = new LinkedList<>();
ll.addFirst("A");
ll.addLast("B");
ll.peekFirst();   // "A"
ll.pollLast();    // removes "B"
```

### Key differences

| Feature | ArrayList | LinkedList |
|---|---|---|
| Access | O(1) | O(n) |
| Insert/Delete (middle) | O(n) | O(1) |
| Memory | Less | More (node pointers) |
| Use case | Read-heavy | Insert/delete-heavy |

---

## 2. Set — No duplicates

### HashSet
- Backed by a HashMap. O(1) add/remove/contains. **No ordering guarantee**.

```java
Set<String> set = new HashSet<>();
set.add("Apple");
set.add("Apple"); // ignored — duplicate
set.contains("Apple"); // true
set.size();            // 1
```

### LinkedHashSet
- Maintains **insertion order**. Slightly slower than HashSet.

```java
Set<String> lhs = new LinkedHashSet<>();
lhs.add("Banana");
lhs.add("Apple");
// Iterates as: Banana, Apple
```

### TreeSet
- Backed by a Red-Black tree. Elements stored in **sorted order**. O(log n) operations.

```java
TreeSet<Integer> ts = new TreeSet<>();
ts.add(5); ts.add(1); ts.add(3);
ts.first();          // 1
ts.last();           // 5
ts.headSet(3);       // [1]
ts.tailSet(3);       // [3, 5]
```

---

## 3. Queue & Deque — FIFO / Double-ended

### PriorityQueue
- Elements dequeued by **natural order or comparator** (min-heap by default).

```java
PriorityQueue<Integer> pq = new PriorityQueue<>();
pq.offer(30); pq.offer(10); pq.offer(20);
pq.poll(); // 10 (smallest first)
pq.peek(); // 20 (next without removing)
```

### ArrayDeque
- Double-ended queue. Faster than `Stack` and `LinkedList` for stack/queue use.

```java
ArrayDeque<String> dq = new ArrayDeque<>();
dq.push("A");      // add to front (stack push)
dq.offerLast("B"); // add to back
dq.pop();          // remove from front
dq.pollLast();     // remove from back
```

---

## 4. Map — Key-Value pairs (not a Collection)

### HashMap
- O(1) average for get/put. **No order**. Allows one `null` key.

```java
Map<String, Integer> map = new HashMap<>();
map.put("Alice", 90);
map.put("Bob", 85);
map.get("Alice");           // 90
map.containsKey("Bob");     // true
map.getOrDefault("X", 0);  // 0
map.putIfAbsent("Alice", 99); // won't overwrite

// Iterating
for (Map.Entry<String, Integer> e : map.entrySet()) {
    System.out.println(e.getKey() + " → " + e.getValue());
}
```

### LinkedHashMap
- Maintains **insertion order** (or access order if configured).

```java
Map<String, Integer> lhm = new LinkedHashMap<>();
lhm.put("B", 2); lhm.put("A", 1);
// Iterates: B, A (insertion order)
```

### TreeMap
- Sorted by keys in natural/comparator order. O(log n) operations.

```java
TreeMap<String, Integer> tm = new TreeMap<>();
tm.put("Zebra", 1); tm.put("Apple", 2);
tm.firstKey();  // "Apple"
tm.lastKey();   // "Zebra"
tm.subMap("A", "M"); // keys from A to M
```

### Hashtable
- Legacy, synchronized version of HashMap. Prefer `ConcurrentHashMap` in modern code.

## 6. Thread-Safe Collections (java.util.concurrent)

| Class | Description |
|---|---|
| `ConcurrentHashMap` | Thread-safe HashMap, better than Hashtable |
| `CopyOnWriteArrayList` | Thread-safe ArrayList for read-heavy workloads |
| `CopyOnWriteArraySet` | Thread-safe Set |
| `BlockingQueue` (interface) | Thread-safe queue with blocking operations |
| `ArrayBlockingQueue` | Bounded blocking queue |
| `LinkedBlockingQueue` | Optionally bounded blocking queue |

```java
Map<String, Integer> safe = new ConcurrentHashMap<>();
safe.put("A", 1); // safe across threads
```

## 7. Choosing the Right Collection

| Need | Use |
|---|---|
| Ordered list, fast reads | `ArrayList` |
| Frequent insert/delete | `LinkedList` |
| Unique elements, fast lookup | `HashSet` |
| Unique + insertion order | `LinkedHashSet` |
| Unique + sorted | `TreeSet` |
| Key-value, fast lookup | `HashMap` |
| Key-value + insertion order | `LinkedHashMap` |
| Key-value + sorted keys | `TreeMap` |
| Priority-based processing | `PriorityQueue` |
| Stack or Deque | `ArrayDeque` |
| Thread-safe map | `ConcurrentHashMap` |

### Q1. How does HashMap work internally?

HashMap ke andar ek array hota hai (bucket array). Har bucket me `Node<K,V>` object rehta hai.  
`put(key, value)` karte time:
1. Key ka `hashCode()` nikalte hain.
2. Us hash ko array ke size ke hisaab se index me convert karte hain (`index = (n-1) & hash`).
3. Agar bucket khali hai to wahan node daal dete hain.
4. Agar bucket me already koi node hai (collision), to wahan linked list ya tree me traverse karte hain aur `equals()` se check karte hain ki key already exist karti hai ya nahi.
5. `get(key)` bhi same index calculate karta hai aur wahan se dhundta hai.

**Java 8+ me change:**  
Jab ek bucket me 8 se zyada nodes aa jaye aur array size 64+ ho, to linked list Red-Black tree me convert ho jati hai (O(n) → O(log n)). Agar tree me nodes 6 se kam ho jaye to wapas list.

### Q2. Can we use Custom Object as key in HashMap?

**Haan, but condition ke saath.**  
- Tumhe `hashCode()` aur `equals()` method override karne padenge.
- Agar `a.equals(b)` true hai, to `a.hashCode() == b.hashCode()` bhi true hona chahiye.
- **Key immutable hona chahiye** (ya effectively immutable) — warna key insert karne ke baad agar uski property change kar di to `get()` kaam nahi karega.

Example:
```java
class Student {
    private final int rollNo;  // final = immutable
    private final String name;
    
    @Override
    public boolean equals(Object o) { ... }
    
    @Override
    public int hashCode() {
        return Objects.hash(rollNo, name);
    }
}
```

### Q3. What is HashMap in Java?

HashMap ek `Map` interface ka implementation hai jo key-value pairs store karta hai.  
- Ek `null` key allow hai  
- Multiple `null` values allow hain  
- Order guaranteed nahi hai  
- Thread-safe nahi hai  
- Average case me `put`/`get` O(1)  

---

### Q4. Why is String considered a good key for HashMap?

- **Immutable** → hash code ek baar calculate hota hai, cache ho jata hai, badalta nahi.
- `hashCode()` aur `equals()` already properly override hain.
- String pool ki vajah se memory bachti hai.
- Natural use case — user ID, email, etc.

---

### Q5. Is HashMap thread-safe in Java?

**Bilkul nahi.**  
Agar multiple threads ek saath `put`, `remove`, ya `get` kare bina synchronization ke, to problems ho sakti hain:
- Infinite loop (old Java versions me)
- Lost updates
- `ConcurrentModificationException` (agar ek thread iterate kar raha hai aur doosra modify kar raha hai)

**Solution:**  
- `Collections.synchronizedMap(new HashMap<>())` — locking heavy hai  
- `ConcurrentHashMap` — **best** (bucket-level locking, Java 8 me CAS + synchronized)

### Q6. Explain HashMap before and after Java 8 changes

| Feature | Java 7 aur pehle | Java 8+ |
|---------|----------------|---------|
| Collision handling | Sirf linked list | Linked list + Red-Black tree |
| Worst-case get | O(n) | O(log n) |
| Tree threshold | N/A | 8 (list to tree), 6 (tree to list) |
| Insertion position | Head insertion (kuch versions me) | Tail insertion |
| Node name | `Entry` | `Node` + `TreeNode` |

**Kyun change kiya?**  
DoS attack se bachne ke liye (jab koi intentionally same hashcode wale bahut saare keys daal de) aur performance improve karne ke liye.

### Q7. Explain equals() and hashCode() with example

**Contract:**  
- Agar `a.equals(b)` true → `a.hashCode() == b.hashCode()` must be true.
- Agar `a.hashCode() == b.hashCode()` hai to zaroori nahi ki `a.equals(b)` true ho (collision ho sakta hai).
- `hashCode()` consistent hona chahiye — same object ke liye same value return kare, jab tak object modify na ho.

**Example:**
```java
class Person {
    String aadhar;
    
    @Override
    public boolean equals(Object obj) {
        if (this == obj) return true;
        if (obj == null) return false;
        Person p = (Person) obj;
        return this.aadhar.equals(p.aadhar);
    }
    
    @Override
    public int hashCode() {
        return aadhar.hashCode();
    }
}
```
Yahan aadhar unique hai, to equals aur hashCode dono aadhar pe based hain.

### Q8. How to sort a HashMap in Java?

HashMap khud sort nahi hota. Tum:
1. **Keys ke hisaab se sort karna hai** → `TreeMap` use karo ya `LinkedHashMap` with sorting logic.
2. **Values ke hisaab se sort karna hai** → entries ko list me daalo, `Collections.sort()` karo, fir `LinkedHashMap` me daal do.

Example (values ke hisaab se):
```java
Map<String, Integer> map = new HashMap<>();
List<Map.Entry<String, Integer>> list = new ArrayList<>(map.entrySet());
list.sort(Map.Entry.comparingByValue());
Map<String, Integer> sorted = new LinkedHashMap<>();
for (var entry : list) sorted.put(entry.getKey(), entry.getValue());
```

### Q9. Explain internal working of a HashMap

Yeh Q1 aur Q6 ka combination hai.  
Short me:  
- Array of `Node<K,V>`  
- Hashing → index calculation  
- Collision → chaining (linked list / tree)  
- Load factor = 0.75  
- Resizing (2x) when threshold crossed  
- Java 8+ tree conversion for performance

### Q10. How will you retrieve the Value object if two Keys have the same hashcode?

Same hashcode ka matlab **collision** — dono keys ek hi bucket me jayenge.

**Retrieval process:**  
1. Bucket me jakar linked list (ya tree) traverse karte hain.
2. Har node pe `equals()` method call karte hain to find exact key.
3. Jis node ka key `equals()` se match ho jaye, uska value return kar dete hain.

Interview me bolna: *"hashCode() bucket batata hai, equals() exact key identify karta hai."*

### Q11. Fail-fast vs Fail-safe iterators

| Feature | Fail-fast | Fail-safe |
|---------|-----------|-----------|
| Example | HashMap, ArrayList iterator | ConcurrentHashMap, CopyOnWriteArrayList |
| Behavior | Jab modification detect kare, throw `ConcurrentModificationException` | Modification detect kare to exception nahi, original collection ki snapshot pe iterate karta hai |
| Performance | Better | Slightly slower (copy ya separate traversal) |
| Memory | Kam | Zyada (snapshot) |

**SDE-2 level:** Bolna ki fail-fast `modCount` variable check karta hai. Fail-safe concurrent collections me hota hai.

### Q12. What is Stack class in Java?

`Stack<E>` ek legacy class hai jo Vector ko extend karti hai.  
LIFO (Last In First Out) data structure.  
Methods: `push()`, `pop()`, `peek()`, `empty()`, `search()`.

**But** — Stack class ko use mat karo (legacy, poorly designed). Instead use `Deque<E> stack = new ArrayDeque<>();`

---

### Q13. What is Vector class in Java?

Vector bhi legacy class hai — synchronized, thread-safe.  
ArrayList jaisa hai, but har method synchronized hai.  
**Performance slow** because of synchronization overhead.  
Use nahi karte aaj kal — instead use `ArrayList` with `Collections.synchronizedList()` ya `CopyOnWriteArrayList`.

---

### Q14. What is WeakHashMap in Java?

`WeakHashMap<K,V>` me keys **weak references** hote hain.  
Matlab — agar key ka koi strong reference nahi bacha, to garbage collector us key-value pair ko automatically remove kar sakta hai.  
**Use case:** caching — jahan aap nahi chahte ki cache memory ko unnecessarily hold kare.

Example: `WeakHashMap<Student, Integer> marks` — agar Student object kisi aur jagah use nahi ho raha, to cache se auto remove ho jayega.

---

### Q15. What is IdentityHashMap in Java?

`IdentityHashMap` me **reference equality** use hoti hai (`==`), `equals()` nahi.  
Matlab — do alag objects same content ke hain to bhi alag treat honge.  
Hash code bhi `System.identityHashCode()` use karta hai.

**Use case:** Serialization, framework internals, jahan aapko object identity se compare karna ho, content se nahi.

### Q16. How ConcurrentHashMap works before & after Java 8?

| Aspect | Java 7 (before 8) | Java 8+ |
|--------|-------------------|---------|
| Locking | Segment locking (16 segments, each segment lock) | Bucket-level locking (CAS + synchronized on first node) |
| Concurrency level | Fixed (default 16) | Dynamic |
| Read performance | Very good (no lock) | Even better |
| Write performance | Segment lock | Fine-grained CAS |
| Collision | Linked list per segment | Linked list + Tree per bucket |
| Size calculation | Tries lock, then sums | Uses `baseCount` + `CounterCell` (JDK 8 improvement) |

**Java 8 me kyun change kiya?**  
Better concurrency, less memory overhead, no fixed concurrency level.

---

### Q17. What is ConcurrentHashMap?

`ConcurrentHashMap` thread-safe map hai jo high concurrency ke liye designed hai.  
- Multiple threads ek saath read kar sakte hain (almost no locking)  
- Write operations pe sirf specific bucket lock hota hai, poori map nahi  
- Null key/value allow nahi karta (HashMap se difference)  
- Iterators fail-safe hote hain  

---

### Q18. What is hash collision?

Jab do alag keys ka `hashCode()` same ho, to woh same bucket me jayenge. Is situation ko **hash collision** bolte hain.  
HashMap collision ko handle karta hai **chaining** se — ek bucket me linked list ya tree.

---

### Q19. Does HashMap allow duplicate values?

**Haan.** Duplicate values allowed hain.  
HashMap duplicate keys nahi allow karta (new value old value ko replace karegi), but different keys ke same value ho sakte hain.

---

### Q20. Can you store null keys in HashMap?

**Haan.** Ek `null` key allowed hai.  
Multiple `null` values bhi allowed hain.

---

### Q21. How does HashMap handle null keys in Java?

Null key ka hash code 0 define hota hai.  
Woh hamesha bucket index 0 me store hota hai.  
`put(null, value)` karte time special logic hota hai — `hashCode()` call nahi karta, seedha 0 index me daal deta hai.  
`get(null)` bhi same — seedha 0 index me dhundhta hai.

---

### Q22. How does get() method of HashMap work?

1. Key ka hash calculate karo.
2. Index = `(n-1) & hash` (n = array length)
3. Us bucket me jakar:
   - Pehle node check karo (hash compare)
   - Agar match ho to `equals()` check karo
   - Agar list/tree ho to traverse karo
4. Match mila to value return, nahi to `null`

---

### Q23. How many entries you can put into a HashMap?

Theoretically: `Integer.MAX_VALUE` (2^31 - 1) entries.  
Practically: Memory limit pe depend karta hai.  
Jab size threshold cross kare, resize hota hai. Agar resize ke time memory nahi hai to `OutOfMemoryError` aayega.

### Q25. Explain working of HashSet class in Java

**HashSet internally HashMap use karta hai.**  
- `HashSet<E>` ke andar ek `HashMap<E, Object>` rehta hai.
- `add(e)` → map.put(e, PRESENT) (PRESENT ek dummy object hai)
- `remove(e)` → map.remove(e)
- `contains(e)` → map.containsKey(e)
