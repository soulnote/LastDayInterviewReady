
## Q1. Between int or Integer who will consume more memory?

**Answer:** **Integer** zyada memory consume karega.

**Memory breakdown (JVM dependent, approximate):**

| Type | Memory | Reason |
|------|--------|--------|
| **int** | 4 bytes | Primitive — sirf value store hoti hai |
| **Integer** | 16-24 bytes | Object overhead (12-16 bytes header) + 4 bytes value + padding |

**Detailed calculation (64-bit JVM with compressed oops):**
- `int` → **4 bytes**
- `Integer` → 12 bytes (object header) + 4 bytes (int value) = **16 bytes** (plus reference overhead)

**SDE-2 level nuance:**
```java
// Array comparison
int[] intArray = new int[1000];        // ~4KB
Integer[] integerArray = new Integer[1000];  // ~8KB (references) + object memory
```

**Performance impact:**
- Autoboxing creates **temporary objects** → GC pressure
- Use `int` in performance-critical code, collections require `Integer`

**Interview me bolna:** *"Integer consumes 4x more memory than int due to object header overhead. For large datasets, prefer primitive arrays or libraries like Eclipse Collections for memory efficiency."*

## Q2. Symptoms of a Memory Leak

**Definition:** Objects **unreferenced** nahi hote but **useless** hote hain — GC unko collect nahi kar sakta.

**Symptoms (SDE-2 level):**

| Symptom | Kaise pehchanoge |
|---------|------------------|
| **1. Degraded performance** | Application slow ho jaye gradually, especially after each operation |
| **2. OutOfMemoryError** | `java.lang.OutOfMemoryError: Java heap space` |
| **3. High GC activity** | GC logs me frequent Full GC, but memory kam nahi ho rahi |
| **4. CPU spike** | GC continuously running (GC threads consuming CPU) |
| **5. Response time degradation** | Response time badhta jaye, restart karne se thik ho |
| **6. Unusual object retention** | Heap dump me unexpected objects zyada count me |
| **7. Memory usage graph** | Sawtooth pattern nahi, gradually increasing line |

**Example symptom in logs:**
```
[Full GC (System.gc()) 1024M->1010M(1024M), 1.5 secs]
[Full GC (System.gc()) 1024M->1012M(1024M), 1.6 secs]
[Full GC (System.gc()) 1024M->1015M(1024M), 1.7 secs]
// Memory not going down!
```

## Q3. Examples and UseCases of Memory Leaks

**Real-world examples (SDE-2 level must know):**

### 1. **Listener/Callback not unregistered**
```java
public class Button {
    private List<ClickListener> listeners = new ArrayList<>();
    
    public void addListener(ClickListener listener) {
        listeners.add(listener);  // Kabhi remove nahi kiya
    }
}
// Activity/Fragment destroyed but listener still referenced → leak
```

### 2. **Inner classes holding outer class reference**
```java
public class Outer {
    private int[] largeData = new int[1_000_000];
    
    public void createInner() {
        class Inner {
            // Implicitly holds reference to Outer
        }
        return new Inner();  // Outer can't be GC'd
    }
}
```

### 3. **HashMap/Set with mutable keys**
```java
Map<Student, String> map = new HashMap<>();
Student s = new Student(1, "John");
map.put(s, "Value");
s.setId(2);  // Hash code changed → object lost in map, never GC'd
```

### 4. **ThreadLocal misuse** (Common in web apps)
```java
ThreadLocal<byte[]> threadLocal = new ThreadLocal<>();
threadLocal.set(new byte[1024 * 1024]);  // 1MB
// Thread pool reuses threads → threadLocal persists forever!
```

### 5. **Static collections**
```java
public class Cache {
    private static List<byte[]> cache = new ArrayList<>();
    
    public void addData(byte[] data) {
        cache.add(data);  // Kabhi clear nahi kiya → grows forever
    }
}
```

### 6. **ClassLoader leaks** (In application servers)
```java
// Custom ClassLoader ke saath deployed application
// Undeploy karte time ClassLoader + all loaded classes still referenced
// PermGen/Metaspace me leak
```

## Q4. How to Detect a Memory Leak in Java?

**Tools & Techniques (SDE-2 level):**

| Step | Tool | Kaise use kare |
|------|------|----------------|
| **1. Monitor GC logs** | JVM flags | `-XX:+PrintGCDetails -XX:+PrintGCTimeStamps` |
| **2. Visualize heap** | JVisualVM, JConsole | Real-time memory graph dekho |
| **3. Heap dump analysis** | jmap, Eclipse MAT, YourKit | `jmap -dump:live,format=b,file=heap.hprof <pid>` |
| **4. Object retention** | MAT (Eclipse) | Dominator tree se dekho kaun objects hold kar raha |
| **5. Leak suspects** | MAT | "Leak Suspects" report automatic |
| **6. Compare heap dumps** | MAT | Dump1 (start) vs Dump2 (after operation) |
| **7. Profiling** | Async Profiler, JProfiler | Track allocations in real-time |

**Command line detection:**
```bash
# Get PID
jps -l

# Monitor GC (1 sec interval)
jstat -gcutil <pid> 1000

# Take heap dump
jmap -dump:live,format=b,file=leak.hprof <pid>

# Analyze with MAT (Eclipse)
# OR use jhat (deprecated)
jhat -port 7000 leak.hprof
```

**Sample GC log analysis:**
```
# Normal pattern - sawtooth
[GC 512M->256M(1024M)]  # After GC, memory drops

# Leak pattern - increasing baseline
[GC 512M->480M(1024M)]
[GC 600M->590M(1024M)]
[GC 700M->695M(1024M)]  # Baseline increasing!
```

## Q5. How Are Strings Represented in Memory?

**Java 8 vs Java 9+ (major change):**

### Java 8 (and earlier):
```java
String s = "Hello";
// Memory layout:
// - char[] (char array - 2 bytes per char)
// - int hash (hash code cache)
// - int offset (shared strings - deprecated)
// - int count (length)
```

### Java 9+ (Compact Strings):
```java
String s = "Hello";
// Memory layout:
// - byte[] value (byte array)
// - byte coder (0 = LATIN1, 1 = UTF16)
// - int hash
```

**Memory comparison:**
| String content | Java 8 | Java 9+ |
|----------------|--------|---------|
| ASCII ("Hello") | 10 bytes (char[] 2*5) | 5 bytes (byte[] 1*5) |
| Unicode ("你好") | 8 bytes (char[] 2*2) | 8 bytes (byte[] 2*2) |

**String interning:**
```java
String s1 = "Hello";           // String pool
String s2 = new String("Hello"); // Heap (separate object)

// String pool: unique strings stored, memory efficient
String s3 = s2.intern();  // Returns reference from pool
// s1 == s3 (true)
```

**String memory optimization:**
```java
// Bad - creates many objects
for (int i = 0; i < 1000; i++) {
    result += data;  // Creates new StringBuilder each time!
}

// Good
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 1000; i++) {
    sb.append(data);
}
```

## Q6. Is It Possible to Resurrect an Object That became Eligible for Garbage Collection?

**Answer:** **Yes, but only once and in a very specific way.**

**How? Using `finalize()` method (deprecated):**

```java
public class ResurrectObject {
    private static ResurrectObject savedRef = null;
    
    @Override
    protected void finalize() throws Throwable {
        super.finalize();
        savedRef = this;  // Resurrection! Object becomes reachable again
        System.out.println("Object resurrected!");
    }
    
    public static void main(String[] args) throws Exception {
        ResurrectObject obj = new ResurrectObject();
        obj = null;  // Eligible for GC
        
        System.gc();  // Request GC
        Thread.sleep(100);  // Wait for finalization
        
        if (savedRef != null) {
            System.out.println("Object is alive again!");
            savedRef = null;  // Now it can be collected
            System.gc();  // Second time - finalize() won't be called again
        }
    }
}
```

**Important limitations:**
- `finalize()` is **deprecated** (Java 9+) and **removed** in recent versions
- Resurrection happens **only once** per object
- `finalize()` runs on a **separate low-priority thread**
- **Performance nightmare** — delays GC significantly

**Modern alternative:** `Cleaner` (Java 9+) or `PhantomReference` — no resurrection possible.

**SDE-2 level point:** *"Resurrection is a relic of finalize() and is generally considered bad practice. It can cause memory leaks and unpredictable behavior. Never rely on it in production."*

## Q7. What Are the Default Garbage Collectors in Different Java Versions?

| Java Version | Default GC (Client/Server) | Parallel Threads |
|--------------|---------------------------|------------------|
| **Java 8** | Parallel GC (Throughput) | Parallel threads for young + old gen |
| **Java 9** | G1GC (Garbage First) | Region-based |
| **Java 10** | G1GC | Improved G1 |
| **Java 11** | G1GC | Stable G1 |
| **Java 12** | G1GC | Improved |
| **Java 13** | G1GC + Experimental ZGC | ZGC (low latency) experimental |
| **Java 14** | G1GC + ZGC (experimental) | ZGC production ready in 15 |
| **Java 15+** | G1GC (default), ZGC available | Both available |
| **Java 17+** | G1GC (default), ZGC, Shenandoah | Multiple low-latency options |

**How to check:**
```bash
java -XX:+PrintFlagsFinal -version | grep Use
# Look for: UseG1GC, UseParallelGC, UseSerialGC, etc.
```

**Change GC:**
```bash
# Use G1GC (default in Java 9+)
java -XX:+UseG1GC -jar app.jar

# Use Parallel GC (Java 8 default)
java -XX:+UseParallelGC -jar app.jar

# Use ZGC (low latency, Java 15+)
java -XX:+UseZGC -jar app.jar

# Use Shenandoah
java -XX:+UseShenandoahGC -jar app.jar
```

**Selection guide (SDE-2 level):**
| Requirement | GC Choice |
|-------------|-----------|
| High throughput (batch jobs) | Parallel GC |
| Low latency (web servers) | G1GC or ZGC |
| Very low latency (< 10ms) | ZGC or Shenandoah |
| Small heap (< 4GB) | Serial GC |
| Large heap (> 32GB) | ZGC or Shenandoah |

## Q8. What Are Strong, Weak, Soft, and Phantom References?

**Reference types hierarchy (SDE-2 level critical):**

| Reference Type | GC Behavior | Use Case |
|----------------|-------------|----------|
| **Strong** | Never collected (as long as reachable) | Normal objects |
| **Soft** | Collected **only when memory is low** (before OOME) | Memory-sensitive cache |
| **Weak** | Collected **as soon as GC runs** | WeakHashMap, canonical mappings |
| **Phantom** | Collected but queued after finalization | Pre-mortem cleanup |

### Detailed Explanation:

**1. Strong Reference (Default):**
```java
Object obj = new Object();  // Strong reference
obj = null;  // Now eligible for GC
// JVM will never collect as long as reference exists
```

**2. Soft Reference:**
```java
SoftReference<byte[]> softRef = new SoftReference<>(new byte[1024*1024]);
// GC will collect only if memory is critically low
// Good for caches - survive multiple GC cycles
```

**3. Weak Reference:**
```java
WeakReference<Object> weakRef = new WeakReference<>(new Object());
// GC will collect on next cycle, regardless of memory
// Used in WeakHashMap
```

**4. Phantom Reference:**
```java
ReferenceQueue<Object> queue = new ReferenceQueue<>();
PhantomReference<Object> phantomRef = new PhantomReference<>(new Object(), queue);
// get() always returns null
// Used to detect when object is GC'd (cleanup resources)
```

### Practical example with ReferenceQueue:
```java
public class ReferenceDemo {
    public static void main(String[] args) throws Exception {
        ReferenceQueue<Object> queue = new ReferenceQueue<>();
        
        // Create weak reference with queue
        WeakReference<Object> weakRef = new WeakReference<>(new Object(), queue);
        
        // Force GC
        System.gc();
        
        // Wait for enqueuing
        Reference<?> ref = queue.remove(1000);
        if (ref != null) {
            System.out.println("Object has been collected!");
        }
    }
}
```

### WeakHashMap example (real use case):
```java
WeakHashMap<Key, Value> cache = new WeakHashMap<>();
Key key = new Key("id1");
cache.put(key, new Value("data"));
key = null;  // Strong reference removed
// Next GC will remove entry from WeakHashMap automatically!
```

### SDE-2 level comparison:

| Aspect | Strong | Soft | Weak | Phantom |
|--------|--------|------|------|---------|
| Collected when? | Never directly | Memory low | Next GC | After finalization |
| `get()` returns | Object | Object (may be null) | Object (may be null) | Always null |
| Need ReferenceQueue? | No | Optional | Optional | Yes (almost always) |
| Use case | Normal objects | Cache | Metadata | Cleanup |

## Q9. What Happens in the Memory When We Use the new Keyword?

**Step-by-step memory process (SDE-2 level):**

```java
MyClass obj = new MyClass();
```

**What JVM does:**

| Step | Operation | Memory Region | What happens |
|------|-----------|---------------|--------------|
| 1 | Load class | Metaspace (Java 8+) | Check if `MyClass` is loaded, if not → load |
| 2 | Allocate memory | Heap (Eden) | Allocate space for object (header + fields) |
| 3 | Zero initialization | Heap | Set all fields to default values (0, null, false) |
| 4 | Set object header | Heap | Mark word (GC info) + klass pointer |
| 5 | Constructor call | Heap | Initialize instance variables, call constructor |
| 6 | Return reference | Stack | Store reference in local variable `obj` |

**Detailed memory layout (64-bit with compressed oops):**
```
Object in Heap:
┌─────────────────────────────┐
│  Object Header (12 bytes)   │ ← Mark word (8 bytes) + Klass pointer (4 bytes)
├─────────────────────────────┤
│  Instance fields (variable)  │
├─────────────────────────────┤
│  Padding (align to 8 bytes)  │
└─────────────────────────────┘
```

**Example with fields:**
```java
class Employee {
    int id;           // 4 bytes
    String name;      // 4 bytes (reference)
    double salary;    // 8 bytes
}
// Total: 12 (header) + 16 (fields) = 28 bytes → padded to 32 bytes
```

**SDE-2 level nuance:**
- **Escape analysis** can allocate objects on stack (Java 8+)
- **TLAB (Thread Local Allocation Buffer)** — each thread has its own Eden region for faster allocation
- **Bulk allocation** — JVM can allocate multiple objects at once

## Q10. How much memory a single class will take?

**Tricky question!** Class ka memory do parts me divide hota hai:

### Part 1: Class metadata (Metaspace - off-heap)
```java
public class MyClass {
    // Static fields
    private static int counter;      // 4 bytes (Metaspace)
    private static String config;    // Reference (Metaspace)
    
    // Methods
    public void doSomething() {}     // Method bytecode (Metaspace)
}
```

### Part 2: Object instance (Heap - per object)
```java
MyClass obj = new MyClass();  // Heap memory for this instance
```

### Exact calculation for a class:

| Component | Memory Location | Size (approx) |
|-----------|----------------|---------------|
| **Class metadata** | Metaspace | 100-200 KB (depends) |
| - Class name, superclass | Metaspace | Variable |
| - Method bytecode | Metaspace | Per method |
| - Constant pool | Metaspace | Per constant |
| - Static fields | Metaspace | Field sizes |
| - vtable (virtual methods) | Metaspace | 8 bytes per method |
| **Object instance** | Heap | Per instance |
| - Object header | Heap | 12-16 bytes |
| - Instance fields | Heap | Field sizes |

**Example calculation:**
```java
class Account {
    private static int totalAccounts;  // Metaspace: 4 bytes
    private int accountId;              // Heap: 4 bytes per instance
    private double balance;             // Heap: 8 bytes per instance
    private String owner;               // Heap: 4 bytes (reference)
}
```

**Memory per object:**
```
12 bytes (header) 
+ 4 bytes (accountId)
+ 8 bytes (balance) 
+ 4 bytes (owner reference)
= 28 bytes → padded to 32 bytes (aligned to 8)
```

**SDE-2 level answer:** 
*"Class memory do parts me hoti hai — metadata (Metaspace) jo ek baar load hoti hai (~100-200 KB), aur instance data (Heap) jo har object ke liye allocate hoti hai. Static fields Metaspace me rehte hain, instance fields heap me. Exact size JVM implementation, bitness, aur compressed oops pe depend karta hai."*

## Q11. What causes OutOfMemoryError even when memory is available?

**Paradox — memory available hai phir bhi OOM kyun? (SDE-2 level critical)**

### Reasons:

| Cause | Explanation | Example |
|-------|-------------|---------|
| **1. Fragmentation** | Heap fragmented → contiguous block nahi mil raha | Large array allocation fails despite free space |
| **2. Native memory limit** | Process memory limit cross ho gayi | `ulimit -v` or Docker memory limit |
| **3. Metaspace full** | Class metadata area full | Loading too many classes dynamically |
| **4. Direct memory full** | NIO buffers (`ByteBuffer.allocateDirect`) | Network buffers exhausted |
| **5. Swap space** | Physical RAM full, swap full | System-wide memory pressure |
| **6. GC overhead limit** | 98% time GC, but <2% memory recovered | JVM throws OOM to prevent infinite loop |
| **7. Array size limit** | Requested array size > JVM limit | `new int[Integer.MAX_VALUE]` |
| **8. Stack memory** | Thread stack allocation fails | Too many threads |

### Detailed examples:

**1. Fragmentation OOM:**
```java
// Heap has 1GB free but fragmented
byte[] array = new byte[500 * 1024 * 1024];  // 500MB contiguous needed
// OOM even though 1GB free (but not contiguous)
```

**2. Native memory OOM:**
```bash
# Docker container with 1GB limit
docker run -m 1g java -jar app.jar
# JVM heap = 512MB, but native code needs more → OOM
```

**3. GC overhead limit:**
```java
// JVM throws OOM when:
// - GC takes >98% of total time
// - And recovers <2% of heap
// JVM says: "Your app is stuck, let me throw OOM"
```

**4. Metaspace OOM:**
```java
// Dynamic class loading
while (true) {
    Class<?> clazz = generateNewClass();  // New class each iteration
    // Metaspace fills up → OOM
}
```

### Diagnostic commands:
```bash
# Check process limits
ulimit -a

# Check memory mapping
pmap -x <pid>

# Check heap fragmentation
jcmd <pid> GC.heap_info

# Check native memory
jcmd <pid> VM.native_memory summary
```

**SDE-2 level answer:**
*"OOME tab bhi aa sakta hai jab memory available ho — reasons include heap fragmentation (contiguous block nahi milna), native memory limit (Docker/ulimit), metaspace exhaustion, GC overhead limit, ya array size limit cross hona. Production me we use `-XX:+HeapDumpOnOutOfMemoryError` and analyze heap dumps to identify the real cause."*
