## Q1. What are issues without volatile keyword in Java?

**Answer:**  
Without `volatile`, threads may see **stale/cached values** of variables due to CPU caching and JVM optimizations, causing **visibility problems**.

**The Problem - Visibility Issue:**

```java
// Without volatile - Thread might never see the change!
class SharedFlag {
    private boolean running = true;  // No volatile
    
    public void stop() {
        running = false;  // Main thread changes value
    }
    
    public void run() {
        while (running) {  // Worker thread may cache running = true
            // Do work - may never see running = false!
        }
        System.out.println("Thread stopped");  // May never print!
    }
}

// Demonstration
public static void main(String[] args) throws InterruptedException {
    SharedFlag flag = new SharedFlag();
    Thread worker = new Thread(flag::run);
    worker.start();
    
    Thread.sleep(1000);
    flag.stop();  // Main thread sets running = false
    
    worker.join();  // May hang forever!
}
```

**Why This Happens:**

```
CPU Architecture:
┌─────────────────────────────────────────────────────────┐
│                      Main Memory                         │
│              running = true (initial)                    │
└─────────────────────────────────────────────────────────┘
         │                           │
         ▼                           ▼
┌─────────────────┐         ┌─────────────────┐
│   CPU Core 1    │         │   CPU Core 2    │
│  ┌───────────┐  │         │  ┌───────────┐  │
│  │  L1 Cache │  │         │  │  L1 Cache │  │
│  │running=true│  │         │  │running=true│  │
│  └───────────┘  │         │  └───────────┘  │
│  Thread: Main   │         │  Thread: Worker │
└─────────────────┘         └─────────────────┘

Main thread updates: running = false
But worker thread's cache still has running = true!
```

**Solution with volatile:**

```java
class SharedFlagWithVolatile {
    private volatile boolean running = true;  // Volatile keyword
    
    public void stop() {
        running = false;  // Write to main memory immediately
    }
    
    public void run() {
        while (running) {  // Always reads from main memory
            // Do work
        }
        System.out.println("Thread stopped");  // Will print!
    }
}
```

**Another Example - Broken Counter:**

```java
// Without volatile - Broken!
class Counter {
    private int count = 0;  // No volatile
    
    public void increment() {
        count++;  // Not atomic anyway, but visibility also issue
    }
    
    public int getCount() {
        return count;  // May return stale value
    }
}

// With volatile - Still not atomic, but visibility fixed
class CounterWithVolatile {
    private volatile int count = 0;
    
    public synchronized void increment() {  // Need both volatile + sync
        count++;
    }
    
    public int getCount() {
        return count;  // Always fresh value
    }
}
```

**Visibility Issues in Real Code:**

```java
// Real-world example: Configuration flag
class ConfigService {
    private volatile boolean configLoaded = false;  // volatile needed!
    private Config config;
    
    public void loadConfig() {
        config = loadFromDatabase();
        configLoaded = true;  // Without volatile, other threads may not see this
    }
    
    public Config getConfig() {
        while (!configLoaded) {  // Without volatile, infinite loop possible
            Thread.yield();
        }
        return config;
    }
}

// Another example: Shutdown hook
class Application {
    private volatile boolean shutdown = false;  // Must be volatile
    
    public void start() {
        Runtime.getRuntime().addShutdownHook(new Thread(() -> {
            shutdown = true;  // This needs to be visible to worker threads
        }));
        
        while (!shutdown) {
            // Process work
        }
    }
}
```

**What volatile DOES NOT guarantee:**

```java
class VolatileCounter {
    private volatile int count = 0;
    
    // ❌ This is NOT thread-safe even with volatile!
    public void increment() {
        count++;  // Still not atomic: read-modify-write
    }
    
    // ✅ Use atomic classes instead
    private AtomicInteger atomicCount = new AtomicInteger();
    
    public void safeIncrement() {
        atomicCount.incrementAndGet();  // Atomic operation
    }
}
```

**Memory Visibility Guarantees:**

| Operation | Without volatile | With volatile |
|-----------|-----------------|---------------|
| Read | May read stale value | Always reads from main memory |
| Write | May stay in cache | Immediately flushed to main memory |
| Reordering | Compiler/CPU can reorder | Prevents reordering (happens-before) |

**Interview Tip:** "Without volatile, threads may cache variable values, causing visibility issues. Volatile ensures reads/writes go to main memory immediately. Use volatile for flags, status variables. Not for compound actions (like i++). Provides happens-before guarantee: write to volatile happens before subsequent read."

## Q2. What is volatile keyword in Java?

**Answer:**  
`volatile` is a Java keyword that ensures **visibility** of changes to variables across threads and prevents **instruction reordering**.

**Key Guarantees:**

1. **Visibility:** Write to volatile variable happens-before any subsequent read
2. **Atomicity:** Only for single read/write (not compound operations)
3. **Prevents reordering:** Compiler/CPU cannot reorder volatile operations

**Basic Usage:**

```java
class VolatileExample {
    private volatile boolean flag = false;
    private volatile int counter = 0;
    private volatile ReferenceType ref = null;
    
    public void writer() {
        flag = true;      // Write to volatile - flushed to main memory
        counter = 100;    // Also visible due to happens-before
        ref = new Object(); // Reference assignment is atomic
    }
    
    public void reader() {
        while (!flag) {   // Always reads from main memory
            // wait
        }
        // Guaranteed to see counter = 100 and ref != null
        System.out.println(counter);
    }
}
```

**Happens-Before Relationship:**

```java
class HappensBeforeExample {
    private int a = 0;
    private volatile boolean flag = false;
    
    public void writer() {
        a = 42;           // Normal write
        flag = true;      // Volatile write
    }
    
    public void reader() {
        if (flag) {       // Volatile read
            // Guaranteed to see a = 42 due to happens-before!
            System.out.println(a);  // Always prints 42
        }
    }
}
```

**Preventing Instruction Reordering:**

```java
class ReorderingExample {
    private volatile boolean initialized = false;
    private Config config;
    
    // Without volatile, compiler might reorder:
    // config = new Config();  ← Could be reordered after initialized = true
    // initialized = true;     ← Other thread sees initialized true but config null!
    
    public void initialize() {
        config = new Config();     // Normal write
        initialized = true;        // Volatile write - prevents reordering
    }
    
    public void use() {
        if (initialized) {         // Volatile read
            config.use();          // Safe - config guaranteed initialized
        }
    }
}
```

**Double-Checked Locking (Singleton Pattern):**

```java
// ✅ Correct implementation with volatile
class Singleton {
    private static volatile Singleton instance;  // volatile is CRITICAL!
    
    private Singleton() {}
    
    public static Singleton getInstance() {
        if (instance == null) {           // First check (no lock)
            synchronized (Singleton.class) {
                if (instance == null) {   // Second check (with lock)
                    instance = new Singleton();  // volatile prevents reordering
                }
            }
        }
        return instance;
    }
}

// Without volatile, this can break due to reordering:
// 1. Allocate memory
// 2. Set instance reference (half-constructed object visible!)
// 3. Initialize object
// Another thread sees non-null instance but uninitialized object!
```

**When to Use volatile:**

```java
// ✅ Good use cases for volatile:

// 1. Status flags
private volatile boolean shutdown = false;

// 2. Configuration changes
private volatile Config currentConfig;

// 3. Published objects (immutable)
private volatile ImmutableData data;

// 4. Simple counters (single-threaded write, multiple readers)
private volatile int lastId;

// ❌ Bad use cases for volatile:

// 1. Compound operations
private volatile int count;
public void increment() {
    count++;  // NOT atomic - use AtomicInteger instead
}

// 2. Dependencies between volatile variables
private volatile int x, y;
public void update() {
    x = 1;  // No happens-before between x and y
    y = 2;  // Other thread may see y=2 but x=0!
}

// 3. When atomicity needed
private volatile List<String> list;  // list reference is volatile, but list content isn't
```

**Volatile vs Synchronized:**

| Feature | volatile | synchronized |
|---------|----------|--------------|
| **Visibility** | ✅ Yes | ✅ Yes |
| **Atomicity** | ❌ Only single read/write | ✅ Yes (block level) |
| **Mutual exclusion** | ❌ No | ✅ Yes |
| **Performance** | Fast (no locking) | Slower (lock overhead) |
| **Use case** | Flags, status variables | Compound operations |

**Performance Impact:**

```java
// Volatile read/write is slightly more expensive than normal
// But much cheaper than synchronized

// Approximate cost (relative):
// Normal variable: 1x
// Volatile: 2-3x
// Synchronized: 50-100x (contended)

private volatile int volatileCounter;
private int normalCounter;

public void volatileIncrement() {
    volatileCounter++;  // Not atomic, just demo
}

public synchronized void syncIncrement() {
    normalCounter++;
}
```

**Interview Tip:** "Volatile ensures visibility (no stale reads) and prevents reordering. Use for flags, status variables, published objects. Not for compound actions (i++ needs AtomicInteger). Provides happens-before guarantee: write to volatile happens before subsequent read. Essential for double-checked locking."

## Q3. Why we need multiple threads?

**Answer:**  
Multiple threads allow **parallel execution** of tasks, improving **performance**, **responsiveness**, and **resource utilization**.

**Main Reasons:**

**1. Improved Performance (Parallel Processing):**

```java
// Single-threaded - Slow for large tasks
class SingleThreadedProcessor {
    public void processLargeList(List<Data> dataList) {
        for (Data data : dataList) {
            process(data);  // Sequential - takes 10 seconds for 1M items
        }
    }
}

// Multi-threaded - Faster on multi-core CPUs
class MultiThreadedProcessor {
    public void processLargeList(List<Data> dataList) {
        int cores = Runtime.getRuntime().availableProcessors();
        ExecutorService executor = Executors.newFixedThreadPool(cores);
        
        List<CompletableFuture<Void>> futures = new ArrayList<>();
        for (Data data : dataList) {
            futures.add(CompletableFuture.runAsync(() -> process(data), executor));
        }
        CompletableFuture.allOf(futures.toArray(new CompletableFuture[0])).join();
        // With 8 cores: 10 seconds → 1.25 seconds
    }
}
```

**2. Responsive UI (Event Dispatching Thread):**

```java
// ❌ Bad - Freezes UI
button.addActionListener(e -> {
    // Long operation on EDT (Event Dispatch Thread)
    Thread.sleep(5000);  // UI freezes for 5 seconds!
    label.setText("Done");
});

// ✅ Good - Background thread
button.addActionListener(e -> {
    new Thread(() -> {
        // Long operation in background
        Thread.sleep(5000);
        
        // Update UI on EDT
        SwingUtilities.invokeLater(() -> label.setText("Done"));
    }).start();
});
```

**3. Better Resource Utilization (I/O Operations):**

```java
// Single-threaded I/O - Wastes CPU waiting
class SingleThreadedDownloader {
    public void downloadFiles(List<String> urls) {
        for (String url : urls) {
            download(url);  // CPU idle while waiting for network
        }
    }
}

// Multi-threaded I/O - Utilizes CPU better
class MultiThreadedDownloader {
    public void downloadFiles(List<String> urls) {
        ExecutorService executor = Executors.newFixedThreadPool(10);
        
        for (String url : urls) {
            executor.submit(() -> download(url));
            // Multiple downloads in parallel - better throughput
        }
    }
}
```

**4. Asynchronous Processing:**

```java
@Service
public class OrderService {
    
    // Without threads - user waits for everything
    public Order processOrderSync(OrderRequest request) {
        validateOrder(request);      // 100ms
        processPayment(request);     // 500ms
        updateInventory(request);    // 200ms
        sendConfirmation(request);   // 300ms
        // Total: 1.1 seconds - user waits
        return order;
    }
    
    // With threads - user gets immediate response
    @Async
    public CompletableFuture<Order> processOrderAsync(OrderRequest request) {
        validateOrder(request);
        
        // Parallel execution of independent tasks
        CompletableFuture<Void> payment = CompletableFuture.runAsync(() -> processPayment(request));
        CompletableFuture<Void> inventory = CompletableFuture.runAsync(() -> updateInventory(request));
        
        CompletableFuture.allOf(payment, inventory).join();
        sendConfirmation(request);  // Async, don't wait
        
        return CompletableFuture.completedFuture(order);
        // User gets response immediately, background tasks continue
    }
}
```

**5. Server Handling Multiple Clients:**

```java
// Single-threaded server - One client at a time
class SingleThreadedServer {
    public void start() {
        ServerSocket serverSocket = new ServerSocket(8080);
        while (true) {
            Socket client = serverSocket.accept();
            handleClient(client);  // Other clients wait!
        }
    }
}

// Multi-threaded server - Handles many clients
class MultiThreadedServer {
    public void start() {
        ExecutorService pool = Executors.newCachedThreadPool();
        ServerSocket serverSocket = new ServerSocket(8080);
        
        while (true) {
            Socket client = serverSocket.accept();
            pool.submit(() -> handleClient(client));  // Non-blocking
        }
    }
}
```

**6. Producer-Consumer Pattern:**

```java
class ProducerConsumerExample {
    private final BlockingQueue<Task> queue = new LinkedBlockingQueue<>();
    
    // Producer thread(s)
    public void produce() {
        new Thread(() -> {
            while (true) {
                Task task = createTask();
                queue.put(task);  // Producer continues
            }
        }).start();
    }
    
    // Consumer thread(s) - can have multiple
    public void consume() {
        new Thread(() -> {
            while (true) {
                Task task = queue.take();  // Process in background
                process(task);
            }
        }).start();
    }
}
```

**Performance Comparison:**

| Scenario | Single Thread | Multiple Threads | Improvement |
|----------|---------------|-----------------|-------------|
| CPU-intensive (8-core) | 100% of 1 core | ~800% utilization | 8x faster |
| I/O-intensive (network) | Sequential | Parallel | 10-100x faster |
| UI responsiveness | Frozen | Responsive | User experience |
| Server throughput | 1 client at a time | 1000s of clients | 1000x+ |

**When NOT to use multiple threads:**

```java
// ❌ Not beneficial for:
// 1. Simple sequential tasks
// 2. Single-core CPU (overhead > benefit)
// 3. Very small tasks (thread overhead dominates)
// 4. Highly coupled tasks with dependencies

// Example - Not worth threading:
for (int i = 0; i < 10; i++) {
    computeSomethingSmall();  // Overhead of creating threads > benefit
}

// Example - Worth threading:
for (int i = 0; i < 10000; i++) {
    computeHeavyOperation();  // Benefits from parallelization
}
```

**Interview Tip:** "Multiple threads needed for: **performance** (multi-core CPUs), **responsiveness** (non-blocking UI), **resource utilization** (I/O overlap), **throughput** (server handling many clients), **asynchronous processing** (fire-and-forget). Threads enable parallelism but add complexity (synchronization, deadlocks)."

## Q4. What are different ways to create Threads in Java?

**Answer:**  
Java provides **3 main ways** to create threads, plus modern approaches with `ExecutorService`.

**Method 1: Extending Thread Class**

```java
class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("Thread running: " + Thread.currentThread().getName());
    }
}

// Usage
MyThread t1 = new MyThread();
t1.start();  // Starts new thread

// With constructor parameters
class WorkerThread extends Thread {
    private String taskName;
    
    public WorkerThread(String taskName) {
        this.taskName = taskName;
    }
    
    @Override
    public void run() {
        System.out.println("Processing: " + taskName);
    }
}
```

**Method 2: Implementing Runnable Interface (Recommended)**

```java
class MyRunnable implements Runnable {
    @Override
    public void run() {
        System.out.println("Runnable running: " + Thread.currentThread().getName());
    }
}

// Usage
Thread t1 = new Thread(new MyRunnable());
t1.start();

// Using lambda (Java 8+)
Runnable task = () -> System.out.println("Lambda thread running");
new Thread(task).start();

// Anonymous class
new Thread(new Runnable() {
    @Override
    public void run() {
        System.out.println("Anonymous thread");
    }
}).start();
```

**Method 3: Implementing Callable Interface (Returns value)**

```java
class MyCallable implements Callable<String> {
    @Override
    public String call() throws Exception {
        Thread.sleep(1000);
        return "Result from thread";
    }
}

// Usage with ExecutorService
ExecutorService executor = Executors.newSingleThreadExecutor();
Future<String> future = executor.submit(new MyCallable());
String result = future.get();  // Blocks until result available
System.out.println(result);
executor.shutdown();

// Lambda version
Callable<String> task = () -> {
    Thread.sleep(1000);
    return "Hello";
};
```

**Method 4: Using ExecutorService (Modern Approach)**

```java
// Fixed thread pool
ExecutorService fixedPool = Executors.newFixedThreadPool(10);
fixedPool.submit(() -> System.out.println("Task 1"));
fixedPool.submit(() -> System.out.println("Task 2"));
fixedPool.shutdown();

// Cached thread pool (creates threads as needed)
ExecutorService cachedPool = Executors.newCachedThreadPool();
cachedPool.submit(() -> heavyTask());

// Single thread executor
ExecutorService singleExecutor = Executors.newSingleThreadExecutor();
singleExecutor.submit(() -> sequentialTask());

// Scheduled thread pool
ScheduledExecutorService scheduledPool = Executors.newScheduledThreadPool(5);
scheduledPool.schedule(() -> task(), 5, TimeUnit.SECONDS);
scheduledPool.scheduleAtFixedRate(() -> periodicTask(), 0, 10, TimeUnit.SECONDS);
```

**Method 5: Using CompletableFuture (Modern Async)**

```java
// Run async task
CompletableFuture<Void> future = CompletableFuture.runAsync(() -> {
    System.out.println("Async task");
});

// With return value
CompletableFuture<String> result = CompletableFuture.supplyAsync(() -> {
    return "Result";
});

// Chaining
CompletableFuture.supplyAsync(() -> "Hello")
    .thenApply(String::toUpperCase)
    .thenAccept(System.out::println);
```

**Comparison of Approaches:**

| Approach | Returns Value | Throws Exception | Thread Reuse | Recommended |
|----------|---------------|------------------|--------------|-------------|
| **Thread** | No | No | No | ❌ Legacy |
| **Runnable** | No | No | Yes | ✅ Basic |
| **Callable** | Yes | Yes | Yes | ✅ With result |
| **ExecutorService** | Yes (Future) | Yes | Yes | ✅ Production |
| **CompletableFuture** | Yes | Yes | Yes | ✅ Modern |

**Complete Example - Real World:**

```java
public class ThreadCreationDemo {
    
    // 1. Extending Thread (avoid - can't extend other classes)
    static class DownloadThread extends Thread {
        private String url;
        
        DownloadThread(String url) {
            this.url = url;
        }
        
        @Override
        public void run() {
            download(url);
        }
    }
    
    // 2. Implementing Runnable (preferred)
    static class DownloadRunnable implements Runnable {
        private String url;
        
        DownloadRunnable(String url) {
            this.url = url;
        }
        
        @Override
        public void run() {
            download(url);
        }
    }
    
    // 3. Callable with return value
    static class DownloadCallable implements Callable<String> {
        private String url;
        
        DownloadCallable(String url) {
            this.url = url;
        }
        
        @Override
        public String call() throws Exception {
            return downloadAndGetContent(url);
        }
    }
    
    // Modern approach with ExecutorService
    public static void main(String[] args) {
        // Method 1: Extending Thread
        new DownloadThread("file1.zip").start();
        
        // Method 2: Runnable
        new Thread(new DownloadRunnable("file2.zip")).start();
        
        // Method 3: Lambda Runnable
        new Thread(() -> download("file3.zip")).start();
        
        // Method 4: ExecutorService
        ExecutorService executor = Executors.newFixedThreadPool(5);
        executor.submit(() -> download("file4.zip"));
        
        // Method 5: Callable with Future
        Future<String> future = executor.submit(new DownloadCallable("file5.zip"));
        try {
            String content = future.get();  // Wait for result
            System.out.println("Downloaded: " + content);
        } catch (Exception e) {
            e.printStackTrace();
        }
        
        executor.shutdown();
    }
    
    private static void download(String url) {
        System.out.println("Downloading: " + url);
    }
    
    private static String downloadAndGetContent(String url) {
        return "Content of " + url;
    }
}
```

**Best Practices:**

```java
// ✅ DO: Prefer Runnable over Thread (more flexible)
public class GoodPractice implements Runnable {
    @Override
    public void run() { /* work */ }
}

// ✅ DO: Use ExecutorService for production code
ExecutorService executor = Executors.newFixedThreadPool(10);

// ✅ DO: Use try-finally to shutdown
ExecutorService executor = Executors.newFixedThreadPool(10);
try {
    executor.submit(task);
} finally {
    executor.shutdown();
}

// ❌ DON'T: Start thread in constructor (may expose partially constructed object)
class BadPractice extends Thread {
    BadPractice() {
        this.start();  // Dangerous!
    }
}

// ❌ DON'T: Use Thread.stop(), suspend(), resume() (deprecated, unsafe)
```

**Interview Tip:** "3 main ways: **Thread** (extend), **Runnable** (implement), **Callable** (with return). Runnable preferred over Thread (flexibility). For production: **ExecutorService** for thread pooling, **CompletableFuture** for async pipelines. Never call `run()` directly (call `start()`). Use lambda for concise Runnable."

## Q5. List all thread methods in Java?

**Answer:**  
Thread class provides methods for **controlling thread execution, state management, and synchronization**.

**Thread Lifecycle Methods:**

```java
// 1. start() - Begins thread execution (calls run() in new thread)
Thread t = new Thread(() -> System.out.println("Running"));
t.start();  // Asynchronous

// 2. run() - Contains code to execute (called by start(), never call directly)
@Override
public void run() {
    // Thread logic here
}

// 3. sleep() - Pauses current thread for specified time
try {
    Thread.sleep(1000);  // Pause for 1 second
    Thread.sleep(1000, 500000);  // 1 second + 500,000 nanoseconds
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();
}
```

**Thread Control Methods:**

```java
// 4. join() - Waits for thread to die
Thread t = new Thread(() -> {
    Thread.sleep(2000);
});
t.start();
t.join();  // Current thread waits for t to finish
t.join(1000);  // Waits max 1 second

// 5. interrupt() - Interrupts a waiting/sleeping thread
Thread t = new Thread(() -> {
    try {
        Thread.sleep(10000);
    } catch (InterruptedException e) {
        System.out.println("Interrupted!");
    }
});
t.start();
t.interrupt();  // Causes InterruptedException

// 6. isInterrupted() - Checks if thread was interrupted
if (t.isInterrupted()) {
    System.out.println("Thread was interrupted");
}

// 7. interrupted() - Static method, checks and clears interrupt status
if (Thread.interrupted()) {
    System.out.println("Current thread interrupted, status cleared");
}
```

**Thread State Methods:**

```java
// 8. getState() - Returns current thread state
Thread.State state = t.getState();
// Possible states: NEW, RUNNABLE, BLOCKED, WAITING, TIMED_WAITING, TERMINATED

// 9. isAlive() - Checks if thread is alive (not terminated)
if (t.isAlive()) {
    System.out.println("Thread still running");
}

// 10. isDaemon() - Checks if thread is daemon
boolean isDaemon = t.isDaemon();

// 11. setDaemon() - Marks thread as daemon (must be called before start)
t.setDaemon(true);  // JVM exits when only daemon threads remain
```

**Priority Methods:**

```java
// 12. getPriority() - Returns thread priority (1-10)
int priority = t.getPriority();

// 13. setPriority() - Sets thread priority
t.setPriority(Thread.MAX_PRIORITY);   // 10
t.setPriority(Thread.NORM_PRIORITY);  // 5 (default)
t.setPriority(Thread.MIN_PRIORITY);   // 1
```

**Static Utility Methods:**

```java
// 14. currentThread() - Returns currently executing thread
Thread current = Thread.currentThread();

// 15. yield() - Hints scheduler to let other threads run
Thread.yield();  // Not guaranteed, just hint

// 16. dumpStack() - Prints stack trace
Thread.dumpStack();  // Useful for debugging

// 17. holdsLock() - Checks if current thread holds lock
Object lock = new Object();
synchronized (lock) {
    boolean holds = Thread.holdsLock(lock);  // true
}

// 18. getAllStackTraces() - Gets stack traces of all live threads
Map<Thread, StackTraceElement[]> traces = Thread.getAllStackTraces();
```

**Deprecated (Don't Use):**

```java
// ❌ DON'T USE - Unsafe and deprecated
t.stop();        // Causes immediate termination (unsafe)
t.suspend();     // Suspends thread (deadlock risk)
t.resume();      // Resumes suspended thread
```

**Complete Example:**

```java
public class ThreadMethodsDemo {
    public static void main(String[] args) throws InterruptedException {
        // Create thread
        Thread worker = new Thread(() -> {
            System.out.println("Worker started");
            
            // Check interrupt status
            while (!Thread.currentThread().isInterrupted()) {
                try {
                    Thread.sleep(100);
                    System.out.println("Working...");
                } catch (InterruptedException e) {
                    System.out.println("Got interrupted!");
                    Thread.currentThread().interrupt();  // Restore status
                    break;
                }
            }
            
            System.out.println("Worker finished");
        });
        
        // Configure thread
        worker.setName("Worker-1");
        worker.setPriority(Thread.NORM_PRIORITY);
        worker.setDaemon(false);
        
        // Check states
        System.out.println("State before start: " + worker.getState());  // NEW
        
        // Start thread
        worker.start();
        System.out.println("State after start: " + worker.getState());   // RUNNABLE
        
        // Main thread yields
        Thread.yield();
        
        // Wait for a bit
        Thread.sleep(500);
        
        // Interrupt worker
        worker.interrupt();
        
        // Wait for completion with timeout
        worker.join(2000);
        
        System.out.println("Is alive: " + worker.isAlive());     // false
        System.out.println("Final state: " + worker.getState()); // TERMINATED
        
        // Check if current thread holds lock
        Object lock = new Object();
        synchronized (lock) {
            System.out.println("Holds lock: " + Thread.holdsLock(lock));  // true
        }
    }
}
```

**Thread States and Transitions:**

```
                    ┌─────────────────────────────────────────┐
                    │                                         │
                    ▼                                         │
    ┌──────┐    start()    ┌──────────┐    scheduler   ┌───────────┐
    │ NEW  │──────────────►│ RUNNABLE │◄──────────────►│ RUNNING   │
    └──────┘                └──────────┘                └───────────┘
                                │                             │
                                │                             │
                    wait/join/   │                             │ sleep/join/
                    park         │                             │ wait(timeout)
                                ▼                             ▼
                          ┌──────────┐                   ┌───────────────┐
                          │ WAITING  │                   │ TIMED_WAITING │
                          └──────────┘                   └───────────────┘
                                │                             │
                                │ notify/notifyAll/          │ timeout/
                                │ interrupt                   │ interrupt
                                ▼                             ▼
                          ┌──────────┐      finish       ┌────────────┐
                          │  BLOCKED │──────────────────►│ TERMINATED │
                          └──────────┘                    └────────────┘
```

**Interview Tip:** "Key methods: `start()` (begin execution), `join()` (wait for completion), `sleep()` (pause), `interrupt()` (signal interruption), `yield()` (hint to scheduler). Check state with `getState()`. Never use `stop()`, `suspend()`, `resume()` (deprecated). Use `Thread.currentThread()` to get current thread reference."

## Q6. What is ThreadLocal class in Java?

**Answer:**  
`ThreadLocal` provides **thread-local variables** - each thread has its own independent copy of the variable.

**Basic Concept:**

```java
class ThreadLocalExample {
    // Each thread gets its own copy
    private static ThreadLocal<Integer> threadLocal = ThreadLocal.withInitial(() -> 0);
    
    public static void main(String[] args) {
        Runnable task = () -> {
            int value = threadLocal.get();
            threadLocal.set(value + 1);
            System.out.println(Thread.currentThread().getName() + ": " + threadLocal.get());
        };
        
        Thread t1 = new Thread(task, "Thread-1");
        Thread t2 = new Thread(task, "Thread-2");
        
        t1.start();  // Thread-1: 1
        t2.start();  // Thread-2: 1 (separate copy!)
    }
}
```

**Real-world Example - Transaction Management:**

```java
public class TransactionManager {
    private static final ThreadLocal<Transaction> currentTx = new ThreadLocal<>();
    
    public static void beginTransaction() {
        if (currentTx.get() != null) {
            throw new IllegalStateException("Transaction already active");
        }
        currentTx.set(new Transaction());
    }
    
    public static void commit() {
        Transaction tx = currentTx.get();
        if (tx == null) {
            throw new IllegalStateException("No active transaction");
        }
        tx.commit();
        currentTx.remove();
    }
    
    public static void rollback() {
        Transaction tx = currentTx.get();
        if (tx != null) {
            tx.rollback();
            currentTx.remove();
        }
    }
    
    public static Transaction getCurrentTransaction() {
        return currentTx.get();
    }
}

// Usage
@Transactional
public void businessMethod() {
    TransactionManager.beginTransaction();
    try {
        // Multiple DAO calls use same transaction via ThreadLocal
        userDao.update();
        orderDao.create();
        TransactionManager.commit();
    } catch (Exception e) {
        TransactionManager.rollback();
        throw e;
    }
}
```

**Interview Tip:** "ThreadLocal = each thread has independent variable copy. Used for: user context propagation (without passing parameter), thread-safe formatters, transaction management, request tracking. **CRITICAL**: Always call `remove()` in finally block to prevent memory leaks in thread-pooled environments (web servers)."

## Q7. Explain differences between synchronized keyword, Lock, and Semaphore?

**Answer:**

| Feature | synchronized | Lock (ReentrantLock) | Semaphore |
|---------|--------------|---------------------|-----------|
| **Type** | Keyword | Interface | Counting semaphore |
| **Lock acquisition** | Automatic | Manual (lock()/unlock()) | Manual (acquire()/release()) |
| **Lock release** | Automatic (on block exit) | Manual (finally block) | Manual (finally block) |
| **Try lock** | ❌ No | ✅ tryLock() | ✅ tryAcquire() |
| **Timeout** | ❌ No | ✅ tryLock(timeout) | ✅ tryAcquire(timeout) |
| **Fairness** | ❌ No | ✅ Can be fair | ✅ Can be fair |
| **Interruptible** | ❌ No | ✅ lockInterruptibly() | ✅ acquireInterruptibly() |
| **Multiple permits** | ❌ No (1 thread) | ❌ No (1 thread) | ✅ Yes (N permits) |
| **Performance** | Fast (biased locking) | Slightly slower | Depends on permits |

**1. synchronized Keyword:**

```java
// Method synchronization
public synchronized void method() {
    // Only one thread at a time
}

// Block synchronization
public void method() {
    synchronized(this) {  // Lock on current object
        // Critical section
    }
}

// Static synchronization
public static synchronized void staticMethod() {
    // Lock on Class object
}

// Example
class Counter {
    private int count = 0;
    
    public synchronized void increment() {  // Method level
        count++;
    }
    
    public void decrement() {
        synchronized(this) {  // Block level
            count--;
        }
    }
}
```

**2. Lock Interface (ReentrantLock):**

```java
class LockExample {
    private final Lock lock = new ReentrantLock();
    private int count = 0;
    
    // Basic usage
    public void increment() {
        lock.lock();  // Acquire lock
        try {
            count++;
        } finally {
            lock.unlock();  // Always unlock in finally!
        }
    }
    
    // Try lock (non-blocking)
    public boolean tryIncrement() {
        if (lock.tryLock()) {  // Don't block if not available
            try {
                count++;
                return true;
            } finally {
                lock.unlock();
            }
        }
        return false;
    }
    
    // Try lock with timeout
    public boolean tryIncrementWithTimeout() throws InterruptedException {
        if (lock.tryLock(1, TimeUnit.SECONDS)) {
            try {
                count++;
                return true;
            } finally {
                lock.unlock();
            }
        }
        return false;
    }
    
    // Interruptible lock
    public void interruptibleIncrement() throws InterruptedException {
        lock.lockInterruptibly();  // Can be interrupted while waiting
        try {
            count++;
        } finally {
            lock.unlock();
        }
    }
    
    // Fair lock (first-come-first-served)
    private final Lock fairLock = new ReentrantLock(true);  // Fair
    
    // Check lock status
    public void checkLock() {
        boolean isLocked = lock.tryLock();  // Check if available
        if (isLocked) lock.unlock();
        
        boolean isHeld = ((ReentrantLock) lock).isHeldByCurrentThread();
        int queueLength = ((ReentrantLock) lock).getQueueLength();
    }
}
```

**3. Semaphore (Counting Semaphore):**

```java
class SemaphoreExample {
    // Binary semaphore (acts like lock)
    private final Semaphore binarySemaphore = new Semaphore(1);  // 1 permit
    
    // Counting semaphore - limit concurrent access
    private final Semaphore connectionPool = new Semaphore(5);  // 5 permits
    
    // Fair semaphore
    private final Semaphore fairSemaphore = new Semaphore(3, true);  // Fair
    
    // Binary semaphore usage (similar to lock)
    public void binarySemaphoreExample() {
        binarySemaphore.acquire();  // Get permit (block if not available)
        try {
            // Critical section
        } finally {
            binarySemaphore.release();  // Return permit
        }
    }
    
    // Connection pool - limit to 5 concurrent connections
    public void useConnection() {
        try {
            connectionPool.acquire();  // Wait for available permit
            // Use database connection
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        } finally {
            connectionPool.release();  // Return permit
        }
    }
    
    // Non-blocking acquire
    public boolean tryAcquire() {
        if (connectionPool.tryAcquire()) {
            try {
                // Use connection
                return true;
            } finally {
                connectionPool.release();
            }
        }
        return false;
    }
    
    // Acquire multiple permits
    public void acquireMultiple() throws InterruptedException {
        semaphore.acquire(3);  // Acquire 3 permits
        try {
            // Use resources
        } finally {
            semaphore.release(3);
        }
    }
    
    // Available permits
    public int availablePermits() {
        return connectionPool.availablePermits();
    }
}
```

```java
// 4. Real-world - Producer-Consumer with Semaphore
class BoundedBuffer<T> {
    private final Semaphore emptySlots;
    private final Semaphore fullSlots;
    private final Lock lock = new ReentrantLock();
    private final Queue<T> buffer = new LinkedList<>();
    
    public BoundedBuffer(int capacity) {
        this.emptySlots = new Semaphore(capacity);
        this.fullSlots = new Semaphore(0);
    }
    
    public void put(T item) throws InterruptedException {
        emptySlots.acquire();  // Wait for empty slot
        lock.lock();
        try {
            buffer.add(item);
        } finally {
            lock.unlock();
        }
        fullSlots.release();  // Signal item available
    }
    
    public T take() throws InterruptedException {
        fullSlots.acquire();  // Wait for item
        lock.lock();
        T item;
        try {
            item = buffer.poll();
        } finally {
            lock.unlock();
        }
        emptySlots.release();  // Signal empty slot
        return item;
    }
}
```

**When to Use Which:**

```java
// ✅ Use synchronized when:
// - Simple critical sections
// - No need for advanced features (timeout, interrupt)
// - Legacy code

// ✅ Use ReentrantLock when:
// - Need tryLock() or timeout
// - Need interruptible lock
// - Need fair locking
// - Complex lock logic (multiple conditions)

// ✅ Use Semaphore when:
// - Need to limit concurrent access (N threads)
// - Pool of resources
// - Rate limiting
// - Binary semaphore as lock (rare)
```

**Interview Tip:** "synchronized = simplest, automatic. Lock = advanced features (tryLock, timeout, interruptible). Semaphore = control concurrent access count (N permits). Always unlock in finally block for Lock/Semaphore. synchronized locks are reentrant (same thread can re-acquire). Prefer synchronized for simple cases, Lock/Semaphore for complex synchronization."
