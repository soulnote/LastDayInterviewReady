
# 🎯 Q1. What is OOP?

## 1. Concept Explanation (Beginner → Advanced)

**Beginner:**  
OOP = Object-Oriented Programming. Ek **programming paradigm** jahan hum **real-world cheezo ko objects** mein represent karte hain. Har object ke paas **data** (properties/attributes) aur **behaviors** (methods) hote hain.

**Intermediate:**  
OOP ek **way of thinking** hai jahan hum code ko **objects** ke form me organize karte hain jo **real-world entities** ko model karte hain. Har object apni **state** (data) aur **behavior** (methods) ko encapsulate karta hai.

**Advanced (Core Philosophy):**

```
OOP vs Procedural Programming:
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│  PROCEDURAL (C, Pascal):                                                   │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  // Data and functions are separate                                 │   │
│  │  struct Student {                                                   │   │
│  │      String name;                                                   │   │
│  │      int age;                                                       │   │
│  │  };                                                                 │   │
│  │                                                                      │   │
│  │  void displayStudent(struct Student s) { ... }                      │   │
│  │  void calculateGrade(struct Student s) { ... }                      │   │
│  │                                                                      │   │
│  │  // Problem: Any function can modify any data                       │   │
│  │  // Not secure, hard to maintain                                    │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  OOP (Java, C++, Python):                                                  │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  // Data and methods are TOGETHER inside object                     │   │
│  │  class Student {                                                    │   │
│  │      private String name;      // data (encapsulated)               │   │
│  │      private int age;                                               │   │
│  │                                                                      │   │
│  │      public void display() { ... }  // behavior (with data)         │   │
│  │      public void calculateGrade() { ... }                           │   │
│  │  }                                                                  │   │
│  │                                                                      │   │
│  │  // Benefit: Data is protected, functions are associated            │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 2. Real-World Analogy

```
REAL-WORLD ANALOGY:
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│  Real World:                                                                │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  "Car" is an object:                                                │   │
│  │  - State: color, model, speed, fuelLevel                          │   │
│  │  - Behavior: accelerate(), brake(), turn()                         │   │
│  │                                                                      │   │
│  │  "Driver" interacts with Car through its methods                    │   │
│  │  - Driver doesn't need to know HOW engine works                     │   │
│  │  - Just knows: press accelerator → car moves                       │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  Programming Analogy:                                                      │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  Car myCar = new Car("Red", "Tesla");                               │   │
│  │  myCar.accelerate();  // I don't care HOW it works internally       │   │
│  │  myCar.brake();                                                      │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 3. Interview Tips

**What interviewers expect:**

✅ *"OOP is a programming paradigm that organizes code around 'objects' rather than functions and logic. An object contains both data (fields) and behavior (methods). For example, a 'BankAccount' object has data like balance and methods like deposit() and withdraw(). This makes code more modular, reusable, and easier to maintain. The four main pillars are Encapsulation, Inheritance, Polymorphism, and Abstraction."*

# 🎯 Q3. What are the main features of OOP?

## 1. The 4 Pillars

```
FOUR PILLARS OF OOP:
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│  PILLAR 1: ENCAPSULATION                                                   │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  Definition: Bundle data + methods together, hide internal details  │   │
│  │  Real-world: ATM machine - you insert card, enter PIN,              │   │
│  │              but don't know how it verifies internally              │   │
│  │  Code: private fields + public getters/setters                      │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  PILLAR 2: INHERITANCE                                                     │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  Definition: Child class inherits properties/methods from parent    │   │
│  │  Real-world: Dog inherits from Animal (both eat, sleep)             │   │
│  │  Code: class Dog extends Animal { ... }                             │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  PILLAR 3: POLYMORPHISM                                                    │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  Definition: Same method name, different implementations            │   │
│  │  Real-world: "play()" - Dog plays fetch, Cat plays with yarn        │   │
│  │  Code: animal.play() - behaves differently based on actual type     │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  PILLAR 4: ABSTRACTION                                                     │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  Definition: Hide complex implementation, show only essential       │   │
│  │  Real-world: Car steering wheel - you turn it, don't know about     │   │
│  │              rack and pinion, power steering, etc.                  │   │
│  │  Code: Interface/Abstract class with method signatures only         │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 2. Quick Comparison

| Pillar | What it does | Key Word | Real Example |
|--------|--------------|----------|--------------|
| **Encapsulation** | Protects data | `private` | Bank balance can't be directly modified |
| **Inheritance** | Code reuse | `extends` | Dog extends Animal |
| **Polymorphism** | One interface, many forms | `override` | Animal.makeSound() → Dog barks, Cat meows |
| **Abstraction** | Hide complexity | `abstract`, `interface` | Driver only knows steering wheel, not engine |

# 🎯 Q4. What is Encapsulation in OOP?

## 1. Deep Explanation

```
ENCAPSULATION - DETAILED:
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│  Definition: Bundling of data (variables) and methods (functions) that      │
│              operate on that data into a single unit (class), and           │
│              restricting direct access to some of the object's components.  │
│                                                                             │
│  WITHOUT Encapsulation (BAD):                                              │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  class BankAccount {                                                │   │
│  │      double balance;  // public!                                    │   │
│  │  }                                                                  │   │
│  │                                                                      │   │
│  │  BankAccount acc = new BankAccount();                               │   │
│  │  acc.balance = -1000;  // Direct modification! Can go negative!     │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  WITH Encapsulation (GOOD):                                                │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  class BankAccount {                                                │   │
│  │      private double balance;  // hidden!                           │   │
│  │                                                                      │   │
│  │      public void deposit(double amount) {                           │   │
│  │          if (amount > 0) {                                          │   │
│  │              balance += amount;                                     │   │
│  │          }                                                          │   │
│  │      }                                                              │   │
│  │                                                                      │   │
│  │      public void withdraw(double amount) {                          │   │
│  │          if (amount > 0 && amount <= balance) {                     │   │
│  │              balance -= amount;                                     │   │
│  │          }                                                          │   │
│  │      }                                                              │   │
│  │                                                                      │   │
│  │      public double getBalance() {                                   │   │
│  │          return balance;  // read-only access                       │   │
│  │      }                                                              │   │
│  │  }                                                                  │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 2. Real-World Code Example

```java
public class Employee {
    // Private fields (encapsulated data)
    private String name;
    private double salary;
    private String department;
    
    // Constructor
    public Employee(String name, double salary, String department) {
        this.name = name;
        setSalary(salary);  // Using setter for validation
        this.department = department;
    }
    
    // Public getters (controlled access)
    public String getName() {
        return name;
    }
    
    public double getSalary() {
        return salary;
    }
    
    // Setter with validation (business logic)
    public void setSalary(double salary) {
        if (salary < 0) {
            throw new IllegalArgumentException("Salary cannot be negative");
        }
        if (salary < 30000) {
            throw new IllegalArgumentException("Salary below minimum wage");
        }
        this.salary = salary;
    }
    
    // Business method that uses encapsulated data
    public void giveRaise(double percentage) {
        if (percentage > 0 && percentage <= 30) {
            this.salary += this.salary * (percentage / 100);
        }
    }
}
```

## 3. Benefits

```
BENEFITS OF ENCAPSULATION:
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│  1. DATA PROTECTION                                                         │
│     - Prevent invalid values (e.g., negative salary)                        │
│     - Maintain data integrity                                               │
│                                                                             │
│  2. FLEXIBILITY                                                             │
│     - Change internal implementation without affecting outside code         │
│     - Example: Change salary storage from double to BigDecimal              │
│                                                                             │
│  3. MAINTAINABILITY                                                         │
│     - All code that modifies data is in one place                           │
│     - Easy to add validation, logging, auditing                             │
│                                                                             │
│  4. TESTABILITY                                                             │
│     - Each class can be tested independently                                │
│     - Mock objects can replace real dependencies                            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

# 🎯 Q5. What is Inheritance in OOP?

## 1. Deep Explanation

```
INHERITANCE - DETAILED:
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│  Definition: Mechanism where one class (child/subclass) acquires the        │
│              properties and behaviors of another class (parent/superclass).  │
│                                                                             │
│  IS-A Relationship:                                                         │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  Dog IS-A Animal                                                    │   │
│  │  Car IS-A Vehicle                                                   │   │
│  │  SavingsAccount IS-A BankAccount                                    │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  Inheritance Hierarchy:                                                     │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                                                                      │   │
│  │                      Animal (Parent)                                 │   │
│  │                  /          |          \                            │   │
│  │                 /           |           \                           │   │
│  │            Mammal         Bird         Reptile                       │   │
│  │           /      \        /    \                                    │   │
│  │        Dog      Cat    Eagle  Parrot                                │   │
│  │                                                                      │   │
│  │  - Dog inherits from Mammal, which inherits from Animal             │   │
│  │  - Dog has all features of Animal + Mammal + its own                │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 2. Code Example

```java
// Parent class
class Animal {
    protected String name;
    
    public Animal(String name) {
        this.name = name;
    }
    
    public void eat() {
        System.out.println(name + " is eating");
    }
    
    public void sleep() {
        System.out.println(name + " is sleeping");
    }
}

// Child class
class Dog extends Animal {
    private String breed;
    
    public Dog(String name, String breed) {
        super(name);  // Call parent constructor
        this.breed = breed;
    }
    
    // New method (specific to Dog)
    public void bark() {
        System.out.println(name + " is barking");
    }
    
    // Override parent method
    @Override
    public void eat() {
        System.out.println(name + " the dog is eating dog food");
    }
}

// Usage
public class Main {
    public static void main(String[] args) {
        Dog dog = new Dog("Buddy", "Golden Retriever");
        
        dog.eat();    // Overridden method
        dog.sleep();  // Inherited from Animal
        dog.bark();   // Dog's own method
    }
}
```

```
TYPES OF INHERITANCE:
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │

│     ┌─────────┐                                                            │
│     │ Class A │                                                            │
│     └────┬────┘                                                            │
│          │                                                                 │

│                                                                             │
│  2. MULTILEVEL INHERITANCE                                                  │
│     ┌─────────┐                                                            │
│     │ Class A │                                                            │
│     └────┬────┘                                                            │
│          │                                                                 │
│          ▼                                                                 │
│     ┌─────────┐                                                            │
│     │ Class B │                                                            │
│     └────┬────┘                                                            │
│          │                                                                 │
│          ▼                                                                 │
│     ┌─────────┐                                                            │
│     │ Class C │  (Java supports)                                           │
│     └─────────┘                                                            │
│                                                                             │
│  3. HIERARCHICAL INHERITANCE                                                │
│               ┌─────────┐                                                  │
│               │ Class A │                                                  │

│          ┌────────┼────────┐                                               │
```

# 🎯 Q6. What is Abstraction in OOP?

## 1. Deep Explanation

```
ABSTRACTION - DETAILED:
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│  Definition: Hiding complex implementation details and showing only         │
│              essential features to the user.                                │
│                                                                             │
│  Real-World Example:                                                        │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  TV Remote:                                                          │   │
│  │  - You see: ON, OFF, Volume buttons (essential features)            │   │
│  │  - Hidden: Circuit board, infrared signals, encoding logic          │   │
│  │                                                                      │   │
│  │  You don't need to know HOW it works to USE it!                     │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  Code Example:                                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  // Abstract class (partial abstraction)                            │   │
│  │  abstract class Vehicle {                                           │   │
│  │      abstract void start();  // No implementation                   │   │
│  │      abstract void stop();                                          │   │
│  │                                                                      │   │
│  │      void honk() {  // Concrete method                              │   │
│  │          System.out.println("Beep beep!");                          │   │
│  │      }                                                              │   │
│  │  }                                                                  │   │
│  │                                                                      │   │
│  │  // Interface (100% abstraction)                                    │   │
│  │  interface Payment {                                                │   │
│  │      void pay(double amount);  // Only declaration                  │   │
│  │  }                                                                  │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 2. Abstraction vs Encapsulation (Important)

```
ABSTRACTION vs ENCAPSULATION:
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│  Aspect              | Abstraction              | Encapsulation             │
│  ────────────────────┼──────────────────────────┼──────────────────────────│
│  **Purpose**         | Hide complexity          | Hide data                │
│  **Focus**           | What object does         | How object does it       │
│  **Level**           | Design level (interface) | Implementation level     │
│  **Implementation**  | Abstract classes,        | Private fields,          │
│                      | Interfaces               | getters/setters          │
│  **Example**         | Car steering wheel       | Car engine internal      │
│                      | (WHAT you can do)        | (HOW it works)           │
│                                                                             │
│  Analogy:                                                                   │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  Restaurant:                                                        │   │
│  │  - Abstraction: Menu (WHAT you can order)                          │   │
│  │  - Encapsulation: Kitchen (HOW food is prepared - hidden)          │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

# 🎯 Q7. What is Polymorphism in OOP?

## 1. Deep Explanation

# 🎯 Q8. What is meant by static polymorphism?

## Answer

**Static Polymorphism (Compile-time Polymorphism):** Method calls are resolved at **compile time**, not at runtime.

**Achieved through:** Method Overloading

```java
public class MathOperations {
    
    // Same method name, different parameters
    public int add(int a, int b) {
        return a + b;
    }
    
    public int add(int a, int b, int c) {
        return a + b + c;
    }
    
    public double add(double a, double b) {
        return a + b;
    }
    
    public static void main(String[] args) {
        MathOperations math = new MathOperations();
        
        // Compiler knows which method to call based on arguments
        math.add(5, 10);        // Calls add(int, int)
        math.add(5, 10, 15);    // Calls add(int, int, int)
        math.add(5.5, 10.5);    // Calls add(double, double)
    }
}
```

**Key Points:**
- **Compile-time decision** - Compiler knows which method to call
- **Better performance** - No runtime overhead
- **Method overloading** - Same name, different parameters
- **Cannot change behavior at runtime**

# 🎯 Q9. What is meant by dynamic polymorphism?

## Answer

**Dynamic Polymorphism (Runtime Polymorphism):** Method calls are resolved at **runtime** based on the actual object type.

**Achieved through:** Method Overriding

```java
class Payment {
    void processPayment(double amount) {
        System.out.println("Processing payment of $" + amount);
    }
}

class CreditCardPayment extends Payment {
    @Override
    void processPayment(double amount) {
        System.out.println("Processing credit card payment of $" + amount);
        // Additional credit card logic
    }
}

class PayPalPayment extends Payment {
    @Override
    void processPayment(double amount) {
        System.out.println("Processing PayPal payment of $" + amount);
        // Additional PayPal logic
    }
}

public class PaymentSystem {
    public static void main(String[] args) {
        Payment payment;
        
        // Runtime decides which method to call
        payment = new CreditCardPayment();
        payment.processPayment(100);  // Calls CreditCardPayment version
        
        payment = new PayPalPayment();
        payment.processPayment(200);  // Calls PayPalPayment version
    }
}
```

**Key Points:**
- **Runtime decision** - JVM decides at runtime using Vtable
- **Requires inheritance** - Parent-child relationship
- **Method overriding** - Child provides specific implementation
- **Enables flexibility** - Can change behavior without changing code

# 🎯 Q11. What is method hiding?

## Answer

**Method Hiding:** When a child class defines a **static method** with the same signature as a static method in the parent class.

```java
class Parent {
    static void display() {
        System.out.println("Parent static method");
    }
    
    void show() {
        System.out.println("Parent instance method");
    }
}

class Child extends Parent {
    // This is METHOD HIDING (not overriding!)
    static void display() {
        System.out.println("Child static method");
    }
    
    // This is METHOD OVERRIDING
    @Override
    void show() {
        System.out.println("Child instance method");
    }
}

public class Test {
    public static void main(String[] args) {
        Parent p = new Child();
        
        p.display();  // Output: "Parent static method" (HIDING)
        p.show();     // Output: "Child instance method" (OVERRIDING)
        
        // For static methods, reference type (Parent) decides, not object type
        // For instance methods, object type (Child) decides
    }
}
```

**Key Differences from Overriding:**

| Aspect | Overriding | Method Hiding |
|--------|-----------|---------------|
| **Method type** | Instance method | Static method |
| **Polymorphism** | Runtime | Compile-time |
| **Decision based on** | Object type | Reference type |
| **@Override annotation** | Works | Not applicable |

# 🎯 Q12. What is variable hiding?

## Answer

**Variable Hiding:** When a child class declares a variable with the **same name** as a variable in the parent class.

```java
class Parent {
    String name = "Parent";
    int age = 50;
}

class Child extends Parent {
    String name = "Child";  // Hides Parent's name
    int age = 25;           // Hides Parent's age
    
    void display() {
        System.out.println("name: " + name);           // "Child"
        System.out.println("super.name: " + super.name); // "Parent"
        
        System.out.println("age: " + age);             // 25
        System.out.println("super.age: " + super.age); // 50
    }
}

public class Test {
    public static void main(String[] args) {
        Parent p = new Child();
        System.out.println(p.name);  // "Parent" (reference type decides)
        
        Child c = new Child();
        System.out.println(c.name);  // "Child" (object type decides)
    }
}
```

**Important:**
- Variables are **not polymorphic** like methods
- Reference type decides which variable is accessed
- Use `super` keyword to access parent variable

# 🎯 Q13. What is abstract class?

## Answer

**Abstract Class:** A class that **cannot be instantiated** and may contain **abstract methods** (without body) and **concrete methods** (with body).

```java
// Abstract class
abstract class Vehicle {
    protected String brand;
    
    // Constructor (can have!)
    public Vehicle(String brand) {
        this.brand = brand;
    }
    
    // Abstract method (no body)
    public abstract void startEngine();
    
    // Concrete method (with body)
    public void honk() {
        System.out.println(brand + " is honking!");
    }
    
    // Concrete method
    public void displayInfo() {
        System.out.println("Vehicle brand: " + brand);
    }
}

// Concrete class (must implement all abstract methods)
class Car extends Vehicle {
    private int numberOfDoors;
    
    public Car(String brand, int numberOfDoors) {
        super(brand);
        this.numberOfDoors = numberOfDoors;
    }
    
    @Override
    public void startEngine() {
        System.out.println(brand + " car engine started with key!");
    }
}

class ElectricCar extends Vehicle {
    public ElectricCar(String brand) {
        super(brand);
    }
    
    @Override
    public void startEngine() {
        System.out.println(brand + " electric car started silently!");
    }
}

// Usage
public class Test {
    public static void main(String[] args) {
        // Vehicle v = new Vehicle("Toyota"); // ERROR! Cannot instantiate
        
        Vehicle car = new Car("Toyota", 4);
        car.startEngine();  // "Toyota car engine started with key!"
        car.honk();         // "Toyota is honking!"
        
        Vehicle ev = new ElectricCar("Tesla");
        ev.startEngine();   // "Tesla electric car started silently!"
    }
}
```

```
USE ABSTRACT CLASS WHEN:

│  ✅ You have common fields/methods that all subclasses need                │
│  ✅ You want to provide default implementations                           │
│  ✅ You need non-public members (protected, private)                       │

│  ✅ You have constructors (interfaces don't have)                          │
```

# 🎯 Q14. What is interface?

## Answer

**Interface:** A **contract** that defines **what** a class should do, but not **how**. All methods are **abstract by default** (until Java 8).

```java
// Interface (100% abstraction before Java 8)
interface Payment {
    // Public, static, final by default
    double TAX_RATE = 0.18;  // Constant
    
    // Abstract method (no body)
    void pay(double amount);
    
    // Default method (Java 8+)
    default void printReceipt() {
        System.out.println("Receipt printed");
    }
    
    // Static method (Java 8+)
    static void validatePayment() {
        System.out.println("Payment validated");
    }
}

// Another interface
interface Refundable {
    void refund(double amount);
}

// Class implementing multiple interfaces
class CreditCardPayment implements Payment, Refundable {
    @Override
    public void pay(double amount) {
        System.out.println("Paying $" + amount + " via Credit Card");
        System.out.println("Tax: $" + (amount * Payment.TAX_RATE));
    }
    
    @Override
    public void refund(double amount) {
        System.out.println("Refunding $" + amount + " to Credit Card");
    }
}

class PayPalPayment implements Payment {
    private String email;
    
    public PayPalPayment(String email) {
        this.email = email;
    }
    
    @Override
    public void pay(double amount) {
        System.out.println("Paying $" + amount + " via PayPal to " + email);
    }
}

// Usage
public class Test {
    public static void main(String[] args) {
        Payment payment1 = new CreditCardPayment();
        payment1.pay(100);
        payment1.printReceipt();  // Default method
        
        Payment payment2 = new PayPalPayment("user@example.com");
        payment2.pay(50);
        
        Payment.validatePayment();  // Static method
    }
}
```

# 🎯 Q16. Is it possible for a single class to implement two different interfaces?

## Answer

**YES!** Java supports **multiple interface inheritance**.

```java
interface Printable {
    void print();
}

interface Showable {
    void show();
}

// Class implementing BOTH interfaces
class Document implements Printable, Showable {
    @Override
    public void print() {
        System.out.println("Printing document...");
    }
    
    @Override
    public void show() {
        System.out.println("Showing document...");
    }
}

// What if both interfaces have same method? (No problem!)
interface A {
    void display();
}

interface B {
    void display();  // Same method signature
}

class C implements A, B {
    @Override
    public void display() {  // Single implementation works for both
        System.out.println("Displaying...");
    }
}

// What if methods have same name but different return types? (PROBLEM!)
interface X {
    int getValue();
}

interface Y {
    String getValue();  // Different return type - COMPILE ERROR if implemented
}

// Solution: Can't have both
```

**Key Points:**
- A class can implement **any number of interfaces**
- If methods conflict (same name, same parameters), single implementation works
- If methods conflict with different return types → compilation error

# 🎯 Q17. Why default methods are allowed in interfaces?

## Answer

**Default methods** were introduced in Java 8 primarily for **backward compatibility** and **interface evolution**.

```java
// BEFORE Java 8 - Adding new method breaks ALL implementations!
interface Payment {
    void pay(double amount);
    // void printReceipt();  // Adding this breaks all existing classes
}

// AFTER Java 8 - Default method provides backward compatibility
interface Payment {
    void pay(double amount);
    
    default void printReceipt() {
        System.out.println("Printing receipt...");
    }
}

// Existing classes work without changes!
class CreditCardPayment implements Payment {
    @Override
    public void pay(double amount) {
        // Implementation
    }
    // printReceipt() automatically available
}
```

## Why Default Methods?

```
REASONS FOR DEFAULT METHODS:
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│  1. BACKWARD COMPATIBILITY                                                  │
│     - Add new methods to interfaces without breaking existing code         │
│     - Collections Framework (Java 8 added stream() to Collection)          │
│                                                                             │
│  2. MULTIPLE INHERITANCE OF BEHAVIOR                                       │
│     - Classes can inherit behavior from multiple interfaces                │

│                                                                             │
│  3. OPTIONAL METHODS                                                        │
│     - Provide default implementation that most classes can use             │

│                                                                             │
│  4. API EVOLUTION                                                          │

│                                                                             │
```

## Diamond Problem with Default Methods

```java
interface A {
    default void display() {
        System.out.println("A");
    }
}

interface B {
    default void display() {
        System.out.println("B");
    }
}

class C implements A, B {
    // MUST override to resolve conflict
    @Override
    public void display() {
        A.super.display();  // Explicitly choose A's method
        // OR B.super.display();
        // OR provide new implementation
    }
}
```

# 🎯 Q18. What are SOLID principles?

## Answer

**SOLID** = 5 design principles for maintainable and scalable object-oriented code.

```
SOLID PRINCIPLES:
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│  S - Single Responsibility Principle (SRP)                                 │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  "A class should have only ONE reason to change"                    │   │
│  │                                                                      │   │
│  │  BAD: User class handles database AND email AND logging             │   │
│  │  GOOD: UserService (DB), EmailService (email), Logger (logging)     │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  O - Open/Closed Principle (OCP)                                          │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  "Open for extension, closed for modification"                      │   │
│  │                                                                      │   │
│  │  BAD: Add new payment type → modify existing code                   │   │
│  │  GOOD: Payment interface → add new class without modifying existing │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  L - Liskov Substitution Principle (LSP)                                  │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  "Subtypes must be substitutable for their base types"              │   │
│  │                                                                      │   │
│  │  BAD: Bird class, Penguin extends Bird → fly() breaks               │   │
│  │  GOOD: FlyingBird extends Bird, Penguin doesn't extend FlyingBird   │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  I - Interface Segregation Principle (ISP)                                │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  "Clients should not be forced to depend on methods they don't use" │   │
│  │                                                                      │   │
│  │  BAD: Worker interface with work(), eat(), sleep()                  │   │
│  │  GOOD: Workable, Eatable, Sleepable interfaces                      │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  D - Dependency Inversion Principle (DIP)                                 │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  "Depend on abstractions, not concretions"                          │   │
│  │                                                                      │   │
│  │  BAD: OrderService directly creates MySQLDatabase object            │   │
│  │  GOOD: OrderService depends on Database interface                   │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Code Examples for SOLID

```java

    void save(User user) { ... }
    User findById(int id) { ... }
}

    void sendWelcomeEmail(User user) { ... }

// O - Open/Closed
```

# 🎯 Q19. Can you explain how "Inheritance" and "Composition" are different?

## Answer

```
INHERITANCE vs COMPOSITION:
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│  INHERITANCE (IS-A relationship):                                          │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  class Dog extends Animal {  // Dog IS-A Animal                     │   │
│  │                                                                      │   │
│  │  Pros:                                                              │   │
│  │  - Code reuse through parent                                        │   │
│  │  - Polymorphism support                                             │   │
│  │                                                                      │   │
│  │  Cons:                                                              │   │
│  │  - Tight coupling (child depends on parent)                         │   │
│  │  - Fragile base class problem                                       │   │
│  │  - Can't change behavior at runtime                                 │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  COMPOSITION (HAS-A relationship):                                         │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  class Car {                                                        │   │
│  │      private Engine engine;  // Car HAS-A Engine                    │   │
│  │      private Wheels wheels;  // Car HAS-A Wheels                    │   │
│  │  }                                                                  │   │
│  │                                                                      │   │
│  │  Pros:                                                              │   │
│  │  - Loose coupling                                                   │   │
│  │  - Can change behavior at runtime (dependency injection)            │   │
│  │  - More flexible                                                    │   │
│  │                                                                      │   │
│  │  Cons:                                                              │   │
│  │  - More boilerplate code                                            │   │
│  │  - No polymorphism (without interfaces)                             │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## Code Comparison

```java
// INHERITANCE (IS-A)
class Animal {
    void eat() { System.out.println("Eating"); }
}

class Dog extends Animal {
    void bark() { System.out.println("Barking"); }
}

// COMPOSITION (HAS-A)
class Engine {
    void start() { System.out.println("Engine started"); }
}

class Car {
    private Engine engine;  // Composition
    
    public Car(Engine engine) {
        this.engine = engine;
    }
    
    void start() {
        engine.start();  // Delegation
        System.out.println("Car started");
    }
}

public class Test {

        car.start();
        
        // Swap engine without modifying Car class!
        Car evCar = new Car(new ElectricEngine());
        evCar.start();
    }
}
```

## When to Use Which?

```
DECISION TREE:
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│  Use INHERITANCE when:                                                     │
│  ✅ Genuine IS-A relationship (Dog IS-A Animal)                            │
│  ✅ You need to override methods                                           │
│  ✅ You need polymorphism (parent reference to child)                      │
│  ✅ The relationship is permanent (not changing at runtime)                │
│                                                                             │
│  Use COMPOSITION when:                                                     │
│  ✅ HAS-A relationship (Car HAS-A Engine)                                  │
│  ✅ You need flexibility (swap implementations)                            │
│  ✅ You want loose coupling                                                │
│  ✅ You're designing for testability (dependency injection)                │
│                                                                             │
│  BEST PRACTICE: "Prefer composition over inheritance"                      │
│  - Use inheritance sparingly                                               │
│  - Composition gives more flexibility                                      │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

# 🎯 Q24. Inheritance - We have 3 classes A, B and C, Class C extends Class B and Class B extends Class A, Each class has an method add(), is there a way to call A's add() method from class C?

## Answer

**YES!** Use `super` keyword multiple times or direct casting.

```java
class A {
    void add() {
        System.out.println("A's add()");
    }
}

class B extends A {
    @Override
    void add() {
        System.out.println("B's add()");
    }
}

class C extends B {
    @Override
    void add() {
        System.out.println("C's add()");
    }
    
    void callAllAddMethods() {
        // Call C's own add()
        this.add();  // or just add()
        
        // Call B's add() (immediate parent)
        super.add();
        
        // Call A's add() - TWO WAYS:
        
        // WAY 1: Call super of super (not directly possible, but...)
        // Need to go through B's method that calls A
        // Not clean
        
        // WAY 2: Cast to A (Recommended)
        ((A) this).add();  // This calls A's add()!
        
        // WAY 3: Create a method in B to call A
        // super.super.add() - NOT allowed in Java!
    }
}

// DEMONSTRATION
public class Test {
    public static void main(String[] args) {
        C c = new C();
        
        c.add();           // Output: "C's add()"
        ((A) c).add();     // Output: "A's add()" (direct cast)
        
        // Or using reference
        A aRef = c;
        aRef.add();        // Output: "A's add()" (polymorphism works)
    }
}
```

**Key Points:**
- `super.super.add()` is **NOT allowed** in Java
- Casting to parent type works because of polymorphism
- Or create a method in B that calls A's add()

```java
        System.out.println("Stopping " + brand + " " + model);
        engine.stop();
    }
    
    public void displayInfo() {
```

# 🎯 Q28. Implementing only few methods from multiple interfaces

## Answer

**Solution:** Use **Adapter Pattern** or **Default Methods** (Java 8+)

```java
// Interfaces with many methods
interface DatabaseOperations {
    void save();
    void update();
    void delete();
    void findById();
    void findAll();
    void batchSave();
    void transactionBegin();
    void transactionCommit();
    void transactionRollback();
}

interface NotificationService {
    void sendEmail();
    void sendSMS();
    void sendPush();
    void sendSlack();
    void sendWhatsApp();
}

// PROBLEM: Implementing class forced to implement ALL methods
class MyService implements DatabaseOperations, NotificationService {
    // Must implement ALL 9 + 5 = 14 methods!
    // Even if I only need 2 methods!
    
    @Override public void save() { /* needed */ }
    @Override public void update() { /* needed */ }
    @Override public void delete() { /* not needed - but must implement! */ }
    @Override public void findById() { /* not needed */ }
    // ... and so on (waste of code)
}

// SOLUTION 1: Adapter Pattern (Java 7 style)
// Create abstract adapter classes with empty implementations
abstract class DatabaseAdapter implements DatabaseOperations {
    @Override public void save() {}
    @Override public void update() {}
    @Override public void delete() {}
    @Override public void findById() {}
    @Override public void findAll() {}
    @Override public void batchSave() {}
    @Override public void transactionBegin() {}
    @Override public void transactionCommit() {}
    @Override public void transactionRollback() {}
}

abstract class NotificationAdapter implements NotificationService {
    @Override public void sendEmail() {}
    @Override public void sendSMS() {}
    @Override public void sendPush() {}
    @Override public void sendSlack() {}
    @Override public void sendWhatsApp() {}
}

// Now implement only needed methods
class MyBetterService extends DatabaseAdapter implements NotificationService {
    
    // Only implement what I need
    @Override
    public void save() {
        System.out.println("Saving data");
    }
    
    @Override
    public void update() {
        System.out.println("Updating data");
    }
    
    // Notification methods - only implement needed ones
    @Override
    public void sendEmail() {
        System.out.println("Sending email");
    }
    
    // Other methods use default empty implementations from adapter
}

// SOLUTION 2: Using Default Methods (Java 8+)
interface ModernDatabaseOps {
    void save();
    void update();
    
    // Default implementations for optional methods
    default void delete() {
        throw new UnsupportedOperationException("Delete not supported");
    }
    
    default void findById() {
        throw new UnsupportedOperationException("Find by ID not supported");
    }
    
    default void batchSave() {
        throw new UnsupportedOperationException("Batch save not supported");
    }
}

interface ModernNotificationService {
    void sendEmail();
    void sendSMS();
    
    default void sendPush() {
        throw new UnsupportedOperationException("Push not supported");
    }
    
    default void sendSlack() {
        throw new UnsupportedOperationException("Slack not supported");
    }
}

// Now implement only required methods
class ModernService implements ModernDatabaseOps, ModernNotificationService {
    
    @Override
    public void save() {
        System.out.println("Saving");
    }
    
    @Override
    public void update() {
        System.out.println("Updating");
    }
    
    @Override
    public void sendEmail() {
        System.out.println("Email sent");
    }
    
    @Override
    public void sendSMS() {
        System.out.println("SMS sent");
    }
    
    // delete(), findById(), sendPush(), etc. use default (throw exception)
}

// SOLUTION 3: Interface Segregation Principle (Best Practice)
// Split large interfaces into smaller, focused ones
interface Saveable {
    void save();
}

interface Updatable {
    void update();
}

interface Deletable {
    void delete();
}

interface Findable {
    void findById();
}

interface Emailable {
    void sendEmail();
}

interface Smsable {
    void sendSMS();
}

// Now implement ONLY what you need
class SegregatedService implements Saveable, Updatable, Emailable {
    
    @Override
    public void save() {
        System.out.println("Saving");
    }
    
    @Override
    public void update() {
        System.out.println("Updating");
    }
    
    @Override
    public void sendEmail() {
        System.out.println("Email sent");
    }
    
    // No unnecessary methods!
}
```
