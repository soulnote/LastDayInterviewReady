## **Core Java**

### **Q1. Order of execution - static initializers, instance initializers, and constructors**

**Answer:** (Previously answered in Q21, let me give you a **quick refresher**)

**Order:**
```
1. Parent class static initializer
2. Child class static initializer
3. Parent class instance initializer
4. Parent class constructor
5. Child class instance initializer
6. Child class constructor
```

**Quick example:**
```java
class A {
    static { System.out.println("1 - A static"); }
    { System.out.println("3 - A instance"); }
    A() { System.out.println("4 - A constructor"); }
}

class B extends A {
    static { System.out.println("2 - B static"); }
    { System.out.println("5 - B instance"); }
    B() { System.out.println("6 - B constructor"); }
}

// new B() → prints 1,2,3,4,5,6
```

**Interview tip:** "Static blocks execute once during class loading (parent first). Instance blocks execute before constructor (parent then child)."

### **Q2. Millions of temporary StringBuilder objects causing GC pauses**

**Answer:**  
Yeh ek **real performance problem** hai. Temporary StringBuilder objects create ho rahe hain, GC frequently run ho raha hai, aur application pause ho rahi hai.

**Root Cause:**
- StringBuilder objects short-lived hote hain (young generation mein create → quickly become garbage)
- Millions of objects → Young GC frequently triggers → Stop-The-World pauses

**Solutions:**

**1. Reuse StringBuilder objects (Object Pooling):**
```java
// ❌ Bad - New object each time
public String process(String input) {
    StringBuilder sb = new StringBuilder();  // New object every call
    sb.append(input);
    return sb.reverse().toString();
}

// ✅ Good - Reuse with ThreadLocal
public class StringBuilderHolder {
    private static final ThreadLocal<StringBuilder> sbPool = 
        ThreadLocal.withInitial(() -> new StringBuilder(1024));  // Pre-allocate capacity
    
    public static StringBuilder get() {
        StringBuilder sb = sbPool.get();
        sb.setLength(0);  // Clear without creating new object
        return sb;
    }
}

// Usage
StringBuilder sb = StringBuilderHolder.get();
sb.append("Hello").append("World");
```

**2. Pre-allocate capacity to avoid internal resizing:**
```java
// ❌ Bad - Default capacity 16, resizes multiple times
StringBuilder sb = new StringBuilder();

// ✅ Good - Estimate required capacity
StringBuilder sb = new StringBuilder(1024);  // Pre-allocate 1KB
// Reduces internal array reallocation
```

**3. Use String concatenation for simple cases (Java 9+ optimized):**
```java
// Java 9+ - String concatenation uses invokedynamic, sometimes better than StringBuilder
String result = "Hello " + name + "! Welcome to " + city;
// Compiler optimizes this
```

**4. Use compile-time constants:**
```java
// ❌ Bad
String log = "User: " + userId + " Action: " + action;

// ✅ Good for repeated patterns
private static final String LOG_PATTERN = "User: %s Action: %s";
String log = String.format(LOG_PATTERN, userId, action);
// Or use MessageFormat
```

**5. Increase Young Generation size (if memory allows a bit):**
```bash
# JVM tuning
-XX:NewSize=512m -XX:MaxNewSize=512m
# Reduces frequency of Young GC
```

**6. Use StringBuilder directly without wrapper methods:**
```java
// ❌ Bad - Creates StringBuilder internally
public String buildString() {
    return String.join(", ", list);  // Creates StringBuilder internally
}

// ✅ Good - Reuse existing
public void buildString(StringBuilder sb, List<String> list) {
    for (int i = 0; i < list.size(); i++) {
        if (i > 0) sb.append(", ");
        sb.append(list.get(i));
    }
}
```

**Best Solution (Production-ready):**
```java
public class OptimizedStringBuilder {
    // ThreadLocal pool with different sizes
    private static final ThreadLocal<StringBuilder> smallPool = 
        ThreadLocal.withInitial(() -> new StringBuilder(256));
    private static final ThreadLocal<StringBuilder> largePool = 
        ThreadLocal.withInitial(() -> new StringBuilder(4096));
    
    public static StringBuilder get(int estimatedSize) {
        StringBuilder sb = (estimatedSize < 512) ? smallPool.get() : largePool.get();
        sb.setLength(0);
        if (sb.capacity() < estimatedSize) {
            sb.ensureCapacity(estimatedSize);
        }
        return sb;
    }
}
```

**Interview tip:** "Main solution - **Reuse with ThreadLocal**. GC pauses ka main reason is short-lived objects. Reusing objects eliminates GC pressure entirely. Pre-allocation of capacity avoids internal array resizing."

### **Q3. Bean created for abstract class and interface? Why @Repository on JpaRepository interface?**

**Answer:**  
**No** - abstract classes aur interfaces ke instances **directly nahi bana sakte**. But Spring **dynamic proxy** banata hai.

**Why interface ka bean possible hai?**
```java
// JpaRepository is an interface
public interface UserRepository extends JpaRepository<User, Long> {
    // No implementation
}

// But Spring creates a bean!
@Autowired
UserRepository userRepo;  // Works!
```

**How Spring does it:**
1. Spring uses **JDK Dynamic Proxy** or **CGLIB**
2. At runtime, Spring creates implementation class dynamically
3. That implementation class becomes a bean

**Abstract class scenario:**
```java
// ❌ Spring CANNOT create bean directly
public abstract class BaseService {
    public abstract void doWork();
}

// ✅ Spring can create bean if using concrete subclass
@Service
public class ConcreteService extends BaseService {
    public void doWork() { }
}
```

**Why @Repository on JpaRepository works:**
```java
// @Repository is optional on JpaRepository because:
@Repository
public interface UserRepository extends JpaRepository<User, Long> { }

// Spring Data JPA's infrastructure automatically:
// 1. Creates proxy implementation at runtime
// 2. Registers that proxy as a bean
// 3. @Repository adds exception translation (PersistenceExceptionTranslationPostProcessor)
```

**Behind the scenes:**
```java
// Spring internally does something like:
public class JpaRepositoryFactory {
    public <T> T getRepository(Class<T> repositoryInterface) {
        // Creates dynamic proxy
        return (T) Proxy.newProxyInstance(
            repositoryInterface.getClassLoader(),
            new Class[]{repositoryInterface},
            new JpaRepositoryInvocationHandler()
        );
    }
}
```

**Interview tip:** "Interfaces ke liye bean possible hai because Spring creates **dynamic proxy** at runtime. Abstract classes ke liye direct bean possible nahi hai unless you provide concrete implementation. `@Repository` on JpaRepository is marker for exception translation."

### **Q4. Store country-state-city data structure**

**Answer:**  
Yeh **hierarchical data** hai - multiple solutions exist depending on use case.

**Solution 1: Nested Maps (Most straightforward)**
```java
Map<String, Map<String, List<String>>> countryStateCity = new HashMap<>();

// Data structure: Country → State → List<City>

// Adding data
countryStateCity.computeIfAbsent("India", k -> new HashMap<>())
    .computeIfAbsent("Maharashtra", k -> new ArrayList<>())
    .add("Mumbai");
countryStateCity.get("India").get("Maharashtra").add("Pune");
countryStateCity.get("India").get("Maharashtra").add("Nagpur");

countryStateCity.computeIfAbsent("India", k -> new HashMap<>())
    .computeIfAbsent("Karnataka", k -> new ArrayList<>())
    .add("Bangalore");
countryStateCity.get("India").get("Karnataka").add("Mysore");

// Query - Get all cities in a state
List<String> cities = countryStateCity.get("India").get("Maharashtra");
// Returns [Mumbai, Pune, Nagpur]
```

**Solution 2: Custom Class Hierarchy (Type-safe)**
```java
class City {
    private String name;
    // constructor, getters
}

class State {
    private String name;
    private List<City> cities;
    
    public State(String name) {
        this.name = name;
        this.cities = new ArrayList<>();
    }
    
    public void addCity(City city) {
        cities.add(city);
    }
    
    public List<City> getCities() {
        return cities;
    }
}

class Country {
    private String name;
    private Map<String, State> states;  // State name → State object
    
    public Country(String name) {
        this.name = name;
        this.states = new HashMap<>();
    }
    
    public void addState(State state) {
        states.put(state.getName(), state);
    }
    
    public List<City> getCitiesByState(String stateName) {
        State state = states.get(stateName);
        return state != null ? state.getCities() : Collections.emptyList();
    }
}

// Usage
Country india = new Country("India");
State maharashtra = new State("Maharashtra");
maharashtra.addCity(new City("Mumbai"));
maharashtra.addCity(new City("Pune"));
india.addState(maharashtra);
```

**Solution 3: Using Java 8+ Streams with FlatMap (for search across states)**
```java
// Find all cities containing "pur" across all states in India
List<City> matchedCities = countryStateCity.entrySet().stream()
    .flatMap(country -> country.getValue().entrySet().stream())
    .flatMap(state -> state.getValue().stream()
        .filter(city -> city.contains("pur"))
        .map(cityName -> new City(cityName, state.getKey())))
    .collect(Collectors.toList());
```

**Solution 4: Database approach (if data is large)**
```sql
-- Tables: country(id, name), state(id, country_id, name), city(id, state_id, name)
SELECT city.name FROM city 
JOIN state ON city.state_id = state.id 
WHERE state.name = 'Maharashtra';
```

**Solution 5: Trie/Prefix tree (for autocomplete/search)**
```java
// For fast prefix search like "Mu" → Mumbai, Muscat, etc.
class LocationTrie {
    private TrieNode root;
    
    public List<String> searchByPrefix(String prefix) {
        // Returns all locations starting with prefix
    }
}
```

**Interview tip:** "Simple requirement ke liye **nested Map** sufficient hai. Large data or complex queries ke liye **database** better. Type safety chahiye toh **custom classes**. Memory efficient chahiye toh **normalized structure** with IDs."

### **Q5. Security for desktop application without Spring/frameworks**

**Answer:**  
Desktop application security mein multiple layers hoti hain. Without Spring, aap **Java built-in features** use karoge.

**1. Authentication (Login mechanism):**
```java
public class AuthenticationService {
    // Store password hashes, NOT plain text
    private Map<String, String> userPasswordHash = new ConcurrentHashMap<>();
    
    public boolean authenticate(String username, char[] password) {
        String storedHash = userPasswordHash.get(username);
        if (storedHash == null) return false;
        
        // Use bcrypt or PBKDF2 (not simple hash)
        return BCrypt.checkpw(new String(password), storedHash);
        
        // Clear password from memory immediately
        Arrays.fill(password, ' ');
    }
    
    public void createUser(String username, char[] password) {
        String hashedPassword = BCrypt.hashpw(new String(password), BCrypt.gensalt(12));
        userPasswordHash.put(username, hashedPassword);
    }
}
```

**2. Secure configuration storage (encrypt sensitive data):**
```java
public class SecureConfig {
    private static final String ALGORITHM = "AES/GCM/NoPadding";
    private static final int KEY_SIZE = 256;
    
    // Store encrypted config
    public void saveConfig(String key, String value, SecretKey encryptionKey) {
        Cipher cipher = Cipher.getInstance(ALGORITHM);
        cipher.init(Cipher.ENCRYPT_MODE, encryptionKey);
        byte[] encrypted = cipher.doFinal(value.getBytes(StandardCharsets.UTF_8));
        
        // Save encrypted bytes to file
        saveToFile(key, encrypted, cipher.getIV());
    }
}
```

**3. Input validation (prevent injection attacks):**
```java
public class InputValidator {
    private static final Pattern SAFE_PATTERN = Pattern.compile("^[a-zA-Z0-9\\s\\-_.]+$");
    
    public static boolean isValidInput(String input) {
        if (input == null || input.length() > 256) return false;
        return SAFE_PATTERN.matcher(input).matches();

    // Prevent SQL injection (if using embedded DB)

        return input.replace("'", "''");  // Escape single quotes

}

**4. Secure communication (if desktop app talks to server):**

```

**5. Role-based access control (RBAC):**

### **Q6. Which is better? `Bob fob = new Bob();` OR `Bob fob = new Derived();`**

**Answer:**  
Yeh **use case dependent** hai. Dono ke alag alag advantages hain.

**`Bob fob = new Bob();` (Base class reference to base object)**
```java
Bob fob = new Bob();
```
- Simple, direct
- Only Bob's methods accessible (unless overridden)
- No polymorphism benefits

**`Bob fob = new Derived();` (Base class reference to derived object)**
```java
Bob fob = new Derived();  // Where Derived extends Bob
```
- **Polymorphism** - can change implementation without changing reference type
- **Loose coupling** - code depends on abstraction (Bob), not concrete Derived
- Can only call Bob's methods (even if Derived has more)

**Which is better?**

| Scenario | Better choice | Why |
|----------|---------------|-----|
| **Simple utility class** | `new Bob()` | No polymorphism needed |
| **Dependency injection** | `new Derived()` (but via interface) | Easy to swap implementations |
| **Testing** | `new Derived()` (mock) | Can inject mock/test doubles |
| **Framework code** | Interface reference | Loose coupling |
| **Performance critical** | `new Bob()` | No vtable overhead (minimal though) |
| **Need Derived-specific methods** | `Derived d = new Derived()` | Can't use base reference |

**Best Practice - Use Interface when possible:**
```java
// Instead of concrete class
List<String> list = new ArrayList<>();  // Good - programming to interface

// Instead of
ArrayList<String> list = new ArrayList<>();  // Less flexible
```

**Real example:**
```java
// ❌ Less flexible
ArrayList<String> names = new ArrayList<>();

// ✅ More flexible - can change to LinkedList anytime
List<String> names = new ArrayList<>();  // Or new LinkedList<>()
```

**Interview tip:** "Programming to **interface/abstract class** is better for maintainability. Use `Bob fob = new Derived()` if Derived is just an implementation detail and Bob defines the contract. Use `Derived d = new Derived()` only if you need Derived-specific methods."

## **Serialization**

### **Q7. Class A serialized, Class B not serialized, A contains reference to B**

**Answer:**  
**Yes, Class B will also be serialized/initialized** - actually, it will cause **NotSerializableException**.

**Explanation:**
```java
class B {
    String data = "B's data";
}

class A implements Serializable {
    private static final long serialVersionUID = 1L;
    B bRef = new B();  // A contains reference to non-serializable B
}

// When you try to serialize:
A a = new A();
ObjectOutputStream oos = new ObjectOutputStream(new FileOutputStream("data.ser"));
oos.writeObject(a);  // ❌ Throws java.io.NotSerializableException: B
```

**Why?**
- Serialization is **recursive** - JVM serializes entire object graph
- A contains reference to B → JVM tries to serialize B
- B doesn't implement Serializable → Exception thrown

**Solutions:**

**1. Make B serializable:**
```java
class B implements Serializable {
    String data = "B's data";
}
```

**2. Mark B as `transient` (skip serialization):**
```java
class A implements Serializable {
    transient B bRef = new B();  // Won't be serialized
}
// After deserialization, bRef will be null
```

**3. Custom serialization logic:**
```java
class A implements Serializable {
    B bRef = new B();
    
    private void writeObject(ObjectOutputStream oos) throws IOException {
        oos.defaultWriteObject();
        // Manually serialize B's important data
        oos.writeObject(bRef.data);
    }
    
    private void readObject(ObjectInputStream ois) throws IOException, ClassNotFoundException {
        ois.defaultReadObject();
        String bData = (String) ois.readObject();
        bRef = new B();
        bRef.data = bData;
    }
}
```

**Interview tip:** "Serialization is transitive. If any referenced object is not Serializable, you get `NotSerializableException`. Fix by making referenced class Serializable, using `transient`, or custom serialization."

## **Database**

### **Q8. Which database to choose for large volumes of data?**

**Answer:**  
Yeh **workload type** par depend karta hai. 'Large volume' ka matlab alag ho sakta hai.

**Decision Framework:**

| Workload Type | Best Choice | Example |
|---------------|-------------|---------|
| **Transactional (OLTP)** | PostgreSQL, Oracle, SQL Server | Banking, E-commerce orders |
| **Analytical (OLAP)** | ClickHouse, Snowflake, BigQuery | Reports, BI dashboards |
| **Time-series** | InfluxDB, TimescaleDB | IoT sensors, metrics |
| **Key-Value lookups** | Redis, DynamoDB | Session cache, user profiles |
| **Document store** | MongoDB, Cosmos DB | JSON data, catalogs |
| **Search** | Elasticsearch | Log search, product search |
| **Graph relationships** | Neo4j, Amazon Neptune | Social networks, fraud detection |
| **Columnar (Big Data)** | Cassandra, HBase | Write-heavy, wide columns |

**Detailed analysis:**

**1. PostgreSQL (Best for general purpose):**
- Handles TB-scale data
- ACID compliant
- Good for complex queries
- Limitations: ~10TB recommended, sharding not built-in

**2. ClickHouse (Best for analytics):**
- Handles PB-scale data
- 100-1000x faster than traditional DB for aggregations
- Columnar storage
- Limitations: Not for frequent updates/deletes

**3. Cassandra (Best for write-heavy, distributed):**
- Linear scalability to hundreds of nodes
- No single point of failure
- 1M+ writes/second possible
- Limitations: No joins, limited aggregations

**4. MongoDB (Best for flexible schema):**
- Horizontal scaling with sharding
- Handles billions of documents
- Rich query language
- Limitations: No ACID across documents (though now improved)

**Decision tree:**
```
Large volume data?
    │
    ├─ Need ACID transactions? → YES → PostgreSQL / Oracle
    │
    ├─ Mostly analytical queries? → YES → ClickHouse / Snowflake
    │
    ├─ High write throughput? → YES → Cassandra / Kafka + DB
    │
    ├─ Unstructured/semi-structured? → YES → MongoDB
    │
    └─ Real-time search? → YES → Elasticsearch
```

**Interview tip:** "No single database fits all. For **transactional** data → PostgreSQL. For **analytics** → ClickHouse. For **write-heavy distributed** → Cassandra. Often companies use **polyglot persistence** - multiple DBs for different needs."

## **Hibernate**

### **Q9. How to save millions of data into database using hibernate?**

**Answer:**  
Hibernate se millions of records save karne ka **default way** bahut slow hai. **Batch processing** use karna padega.

**Problem with default approach:**
```java
// ❌ NEVER do this - extremely slow
for (int i = 0; i < 1_000_000; i++) {
    session.save(new Record(i));  // Each save = separate transaction
}
// Result: 1 million individual INSERT statements, 1 million commits
```

**Solution 1: JDBC Batching with Hibernate:**
```java
// Configure hibernate.cfg.xml
<hibernate-configuration>
    <property name="hibernate.jdbc.batch_size">50</property>
    <property name="hibernate.order_inserts">true</property>
    <property name="hibernate.order_updates">true</property>
    <property name="hibernate.jdbc.batch_versioned_data">true</property>
</hibernate-configuration>

// Code
Session session = sessionFactory.openSession();
Transaction tx = session.beginTransaction();

int batchSize = 50;
for (int i = 0; i < 1_000_000; i++) {
    Record record = new Record(i, "Data " + i);
    session.save(record);
    
    if (i % batchSize == 0) {
        session.flush();  // Send batch to DB
        session.clear();  // Clear persistence context
    }
}
tx.commit();
session.close();
```

**Solution 2: StatelessSession (Best for millions of records):**
```java
// StatelessSession doesn't maintain cache - much more memory efficient
StatelessSession statelessSession = sessionFactory.openStatelessSession();
Transaction tx = statelessSession.beginTransaction();

for (int i = 0; i < 1_000_000; i++) {
    Record record = new Record(i, "Data " + i);
    statelessSession.insert(record);  // No caching, no dirty checking
    
    if (i % 1000 == 0) {
        statelessSession.flush();
    }
}
tx.commit();
statelessSession.close();
```

**Solution 3: Native JDBC Batch (Fastest):**
```java
// Bypass Hibernate entirely for bulk inserts
Connection conn = session.doReturningWork(connection -> connection);
String sql = "INSERT INTO record (id, name) VALUES (?, ?)";
PreparedStatement ps = conn.prepareStatement(sql);

int batchSize = 1000;
for (int i = 0; i < 1_000_000; i++) {
    ps.setInt(1, i);
    ps.setString(2, "Data " + i);
    ps.addBatch();
    
    if (i % batchSize == 0) {
        ps.executeBatch();
        ps.clearBatch();
    }
}
ps.executeBatch();
```

**Solution 4: Use Bulk Insert with multiple values:**
```java
// Single INSERT with 1000 values
String sql = "INSERT INTO record (id, name) VALUES ";
StringBuilder sb = new StringBuilder(sql);
for (int i = 0; i < 1000; i++) {
    sb.append("(?, ?)");
    if (i < 999) sb.append(",");
}
// Execute one big insert instead of 1000 small ones
```

**Performance Comparison:**
| Method | Time for 1M records | Memory usage |
|--------|---------------------|--------------|
| Default Hibernate | ~45 minutes | High (OOM risk) |
| Hibernate + Batching | ~3-5 minutes | Medium |
| StatelessSession | ~2-3 minutes | Low |
| JDBC Batch | ~30-60 seconds | Very low |

**Best Practice Code:**
```java
public void bulkInsert(List<Record> records) {
    int batchSize = 500;
    Session session = sessionFactory.openSession();
    Transaction tx = null;
    
    try {
        tx = session.beginTransaction();
        
        for (int i = 0; i < records.size(); i++) {
            session.save(records.get(i));
            
            if (i % batchSize == 0 && i > 0) {
                session.flush();
                session.clear();
            }
        }
        
        tx.commit();
    } catch (Exception e) {
        if (tx != null) tx.rollback();
        throw e;
    } finally {
        session.close();
    }
}
```

**Interview tip:** "Hibernate bulk inserts ke liye not optimized. Use **StatelessSession** or **JDBC batch**. Set `batch_size` to 50-500. Disable second-level cache. Use `session.flush()` and `clear()` periodically. For extreme scale, consider **native JDBC** or **database-specific bulk insert tools** like `COPY` command in PostgreSQL."

## **Rest API**

### **Q10. Controller annotated and database operation performed - what outcome?**

**Answer:**  
Yeh question thoda incomplete hai, but main common scenarios explain karta hoon.

**Assuming typical Spring Boot controller:**
```java
@RestController
public class UserController {
    
    @Autowired
    private UserService userService;
    
    @PostMapping("/user")
    public User createUser(@RequestBody User user) {
        return userService.save(user);  // Database operation
    }
}
```

**Outcome depends on Transaction management:**

**Scenario 1: Without @Transactional**
```java
// Controller has NO transaction
// Each database call = separate transaction
userService.save(user);  // Transaction 1 (auto-commit)
userService.sendEmail(user);  // Transaction 2
// If email fails, user still saved!
```

**Scenario 2: With @Transactional on Service**
```java
@Service
@Transactional
public class UserService {
    public void createUser(User user) {
        saveToDB(user);     // Part of same transaction
        sendEmail(user);    // If fails, DB rollback
    }
}
// Controller call → transaction starts at service boundary
```

**Scenario 3: @Transactional on Controller (not recommended)**
```java
@RestController
@Transactional  // Not recommended - controllers should be thin
public class UserController {
    // Transaction spans entire HTTP request
    // Long-running transaction = bad for concurrency
}
```

**Possible issues:**

1. **LazyInitializationException:**
```java
// Controller returns entity with lazy-loaded collections
@GetMapping("/user/{id}")
public User getUser(@PathVariable Long id) {
    User user = userService.findById(id);
    return user;  // If user has lazy-loaded roles → LazyInitializationException
}
// Fix: Use @Transactional on controller or DTO projection
```

2. **Transaction not started:**
```java
// No @Transactional anywhere
userService.updateUser(user);  // Auto-commit mode
// Each JPA operation in separate transaction
```

3. **Read-only transaction optimization:**
```java
@Transactional(readOnly = true)
public List<User> getUsers() {
    // Hibernate won't do dirty checking - better performance
}
```

**Best Practice:**
```java
@RestController
public class UserController {
    // Controllers: no @Transactional (thin layer)
    
    @Autowired
    private UserService userService;
    
    @PostMapping("/user")
    public ResponseEntity<User> createUser(@Valid @RequestBody User user) {
        User saved = userService.createUser(user);  // Service handles transaction
        return ResponseEntity.created(URI.create("/user/" + saved.getId())).body(saved);
    }
}

@Service
@Transactional  // Transaction at service layer
public class UserService {
    public User createUser(User user) {
        // All database operations in single transaction
        return userRepository.save(user);
    }
}
```

**Interview tip:** "Controller mein database operation normally works, but **transaction boundary** important hai. Without `@Transactional`, each database call auto-commits. With `@Transactional`, multiple operations in same transaction. Keep controllers **thin** - no `@Transactional` on controllers, put on service layer."

## **Collections**

### **Q11. Convert list of student names to set - what happens to duplicates?**

**Answer:**  
**Duplicates automatically removed** - Set doesn't allow duplicate elements.

**Example:**
```java
List<String> studentNames = Arrays.asList(
    "Amit", "Priya", "Raj", "Amit", "Priya", "Sneha", "Raj"
);

// Convert to HashSet
Set<String> uniqueNames = new HashSet<>(studentNames);

System.out.println(uniqueNames);
// Output: [Amit, Priya, Raj, Sneha] - duplicates removed
// Order NOT preserved (HashSet)
```

**Different Set implementations behavior:**

| Set Type | Duplicate handling | Order preserved? | Performance |
|----------|-------------------|------------------|-------------|
| **HashSet** | Removed | ❌ No (unordered) | O(1) average |
| **LinkedHashSet** | Removed | ✅ Yes (insertion order) | O(1) average |
| **TreeSet** | Removed | ✅ Yes (sorted order) | O(log n) |

**Code examples:**
```java
// 1. HashSet - fastest, no order
Set<String> hashSet = new HashSet<>(studentNames);
// Result: [Sneha, Amit, Priya, Raj] (arbitrary order)

// 2. LinkedHashSet - maintains insertion order
Set<String> linkedHashSet = new LinkedHashSet<>(studentNames);
// Result: [Amit, Priya, Raj, Sneha] (same order as first occurrence)

// 3. TreeSet - sorted order (natural ordering)
Set<String> treeSet = new TreeSet<>(studentNames);
// Result: [Amit, Priya, Raj, Sneha] (alphabetical)

// 4. Using Java 8 streams
Set<String> streamSet = studentNames.stream()
    .collect(Collectors.toSet());  // Returns HashSet

// 5. To preserve order with streams
Set<String> orderedSet = studentNames.stream()
    .collect(Collectors.toCollection(LinkedHashSet::new));
```

**What defines duplicate?**
```java
// For String, "Amit".equals("Amit") → duplicate
// For custom objects, based on equals() and hashCode()
class Student {
    String name;
    int rollNo;
    
    @Override
    public boolean equals(Object o) {
        // Two students are equal if rollNo same
        return rollNo == ((Student) o).rollNo;
    }
    
    @Override
    public int hashCode() {
        return Objects.hash(rollNo);
    }
}
```

**Important - Set doesn't throw error on duplicate:**
```java
Set<String> names = new HashSet<>();
names.add("Amit");    // Returns true
names.add("Amit");    // Returns false (duplicate not added, no error)
```

**Interview tip:** "Set automatically removes duplicates based on `equals()` and `hashCode()`. Choose HashSet for performance, LinkedHashSet to preserve order, TreeSet for sorted order. Duplicates are silently ignored, no exception thrown."
