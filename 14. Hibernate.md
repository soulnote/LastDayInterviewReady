## Q1. Explain Hibernate Architecture?

**Answer:**  
Hibernate ek **ORM (Object-Relational Mapping)** framework hai jo Java objects ko database tables ke saath map karta hai. Iski architecture layered hai.

**Core Components Architecture:**

```
┌─────────────────────────────────────────────────────────────┐
│                     Application Layer                        │
│                    (Entity/POJO Objects)                      │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                   Hibernate Configuration                     │
│         (hibernate.cfg.xml / persistence.xml)                │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                     SessionFactory                            │
│            (Thread-safe, created once per app)                │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                       Session                                 │
│           (Lightweight, one per transaction)                  │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  Transaction Management (begin/commit/rollback)      │    │
│  │  First Level Cache (mandatory, session-scoped)       │    │
│  │  Persistence Context (manages entity states)         │    │
│  └─────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                    JDBC / Connection Pool                     │
│                    (HikariCP, C3P0, etc.)                     │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                      Database                                 │
└─────────────────────────────────────────────────────────────┘
```

**Detailed Component Explanation:**

```java
// 1. Configuration - Reads mapping and DB settings
Configuration config = new Configuration();
config.configure("hibernate.cfg.xml");
config.addAnnotatedClass(Employee.class);

// 2. SessionFactory - Heavyweight, thread-safe, one per DB
SessionFactory sessionFactory = config.buildSessionFactory();

// 3. Session - Lightweight, non-thread-safe, one per transaction
Session session = sessionFactory.openSession();

// 4. Transaction - Unit of work
Transaction tx = session.beginTransaction();
Employee emp = session.get(Employee.class, 1L);
emp.setSalary(50000);
tx.commit();  // Commits to DB

// 5. Query - HQL / Criteria / Native SQL
Query<Employee> query = session.createQuery("FROM Employee WHERE dept = :dept", Employee.class);
query.setParameter("dept", "IT");
List<Employee> result = query.list();

session.close();  // Close session
```

**Core Interfaces:**

| Interface | Responsibility | Thread-Safe | Lifetime |
|-----------|---------------|-------------|----------|
| **Configuration** | Load settings, build SessionFactory | No | Startup only |
| **SessionFactory** | Create Sessions, holds metadata | Yes | Application lifetime |
| **Session** | DB operations, persistence context | No | Per transaction/request |
| **Transaction** | Atomic operations, commit/rollback | No | Per operation |
| **Query** | Execute HQL/Criteria/Native SQL | No | Per query |

**Entity States in Hibernate:**
```java
// Transient - Object created but not associated with Session
Employee emp = new Employee("John", 50000);  // Transient

// Persistent - Associated with Session, tracked for changes
Session session = sessionFactory.openSession();
Transaction tx = session.beginTransaction();
session.save(emp);  // Persistent
emp.setSalary(60000);  // Auto-dirty checked at flush

// Detached - Session closed, object still has values
tx.commit();
session.close();  // Now emp is Detached

// Removed - Scheduled for deletion
session.delete(emp);  // Removed state
```

**Interview Tip:** "Hibernate architecture has 6 core components: Configuration, SessionFactory, Session, Transaction, Query, and Cache. SessionFactory is thread-safe and heavy, Session is light and request-scoped. Entities have 4 states: Transient, Persistent, Detached, Removed."

## Q2. Explain Differences between Session vs SessionFactory?

**Answer:**

| Feature | Session | SessionFactory |
|---------|---------|----------------|
| **Creation** | `sessionFactory.openSession()` | `new Configuration().buildSessionFactory()` |
| **Weight** | Lightweight | Heavyweight (expensive to create) |
| **Thread-Safety** | ❌ Not thread-safe | ✅ Thread-safe |
| **Lifetime** | Per request/transaction | Per application (one per DB) |
| **Cache** | Has First Level Cache (mandatory) | Has Second Level Cache (optional) |
| **Connection** | Uses one DB connection | Holds connection pool |
| **Purpose** | CRUD operations, queries | Factory for Sessions, metadata |
| **Close** | Must close after use | Close at application shutdown |

**Detailed Example:**

```java
// SessionFactory - Created once at application startup
@Configuration
public class HibernateConfig {
    private static SessionFactory sessionFactory;
    
    static {
        try {
            Configuration config = new Configuration();
            config.configure("hibernate.cfg.xml");
            config.addAnnotatedClass(Employee.class);
            config.addAnnotatedClass(Department.class);
            sessionFactory = config.buildSessionFactory();  // Heavy operation
        } catch (Throwable ex) {
            throw new ExceptionInInitializerError(ex);
        }
    }
    
    public static SessionFactory getSessionFactory() {
        return sessionFactory;
    }
}

// Session - Created per request
@Repository
public class EmployeeDAO {
    
    public Employee findById(Long id) {
        // Open new session for this operation
        try (Session session = HibernateConfig.getSessionFactory().openSession()) {
            return session.get(Employee.class, id);
        }  // Auto-closed
    }
    
    public void updateSalary(Long id, double newSalary) {
        Session session = HibernateConfig.getSessionFactory().openSession();
        Transaction tx = null;
        try {
            tx = session.beginTransaction();
            Employee emp = session.get(Employee.class, id);
            emp.setSalary(newSalary);
            tx.commit();  // Auto-flush before commit
        } catch (Exception e) {
            if (tx != null) tx.rollback();
            throw e;
        } finally {
            session.close();  // Always close session
        }
    }
}
```

**SessionFactory Methods:**
```java
SessionFactory sf = HibernateUtil.getSessionFactory();

// Open new session
Session session = sf.openSession();

// Get current session (thread-bound)
Session current = sf.getCurrentSession();  // Needs thread binding config

// Get metadata
Metamodel metamodel = sf.getMetamodel();
EntityType<Employee> entity = sf.getMetamodel().entity(Employee.class);

// Cache management
sf.getCache().evictAllEntities();  // Clear second-level cache
sf.getCache().evictEntityRegion(Employee.class);

// Statistics (for monitoring)
Statistics stats = sf.getStatistics();
stats.setStatisticsEnabled(true);
System.out.println("Entity load count: " + stats.getEntityLoadCount());
```

**Best Practices:**
```java
// ✅ Singleton pattern for SessionFactory
public class HibernateUtil {
    private static final SessionFactory sessionFactory = buildSessionFactory();
    
    private static SessionFactory buildSessionFactory() {
        return new Configuration().configure().buildSessionFactory();
    }
    
    public static SessionFactory getSessionFactory() {
        return sessionFactory;
    }
    
    public static void shutdown() {
        getSessionFactory().close();  // Close at app shutdown
    }
}

// ✅ Try-with-resources for Session (Java 7+)
// Note: Session doesn't implement AutoCloseable in older versions
// Use try-finally block instead
```

**Interview Tip:** "SessionFactory = **heavy, thread-safe, one per application** - creates Sessions. Session = **light, NOT thread-safe, one per transaction** - does actual DB work. Never share Session across threads. Always close Session in finally block."

## Q3. What is First Level Cache in Hibernate?

**Answer:**  
First Level Cache is **mandatory, session-scoped cache**. Every Hibernate Session has its own cache that stores entities during the session lifetime.

**Key Characteristics:**
- **Enabled by default** - cannot be disabled
- **Session-scoped** - cache lives only for that session
- **Automatic** - no configuration needed
- **Transparent** - hibernate manages it automatically

**How it works:**

```java
Session session = sessionFactory.openSession();
Transaction tx = session.beginTransaction();

// First query - hits database
Employee emp1 = session.get(Employee.class, 1L);
// SQL: SELECT * FROM employee WHERE id = 1

// Second query with same ID - FROM CACHE, no DB hit
Employee emp2 = session.get(Employee.class, 1L);
// No SQL executed! Returns cached object

System.out.println(emp1 == emp2);  // true (same object reference)

tx.commit();
session.close();
```

**Cache vs Database Interaction:**

```java
Session session = sessionFactory.openSession();

// 1. First query - Cache miss, goes to DB
Employee emp1 = session.get(Employee.class, 1L);  // SQL executed

// 2. Update - Updates cache and marks dirty
emp1.setSalary(75000);

// 3. Query same entity again - Cache hit
Employee emp2 = session.get(Employee.class, 1L);  // No SQL

// 4. Query different entity - Cache miss
Department dept = session.get(Department.class, 5L);  // SQL executed

// 5. Flush - Dirty checking, syncs to DB
tx.commit();  // UPDATE statement executed

session.close();
```

**Cache Operations:**

```java
Session session = sessionFactory.openSession();
Transaction tx = session.beginTransaction();

// Load entity
Employee emp = session.get(Employee.class, 1L);

// Evict single entity from cache
session.evict(emp);  // Removes from first-level cache

// Clear entire cache
session.clear();  // Removes all cached entities

// Contains check (Hibernate internal)
boolean contains = session.contains(emp);  // false after evict

// Refresh from database (overwrites cache)
session.refresh(emp);  // Reloads from DB, ignores cache

// Replicate entity to cache
session.replicate(emp, ReplicationMode.OVERWRITE);

tx.commit();
session.close();
```

**First Level Cache Example with HQL:**

```java
Session session = sessionFactory.openSession();

// HQL bypasses first-level cache for results, but cached entities are used
Employee emp1 = session.get(Employee.class, 1L);  // DB hit

// HQL returns proxy objects, but when accessed, checks cache
List<Employee> results = session.createQuery("FROM Employee WHERE id = 1", Employee.class).list();

// results.get(0) will return cached emp1 object
System.out.println(results.get(0) == emp1);  // true (same object)

session.close();
```

**First Level Cache Limitations:**
```java
// Problem: Cache is session-scoped, doesn't help across requests
Session session1 = sessionFactory.openSession();
Employee emp1 = session1.get(Employee.class, 1L);  // DB hit
session1.close();

Session session2 = sessionFactory.openSession();
Employee emp2 = session2.get(Employee.class, 1L);  // DB hit again!
session2.close();
// Two DB hits even though same entity!
```

**Visual Flow:**
```
First Request:
Session1.open() → Check L1 Cache (empty) → DB Query → Store in L1 Cache → Return

Same Request, Same Entity:
Session1.get() → Check L1 Cache (FOUND!) → Return cached (no DB)

Different Request:
Session2.open() → Check L1 Cache (empty - new session) → DB Query → Store → Return
```

**Interview Tip:** "First Level Cache = **session-scoped, mandatory, automatic**. Same session mein same entity multiple baar fetch karo toh DB hit nahi hoti. `session.evict()` se entity remove kar sakte ho, `session.clear()` se all remove. Different sessions share nahi karte cache."

## Q4. What is Second Level Cache in Hibernate?

**Answer:**  
Second Level Cache is **optional, SessionFactory-scoped cache** that can be shared across multiple sessions.

**Key Characteristics:**
- **Disabled by default** - must be configured
- **SessionFactory-scoped** - shared across all sessions
- **External cache provider** - EhCache, Hazelcast, Infinispan, Caffeine
- **Reduces database hits** across transactions/requests

**Configuration Steps:**

**Step 1: Add Dependencies (EhCache example)**
```xml
<dependency>
    <groupId>org.hibernate.orm</groupId>
    <artifactId>hibernate-ehcache</artifactId>
    <version>6.3.0.Final</version>
</dependency>
```

**Step 2: Configure hibernate.cfg.xml**
```xml
<!-- Enable second-level cache -->
<property name="hibernate.cache.use_second_level_cache">true</property>
<property name="hibernate.cache.region.factory_class">
    org.hibernate.cache.jcache.JCacheRegionFactory
</property>
<property name="hibernate.javax.cache.provider">
    org.ehcache.jsr107.EhcacheCachingProvider
</property>
<property name="hibernate.cache.use_query_cache">true</property>  <!-- Optional -->
```

**Step 3: Annotate Entities**
```java
@Entity
@Cacheable  // Marks entity as cacheable
@Cache(usage = CacheConcurrencyStrategy.READ_WRITE, region = "employeeCache")
public class Employee {
    @Id
    private Long id;
    private String name;
    private double salary;
    
    @ManyToOne
    @Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
    private Department department;
}
```

**How Second Level Cache Works:**

```java
// First session - hits DB, stores in L2 cache
Session session1 = sessionFactory.openSession();
Employee emp1 = session1.get(Employee.class, 1L);  // SQL executed
session1.close();  // L1 cleared, L2 still has data

// Second session - hits L2 cache, no DB hit
Session session2 = sessionFactory.openSession();
Employee emp2 = session2.get(Employee.class, 1L);  // NO SQL!
session2.close();

// Third session with different entity
Session session3 = sessionFactory.openSession();
Department dept = session3.get(Department.class, 5L);  // SQL (if not cached)
session3.close();
```

**Cache Concurrency Strategies:**

| Strategy | Use Case | Behavior |
|----------|----------|----------|
| **READ_ONLY** | Static data (never changes) | Most efficient, can't update |
| **NONSTRICT_READ_WRITE** | Rarely updated data | Weak consistency |
| **READ_WRITE** | Frequently updated | Strong consistency with soft locks |
| **TRANSACTIONAL** | Transactional environments | JTA, full ACID |

```java
// READ_ONLY - Best for reference data
@Entity
@Cacheable
@Cache(usage = CacheConcurrencyStrategy.READ_ONLY)
public class Country {
    @Id
    private String code;
    private String name;
    // Data never changes
}

// READ_WRITE - Most common for updatable entities
@Entity
@Cacheable
@Cache(usage = CacheConcurrencyStrategy.READ_WRITE, region = "employeeCache")
public class Employee {
    // Cache will be updated on write operations
}
```

**Cache Regions:**

```java
// Different regions for different entities
@Entity
@Cacheable
@Cache(usage = CacheConcurrencyStrategy.READ_WRITE, region = "employeeRegion")
public class Employee { }

@Entity
@Cacheable
@Cache(usage = CacheConcurrencyStrategy.READ_ONLY, region = "staticDataRegion")
public class Country { }

// Cache region management
sessionFactory.getCache().evictEntityRegion(Employee.class);  // Clear all employees
sessionFactory.getCache().evictEntityData(Employee.class, 1L);  // Clear specific
sessionFactory.getCache().evictQueryRegion();  // Clear query cache
```

**Query Cache (Second level for queries):**

```java
// Enable query cache (separate from entity cache)
// hibernate.cfg.xml
<property name="hibernate.cache.use_query_cache">true</property>

// Usage
Session session = sessionFactory.openSession();
List<Employee> employees = session.createQuery("FROM Employee WHERE dept = :dept", Employee.class)
    .setParameter("dept", "IT")
    .setCacheable(true)  // Enable query cache
    .setCacheRegion("employeeQueryCache")  // Optional region
    .list();
session.close();

// Same query executed again will use cache
Session session2 = sessionFactory.openSession();
List<Employee> employees2 = session2.createQuery("FROM Employee WHERE dept = :dept", Employee.class)
    .setParameter("dept", "IT")
    .setCacheable(true)
    .list();  // NO SQL!
session2.close();
```

**L2 Cache Eviction:**

```java
// Programmatic eviction
SessionFactory sf = sessionFactory;
Cache cache = sf.getCache();

// Evict specific entity
cache.evictEntityData(Employee.class, 1L);

// Evict all entities of a type
cache.evictEntityRegion(Employee.class);

// Evict entire second-level cache
cache.evictAllRegions();

// Through session
Session session = sessionFactory.openSession();
session.evict(Employee.class);  // Only L1, not L2
// For L2 eviction, must use SessionFactory
```

**Interview Tip:** "Second Level Cache = **SessionFactory-scoped, optional, shared across sessions**. Needs cache provider (EhCache, Hazelcast). Configure with `@Cacheable` and `@Cache` annotations. Different strategies: READ_ONLY, READ_WRITE, NONSTRICT_READ_WRITE, TRANSACTIONAL. Query cache caches query results, not entities."

## Q5. What are differences between First and Second Level Cache?

**Answer:**

| Feature | First Level Cache | Second Level Cache |
|---------|-------------------|---------------------|
| **Scope** | Session level | SessionFactory level |
| **Enabled by default** | ✅ Yes | ❌ No (needs configuration) |
| **Shared across sessions** | ❌ No | ✅ Yes |
| **Shared across transactions** | ❌ No | ✅ Yes |
| **Cache provider** | Built-in (internal) | External (EhCache, Hazelcast) |
| **Configuration required** | No | Yes |
| **Eviction** | Automatic at session close | Manual or time-based |
| **Memory** | Within session | Across application |
| **Entities stored** | All entities accessed | Only annotated with @Cacheable |
| **Dirty checking** | Yes (automatic) | No (manual invalidation) |
| **Performance impact** | Low | Moderate (serialization) |

**Detailed Comparison with Code:**

```java
// FIRST LEVEL CACHE - Session-scoped
Session session1 = sessionFactory.openSession();
Employee e1 = session1.get(Employee.class, 1L);  // DB hit
Employee e2 = session1.get(Employee.class, 1L);  // NO DB hit (L1 cache)
session1.close();

Session session2 = sessionFactory.openSession();
Employee e3 = session2.get(Employee.class, 1L);  // DB hit again! (L1 not shared)
session2.close();

// SECOND LEVEL CACHE - SessionFactory-scoped (after configuration)
Session s1 = sessionFactory.openSession();
Employee e1 = s1.get(Employee.class, 1L);  // DB hit
s1.close();

Session s2 = sessionFactory.openSession();
Employee e2 = s2.get(Employee.class, 1L);  // NO DB hit (from L2 cache)
s2.close();

Session s3 = sessionFactory.openSession();
Employee e3 = s3.get(Employee.class, 1L);  // NO DB hit (still from L2)
s3.close();
```

**Cache Interaction Flow:**

```java
// How both caches work together
Session session = sessionFactory.openSession();

// 1. Check L1 cache
// 2. If not in L1, check L2 cache
// 3. If not in L2, hit database

Employee emp = session.get(Employee.class, 1L);
// L1 miss → L2 miss → DB hit → Store in L2 → Store in L1

Employee empAgain = session.get(Employee.class, 1L);
// L1 hit → return (no L2 check even)

session.close();
// L1 cleared, L2 still has data

Session newSession = sessionFactory.openSession();
Employee emp2 = newSession.get(Employee.class, 1L);
// L1 miss → L2 hit → Store in L1 → return (no DB hit)
newSession.close();
```

**When to use which:**

```java
// L1 Cache - Always present, useful for:
// - Avoiding repeated queries in same transaction
// - Dirty checking for updates

// L2 Cache - Use when:
// - Data is read-heavy, write-light
// - Reference data (countries, status types)
// - Across multiple user requests
// - Application with many reads, few writes

@Entity
@Cacheable  // Enable L2
@Cache(usage = CacheConcurrencyStrategy.READ_ONLY)  // Perfect for reference data
public class Country {
    // Data rarely changes, read by many users
}

@Entity
@Cacheable
@Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
public class Product {
    // Moderate updates, high reads
}

// DON'T use L2 cache for:
@Entity
@Cacheable  // ❌ Bad for frequently updated entities
public class Transaction {
    // Every transaction changes data - cache overhead > benefit
}
```

**Memory Impact:**

```java
// L1 Cache memory - per session
// One user session: stores all entities touched in that session
// Session ends → memory freed

// L2 Cache memory - global
// Stores entities across all sessions
// Can grow large, needs eviction policy
<property name="hibernate.cache.region_prefix">myapp</property>
<property name="hibernate.cache.use_second_level_cache">true</property>
<property name="hibernate.cache.region.factory_class">org.hibernate.cache.ehcache.EhCacheRegionFactory</property>

// Configure eviction (ehcache.xml)
<cache name="employeeCache"
       maxElementsInMemory="10000"
       eternal="false"
       timeToIdleSeconds="300"
       timeToLiveSeconds="600"/>
```

**Interview Tip:** "L1 = session-scoped, mandatory, automatic. L2 = sessionFactory-scoped, optional, needs cache provider. L1 good for same transaction, L2 good for across transactions/requests. Use L2 for read-heavy data (reference data), avoid for frequently updated entities."

## Q6. What are differences between Hibernate and JDBC?

**Answer:**

| Feature | Hibernate | JDBC |
|---------|-----------|------|
| **Abstraction** | High-level ORM | Low-level API |
| **Boilerplate** | Minimal | Extensive |
| **SQL Generation** | Automatic (HQL/JPQL) | Manual |
| **Result Mapping** | Automatic to objects | Manual mapping |
| **Caching** | L1 + L2 cache | No caching |
| **Lazy Loading** | Supported | Not supported |
| **Connection Pool** | Built-in (with HikariCP) | Manual |
| **Portability** | High (DB independent) | Low (DB-specific SQL) |
| **Performance** | Overhead for simple ops | Faster for simple ops |
| **Learning Curve** | Steep | Moderate |

**Code Comparison:**

```java
// JDBC - Manual everything
public Employee getEmployeeJDBC(Long id) throws SQLException {
    String sql = "SELECT id, name, salary FROM employee WHERE id = ?";
    
    try (Connection conn = DriverManager.getConnection(url, user, pass);
         PreparedStatement ps = conn.prepareStatement(sql)) {
        
        ps.setLong(1, id);
        ResultSet rs = ps.executeQuery();
        
        if (rs.next()) {
            Employee emp = new Employee();
            emp.setId(rs.getLong("id"));
            emp.setName(rs.getString("name"));
            emp.setSalary(rs.getDouble("salary"));
            return emp;
        }
    }
    return null;
}

// Hibernate - Automatic mapping
public Employee getEmployeeHibernate(Long id) {
    try (Session session = sessionFactory.openSession()) {
        return session.get(Employee.class, id);
    }
}
```

**CRUD Operations Comparison:**

```java
// JDBC - INSERT
String sql = "INSERT INTO employee (id, name, salary) VALUES (?, ?, ?)";
try (PreparedStatement ps = conn.prepareStatement(sql)) {
    ps.setLong(1, emp.getId());
    ps.setString(2, emp.getName());
    ps.setDouble(3, emp.getSalary());
    ps.executeUpdate();
}

// Hibernate - INSERT
session.save(emp);  // Or session.persist(emp)

// JDBC - UPDATE
String sql = "UPDATE employee SET name = ?, salary = ? WHERE id = ?";
try (PreparedStatement ps = conn.prepareStatement(sql)) {
    ps.setString(1, emp.getName());
    ps.setDouble(2, emp.getSalary());
    ps.setLong(3, emp.getId());
    ps.executeUpdate();
}

// Hibernate - UPDATE
session.update(emp);  // Or just modify persistent object

// JDBC - DELETE
String sql = "DELETE FROM employee WHERE id = ?";
try (PreparedStatement ps = conn.prepareStatement(sql)) {
    ps.setLong(1, id);
    ps.executeUpdate();
}

// Hibernate - DELETE
session.delete(emp);
```

**Complex Query Comparison:**

```java
// JDBC - Joins manual
String sql = "SELECT e.id, e.name, e.salary, d.name as dept_name " +
             "FROM employee e JOIN department d ON e.dept_id = d.id " +
             "WHERE e.salary > ?";
// Manual result mapping required

// Hibernate - HQL
List<Object[]> results = session.createQuery(
    "SELECT e.id, e.name, e.salary, d.name FROM Employee e JOIN e.department d WHERE e.salary > :salary")
    .setParameter("salary", 50000)
    .list();

// Or better - object graph
List<Employee> employees = session.createQuery(
    "FROM Employee e JOIN FETCH e.department WHERE e.salary > :salary", Employee.class)
    .setParameter("salary", 50000)
    .list();
// department already loaded (eager fetch)
```

**Transaction Management:**

```java
// JDBC - Manual
Connection conn = null;
try {
    conn = DriverManager.getConnection(url, user, pass);
    conn.setAutoCommit(false);
    // operations
    conn.commit();
} catch (SQLException e) {
    if (conn != null) conn.rollback();
    throw e;
} finally {
    if (conn != null) conn.close();
}

// Hibernate - Declarative
Session session = sessionFactory.openSession();
Transaction tx = session.beginTransaction();
try {
    // operations
    tx.commit();
} catch (Exception e) {
    tx.rollback();
    throw e;
} finally {
    session.close();
}

// With Spring - Even simpler
@Transactional
public void updateEmployee(Employee emp) {
    // Spring manages transaction boundaries
    session.saveOrUpdate(emp);
}
```

**Performance Considerations:**

| Aspect | JDBC | Hibernate |
|--------|------|-----------|
| **Simple CRUD** | Faster | Slower (overhead) |
| **Batch Operations** | Manual | Built-in batching |
| **Caching** | None | L1 + L2 cache |
| **N+1 Problem** | You handle it | Can happen, but solvable |
| **Connection Pool** | Manual (HikariCP) | Built-in |
| **Statement Caching** | Manual | Automatic |

**When to use which:**

```java
// ✅ Use Hibernate when:
// - Complex domain model with relationships
// - Need object-oriented approach
// - Want database portability
// - Need caching and lazy loading
// - Large application with many entities

// ✅ Use JDBC when:
// - Simple CRUD operations
// - Performance-critical batch processing
// - Need fine-tuned SQL control
// - Stored procedures and DB-specific features
// - Prototyping or simple utilities

// Hybrid approach - best of both
@Entity
public class Employee {
    // Use Hibernate for normal operations
}

// For bulk operations, use native queries
session.createNativeQuery("INSERT INTO employee_log SELECT * FROM employee WHERE hire_date > :date")
    .setParameter("date", LocalDate.now().minusDays(30))
    .executeUpdate();
```

**Interview Tip:** "Hibernate = ORM (automatic mapping, caching, lazy loading) - better for complex domain models. JDBC = low-level API (faster, more control) - better for simple queries and bulk operations. Many apps use both: Hibernate for entity management, native queries for bulk operations."

## Q7. What are differences between get and load methods?

**Answer:**

| Feature | get() | load() |
|---------|-------|--------|
| **Database hit** | Immediate | Lazy (proxy) |
| **Returns if not found** | null | ObjectNotFoundException |
| **Proxy creation** | No | Yes (returns proxy) |
| **Use case** | When you know entity exists | When entity may not exist |
| **Performance** | Always hits DB | Can be lazy, more efficient |
| **Initialization** | Fully initialized | Lazy proxy, initialized on access |

**Code Example:**

```java
Session session = sessionFactory.openSession();

// GET - Hits database immediately
Employee emp1 = session.get(Employee.class, 1L);
// SQL executed immediately: SELECT * FROM employee WHERE id = 1
System.out.println(emp1.getName());  // Already loaded

// LOAD - Returns proxy, no DB hit yet
Employee emp2 = session.load(Employee.class, 2L);
// NO SQL executed yet! (returns proxy object)

System.out.println(emp2.getName());  // Now SQL executed
// SELECT * FROM employee WHERE id = 2

session.close();
```

**Proxy Behavior:**

```java
Session session = sessionFactory.openSession();

// load returns proxy, not the actual entity
Employee proxy = session.load(Employee.class, 1L);
System.out.println(proxy.getClass());  // Employee$HibernateProxy$... (proxy)

// Accessing any property (except id) triggers DB
Long id = proxy.getId();  // No DB hit (id is known)
String name = proxy.getName();  // DB hit now!

// get returns actual entity
Employee actual = session.get(Employee.class, 1L);
System.out.println(actual.getClass());  // Employee (actual class)

session.close();
```

**When Entity Doesn't Exist:**

```java
Session session = sessionFactory.openSession();

// GET - returns null
Employee emp = session.get(Employee.class, 999L);
if (emp == null) {
    System.out.println("Employee not found");
}

// LOAD - throws ObjectNotFoundException
try {
    Employee emp2 = session.load(Employee.class, 999L);
    // Exception NOT thrown here
    emp2.getName();  // ObjectNotFoundException thrown HERE!
} catch (ObjectNotFoundException e) {
    System.out.println("Employee not found");
}

session.close();
```

**Performance Comparison:**

```java
// GET - Always hits DB (even if entity already in cache)
Session s1 = sessionFactory.openSession();
Employee e1 = s1.get(Employee.class, 1L);  // SELECT
Employee e2 = s1.get(Employee.class, 1L);  // FROM L1 cache (no SELECT)
s1.close();

// LOAD - Can avoid DB if only ID needed
Session s2 = sessionFactory.openSession();
Employee proxy = s2.load(Employee.class, 1L);
Long id = proxy.getId();  // No SELECT!

// But to set relationship, no DB needed
Department dept = session.get(Department.class, 5L);
Employee emp = session.load(Employee.class, 1L);
emp.setDepartment(dept);  // No SELECT for Employee!
// Hibernate knows ID, just sets FK

session.close();
```

**Use Cases:**

```java
// ✅ Use get() when:
// - You need to access entity properties immediately
// - Entity may not exist (check for null)
// - You want eager loading

public Employee getEmployeeSafe(Long id) {
    Employee emp = session.get(Employee.class, id);
    if (emp == null) {
        throw new EntityNotFoundException("Employee " + id + " not found");
    }
    return emp;  // Fully loaded
}

// ✅ Use load() when:
// - You only need reference for relationship
// - You're sure entity exists
// - You want lazy loading for performance

// Best use case - setting relationships
public void assignDepartment(Long empId, Long deptId) {
    Employee emp = session.load(Employee.class, empId);  // Proxy, no SELECT
    Department dept = session.load(Department.class, deptId);  // Proxy
    emp.setDepartment(dept);  // Only FK set, no SELECTs!
    session.update(emp);
}

// ❌ DON'T do this - load then access outside session
public Employee getEmployeeWrong(Long id) {
    Session session = sessionFactory.openSession();
    Employee emp = session.load(Employee.class, id);  // Proxy
    session.close();
    return emp;  // LazyInitializationException when accessing properties!
}
```

**Inside SessionFactory.evict():**
```java
// get() and load() behavior with cache
Session session = sessionFactory.openSession();

// First call
Employee e1 = session.get(Employee.class, 1L);  // SELECT
session.evict(e1);  // Remove from L1 cache

// Second call - SELECT again (cache miss)
Employee e2 = session.get(Employee.class, 1L);  // SELECT

// load with cache
Employee e3 = session.load(Employee.class, 1L);  // Proxy, no SELECT
session.evict(e3);  // Evict proxy
Employee e4 = session.load(Employee.class, 1L);  // New proxy

session.close();
```

**Interview Tip:** "get() = **eager, hits DB immediately, returns null if not found**. load() = **lazy, returns proxy, throws ObjectNotFoundException if not found when accessed**. Use load() for setting relationships (FK only), use get() when you need actual data or need to check existence."

## Q8. Explain N+1 problem in Hibernate?

**Answer:**  
N+1 problem is a **performance anti-pattern** where one query fetches parent entities, then N additional queries fetch child entities for each parent.

**The Problem:**

```java
// Entities
@Entity
public class Department {
    @Id
    private Long id;
    private String name;
    
    @OneToMany(mappedBy = "department", fetch = FetchType.LAZY)
    private List<Employee> employees;
}

@Entity
public class Employee {
    @Id
    private Long id;
    private String name;
    
    @ManyToOne
    private Department department;
}

// Problematic code
List<Department> departments = session.createQuery("FROM Department", Department.class).list();
// 1 query: SELECT * FROM department (returns N departments)

for (Department dept : departments) {
    // For each department, execute additional query
    System.out.println(dept.getName() + " has " + dept.getEmployees().size() + " employees");
    // N queries: SELECT * FROM employee WHERE department_id = ?
}
// Total queries = 1 + N
```

**Visual Representation:**
```
N=5 departments:
Query 1: SELECT * FROM department                    (1 query)
Query 2: SELECT * FROM employee WHERE dept_id = 1    (2nd query)
Query 3: SELECT * FROM employee WHERE dept_id = 2    (3rd query)
Query 4: SELECT * FROM employee WHERE dept_id = 3    (4th query)
Query 5: SELECT * FROM employee WHERE dept_id = 4    (5th query)
Query 6: SELECT * FROM employee WHERE dept_id = 5    (6th query)
Total: 1 + 5 = 6 queries for 5 departments!
```

**Solutions to N+1 Problem:**

**Solution 1: JOIN FETCH (Eager loading in query)**

```java
// Single query with join
List<Department> departments = session.createQuery(
    "SELECT DISTINCT d FROM Department d LEFT JOIN FETCH d.employees", Department.class)
    .list();
// ONE query: SELECT d.*, e.* FROM department d LEFT JOIN employee e ON d.id = e.dept_id

// Now accessing employees causes no additional queries
for (Department dept : departments) {
    System.out.println(dept.getEmployees().size());  // No extra query!
}
```

**Solution 2: Entity Graph (JPA 2.1+)**

```java
// Define entity graph
@Entity
@NamedEntityGraph(name = "department.employees", attributeNodes = @NamedAttributeNode("employees"))
public class Department {
    // ...
}

// Usage
EntityGraph graph = session.getEntityGraph("department.employees");
List<Department> departments = session.createQuery("FROM Department", Department.class)
    .setHint("javax.persistence.fetchgraph", graph)
    .list();
// One query with join
```

**Solution 3: Batch Fetching**

```java
// Configure in entity mapping
@Entity
@BatchSize(size = 10)  // Hibernate specific
public class Department {
    @OneToMany(mappedBy = "department", fetch = FetchType.LAZY)
    private List<Employee> employees;
}

// Query becomes: 1 + (N/10) queries instead of 1+N
// With 100 departments, batch size 10 → 1 + 10 = 11 queries instead of 101
```

**Solution 4: Subselect Fetch**

```java
@Entity
public class Department {
    @OneToMany(mappedBy = "department")
    @Fetch(FetchMode.SUBSELECT)  // Hibernate specific
    private List<Employee> employees;
}

// Executes: 1 query for departments + 1 subselect for all employees
// SELECT * FROM department
// SELECT * FROM employee WHERE department_id IN (SELECT id FROM department)
```

**Solution 5: DTO Projection**

```java
// Instead of fetching entities, fetch only needed data
List<Object[]> results = session.createQuery(
    "SELECT d.name, COUNT(e) FROM Department d LEFT JOIN d.employees e GROUP BY d.id", Object[].class)
    .list();

for (Object[] row : results) {
    System.out.println(row[0] + " has " + row[1] + " employees");
}
// ONE query total!
```

**Detecting N+1 Problem:**

```java
// Enable SQL logging
// hibernate.cfg.xml
<property name="hibernate.show_sql">true</property>
<property name="hibernate.format_sql">true</property>

// Or use statistics
Statistics stats = sessionFactory.getStatistics();
stats.setStatisticsEnabled(true);

List<Department> depts = session.createQuery("FROM Department", Department.class).list();
for (Department d : depts) {
    d.getEmployees().size();
}

System.out.println("Query count: " + stats.getQueryExecutionCount());
// If query count > 1, you have N+1 problem!

// Use logging to see actual queries
// Look for repeated SELECT statements with different parameters
```

**Real-world Example:**

```java
// Problem: REST API endpoint returning departments with employees
@GetMapping("/departments")
public List<Department> getDepartments() {
    // ❌ This causes N+1 problem
    return departmentRepository.findAll();  // N+1 queries!
}

// Solution: Use JOIN FETCH
@GetMapping("/departments")
public List<Department> getDepartments() {
    // ✅ Single query
    return departmentRepository.findAllWithEmployees();
}

@Repository
public interface DepartmentRepository extends JpaRepository<Department, Long> {
    
    @Query("SELECT DISTINCT d FROM Department d LEFT JOIN FETCH d.employees")
    List<Department> findAllWithEmployees();
    
    // Or use EntityGraph
    @EntityGraph(attributePaths = {"employees"})
    List<Department> findAll();
}
```

**Comparison of Solutions:**

| Solution | Queries | Pros | Cons |
|----------|---------|------|------|
| **No fix** | 1+N | Simple code | Terrible performance |
| **JOIN FETCH** | 1 | Best for small data | Cartesian product issue, memory heavy |
| **Batch Fetching** | 1+(N/batch) | Good balance | Requires configuration |
| **Subselect** | 2 | Simple annotation | Not good for pagination |
| **DTO Projection** | 1 | Most efficient | Not entity objects |

**Interview Tip:** "N+1 problem: 1 query for parent + N queries for children. Solutions: JOIN FETCH (single query), Batch Fetching (reduces queries), Subselect (2 queries), DTO projection (most efficient). Enable SQL logging to detect. In Spring Boot, use `@EntityGraph` or `JOIN FETCH` in @Query."

## Q9. Difference between JPA vs Hibernate?

**Answer:**

| Feature | JPA | Hibernate |
|---------|-----|-----------|
| **Type** | Specification (interface) | Implementation |
| **Standard** | Java EE standard | Framework |
| **Portability** | High (change providers) | Low (vendor-specific) |
| **Caching** | Basic (@Cacheable) | Advanced (L1, L2, query cache) |

| **Criteria API** | JPA Criteria | Hibernate Criteria (deprecated) + JPA |

| **Custom Types** | Limited | Full support (UserType) |
| **Statistics** | No | Yes |

## Q10. When to use @Version and @Audited in Hibernate?

**Answer:**

### @Version - Optimistic Locking

`@Version` is used for **optimistic locking** to prevent lost updates in concurrent environments.

```java
@Entity
public class Employee {
    @Id
    private Long id;
    private String name;
    private double salary;
    
    @Version
    private Long version;  // Version number for optimistic locking
}
```

**How it works:**

```java
// Two users try to update same employee simultaneously

// User 1 - reads employee
Session s1 = sessionFactory.openSession();
Transaction tx1 = s1.beginTransaction();
Employee e1 = s1.get(Employee.class, 1L);  // version = 1

// User 2 - reads employee (different session)
Session s2 = sessionFactory.openSession();
Transaction tx2 = s2.beginTransaction();
Employee e2 = s2.get(Employee.class, 1L);  // version = 1

// User 1 updates and commits
e1.setSalary(60000);
tx1.commit();
// SQL: UPDATE employee SET salary=60000, version=2 WHERE id=1 AND version=1
// Success: 1 row updated

// User 2 updates and tries to commit
e2.setSalary(65000);
tx2.commit();
// SQL: UPDATE employee SET salary=65000, version=2 WHERE id=1 AND version=1
// Success: 0 rows updated (version no longer 1)
// Hibernate throws: OptimisticLockException
```

**Version Field Types:**

```java
// Supported types
@Version
private int version;      // integer

@Version
private long version;     // long

@Version
private short version;    // short

@Version
private Timestamp lastUpdate;  // timestamp (less reliable)

@Version
private Instant lastModified;  // Java 8 Instant
```

**Customizing Version Behavior:**

```java
@Entity
@OptimisticLocking(type = OptimisticLockType.DIRTY)  // Only check dirty fields
public class Product {
    @Id
    private Long id;
    
    @Version
    private Long version;
    
    private String name;
    private double price;
    
    @OptimisticLock(excluded = true)  // Don't check this field for conflicts
    private String description;
}

// Or use ALL - check all fields
@OptimisticLocking(type = OptimisticLockType.ALL)
public class Account { }
```

**Handling OptimisticLockException:**

```java
@Transactional
public void updateEmployee(Long id, double newSalary) {
    try {
        Employee emp = employeeRepository.findById(id);
        emp.setSalary(newSalary);
        employeeRepository.save(emp);
    } catch (OptimisticLockException e) {
        // Retry logic
        Employee fresh = employeeRepository.findById(id);
        fresh.setSalary(newSalary);
        employeeRepository.save(fresh);
        // Or merge changes
        // Or notify user
    }
}

// With retry pattern
@Retryable(value = OptimisticLockException.class, maxAttempts = 3, backoff = @Backoff(delay = 100))
@Transactional
public void updateWithRetry(Long id, double newSalary) {
    Employee emp = employeeRepository.findById(id);
    emp.setSalary(newSalary);
}
```

### @Audited - Envers (Audit Logging)

`@Audited` automatically tracks **entity changes** over time.

**Configuration:**

```xml
<!-- Add dependency -->
<dependency>
    <groupId>org.hibernate</groupId>
    <artifactId>hibernate-envers</artifactId>
</dependency>

<!-- Configure -->
<property name="hibernate.hbm2ddl.auto">update</property>
<property name="org.hibernate.envers.audit_table_prefix">aud_</property>
<property name="org.hibernate.envers.audit_table_suffix">_audit</property>
```

**Usage:**

```java
@Entity
@Audited  // Audit all fields
public class Employee {
    @Id
    private Long id;
    private String name;
    private double salary;
    
    @NotAudited  // Don't audit this field
    private String temporaryData;
}

// Audit specific fields only
@Entity
@Audited
public class Account {
    @Id
    private Long id;
    
    @Audited
    private double balance;  // Only audit balance changes
    
    @NotAudited
    private String lastLoginIp;
}
```

**Querying Audit History:**

```java

AuditReader reader = AuditReaderFactory.get(entityManager);

// Get entity revisions

// Get entity at specific revision

// Get entity as it was at specific date

// Get all revisions with changes
List<Employee> revisions = reader.createQuery()
    .forRevisionsOfEntity(Employee.class, true, true)
    .add(AuditEntity.id().eq(1L))
    .getResultList();

// Get revision details

    .forRevisionsOfEntity(Employee.class, false, true)
    .addProjection(AuditEntity.revisionNumber())
    .addProjection(AuditEntity.revisionType())
    .addProjection(AuditEntity.revisionProperty("timestamp"));
```

**Custom Revision Entity:**

```java

```

**Use Cases Comparison:**

| Feature | @Version | @Audited |
|---------|----------|----------|
| **Purpose** | Prevent concurrent updates | Track history |
| **Storage** | Single column in same table | Separate audit tables |
| **Performance impact** | Minimal (WHERE version check) | High (writes to audit tables) |
| **Data stored** | Only version number | Complete entity snapshots |
| **Retrieval** | Automatic by Hibernate | Via AuditReader API |
| **Use when** | High concurrency | Compliance, audit trail |

**When to use @Version:**
```java
// ✅ Use @Version when:
// - Multiple users update same records
// - Financial transactions
// - Inventory management
// - Any lost-update-sensitive operation

// Example: Banking
@Entity
public class BankAccount {
    @Version
    private Long version;
    private double balance;
}

// ❌ Don't use when:
// - Read-only data
// - Single-user applications
// - Heavy write conflicts (use pessimistic locking instead)
```

**When to use @Audited:**
```java
// ✅ Use @Audited when:
// - Regulatory compliance (SOX, GDPR)
// - Need "who changed what and when"
// - Audit trail required
// - Debugging data changes

// Example: Compliance requirement
@Entity
@Audited
public class TradeOrder {
    private Long id;
    private String trader;
    private double amount;
    private String status;
    // Every change tracked for audit
}

// ❌ Don't use when:
// - High-volume transactions (performance impact)
// - Temporary data
// - Logging already exists
// - Storage constraints
```

**Interview Tip:** "@Version = **optimistic locking** - prevents lost updates. Hibernate checks version before update, throws OptimisticLockException if mismatch. @Audited = **Envers** - automatically tracks entity changes in separate audit tables. Use @Version for concurrency control, @Audited for compliance and history tracking."

## Q11. What type of cascades does JPA support?

**Answer:**  
Cascade operations propagate entity state changes from parent to child entities.

**Cascade Types:**

| Cascade Type | Description | When to use |
|--------------|-------------|-------------|
| **PERSIST** | Save parent → also save child | Parent owns child lifecycle |
| **MERGE** | Update parent → also update child | Child changes with parent |
| **REMOVE** | Delete parent → also delete child | Child can't exist without parent |
| **REFRESH** | Reload parent → also reload child | Keep consistency |
| **DETACH** | Detach parent → also detach child | Clear persistence context |
| **ALL** | All of the above | Full lifecycle ownership |

**Examples:**

```java
// 1. CascadeType.PERSIST
@Entity
public class Department {
    @Id
    private Long id;
    
    @OneToMany(mappedBy = "department", cascade = CascadeType.PERSIST)
    private List<Employee> employees = new ArrayList<>();
}

// When you save department, employees are automatically saved
Department dept = new Department();
Employee emp = new Employee();
emp.setDepartment(dept);
dept.getEmployees().add(emp);

entityManager.persist(dept);  // Both department AND employee are saved

// 2. CascadeType.REMOVE
@Entity
public class Post {
    @Id
    private Long id;
    
    @OneToMany(mappedBy = "post", cascade = CascadeType.REMOVE)
    private List<Comment> comments;
}

// Deleting post automatically deletes all comments
Post post = entityManager.find(Post.class, 1L);
entityManager.remove(post);  // Post and all its comments deleted

// 3. CascadeType.ALL (most common)
@Entity
public class Order {
    @Id
    private Long id;
    
    @OneToMany(mappedBy = "order", cascade = CascadeType.ALL, orphanRemoval = true)
    private List<OrderItem> items = new ArrayList<>();
}

// All operations propagate
Order order = new Order();
OrderItem item = new OrderItem();
order.addItem(item);

entityManager.persist(order);  // Saves order AND items
entityManager.remove(order);   // Deletes order AND items
```

**Cascade on Different Relationships:**

```java
// OneToMany
@Entity
public class Parent {
    @OneToMany(mappedBy = "parent", cascade = CascadeType.ALL)
    private List<Child> children;
}

// ManyToOne
@Entity
public class Child {
    @ManyToOne(cascade = CascadeType.PERSIST)  // Less common
    private Parent parent;
}

// OneToOne
@Entity
public class User {
    @OneToOne(cascade = CascadeType.ALL)
    private Address address;  // User owns address lifecycle
}

// ManyToMany (be careful - avoid cascade delete)
@Entity
public class Student {
    @ManyToMany(cascade = {CascadeType.PERSIST, CascadeType.MERGE})
    private List<Course> courses;
}
```

**Orphan Removal (Not cascade but related):**

```java
@Entity
public class Parent {
    @OneToMany(mappedBy = "parent", cascade = CascadeType.ALL, orphanRemoval = true)
    private List<Child> children = new ArrayList<>();
    
    public void removeChild(Child child) {
        children.remove(child);
        child.setParent(null);  // Orphan will be deleted automatically
    }
}

// When you remove child from collection, it's deleted from DB
parent.getChildren().remove(child);  // Child deleted on flush
```

**Real-world Example:**

```java

    @Id
    private Long id;
    private String title;

    private List<Comment> comments = new ArrayList<>();
    
    // Tags - PERSIST only (don't delete tags when post deleted)
    @ManyToMany(cascade = CascadeType.PERSIST)
    private Set<Tag> tags = new HashSet<>();
    
    // Author - MERGE only
    @ManyToOne(cascade = CascadeType.MERGE)
    private User author;
}
```

**Cascade Pitfalls:**

```java
// ❌ DANGER - Cascade delete on many-to-many
@Entity
public class Student {
    @ManyToMany(cascade = CascadeType.REMOVE)  // BAD!
    private List<Course> courses;
}
// Deleting student would delete courses that other students also take!

// ✅ CORRECT - No cascade on many-to-many
@Entity
public class Student {
    @ManyToMany
    private List<Course> courses;
}

// ❌ DANGER - Cascading on both sides (circular)
@Entity
public class A {
    @OneToMany(mappedBy = "a", cascade = CascadeType.ALL)
    private List<B> bs;
}

@Entity
public class B {
    @ManyToOne(cascade = CascadeType.ALL)  // BAD - circular cascade
    private A a;
}
```

**Best Practices:**

```java
// 1. Use CascadeType.ALL for composition (child cannot exist without parent)
@Entity
public class Order {
    @OneToMany(mappedBy = "order", cascade = CascadeType.ALL, orphanRemoval = true)
    private List<OrderLine> lines;
}

// 2. Use CascadeType.PERSIST + MERGE for aggregation
@Entity
public class Employee {
    @ManyToOne(cascade = {CascadeType.PERSIST, CascadeType.MERGE})
    private Department department;
}

// 3. No cascade for many-to-many
@Entity
public class Product {
    @ManyToMany
    private List<Category> categories;  // No cascade
}

// 4. Explicitly manage collections
public void addComment(Comment comment) {
    comments.add(comment);
    comment.setPost(this);  // Set both sides
}

public void removeComment(Comment comment) {
    comments.remove(comment);
    comment.setPost(null);
}
```

**Interview Tip:** "Cascade types: PERSIST, MERGE, REMOVE, REFRESH, DETACH, ALL. Use ALL for composition (parent owns child lifecycle), PERSIST+MERGE for aggregation, no cascade for many-to-many. orphanRemoval=true deletes child when removed from collection. Always set both sides of bidirectional relationship."

## Q12. How to change eager loading to lazy loading in Spring Boot?

**Answer:**  
By default, `@ManyToOne` and `@OneToOne` are **EAGER**, `@OneToMany` and `@ManyToMany` are **LAZY**. Here's how to change them.

**1. Using fetch attribute:**

```java
// Change EAGER (default for ManyToOne) to LAZY
@Entity
public class Employee {
    @ManyToOne(fetch = FetchType.LAZY)  // Change from EAGER to LAZY
    @JoinColumn(name = "dept_id")
    private Department department;
}

// Change LAZY (default for OneToMany) to EAGER
@Entity
public class Department {
    @OneToMany(mappedBy = "department", fetch = FetchType.EAGER)  // Change from LAZY to EAGER
    private List<Employee> employees;
}
```

**2. Global configuration (application.properties):**

```properties
spring.jpa.properties.hibernate.format_sql=true
```

**3. Using @EntityGraph for dynamic fetching:**

```java
@Entity
@NamedEntityGraph(name = "Employee.department", attributeNodes = @NamedAttributeNode("department"))
public class Employee {
    @Id
    private Long id;

}
```

## Q13. LazyInitializationException, What it is and the best way to fix it?

**Answer:**  
`LazyInitializationException` occurs when you try to access a **lazy-loaded relationship** outside an **open Hibernate Session** (or JPA Persistence Context).

**The Problem:**

```java
// Controller (no transaction)
@RestController
public class EmployeeController {
    
    @GetMapping("/employee/{id}")
    public Employee getEmployee(@PathVariable Long id) {
        Employee emp = employeeService.getEmployee(id);
        // Session is closed after service method returns
        return emp;
    }
}

// Service (transactional)
@Service
public class EmployeeService {
    
    @Autowired
    private EmployeeRepository repository;
    
    @Transactional  // Session open here
    public Employee getEmployee(Long id) {
        Employee emp = repository.findById(id).orElseThrow();
        // Session closes when method exits
        return emp;  // Proxy for department returned
    }
}

// When Jackson tries to serialize
// employee.getDepartment().getName() → LazyInitializationException!
```

**Solutions:**

**Solution 1: Keep Transaction Open (Anti-pattern but works)**

```java
// Extend transaction to controller (NOT recommended)
@RestController
@Transactional  // ❌ Keep session open for whole request
public class EmployeeController {
    // This works but bad practice - transaction per request
}
```

**Solution 2: Fetch Eagerly in Query (Recommended)**

```java
@Repository
public interface EmployeeRepository extends JpaRepository<Employee, Long> {
    
    // Use JOIN FETCH
    @Query("SELECT e FROM Employee e JOIN FETCH e.department WHERE e.id = :id")
    Optional<Employee> findByIdWithDepartment(@Param("id") Long id);
    
    // Use EntityGraph
    @EntityGraph(attributePaths = {"department"})
    Optional<Employee> findById(Long id);
}

@Service
public class EmployeeService {
    public Employee getEmployee(Long id) {
        return repository.findByIdWithDepartment(id).orElseThrow();
        // Department already loaded - no LazyInitException
    }
}
```

**Solution 3: Use DTOs (Best Practice)**

```java

    private Long id;
    private String name;
    private String departmentName;

        this.id = id;
        this.name = name;
        this.departmentName = departmentName;
```

## Q14. How will you save millions of data into database using Hibernate?

**Answer:**  
Saving millions of records with Hibernate requires **batch processing** and **optimization techniques**.

**Problem with default approach:**

```java
// ❌ NEVER do this - will cause OOM and performance issues
for (int i = 0; i < 1_000_000; i++) {
    Employee emp = new Employee("Emp" + i, i * 1000);
    session.save(emp);  // Each save = separate INSERT + transaction
}
// Result: 
// - 1 million INSERT statements
// - 1 million transactions
// - OutOfMemoryError (session grows)
// - Takes hours/days
```

**Solution 1: JDBC Batching with Hibernate**

```xml
<!-- hibernate.cfg.xml -->
<property name="hibernate.jdbc.batch_size">50</property>
<property name="hibernate.order_inserts">true</property>
<property name="hibernate.order_updates">true</property>
<property name="hibernate.jdbc.batch_versioned_data">true</property>
```

```java
// Batch processing
Session session = sessionFactory.openSession();
Transaction tx = session.beginTransaction();
int batchSize = 50;

for (int i = 0; i < 1_000_000; i++) {
    Employee emp = new Employee("Emp" + i, i * 1000);
    session.save(emp);
    
    if (i % batchSize == 0 && i > 0) {
        session.flush();   // Send batch to DB
        session.clear();   // Clear persistence context (free memory)
    }
}

tx.commit();
session.close();
```

**Solution 2: StatelessSession (Best for Bulk Operations)**

```java
// StatelessSession - No caching, no dirty checking, no L1 cache
StatelessSession statelessSession = sessionFactory.openStatelessSession();
Transaction tx = statelessSession.beginTransaction();

for (int i = 0; i < 1_000_000; i++) {
    Employee emp = new Employee("Emp" + i, i * 1000);
    statelessSession.insert(emp);  // Direct INSERT, no cache
    
    if (i % 1000 == 0) {
        statelessSession.flush();
    }
}

tx.commit();
statelessSession.close();

// Performance: ~50,000 records/second
```

**Solution 3: Native JDBC Batch (Fastest)**

```java
// Bypass Hibernate entirely for bulk inserts
Session session = sessionFactory.openSession();
Connection conn = session.doReturningWork(connection -> connection);

String sql = "INSERT INTO employee (name, salary) VALUES (?, ?)";
PreparedStatement ps = conn.prepareStatement(sql);
int batchSize = 1000;

for (int i = 0; i < 1_000_000; i++) {
    ps.setString(1, "Emp" + i);
    ps.setDouble(2, i * 1000);
    ps.addBatch();
    
    if (i % batchSize == 0 && i > 0) {
        ps.executeBatch();
        ps.clearBatch();
    }
}
ps.executeBatch();  // Final batch

session.close();
// Performance: ~100,000 records/second
```

**Complete Production Solution:**

```java
@Service
public class BulkDataService {
    
    @Autowired
    private SessionFactory sessionFactory;
    
    @Autowired
    private JdbcTemplate jdbcTemplate;
    
    // Method 1: For moderate bulk (100k-500k)
    public void saveWithHibernateBatch(List<Employee> employees) {
        Session session = sessionFactory.openSession();
        Transaction tx = session.beginTransaction();
        int batchSize = 50;
        
        for (int i = 0; i < employees.size(); i++) {
            session.save(employees.get(i));
            
            if (i % batchSize == 0 && i > 0) {
                session.flush();
                session.clear();
            }
        }
        
        tx.commit();
        session.close();
    }
    
    // Method 2: For large bulk (500k-2M)
    public void saveWithStatelessSession(List<Employee> employees) {
        StatelessSession statelessSession = sessionFactory.openStatelessSession();
        Transaction tx = statelessSession.beginTransaction();
        
        for (Employee emp : employees) {
            statelessSession.insert(emp);
        }
        
        tx.commit();
        statelessSession.close();
    }
    
    // Method 3: For huge bulk (2M+)
    public void saveWithJdbcBatch(List<Employee> employees) {
        String sql = "INSERT INTO employee (name, salary, department_id) VALUES (?, ?, ?)";
        
        jdbcTemplate.batchUpdate(sql, employees, 1000, (ps, emp) -> {
            ps.setString(1, emp.getName());
            ps.setDouble(2, emp.getSalary());
            ps.setLong(3, emp.getDepartment().getId());
        });
    }
    
    // Method 4: Using partition and parallel processing
    public void saveInParallel(List<Employee> employees) {
        int partitionSize = 100_000;
        List<List<Employee>> partitions = Lists.partition(employees, partitionSize);
        
        ExecutorService executor = Executors.newFixedThreadPool(4);
        List<CompletableFuture<Void>> futures = partitions.stream()
            .map(partition -> CompletableFuture.runAsync(() -> 
                saveWithStatelessSession(partition), executor))
            .collect(Collectors.toList());
        
        CompletableFuture.allOf(futures.toArray(new CompletableFuture[0])).join();
        executor.shutdown();
    }
}
```
