## **Q1. What are Microservices?**

### 📌 **Simple Definition**

**Microservices = Ek bade application ko chhote, independent services mein todna, jo apne aap deploy, scale, aur develop ho sakte hain**

**Real-life analogy:**
- **Monolith:** Ek restaurant jahan ek hi chef sab kuch banata hai (cooking, billing, cleaning)
- **Microservices:** Food court jahan alag-alag stalls hain - pizza stall, ice cream stall, juice stall (independent but together form complete experience)

## **Q2. Features of Microservices**

```yaml
1. Independent Deployability:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ Ek service change kiya toh sirf wahi redeploy hogi
✅ Baaki services affected nahi hoti
✅ Faster time to market

2. Decentralized Data Management:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ Har service ki apni database hoti hai
✅ Service apne data ka owner hai
✅ Loose coupling

3. Technology Heterogeneity:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ User Service → Java + PostgreSQL
✅ Payment Service → Go + MongoDB
✅ Notification → Node.js + Redis
✅ Har service apni best technology choose kar sakti hai

4. Fault Isolation:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ Ek service down hui toh baaki kaam karte rahenge
✅ Payment service down → Order still placed (just payment pending)

5. Scalability:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ Sirf jiski zyada load hai wahi scale karo
✅ Order service ke 10 instances, User service ke 2 instances

6. Team Autonomy:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ Har team apni service ke liye responsible
✅ Independent development and release cycles
```

## **Q3. Differences: Monolith vs Microservices**

| Aspect | Monolith | Microservices |
|--------|----------|---------------|
| **Deployment** | Whole application ek saath deploy | Individual services independently |
| **Scalability** | Poora application scale karna padta hai | Sirf required services scale hoti hain |
| **Development** | Large team, slow, coordination heavy | Small teams, fast, independent |
| **Technology** | Single technology stack | Polyglot (multiple technologies) |
| **Database** | Single database | Database per service |
| **Fault Tolerance** | Ek module crash = poori app down | Ek service down = baaki kaam karegi |
| **Testing** | End-to-end testing complex | Service testing easier |
| **Complexity** | Simple to develop, complex to scale | Complex to develop, easy to scale |
| **Startup Time** | Slow (whole app starts) | Fast (only required services) |
| **Communication** | In-memory method calls | Network calls (HTTP/RPC) |
| **Best For** | Small teams, simple applications | Large teams, complex applications |

## **Q4. What is Inter Process Communication (IPC)?**

**IPC = Different processes/services ke beech communication karne ka mechanism**

```yaml

2. Asynchronous (Non-blocking):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

- Event Bus
- Publish/Subscribe
```

```java
// ========== 1. REST Communication (Spring Boot) ==========
@Service
public class OrderService {
    
    @Autowired
    private RestTemplate restTemplate;
    
    public Order createOrder(OrderRequest request) {
        // Call Payment Service via REST
        PaymentResponse payment = restTemplate.postForObject(
            "http://payment-service/api/payments",
            request.getPaymentInfo(),
            PaymentResponse.class
        );
        
        // Call Inventory Service
        InventoryResponse inventory = restTemplate.getForObject(
            "http://inventory-service/api/inventory/" + request.getProductId(),
            InventoryResponse.class
        );
        
        return orderRepository.save(new Order(request, payment, inventory));
    }
}

// ========== 2. gRPC Communication ==========
// Define in .proto file
service OrderService {
    rpc CreateOrder(OrderRequest) returns (OrderResponse);
}

// Java Implementation
public class OrderServiceImpl extends OrderServiceGrpc.OrderServiceImplBase {
    
    private final PaymentServiceGrpc.PaymentServiceBlockingStub paymentStub;
    
    @Override
    public void createOrder(OrderRequest request, StreamObserver<OrderResponse> responseObserver) {
        // Call Payment Service via gRPC
        PaymentResponse payment = paymentStub.processPayment(
            PaymentRequest.newBuilder()
                .setAmount(request.getAmount())
                .build()
        );
        
        OrderResponse response = OrderResponse.newBuilder()
            .setOrderId(UUID.randomUUID().toString())
            .setStatus("CREATED")
            .build();
        
        responseObserver.onNext(response);
        responseObserver.onCompleted();
    }
}

// ========== 3. Asynchronous (Kafka) ==========
@Service
public class OrderEventPublisher {
    
    @Autowired
    private KafkaTemplate<String, OrderEvent> kafkaTemplate;
    
    public void publishOrderCreated(Order order) {
        OrderEvent event = OrderEvent.builder()
            .eventId(UUID.randomUUID().toString())
            .eventType("ORDER_CREATED")
            .orderId(order.getId())
            .timestamp(Instant.now())
            .build();
        
        kafkaTemplate.send("order-events", event);
    }
}

// Consumer Service
@Component
public class PaymentEventConsumer {
    
    @KafkaListener(topics = "order-events")
    public void handleOrderCreated(OrderEvent event) {
        if ("ORDER_CREATED".equals(event.getEventType())) {
            // Process payment for this order
            paymentService.processPayment(event.getOrderId());
        }
    }
}
```

```java
// ========== Payment Service (Complete) ==========

// 1. Model/Entity
@Entity
@Table(name = "payments")
public class Payment {
    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    private String id;
    
    private String orderId;
    private BigDecimal amount;
    private String status; // PENDING, COMPLETED, FAILED
    private LocalDateTime createdAt;
    private LocalDateTime processedAt;
}

// 2. DTO for API
@Data
@Builder
public class PaymentRequest {
    @NotNull
    private String orderId;
    
    @NotNull
    @Positive
    private BigDecimal amount;
    
    @NotBlank
    private String paymentMethod; // CREDIT_CARD, DEBIT_CARD, UPI
    
    private CardDetails cardDetails;
}

// 3. REST Controller
@RestController
@RequestMapping("/api/payments")
@Slf4j
public class PaymentController {
    
    @Autowired
    private PaymentService paymentService;
    
    @PostMapping
    public ResponseEntity<PaymentResponse> processPayment(
            @Valid @RequestBody PaymentRequest request,
            @RequestHeader(value = "Idempotency-Key", required = true) String idempotencyKey) {
        
        PaymentResponse response = paymentService.processPayment(request, idempotencyKey);
        return ResponseEntity.status(HttpStatus.CREATED).body(response);
    }
    
    @GetMapping("/{id}")
    public ResponseEntity<PaymentResponse> getPayment(@PathVariable String id) {
        PaymentResponse response = paymentService.getPayment(id);
        return ResponseEntity.ok(response);
    }
}

// 4. Service Layer
@Service
@Slf4j
public class PaymentService {
    
    @Autowired
    private PaymentRepository paymentRepository;
    
    @Autowired
    private KafkaTemplate<String, PaymentEvent> kafkaTemplate;
    
    @Autowired
    private RedisTemplate<String, PaymentResponse> cacheTemplate;
    
    @Transactional
    public PaymentResponse processPayment(PaymentRequest request, String idempotencyKey) {
        // Check idempotency
        Payment existing = paymentRepository.findByIdempotencyKey(idempotencyKey);
        if (existing != null) {
            return mapToResponse(existing);
        }
        
        // Create payment record
        Payment payment = Payment.builder()
            .id(UUID.randomUUID().toString())
            .orderId(request.getOrderId())
            .amount(request.getAmount())
            .status("PROCESSING")
            .createdAt(LocalDateTime.now())
            .idempotencyKey(idempotencyKey)
            .build();
        
        payment = paymentRepository.save(payment);
        
        try {
            // Call external payment gateway
            GatewayResponse gatewayResponse = callPaymentGateway(request);
            
            // Update payment status
            payment.setStatus(gatewayResponse.isSuccess() ? "COMPLETED" : "FAILED");
            payment.setProcessedAt(LocalDateTime.now());
            payment.setTransactionId(gatewayResponse.getTransactionId());
            payment = paymentRepository.save(payment);
            
            // Publish event
            PaymentEvent event = PaymentEvent.builder()
                .paymentId(payment.getId())
                .orderId(payment.getOrderId())
                .status(payment.getStatus())
                .timestamp(Instant.now())
                .build();
            
            kafkaTemplate.send("payment-events", event);
            
            // Cache response
            cacheTemplate.opsForValue().set(payment.getId(), mapToResponse(payment), 
                Duration.ofMinutes(30));
            
            return mapToResponse(payment);
            
        } catch (Exception e) {
            payment.setStatus("FAILED");
            payment.setErrorMessage(e.getMessage());
            paymentRepository.save(payment);
            throw new PaymentProcessingException("Payment failed", e);
        }
    }
}

// 5. Configuration
@Configuration
public class PaymentServiceConfig {
    
    @Bean
    public RestTemplate restTemplate() {
        return new RestTemplateBuilder()
            .setConnectTimeout(Duration.ofSeconds(5))
            .setReadTimeout(Duration.ofSeconds(10))
            .build();
    }
    
    @Bean
    public CircuitBreakerFactory circuitBreakerFactory() {
        return new Resilience4JCircuitBreakerFactory();
    }
}
```

## **Q8. When to Choose Microservices over Monolith?**

### ✅ **Choose Microservices When:**

```yaml
1. Large Team (50+ developers):
   - Multiple teams can work independently
   - Each team owns their services

2. Frequent Deployments (multiple times/day):
   - Independent deployment possible
   - Reduced risk

3. Different Scaling Requirements:
   - Order service needs 10 instances
   - User service needs 2 instances

4. Technology Diversity:
   - Different services need different tech stacks
   - ML service in Python, main app in Java

5. High Availability Requirements:
   - 99.99% uptime needed
   - Fault isolation critical

6. Organizational Structure (Conway's Law):
   - Teams organized by business capabilities
   - Each team owns end-to-end feature
```

### ❌ **Avoid Microservices When:**

```yaml
1. Small Team (<10 developers):
   - Overhead of distributed systems
   - DevOps complexity too high

2. Simple Application:
   - Blog, Portfolio, Small e-commerce
   - Monolith is sufficient

3. Tight Deadlines:
   - Microservices take longer to setup
   - Infrastructure complexity

4. No DevOps Experience:
   - Need CI/CD, container orchestration
   - Monitoring and logging complexity

5. Startup Phase:
   - Focus on product-market fit
   - Monolith allows faster iteration
```

## **Q9. How to Handle Data Consistency?**

### 🔄 **Data Consistency Patterns**

```yaml
Problem: Distributed transactions across services
  Order Service (OrderDB) + Payment Service (PaymentDB) + Inventory Service (InventoryDB)

Solutions:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. Saga Pattern (Most Common):
   - Distributed transaction as sequence of local transactions
   - Each step has compensating transaction

2. Eventual Consistency:
   - Accept temporary inconsistency
   - System becomes consistent over time

3. CQRS (Command Query Responsibility Segregation):
   - Separate read and write models
   - Event sourcing for consistency

4. Two-Phase Commit (2PC):
   - Traditional distributed transaction
   - Heavy, not recommended for microservices
```

### 💻 **Saga Pattern Implementation**

```java
// ========== Saga Pattern - Order Processing ==========

@Service
public class OrderSagaOrchestrator {
    
    @Autowired
    private OrderService orderService;
    
    @Autowired
    private PaymentServiceClient paymentClient;
    
    @Autowired
    private InventoryServiceClient inventoryClient;
    
    @Autowired
    private ShippingServiceClient shippingClient;
    
    @Transactional
    public void createOrderSaga(OrderRequest request) {
        // Step 1: Create Order (with status PENDING)
        Order order = orderService.createOrder(request);
        
        try {
            // Step 2: Reserve Inventory
            InventoryResponse inventory = inventoryClient.reserveInventory(
                request.getProductId(), request.getQuantity());
            
            if (!inventory.isSuccess()) {
                throw new InventoryException("Insufficient inventory");
            }
            
            // Step 3: Process Payment
            PaymentResponse payment = paymentClient.processPayment(
                order.getId(), request.getAmount());
            
            if (!payment.isSuccess()) {
                throw new PaymentException("Payment failed");
            }
            
            // Step 4: Update Order Status to CONFIRMED
            orderService.updateOrderStatus(order.getId(), "CONFIRMED");
            
            // Step 5: Schedule Shipping (Async)
            shippingClient.scheduleShipping(order.getId());
            
        } catch (InventoryException e) {
            // Compensation: Cancel Order
            orderService.updateOrderStatus(order.getId(), "FAILED");
            throw e;
            
        } catch (PaymentException e) {
            // Compensation: Release Inventory
            inventoryClient.releaseInventory(request.getProductId(), request.getQuantity());
            orderService.updateOrderStatus(order.getId(), "FAILED");
            throw e;
        }
    }
}

// ========== Event-Driven Saga (Choreography) ==========

@Service
public class OrderEventProcessor {
    
    @EventListener
    @Async
    public void handleOrderCreated(OrderCreatedEvent event) {
        // Reserve inventory
        inventoryService.reserve(event.getProductId(), event.getQuantity());
        
        // Publish InventoryReserved event
        eventPublisher.publish(new InventoryReservedEvent(event.getOrderId()));
    }
    
    @EventListener
    public void handleInventoryReserved(InventoryReservedEvent event) {
        // Process payment
        paymentService.process(event.getOrderId(), event.getAmount());
        
        // Publish PaymentProcessed event
        eventPublisher.publish(new PaymentProcessedEvent(event.getOrderId()));
    }
}
```

## **Q10. Difference: Saga vs 2PC**

| Aspect | Saga | 2PC (Two-Phase Commit) |
|--------|------|------------------------|
| **Nature** | Distributed transaction | Distributed transaction |
| **Locking** | No locks (eventual consistency) | Locks resources (strict consistency) |
| **Performance** | High (non-blocking) | Low (blocking) |
| **Complexity** | Complex compensation logic | Complex coordinator |
| **Failure Handling** | Compensating transactions | Rollback all participants |
| **Latency** | Low | High |
| **Isolation** | No isolation (dirty reads possible) | Full isolation |
| **Use Case** | Long-running transactions | Short, critical transactions |
| **Example** | Order processing (minutes) | Bank transfer (seconds) |

## **Q11. How to Track Request Journey Across Services?**

### 🔍 **Distributed Tracing Solution**

```yaml
Tools:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

- Jaeger
- Zipkin
- AWS X-Ray
- New Relic
```

## **Q13. What is JWT Token and How Does It Look?**

### 📌 **Definition**

**JWT (JSON Web Token) = Self-contained token jo user information ko securely transmit karne ke liye use hota hai**

### 🔐 **JWT Structure**

```
JWT Format: xxxxx.yyyyy.zzzzz
           (Header.Payload.Signature)

Example JWT:
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
```

### 📦 **JWT Breakdown**

```json
// HEADER (Algorithm & Token Type)
{
  "alg": "HS256",        // Algorithm: HMAC SHA256
  "typ": "JWT"           // Token Type
}
// Base64Url encoded: eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9

// PAYLOAD (Claims - User Data)
{
  "sub": "1234567890",           // Subject (User ID)
  "name": "John Doe",            // User name
  "email": "john@example.com",   // Email
  "roles": ["USER", "ADMIN"],    // Roles
  "iat": 1516239022,             // Issued At (timestamp)
  "exp": 1516242622,             // Expiration (timestamp)
  "iss": "auth-service"          // Issuer
}
// Base64Url encoded: eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ

// SIGNATURE (Verification)
HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secret_key
)
// Result: SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
```

### 💻 **JWT Implementation**

```java
// ========== Generate JWT ==========
@Component
public class JwtTokenProvider {
    
    @Value("${jwt.secret}")
    private String jwtSecret;
    
    @Value("${jwt.expiration}")
    private int jwtExpiration;
    
    public String generateToken(User user) {
        Date now = new Date();
        Date expiryDate = new Date(now.getTime() + jwtExpiration);
        
        return Jwts.builder()
            .setSubject(user.getId())
            .claim("email", user.getEmail())
            .claim("roles", user.getRoles())
            .setIssuedAt(now)
            .setExpiration(expiryDate)
            .signWith(SignatureAlgorithm.HS512, jwtSecret)
            .compact();
    }
    
    public boolean validateToken(String token) {
        try {
            Jwts.parser().setSigningKey(jwtSecret).parseClaimsJws(token);
            return true;
        } catch (SignatureException e) {
            log.error("Invalid JWT signature");
        } catch (ExpiredJwtException e) {
            log.error("Expired JWT token");
        }
        return false;
    }
    
    public String getUserIdFromToken(String token) {
        Claims claims = Jwts.parser()
            .setSigningKey(jwtSecret)
            .parseClaimsJws(token)
            .getBody();
        
        return claims.getSubject();
    }
}

// ========== JWT Authentication Filter ==========
@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {
    
    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                   HttpServletResponse response,
                                   FilterChain chain) throws IOException, ServletException {
        
        String token = extractToken(request);
        
        if (token != null && tokenProvider.validateToken(token)) {
            String userId = tokenProvider.getUserIdFromToken(token);
            
            UsernamePasswordAuthenticationToken authentication = 
                new UsernamePasswordAuthenticationToken(userId, null, getAuthorities(token));
            
            SecurityContextHolder.getContext().setAuthentication(authentication);
        }
        
        chain.doFilter(request, response);
    }
    
    private String extractToken(HttpServletRequest request) {
        String bearerToken = request.getHeader("Authorization");
        if (bearerToken != null && bearerToken.startsWith("Bearer ")) {
            return bearerToken.substring(7);
        }
        return null;
    }
}
```

## **Q15. Why Basic Authentication is NOT Suitable?**

```yaml
Problem with Basic Auth:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Issues:
1. ❌ Credentials sent with EVERY request
2. ❌ No expiration mechanism
3. ❌ Cannot revoke easily
4. ❌ No roles/permissions
5. ❌ Must validate against database each time

Example Basic Auth Header:
Authorization: Basic dXNlcm5hbWU6cGFzc3dvcmQ=

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

            Credentials passed!
                  │
                  ▼
User Service ──Basic Auth──► Payment Service
                  │
            Credentials passed again!
                  │
                  ▼
          (Every service gets username/password)

Why Bad:
1. Password stored in multiple services
2. No way to know if user is still logged in
3. Each service must validate against user database
4. No token revocation
5. Security risk (credentials in logs, network)
```

## **Q27. Choreography-based Saga**

### 🎭 **Definition**

**Choreography = Events ke through coordination, bina central controller ke**

```yaml
Choreography vs Orchestration:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

4. Shipping Service listens → Schedules shipping → Event: SHIPPING_SCHEDULED
5. Notification Service listens → Sends email

If payment fails:
- Payment Service → Event: PAYMENT_FAILED
- Order Service listens → Updates order status to FAILED
- Inventory Service listens → Releases reserved stock
```

## **Q29. What is Fault Tolerance?**

### 📌 **Definition**

**Fault Tolerance = System ka ability to continue working even when some components fail**

```yaml
Fault Tolerance Patterns:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. Circuit Breaker:
   - Stops calling failing service
   - Prevents cascading failures
   - States: CLOSED → OPEN → HALF-OPEN

2. Retry:
   - Automatically retry failed operations
   - With exponential backoff

3. Timeout:
   - Don't wait indefinitely
   - Fail fast

4. Bulkhead:
   - Isolate failures
   - Separate thread pools for different services

5. Fallback:
   - Provide default response
   - Degraded functionality

6. Rate Limiter:
   - Limit requests to downstream services
   - Prevent overload
```

### 💻 **Resilience4j Implementation**

```java
@Service
public class FaultTolerantService {
    
    // ========== Circuit Breaker ==========
    @CircuitBreaker(name = "paymentService", fallbackMethod = "paymentFallback")
    public PaymentResponse callPaymentService(PaymentRequest request) {
        return restTemplate.postForObject(
            "http://payment-service/api/payments", 
            request, 
            PaymentResponse.class
        );
    }
    
    public PaymentResponse paymentFallback(PaymentRequest request, Exception e) {
        log.warn("Payment service failed, using fallback");
        return PaymentResponse.builder()
            .status("PENDING_MANUAL")
            .message("Payment will be processed later")

    }
    
    // ========== Retry with Backoff ==========

    public InventoryResponse callInventoryService(String productId) {
        return restTemplate.getForObject(
            "http://inventory-service/api/inventory/" + productId,
            InventoryResponse.class

    public CompletableFuture<Order> processOrderAsync(OrderRequest request) {
        return CompletableFuture.supplyAsync(() -> orderService.createOrder(request));
    }
    
    // ========== Rate Limiter ==========
    @RateLimiter(name = "notificationService")
    public void sendNotification(Notification notification) {
```

| **Service Discovery** | ✅ Yes | ❌ No |
| **Canary Deployments** | ✅ Yes | ✅ Basic |

### 💻 **Complete Implementation**

```java
// ========== 1. Circuit Breaker with Graceful Fallback ==========
@Service
@Slf4j
public class OrderService {
    
    @Autowired
    private PaymentClient paymentClient;
    
    @CircuitBreaker(name = "paymentService", fallbackMethod = "paymentFallback")
    public Order createOrderWithPayment(OrderRequest request) {
        // Try to process payment
        PaymentResponse payment = paymentClient.processPayment(request.getPaymentInfo());
        
        if (payment.isSuccess()) {
            return createOrder(request, "CONFIRMED");
        } else {
            return createOrder(request, "PENDING_PAYMENT");
        }
    }
    
    // Fallback when Payment Service is down
    public Order paymentFallback(OrderRequest request, Exception e) {
        log.warn("Payment service unavailable, placing order in pending state");
        
        // Create order with PENDING status
        Order order = createOrder(request, "PENDING_PAYMENT");
        
        // Store for later processing
        pendingPaymentQueue.add(order.getId());
        
        // Schedule retry
        schedulePaymentRetry(order.getId());
        
        // Return success to user (with note)
        order.setMessage("Order placed! Payment will be processed shortly.");
        return order;
    }
    
    @Scheduled(fixedDelay = 30000)
    public void retryPendingPayments() {
        List<String> pendingOrders = pendingPaymentQueue.getAll();
        
        for (String orderId : pendingOrders) {
            try {
                Order order = orderRepository.findById(orderId);
                PaymentResponse payment = paymentClient.processPayment(order.getPaymentInfo());
                
                if (payment.isSuccess()) {
                    order.setStatus("CONFIRMED");
                    orderRepository.save(order);
                    pendingPaymentQueue.remove(orderId);
                    
                    // Notify user
                    notificationService.sendEmail(order.getUserId(), 
                        "Your payment has been processed successfully!");
                }
            } catch (Exception e) {
                log.error("Retry failed for order: {}", orderId);
                // Will retry again in next cycle
            }
        }
    }
}

// ========== 2. Degraded Functionality (Show cached data) ==========
@Service
public class ProductCatalogService {
    
    @Autowired
    private RedisTemplate<String, Product> redisTemplate;
    
    @CircuitBreaker(name = "inventoryService", fallbackMethod = "getProductFromCache")
    public Product getProductWithAvailability(Long productId) {
        // Try to get real-time availability
        Product product = productRepository.findById(productId);
        Availability availability = inventoryClient.getAvailability(productId);
        product.setInStock(availability.isInStock());
        product.setQuantity(availability.getQuantity());
        
        // Cache for fallback
        redisTemplate.opsForValue().set("product:" + productId, product, Duration.ofMinutes(5));
        
        return product;
    }
    
    // Fallback: Show cached product (might be slightly stale)
    public Product getProductFromCache(Long productId, Exception e) {
        log.warn("Inventory service down, showing cached availability");
        
        Product product = redisTemplate.opsForValue().get("product:" + productId);
        
        if (product != null) {
            // Add disclaimer
            product.setAvailabilityMessage("Availability information might be slightly outdated");
            return product;
        }
        
        // Ultimate fallback: assume in stock
        product = productRepository.findById(productId);
        product.setInStock(true);
        product.setAvailabilityMessage("Assuming in stock - please check at checkout");
        return product;
    }
}

// ========== 3. Async Processing with Message Queue ==========
@Service
public class AsyncOrderService {
    
    @Autowired
    private KafkaTemplate<String, OrderEvent> kafkaTemplate;
    
    public OrderResponse createOrder(OrderRequest request) {
        // Create order with PROCESSING status
        Order order = orderRepository.save(new Order(request));
        order.setStatus("PROCESSING");
        
        // Publish to queue instead of processing synchronously
        OrderEvent event = OrderEvent.builder()
            .orderId(order.getId())
            .type("ORDER_CREATED")
            .timestamp(Instant.now())
            .build();
        
        kafkaTemplate.send("order-events", event);
        
        // Return immediately
        return OrderResponse.builder()
            .orderId(order.getId())
            .status("PROCESSING")
            .message("Your order is being processed. You'll receive confirmation shortly.")
            .estimatedCompletionTime(LocalDateTime.now().plusMinutes(2))
            .build();
    }
}

@Component
public class OrderEventConsumer {

        try {

            paymentService.processPayment(event.getOrderId());

        }
    }

// ========== 4. Retry with Exponential Backoff ==========
@Service
public class ResilientPaymentClient {
    
    @Retry(name = "paymentRetry", fallbackMethod = "ultimateFallback")
    public PaymentResponse callPaymentWithRetry(PaymentRequest request) {
        // Will retry 3 times with exponential backoff
        return restTemplate.postForObject(
            "http://payment-service/api/payments",
            request,
            PaymentResponse.class
        );
    }
    
    public PaymentResponse ultimateFallback(PaymentRequest request, Exception e) {
        // After all retries failed
        log.error("All payment retries failed", e);
        
        // Store for manual processing
        failedPaymentQueue.add(request);
        
        // Return "pending" response
        return PaymentResponse.builder()
            .status("PENDING_MANUAL_REVIEW")
            .message("Payment will be processed manually. Your order is confirmed.")
            .build();
    }
}

// ========== 5. Bulkhead Pattern (Isolate Failures) ==========
@Configuration
public class BulkheadConfig {
    
    @Bean
    public Bulkhead paymentServiceBulkhead() {
        return Bulkhead.of("paymentService", 
            BulkheadConfig.custom()
                .maxConcurrentCalls(10)  // Only 10 concurrent calls
                .maxWaitDuration(Duration.ofMillis(500))  // Wait 500ms max
                .build()
        );
    }
}

@Service
public class IsolatedService {
    
    @Autowired
    private Bulkhead paymentBulkhead;
    
    public PaymentResponse callPaymentService(PaymentRequest request) {
        // If bulkhead is full, this will fail fast
        return Bulkhead.decorateSupplier(paymentBulkhead, () -> {
            return paymentClient.processPayment(request);
        }).get();
    }
}

// ========== 6. User-Friendly Error Messages ==========
@RestControllerAdvice
public class GracefulErrorHandler {
    
    @ExceptionHandler(CircuitBreakerOpenException.class)
    public ResponseEntity<ErrorResponse> handleCircuitBreaker(Exception e) {
        // Don't show technical error to user
        return ResponseEntity.status(HttpStatus.OK)  // Still 200 OK!
            .body(ErrorResponse.builder()
                .type("TEMPORARY_UNAVAILABLE")
                .message("Some features are temporarily unavailable. We're working on it.")
                .recoveryTime("5 minutes")
                .suggestedAction("Please try again later")
                .build());
    }
    
    @ExceptionHandler(TimeoutException.class)
    public ResponseEntity<OrderResponse> handleTimeout(TimeoutException e) {
        // Return accepted response instead of error
        return ResponseEntity.accepted()
            .body(OrderResponse.builder()
                .status("PROCESSING")
                .message("Request is taking longer than expected. We'll notify you when complete.")
                .build());
    }
```

### 📊 **Complete Fault Tolerance Architecture**

```yaml
User Request Flow with Fallbacks:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

User → API Gateway
         │
         ├─→ Check Circuit Breaker
         │     ├─ CLOSED → Forward to Service
         │     └─ OPEN → Return Fallback Immediately
         │
         ├─→ Try Service Call (with timeout)
         │     ├─ Success → Return Response
         │     ├─ Timeout → Retry (max 3 times)
         │     └─ Failure → Circuit Breaker Opens
         │
         └─→ Fallback Response
               ├─ From Cache (stale data OK)
               ├─ Default Values (assume in stock)
               ├─ Async Processing (queue for later)
               └─ Degraded UI (hide broken features)

User Experience:
✅ Order placed (even if payment is down)
✅ Product shown (even if inventory down)
✅ "Processing" status instead of error
✅ Email notification when fixed
```
