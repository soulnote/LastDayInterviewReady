## **Q1. Why Java is not 100% object oriented?**

**Answer:**  
Java is not 100% object-oriented because it supports **primitive data types** like `int`, `char`, `boolean`, etc., which are not objects. In a pure OOP language (e.g., Smalltalk), everything including numbers and booleans is an object.

**Detailed explanation:**
- Primitives are stored in stack memory, while objects are in heap.
- They don't have methods (e.g., `int` can't have `int.toString()` directly without wrapper).
- **Why Java kept primitives?** Performance & simplicity. Objects have overhead (memory + GC), primitives are faster and more memory-efficient for simple computations.
- Workaround: Wrapper classes (`Integer`, `Boolean`) exist but are not mandatory.

**Interview tip:**  
> "Java is ~99% OOP, but primitives break the '100%' claim. However, with modern JVM optimizations and auto-boxing, the gap has reduced."

## **Q2. Why pointers are not used in Java?**

**Answer:**  
Java does not use pointers **for security and simplicity reasons**. Instead, it uses **references**.

**Detailed explanation:**
- **Security:** Direct pointer access can bypass Java's access control, leading to memory corruption or hacking.
- **Garbage Collection:** Pointers would allow manual memory manipulation, breaking GC's automatic memory management.
- **No pointer arithmetic:** Prevents accidental memory overwrites, buffer overflows (common in C/C++ bugs).
- **What Java uses instead:** References – they point to objects but cannot be incremented/decremented or arbitrarily cast.

**Interview tip:**  
> "Java removed pointers to eliminate common bugs like dangling pointers, memory leaks, and buffer overflows. References are safer and still allow object manipulation."

## **Q3. What is JIT compiler in Java?**

**Answer:**  
JIT (Just-In-Time) compiler improves performance by compiling bytecode into native machine code **at runtime**, instead of interpreting bytecode repeatedly.

**Detailed explanation:**
- **Process:** Java source → bytecode → JVM interprets OR JIT compiles hot methods → native code.
- **How it works:** JVM monitors method call frequency. When a method becomes "hot", JIT compiles it to native code, then directly executes it.
- **Benefit:** Drastically speeds up repeated method calls (10x-100x faster than interpretation).
- **Trade-off:** Initial compilation overhead, but overall throughput improves.

**Interview tip:**  
> "JIT is why Java can be both portable (bytecode) and fast (native execution). Without JIT, Java would be purely interpreted like early JS."

## **Q4. What is Write Once Run Anywhere (WORA) concept in Java?**

**Answer:**  
WORA means Java code compiled once into bytecode can run on any platform (Windows, Linux, Mac) without recompilation, provided that platform has a JVM.

**Detailed explanation:**
- **How it works:** Source → `.java` → `javac` → `.class` (bytecode) → JVM interprets/compiles to platform-specific machine code.
- **JVM as abstraction layer:** JVM handles OS-specific details (file paths, threading, networking).
- **Example:** You compile on Windows, deploy on Linux – no changes needed.
- **Limitations:** Some platform-dependent libraries (e.g., `java.awt` behavior may differ). Also, JNI (Java Native Interface) breaks WORA.

**Interview tip:**  
> "WORA is achieved because bytecode is platform-independent, and JVM acts as a translation layer. This is why Java dominated enterprise – you code once, deploy anywhere."

## **Q5. What is marker interface in Java?**

**Answer:**  
A marker interface has **no methods or constants** – it's an empty interface used to signal/metadata to JVM or compiler about a class's special behavior.

**Detailed explanation:**
- **Examples:** `Serializable`, `Cloneable`, `Remote`.
- **How JVM uses it:**
  - `Serializable` → JVM allows object serialization (conversion to byte stream).
  - `Cloneable` → JVM allows `Object.clone()` (else throws `CloneNotSupportedException`).
- **Modern alternative:** Annotations (`@Override`, `@FunctionalInterface`) are now preferred, but marker interfaces still work.
- **Why used historically:** Type-safety at compile-time and runtime.

**Interview tip:**  
> "Marker interfaces are a Java design pattern to attach metadata to a class. Annotations have largely replaced them, but you'll still see `Serializable` everywhere in legacy code."

## **Q6. How to make class immutable in Java?**

**Answer:**  
An immutable class's state cannot change after object creation. Examples: `String`, `Integer`, `LocalDate`.

**Steps to create:**

1. **Declare class as `final`** – prevents subclassing (subclass could add mutability).
2. **Make all fields `private final`** – ensures they're set once and not accessible directly.
3. **No setter methods** – remove any methods that modify fields.
4. **Initialize all fields via constructor** – deep copy if fields are mutable.
5. **Return defensive copies in getters** – especially for mutable objects like `Date`, `List`.

**Example:**
```java
public final class ImmutablePerson {
    private final String name;
    private final List<String> hobbies;

    public ImmutablePerson(String name, List<String> hobbies) {
        this.name = name;
        this.hobbies = new ArrayList<>(hobbies); // defensive copy
    }

    public String getName() { return name; }
    
    public List<String> getHobbies() {
        return new ArrayList<>(hobbies); // defensive copy
    }
}
```

**Interview tip:**  
> "Always return copies of mutable internal objects in getters. Also, be careful with `Date` objects – use `LocalDate` or clone."

## **Q7. Does Java support multiple inheritances?**

**Answer:**  
Java **does not support multiple inheritance for classes** (to avoid diamond problem), but **supports multiple inheritance for interfaces** (using `default` methods).

**Detailed explanation:**
- **Why no class MI?** Diamond problem – if two parent classes have same method, which one to inherit?
- **Interface MI allowed:** A class can implement multiple interfaces.
- **Diamond problem in interfaces (Java 8+):** If two interfaces have same default method, implementing class **must override** it.
- **Example:**
```java
interface A { default void m1() { } }
interface B { default void m1() { } }
class C implements A, B {
    public void m1() { A.super.m1(); } // resolve manually
}
```

**Interview tip:**  
> "Java trades multiple class inheritance for safety and simplicity. Interfaces + composition give you most benefits without the diamond problem."

## **Q8. Can we use default constructor if an explicit constructor is defined?**

**Answer:**  
**No.** If you define any explicit constructor (with or without parameters), Java does **not** provide the default no-argument constructor.

**Detailed explanation:**
- **Default constructor:** Automatically added only if no constructor is defined.
- **Once you add your own:** The compiler assumes you want full control over object creation.
- **Workaround:** You can explicitly define a no-arg constructor yourself.
- **Impact on subclasses:** If a superclass lacks a no-arg constructor, subclasses must explicitly call `super(...)` with matching arguments.

**Example:**
```java
class Parent {
    Parent(int x) { } // no default constructor now
}
class Child extends Parent {
    Child() { super(10); } // must call explicit super
}
```

**Interview tip:**  
> "This is a common mistake – always define a no-arg constructor explicitly if you need one, especially for frameworks like Hibernate or Spring."

## **Q9. Why Comparable and Comparator interfaces are required?**

**Answer:**  
Both are used for sorting objects, but they serve different purposes:

- **Comparable** – natural/default ordering (single sorting logic).
- **Comparator** – custom/alternative ordering (multiple sorting logics).

**Detailed explanation:**

| Feature | Comparable | Comparator |
|---------|------------|-------------|
| Method | `compareTo(obj)` | `compare(obj1, obj2)` |
| Location | Inside the class | External class or lambda |
| Sorting | `Collections.sort(list)` | `Collections.sort(list, comparator)` |
| Modifies class? | Yes (implements Comparable) | No (external) |
| Multiple logics | No | Yes |

**Example:**
```java
// Comparable
class Employee implements Comparable<Employee> {
    int id;
    public int compareTo(Employee e) { return this.id - e.id; }
}

// Comparator
Comparator<Employee> byName = (e1, e2) -> e1.name.compareTo(e2.name);
Collections.sort(emps, byName);
```

**Interview tip:**  
> "Use Comparable when there's a single natural order (e.g., Student by roll number). Use Comparator for ad-hoc sorting (e.g., sort by name, then by age)."

## **Q10. Comparable implementation in Java**

**Answer:**  
Implement `Comparable<T>` interface and override `compareTo(T obj)` method.

**Contract of `compareTo`:**
- Returns **negative** if `this < obj`
- Returns **zero** if `this == obj`
- Returns **positive** if `this > obj`
- Must be consistent with `equals()` (recommended, but not strictly required)

**Example:**
```java
public class Student implements Comparable<Student> {
    int rollNo;
    String name;
    
    @Override
    public int compareTo(Student other) {
        // natural ordering by roll number
        return Integer.compare(this.rollNo, other.rollNo);
    }
}

// Usage
List<Student> list = ...;
Collections.sort(list); // uses compareTo
```

**Common pitfalls:**
- Don't use subtraction for large numbers (overflow risk):  
  `return this.id - other.id;` ❌ (overflow possible)  
  `return Integer.compare(this.id, other.id);` ✅
- `compareTo` must be **transitive**, **reflexive**, **symmetric**.

**Interview tip:**  
> "Always use wrapper class's `compare` methods or `Comparator.comparing()` for safety. Also, ensure `compareTo` is consistent with `equals` to avoid issues in sorted collections like `TreeSet`."

## **Q11. Comparator implementation in Java**

**Answer:**  
Comparator ek functional interface hai jisme `compare(T o1, T o2)` method override karte hain. Isse multiple sorting logics implement kiye ja sakte hain bina original class ko modify kiye.

**Implementations:**

**1. Traditional way (anonymous class):**
```java
Comparator<Employee> byId = new Comparator<Employee>() {
    @Override
    public int compare(Employee e1, Employee e2) {
        return Integer.compare(e1.id, e2.id);
    }
};
```

**2. Lambda expression (Java 8+):**
```java
Comparator<Employee> byName = (e1, e2) -> e1.name.compareTo(e2.name);
```

**3. Method reference:**
```java
Comparator<Employee> bySalary = Comparator.comparing(Employee::getSalary);
```

**4. Chaining comparators:**
```java
Comparator<Employee> multiLevel = Comparator
    .comparing(Employee::getDept)
    .thenComparing(Employee::getSalary)
    .thenComparing(Employee::getName);
```

**5. Reverse order:**
```java
Comparator<Employee> reverseId = Comparator.comparing(Employee::getId).reversed();
```

**Interview tip:**  
> "Production code mein main hamesha `Comparator.comparing()` with method references use karta hoon - yeh more readable aur less error-prone hai."

## **Q12. What is difference between Comparator vs Comparable?**

**Answer:**

| Feature | Comparable | Comparator |
|---------|------------|------------|
| **Package** | `java.lang` | `java.util` |
| **Method** | `compareTo(T o)` | `compare(T o1, T o2)` |
| **Sorting logic** | Natural/Default ordering | Custom/Alternative ordering |
| **Modifies class** | Haan (class implements it) | Nahi (external) |
| **Multiple logics** | Nahi (sirf ek) | Haan (multiple comparators) |
| **Example** | `String`, `Integer`, `Date` | Sorting by different fields |
| **Usage** | `Collections.sort(list)` | `Collections.sort(list, comparator)` |

**Example:**
```java
// Comparable - ek hi sorting logic
class Product implements Comparable<Product> {
    int price;
    public int compareTo(Product p) {
        return Integer.compare(this.price, p.price);
    }
}

// Comparator - multiple sorting logics
Comparator<Product> byName = (p1,p2) -> p1.name.compareTo(p2.name);
Comparator<Product> byPriceDesc = (p1,p2) -> Integer.compare(p2.price, p1.price);
```

**Interview tip:**  
> "Interviewer pooch sakta hai - 'Aap kab Comparable use karoge aur kab Comparator?' Toh answer: Comparable for single natural ordering (jaise Employee by ID), Comparator for dynamic sorting like UI filters."

## **Q13. Can we override a static method in Java?**

**Answer:**  
**No** - static methods cannot be overridden. Yeh **method hiding** hota hai, overriding nahi.

**Detailed explanation:**
- Static methods belong to **class**, not instance.
- Overriding depends on runtime object type, but static methods resolve at **compile time** based on reference type.
- `@Override` annotation allowed nahi hai static methods par.

**Example:**
```java
class Parent {
    static void m1() { System.out.println("Parent static"); }
    void m2() { System.out.println("Parent instance"); }
}

class Child extends Parent {
    static void m1() { System.out.println("Child static"); } // Hiding, not overriding
    @Override void m2() { System.out.println("Child instance"); } // Overriding
}

Parent p = new Child();
p.m1(); // Output: "Parent static" (compile-time binding)
p.m2(); // Output: "Child instance" (runtime binding)
```

**Interview tip:**  
> "Static methods hide, instance methods override. Avoid calling static methods on object references - always call on class names."

## **Q14. Why is 1 == 1 is true but 1000 == 1000 is false when dealing with Wrapper Classes in Java?**

**Answer:**  
Yeh **Integer Caching** ki wajah se hota hai - JVM `-128` to `127` range ke Integers ko cache karta hai.

**Detailed explanation:**
- `Integer a = 1000;` internally `Integer.valueOf(1000)` call hota hai.
- `valueOf()` method returns cached object for range -128 to 127.
- Is range ke bahar, **naye objects** create hote hain.
- `==` compares **object references**, not values.

**Example:**
```java
Integer a = 1;
Integer b = 1;
System.out.println(a == b);  // true (both from cache)

Integer x = 1000;
Integer y = 1000;
System.out.println(x == y);  // false (different objects)

// Solution: Use .equals()
System.out.println(x.equals(y)); // true

// Cache size change kar sakte ho using JVM flag:
// -Djava.lang.Integer.IntegerCache.high=2000
```

**Interview tip:**  
> "Kabhi bhi wrapper objects ke liye `==` use mat karo - hamesha `.equals()` use karo. Cache sirf performance optimization hai, ispe rely mat karo."

## **Q15. What are variable arguments in Java?**

**Answer:**  
Varargs (variable-length arguments) allow method to accept **zero or more arguments** of same type using `...` syntax.

**Syntax:**
```java
returnType methodName(Type... varName)
```

**Example:**
```java
public int sum(int... numbers) {
    int total = 0;
    for (int num : numbers) {
        total += num;
    }
    return total;
}

// Usage
sum();           // 0
sum(1);          // 1
sum(1, 2, 3);    // 6
sum(new int[]{1,2,3}); // also works
```

**Rules:**
1. Varargs parameter must be **last** parameter.
2. Only **one** varargs parameter per method.
3. Internally it's an array.

**Valid:**
```java
void m1(String s, int... nums) { }  // ✅
void m2(int... nums, String s) { }  // ❌ (must be last)
void m3(int... a, int... b) { }     // ❌ (only one)
```

**Interview tip:**  
> "Varargs ka use `printf()` aur `Collections.addAll()` jaise methods mein hota hai. Performance-wise, varargs creates array each call - so avoid in performance-critical loops."

## **Q18. Why final and Abstract cannot be used at a time in Java?**

**Answer:**  
Kyunki `final` aur `abstract` **contradictory** concepts hain:

- `final` → Class cannot be inherited, method cannot be overridden.
- `abstract` → Class must be inherited, method must be overridden.

**Detailed explanation:**
```java
// Yeh compile nahi hoga
public abstract final class MyClass { }  // ❌ Compilation error

// Method par bhi same
public abstract final void myMethod();  // ❌ Compilation error
```

**Logical reasoning:**
- Abstract method ka koi body nahi hota - isse override karna hi hota hai.
- Final method override nahi ho sakta.
- Abstract class ka object directly create nahi ho sakta - usse inherit karna hi padta hai.
- Final class inherit nahi ho sakti.

**Interview tip:**  
> "Interviewer yeh poochta hai conceptual clarity check karne ke liye. Simple answer - 'final means complete, abstract means incomplete - ek class dono ek saath nahi ho sakti'."

## **Q19. Does an abstract class required to have an abstract method inside?**

**Answer:**  
**No** - abstract class mein ek bhi abstract method hona zaroori nahi hai.

**Detailed explanation:**
- Abstract class without abstract methods is allowed.
- Phir bhi us class ka object nahi bana sakte (abstract keyword ki wajah se).
- Use-case: Jab aap class ko **instantiate hone se rokna chahte ho** but method implementations provide karni ho.

**Example:**
```java
// Valid abstract class with no abstract method
public abstract class BaseController {
    protected Logger logger = LoggerFactory.getLogger(getClass());
    
    public void logInfo(String message) {
        logger.info(message);
    }
    
    public void logError(String message, Exception e) {
        logger.error(message, e);
    }
}

// Child class can use these methods
public class UserController extends BaseController {
    public void createUser() {
        logInfo("Creating user");  // method inherited
    }
}
```

**When to use:**
- Template method pattern mein
- Common utility methods provide karne ke liye but direct instantiation rokni ho
- Framework base classes (Spring ke `WebMvcConfigurerAdapter` - though deprecated)

**Interview tip:**  
> "Abstract class ka matlab 'incomplete' nahi hai - iska matlab 'not fully instantiable' hai. Aap abstract class likh sakte ho without any abstract methods."

## **Q20. Can Constructors be synchronized in Java?**

**Answer:**  
**No** - constructor ko `synchronized` keyword nahi laga sakte. Compilation error aayegi.

**Why?**
- Constructor hamesha **single thread** mein execute hota hai object creation ke time.
- Object creation ke time, object abhi fully initialized nahi hota - dusra thread access nahi kar sakta.
- Synchronized keyword method ya block ke liye hai, constructor ke liye nahi.

**What to do instead?**
```java
// ❌ Yeh allowed nahi hai
public synchronized MyClass() { }  // Compilation error

// ✅ Alternative: synchronized block inside constructor
public MyClass() {
    synchronized(this) {
        // initialization logic
    }
}

// ✅ Or use static factory method with synchronization
public static synchronized MyClass getInstance() {
    return new MyClass();
}
```

**Special case - Singleton:**
```java
public class Singleton {
    private static Singleton instance;
    
    private Singleton() { }  // constructor not synchronized
    
    public static synchronized Singleton getInstance() {  // method synchronized
        if (instance == null) {
            instance = new Singleton();
        }
        return instance;
    }
}
```

**Interview tip:**  
> "Constructor synchronization is unnecessary because each object is created in its own thread's context before any other thread can access it. If you need thread-safe initialization, use synchronized blocks inside constructor or double-checked locking pattern."

## **Q21. Order of execution: static initializers, instance initializers, and constructors**

**Answer:**  
Execution order fixed hai aur aapko **pata hona chahiye** - yeh Java fundamentals hai.

**Order:**
1. **Parent class static initializer** (executed once when class loads)
2. **Child class static initializer** (executed once when class loads)
3. **Parent class instance initializer** (every time object created)
4. **Parent class constructor**
5. **Child class instance initializer** (every time object created)
6. **Child class constructor**

**Example:**
```java
class A {
    static { System.out.println("1 - A static"); }
    { System.out.println("3 - A instance"); }
    A() { System.out.println("4 - A constructor"); }
}

class B extends A {
    static { System.out.println("2 - B static"); }
    { System.out.println("5 - B instance"); }
    B() { System.out.println("6 - B constructor"); }
}

// Creating object: new B();
// Output:
// 1 - A static
// 2 - B static
// 3 - A instance
// 4 - A constructor
// 5 - B instance
// 6 - B constructor
```

**Important notes:**
- Static blocks execute **only once** (class loading time)
- Instance blocks execute **before constructor** (useful for common initialization)
- Agar multiple static blocks, then in **declaration order**

**Interview tip:**  
> "Yeh question interview mein frequently aata hai. Yaad rakho: Static first (parent then child), then instance (parent then child), then constructor (parent then child)."

## **Q22. In what situations is the "super" keyword employed?**

**Answer:**  
`super` keyword use hota hai parent class ke members ko access karne ke liye jab child class mein same name ka member ho.

**3 main situations:**

**1. Access parent class variable (when shadowed):**
```java
class Parent {
    String name = "Parent";
}
class Child extends Parent {
    String name = "Child";
    void print() {
        System.out.println(name);      // "Child"
        System.out.println(super.name); // "Parent"
    }
}
```

**2. Call parent class constructor (must be first statement):**
```java
class Employee {
    Employee(int id, String name) {
        this.id = id;
        this.name = name;
    }
}
class Manager extends Employee {
    Manager(int id, String name, int teamSize) {
        super(id, name);  // Must be first line
        this.teamSize = teamSize;
    }
}
```

**3. Call parent class method (when overridden):**
```java
class Animal {
    void sound() { System.out.println("Animal makes sound"); }
}
class Dog extends Animal {
    @Override
    void sound() {
        super.sound();  // Call parent method
        System.out.println("Dog barks");
    }
}
```

**Interview tip:**  
> "super() constructor call hamesha **first statement** hona chahiye. Agar parent class mein default constructor nahi hai toh explicit super call mandatory hai."

## **Q23. How to write multiple catch statements under a single try block?**

**Answer:**  
Multiple catch blocks sequentially likhe jaate hain - ek try ke baad multiple catch.

**Syntax:**
```java
try {
    // risky code
} catch (IOException e) {
    // handle IO exception
} catch (SQLException e) {
    // handle SQL exception
} catch (Exception e) {
    // handle any other exception (generic - must be last)
}
```

**Important rules:**

**1. Order matters - specific to general:**
```java
// ✅ Correct
catch (NullPointerException e) { }
catch (RuntimeException e) { }
catch (Exception e) { }

// ❌ Wrong - compilation error (unreachable code)
catch (Exception e) { }
catch (NullPointerException e) { }  // Never reaches
```

**2. Java 7+ - Multi-catch (same handling logic):**
```java
try {
    // code
} catch (IOException | SQLException | ParseException e) {
    // All three handled same way
    logger.error("Database/IO error", e);
}
```

**3. Multi-catch restrictions:**
```java
// ❌ Cannot combine unrelated exceptions in inheritance hierarchy
catch (IOException | FileNotFoundException e) { }  // Error - FileNotFoundException is subclass

// ✅ Correct way
catch (FileNotFoundException e) { }
catch (IOException e) { }
```

**Interview tip:**  
> "Production code mein multi-catch use karo jab handling logic same ho. Yaad rakho - specific exceptions pehle, generic baad mein."

## **Q24. What is implicit casting or widening conversion?**

**Answer:**  
Implicit casting (widening) - automatic type conversion jab **smaller** data type ko **larger** data type mein assign karte ho. Koi data loss nahi hota.

**Conversion hierarchy:**
```
byte → short → int → long → float → double
char → int → long → float → double
```

**Examples:**
```java
int x = 100;
long y = x;           // ✅ implicit - int to long
float f = y;          // ✅ implicit - long to float
double d = f;         // ✅ implicit - float to double

char c = 'A';
int i = c;            // ✅ implicit - char to int (65)

byte b = 10;
int i2 = b;           // ✅ implicit
```

**When it happens:**
- Assignment conversion
- Method invocation conversion
- Arithmetic expression conversion

## **Q25. What is explicit casting or narrowing conversion?**

**Answer:**  
Explicit casting (narrowing) - manual type conversion jab **larger** data type ko **smaller** data type mein assign karte ho. **Data loss possible hai**.

**Syntax:** `(targetType) value`

**Examples:**
```java
double d = 100.75;

// Object casting
Object obj = "Hello";
```

```java
// 1. Precision loss
double pi = 3.14159;
int approx = (int) pi;  // 3 - fractional part lost

```

```java

    String str = (String) obj;  // safe
}
```

## **Q27. How Can Java Reflection Be Used to Break Encapsulation?**

**Answer:**  
Reflection private fields, methods, constructors ko access kar sakta hai even if they are `private`.

**Breaking encapsulation examples:**

**1. Accessing private field:**
```java
class User {
    private String password = "secret123";
}

User user = new User();
Field field = User.class.getDeclaredField("password");
field.setAccessible(true);  // Break encapsulation
String password = (String) field.get(user);  // "secret123"
field.set(user, "hacked");  // Modify private field
```

**2. Calling private method:**
```java
class BankAccount {
    private void validateTransaction() {
        // security logic
    }
}

BankAccount acc = new BankAccount();
Method method = BankAccount.class.getDeclaredMethod("validateTransaction");
method.setAccessible(true);
method.invoke(acc);  // Private method called
```

**3. Instantiating private constructor (Singleton break):**
```java
class Singleton {
    private static Singleton instance = new Singleton();
    private Singleton() { }
}

Constructor<Singleton> constructor = Singleton.class.getDeclaredConstructor();
constructor.setAccessible(true);
Singleton newInstance = constructor.newInstance();  // Breaks singleton
```

**How to protect against reflection:**
```java
// 1. Security Manager (legacy)
System.setSecurityManager(new SecurityManager());

// 2. Check in constructor
private Singleton() {
    if (instance != null) {
        throw new RuntimeException("Use getInstance()");
    }
}

// 3. Use Enum (reflection-safe)
public enum SafeSingleton {
    INSTANCE;
}
```

**Interview tip:**  
> "Reflection encapsulation break kar sakta hai, isliye frameworks (Spring, Hibernate) iska use karte hain. Production code mein direct reflection avoid karo unless library likh rahe ho."

## **Q30. How Do You Decide If an Inner Class Is Required?**

**Answer:**  
Inner class tab use karo jab class **logically** outer class ka part ho aur **only outer class** use kare.

**Decision framework:**

**✅ Use inner class when:**
1. **Logical grouping:** Class only meaningful with outer class
```java
class Car {
    class Engine {  // Engine exists only within Car
        void start() { }
    }
}
```

2. **Access to private members:** Need outer class private data
```java
class LinkedList {
    private Node head;
    private class Node {  // Only LinkedList needs Node
        int data;
        Node next;
    }
}
```

3. **Encapsulation:** Hide class from outside world
```java
public class DataProcessor {
    private class Validator {  // Hidden from other packages
        boolean validate(String data) { }
    }
}
```

**❌ Don't use inner class when:**
1. **Reusable elsewhere:** Make it separate top-level class
2. **Too large:** >50 lines - impacts readability
3. **Needs to be instantiated independently**
4. **Testing difficulty:** Inner classes harder to unit test

**Real-world examples:**
```java
// Good - Map.Entry is inner interface
public interface Map<K,V> {
    interface Entry<K,V> {  // Logically part of Map
        K getKey();
        V getValue();
    }
}

// Good - Builder pattern
public class User {
    public static class Builder {  // Static nested
        // Builds User object
    }
}

// Bad - Don't do this
public class Order {
    public class DatabaseHelper {  // ❌ Not logically part of Order
        // Should be separate class
    }
}
```

**Interview tip:**  
> "Inner class ka decision rule: 'Can this class exist independently?' Agar answer 'no' hai (jaise Car ka Engine), tab inner class use karo. Agar 'yes' hai, tab top-level class banao."

## **Q31. How do you write an interface with default and static method?**

**Answer:**  
Java 8+ mein interfaces mein `default` aur `static` methods allowed hain.

**Syntax:**
```java
public interface Vehicle {
    
    // Abstract method (traditional)
    void start();
    
    // Default method - with implementation
    default void honk() {
        System.out.println("Beep beep!");
    }
    
    // Static method
    static void service() {
        System.out.println("Vehicle servicing");
    }
}

// Implementation
class Car implements Vehicle {
    @Override
    public void start() {
        System.out.println("Car starting");
    }
    
    // Optional: override default method
    @Override
    public void honk() {
        System.out.println("Car horn: PONK PONK");
    }
}

// Usage
Car car = new Car();
car.start();      // Car starting
car.honk();       // Car horn: PONK PONK
Vehicle.service(); // Vehicle servicing (static - call on interface)
```

**Key points:**
- `default` methods - instances par call hoti hain, override kar sakte ho
- `static` methods - interface name se call hoti hain, override nahi kar sakte
- Multiple inheritance ka diamond problem? Agar do interfaces mein same default method, to implementing class ko override karna padega

**Interview tip:**  
> "Default methods allow backward compatibility - existing interfaces mein naye methods add kar sakte ho without breaking implementations."

## **Q32. What is object cloning in Java?**

**Answer:**  
Object cloning means creating **exact copy** of an object. Java mein `clone()` method of `Object` class use hoti hai.

**Shallow vs Deep Clone:**

**1. Shallow Clone (default):**
```java
class Address {
    String city;
    Address(String city) { this.city = city; }
}

class Employee implements Cloneable {
    String name;
    Address address;
    
    @Override
    protected Object clone() throws CloneNotSupportedException {
        return super.clone();  // Shallow copy
    }
}

// Result: employee1.address == employee2.address (same object reference)
```

**2. Deep Clone:**
```java
@Override
protected Object clone() throws CloneNotSupportedException {
    Employee cloned = (Employee) super.clone();
    cloned.address = new Address(this.address.city);  // Create new Address
    return cloned;
}
```

**Requirements:**
- Implement `Cloneable` interface (marker interface)
- Override `clone()` method
- Call `super.clone()`

**Better alternatives:**
```java
// 1. Copy constructor
public Employee(Employee other) {
    this.name = other.name;
    this.address = new Address(other.address.city);
}

// 2. Copy factory method
public static Employee copyOf(Employee original) { ... }

// 3. Serialization (for deep copy)
```

**Interview tip:**  
> "`clone()` is broken by design (Joshua Bloch - Effective Java). Production code mein copy constructor ya factory method use karo instead of clone()."

## **Q36. Can You Access Non-Static Variables in the Static Context?**

**Answer:**  
**No** - static context (static method, static block) mein non-static variables directly access nahi kar sakte.

**Why?**
- Static members belong to **class**, exist even without object
- Non-static members belong to **instance**, require object
- Compilation error: "non-static variable cannot be referenced from a static context"

**Example:**
```java
class Example {
    int instanceVar = 10;
    static int staticVar = 20;
    
    static void staticMethod() {
        System.out.println(staticVar);      // ✅ Works
        System.out.println(instanceVar);    // ❌ Compilation error
        
        // Solution: Create object reference
        Example obj = new Example();
        System.out.println(obj.instanceVar); // ✅ Works
    }
}
```

**Common mistake:**
```java
class Main {
    int x = 10;
    
    public static void main(String[] args) {
        System.out.println(x);  // ❌ Error - main is static
    }
}
```

**Workaround:**
```java
class Main {
    int x = 10;
    
    public static void main(String[] args) {
        Main obj = new Main();
        System.out.println(obj.x);  // ✅
    }
}
```

**Interview tip:**  
> "Static methods can only directly access static members. Instance members require object reference. Yeh fundamental Java rule hai - yaad rakho."

## **Q37. What is ClassLoader in Java?**

## **Q39. How Does the Static Keyword Impact Garbage Collection in Java?**

**Answer:**  
Static variables **hinder garbage collection** - wo JVM shutdown tak memory mein rehte hain.

**Detailed explanation:**

**1. Static variables are GC roots:**
```java
class Cache {
    static List<Data> cacheList = new ArrayList<>();  // Never GC'd
    
    void addData(Data d) {
        cacheList.add(d);  // These Data objects also never GC'd!
    }
}
```

**2. Memory leak example:**
```java
class LeakyClass {
    static List<byte[]> leakedData = new ArrayList<>();
    
    void addLeak() {
        leakedData.add(new byte[1024 * 1024]);  // 1MB never freed
    }
}
// Memory grows until OutOfMemoryError
```

**3. Impact on instances:**
```java
class Holder {
    static Holder staticRef;
    Object data = new Object();
}

Holder obj = new Holder();
Holder.staticRef = obj;  // obj now never GC'd
obj = null;  // Still reachable via staticRef - memory leak
```

**How to fix:**
```java
// 1. Clear when not needed
static List<Data> cache = new ArrayList<>();
public void clearCache() {
    cache.clear();  // Allow GC
}

// 2. Use WeakReference
static WeakReference<Data> weakData = new WeakReference<>(new Data());

// 3. Avoid static collections for large/cached data
// Use proper cache framework (Caffeine, EhCache) with eviction policies
```

**Interview tip:**  
> "Static variables are GC roots - they prevent garbage collection. In web applications, static caches cause memory leaks if not managed. Always provide cleanup mechanism for static collections."

## **Q40. Why doesn't Java support operator overloading?**

**Answer:**  
James Gosling (Java creator) intentionally excluded operator overloading for **simplicity and clarity**.

**Reasons:**

1. **Code readability suffers:**
```java
// C++ - What does this do?
a = b + c;  // Could be addition, string concat, or custom logic

// Java - Clear intent
a = b.add(c);  // Method call is explicit
```

2. **Maintenance nightmare:**
```java
// In C++, someone could define:
ComplexNumber operator+(ComplexNumber a, Customer b) { ... }
// Leads to unexpected behavior
```

3. **Java design principle - no surprises:**
```java
// In Java, + only works for:
// - Primitives: int, long, float, double
// - String concatenation (special case)
int sum = 5 + 10;           // Clear
String msg = "Hello" + "World";  // Clear (compiler optimization)
```

4. **Exception - String concatenation:**
```java
// Only allowed operator overloading in Java
String result = "Value: " + 42;  // Compiler converts to StringBuilder
```

**Alternatives in Java:**
```java
// Instead of operator overloading, use methods
BigDecimal a = new BigDecimal("10.5");
BigDecimal b = new BigDecimal("20.3");
BigDecimal sum = a.add(b);     // Instead of a + b
BigDecimal diff = a.subtract(b);
BigDecimal product = a.multiply(b);
```

**Interview tip:**  
> "Java's simplicity over convenience. If you need operator-like behavior, implement methods like `add()`, `subtract()`, `compareTo()` - they're self-documenting and maintainable."

## **Q41. Difference between this and super method?**

**Answer:**

| Feature | `this()` | `super()` |
|---------|----------|-----------|
| **Purpose** | Call another constructor in same class | Call parent class constructor |
| **Target** | Same class constructor | Parent class constructor |
| **Must be first statement** | ✅ Yes | ✅ Yes |
| **Can be used together** | ❌ No (only one can be first) | ❌ No |
| **Recursive call** | ❌ Not allowed (circular) | N/A |
| **Default if not written** | No (compiler doesn't add) | Yes (calls parent no-arg) |

**Examples:**

**`this()` - constructor chaining:**
```java
class Employee {
    int id;
    String name;
    
    Employee() {
        this(0, "Unknown");  // Call parameterized constructor
    }
    
    Employee(int id, String name) {
        this.id = id;
        this.name = name;
    }
}
```

**`super()` - parent constructor call:**
```java
class Parent {
    Parent(String value) {
        System.out.println("Parent: " + value);
    }
}

class Child extends Parent {
    Child() {
        super("Hello");  // Must call parent constructor explicitly
        // If parent has no default constructor, this is mandatory
    }
}
```

**Can't use both:**
```java
class Test {
    Test() {
        this(10);   // ✅
        // super(); // ❌ Can't have both - this() already first
    }
    
    Test(int x) {
        super();    // ✅ (implicit or explicit)
    }
}
```

**Interview tip:**  
> "`this()` and `super()` both must be first statement in constructor. So you can't use both in same constructor. Use `this()` for constructor chaining, `super()` for parent initialization."

## **Q42. Can we override Static method in Java?**

**Answer:**  
**No** - static methods cannot be overridden. They are **hidden** (method hiding).

**Demonstration:**
```java
class Parent {
    static void display() {
        System.out.println("Parent static");
    }
    
    void normal() {
        System.out.println("Parent normal");
    }
}

class Child extends Parent {
    static void display() {  // This is HIDING, not overriding
        System.out.println("Child static");
    }
    
    @Override
    void normal() {  // This is OVERRIDING
        System.out.println("Child normal");
    }
}

// Test
Parent p = new Child();
p.display();  // Output: "Parent static" (compile-time binding)
p.normal();   // Output: "Child normal" (runtime binding)
```

**Key differences:**
| Overriding (instance) | Hiding (static) |
|----------------------|-----------------|
| Runtime polymorphism | Compile-time binding |
| `@Override` annotation works | No compile check |
| Based on object type | Based on reference type |
| Virtual method invocation | No virtual invocation |

**@Override annotation on static:**
```java
class Child extends Parent {
    @Override
    static void display() { }  // ❌ Compilation error
}
```

**Interview tip:**  
> "Static methods belong to class, not instance. Overriding requires runtime object type - static methods resolve at compile time. So they can't be overridden, only hidden."

## **Q44. What is copy constructor and clone() method?**

**Answer:**  
Dono object copy karne ke tareeke hain, but implementation different hai.

**Copy Constructor:**
```java
class Employee {
    int id;
    String name;
    
    // Copy constructor
    public Employee(Employee other) {
        this.id = other.id;
        this.name = other.name;
    }
}

// Usage
Employee e1 = new Employee(1, "John");
Employee e2 = new Employee(e1);  // Copy
```

**clone() method:**
```java
class Employee implements Cloneable {
    int id;
    String name;
    
    @Override
    protected Object clone() throws CloneNotSupportedException {
        return super.clone();  // Shallow copy
    }
}

// Usage
Employee e1 = new Employee(1, "John");
Employee e2 = (Employee) e1.clone();  // Clone
```

**Comparison:**

| Feature | Copy Constructor | clone() |
|---------|------------------|---------|
| **Control** | Full control | Field-by-field copy |
| **Deep copy** | Easy to implement | Need manual override |
| **Checked exception** | No | CloneNotSupportedException |
| **Requires interface** | No | Cloneable (marker) |
| **Works with final fields** | Yes | No (clone modifies) |
| **Recommended** | ✅ Yes (Effective Java) | ❌ No (broken design) |

**Deep copy with copy constructor:**
```java
class Department {
    String name;
    Department(String name) { this.name = name; }
    Department(Department other) { this.name = other.name; }
}

class Employee {
    String name;
    Department dept;
    
    // Deep copy constructor
    Employee(Employee other) {
        this.name = other.name;
        this.dept = new Department(other.dept);  // Deep copy
    }
}
```

**Interview tip:**  
> "Joshua Bloch in Effective Java says: 'clone() is broken'. Production code mein copy constructor use karo - more control, no checked exceptions, works with final fields."

## **Q46. Why buffered input stream is good with performance?**

**Answer:**  
BufferedInputStream **reduces system calls** (native I/O operations), jo bahut expensive hoti hain.

**Without Buffering (FileInputStream):**
```java
FileInputStream fis = new FileInputStream("file.txt");
int data;
while ((data = fis.read()) != -1) {  // Each read = 1 system call
    process(data);
}
// Reading 1MB file (1,048,576 bytes) = 1,048,576 system calls!
```

**With Buffering (BufferedInputStream):**
```java
BufferedInputStream bis = new BufferedInputStream(
    new FileInputStream("file.txt"), 8192  // 8KB buffer
);
int data;
while ((data = bis.read()) != -1) {  // Reads from buffer most times
    process(data);
}
// 1MB file = ~128 system calls (8192 bytes per call)
```

**How it works:**
```
Without Buffer:
Application → read() → OS Kernel → Disk → returns 1 byte → Application
(1 million calls for 1MB)

With Buffer:
Application → read() → Buffer (has 8KB) → if data in buffer, return instantly
Buffer empty? → read() → OS Kernel → Disk → fills 8KB buffer → Application
(128 calls for 1MB)
```

**Performance comparison:**
| Operation | Without Buffer | With Buffer (8KB) |
|-----------|---------------|-------------------|
| System calls for 1MB | ~1,048,576 | ~128 |
| Time (typical) | ~2-5 seconds | ~50-100ms |

**For text reading - BufferedReader:**
```java
BufferedReader br = new BufferedReader(new FileReader("file.txt"));
String line;
while ((line = br.readLine()) != null) {  // Reads line efficiently
    process(line);
}
```

**Interview tip:**  
> "Buffering works on principle of batching - system calls are expensive (context switch to kernel). Buffer accumulates data and reduces calls from per-byte to per-buffer. Always use buffered streams for I/O-heavy operations."

## **Q47. What is Purpose of Generics in Java?**

**Answer:**  
Generics ka main purpose hai **type safety** provide karna **compile-time** par aur **explicit casting** ko avoid karna.

**Core Purposes:**

**1. Compile-time Type Safety:**
```java
// Without Generics (Before Java 5)
List list = new ArrayList();
list.add("Hello");
list.add(123);  // No compile error - can add any type
String s = (String) list.get(0);  // Fine
String t = (String) list.get(1);  // ❌ ClassCastException at runtime!

// With Generics (Java 5+)
List<String> list = new ArrayList<>();
list.add("Hello");
// list.add(123);  // ❌ Compilation error - can't add Integer
String s = list.get(0);  // No casting needed
```

**2. Eliminate Explicit Casting:**
```java
// Without Generics
List list = new ArrayList();
list.add("Hello");
String s = (String) list.get(0);  // Explicit cast needed

// With Generics
List<String> list = new ArrayList<>();
list.add("Hello");
String s = list.get(0);  // No cast - cleaner code
```

**3. Code Reusability with Type Parameters:**
```java
// Generic class
public class Box<T> {
    private T content;
    
    public void set(T content) { this.content = content; }
    public T get() { return content; }
}

// Reuse for different types
Box<String> stringBox = new Box<>();
Box<Integer> intBox = new Box<>();
Box<Employee> employeeBox = new Box<>();
```

**4. Algorithms on Multiple Types:**
```java
// Generic method
public static <T extends Comparable<T>> T max(T a, T b) {
    return a.compareTo(b) > 0 ? a : b;
}

// Works with any Comparable type
max(10, 20);           // Integer
max("Apple", "Banana"); // String
max(3.14, 2.71);       // Double
```

**5. Type Restrictions (Bounded Type Parameters):**
```java
// Restrict to Number subclasses only
public static <T extends Number> double sum(List<T> numbers) {
    double total = 0;
    for (T num : numbers) {
        total += num.doubleValue();
    }
    return total;
}

// Can call with Integer, Double, Float, etc.
sum(List.of(1, 2, 3));      // Works
// sum(List.of("a", "b"));  // ❌ Compilation error
```

**Benefits Summary:**
| Without Generics | With Generics |
|-----------------|---------------|
| Runtime ClassCastException | Compile-time error |
| Explicit casting everywhere | No casting |
| Code duplication for each type | Single implementation |
| Type safety only via documentation | Compiler-enforced safety |

**Interview tip:**  
> "Generics provide **compile-time type safety** - bugs early catch hote hain, runtime mein nahi. Also, **type erasure** hota hai - generic information runtime pe remove ho jati hai (for backward compatibility). Isliye `new T()` ya `instanceof T` possible nahi hai."

## **Q48. Explain Different Types of Encryption?**

**Answer:**  
Encryption ke **3 main types** hote hain: **Symmetric**, **Asymmetric**, aur **Hashing** (though hashing technically encryption nahi hai, often compared).

---

### **1. Symmetric Encryption (Secret Key Encryption)**

**How it works:** Same key used for encryption AND decryption.

```
Plaintext → [Encrypt with Key] → Ciphertext → [Decrypt with same Key] → Plaintext
```

**Algorithms:**
- **AES** (Advanced Encryption Standard) - Most common, 128/192/256-bit keys
- **DES** (Data Encryption Standard) - Deprecated, 56-bit key (weak)
- **3DES** (Triple DES) - Legacy systems
- **ChaCha20** - Modern alternative to AES

**Example:**
```java
// AES encryption in Java
SecretKeySpec key = new SecretKeySpec("0123456789abcdef".getBytes(), "AES");
Cipher cipher = Cipher.getInstance("AES");
cipher.init(Cipher.ENCRYPT_MODE, key);
byte[] encrypted = cipher.doFinal("Secret Message".getBytes());
```

**Pros & Cons:**
| Pros | Cons |
|------|------|
| Very fast | Key distribution problem |
| Suitable for large data | Need secure channel to share key |
| Low computational overhead | Many parties need same key |

**Use Cases:** File encryption, Database encryption, VPN (once key exchanged), Wi-Fi (WPA2)

---

### **2. Asymmetric Encryption (Public Key Encryption)**

**How it works:** Two keys - **Public key** (encryption) and **Private key** (decryption).

```
Plaintext → [Encrypt with Public Key] → Ciphertext → [Decrypt with Private Key] → Plaintext
```

**Algorithms:**
- **RSA** (Rivest-Shamir-Adleman) - Most common, 2048/4096-bit keys
- **ECC** (Elliptic Curve Cryptography) - Newer, smaller keys, more efficient
- **DSA** (Digital Signature Algorithm) - Signatures only
- **Diffie-Hellman** - Key exchange

**Example:**
```java
// RSA in Java
KeyPairGenerator keyGen = KeyPairGenerator.getInstance("RSA");
keyGen.initialize(2048);
KeyPair pair = keyGen.generateKeyPair();
PublicKey publicKey = pair.getPublic();
PrivateKey privateKey = pair.getPrivate();

Cipher cipher = Cipher.getInstance("RSA");
cipher.init(Cipher.ENCRYPT_MODE, publicKey);
byte[] encrypted = cipher.doFinal("Secret".getBytes());

cipher.init(Cipher.DECRYPT_MODE, privateKey);
byte[] decrypted = cipher.doFinal(encrypted);
```

**Pros & Cons:**
| Pros | Cons |
|------|------|
| No key distribution problem | Very slow (100-1000x slower than symmetric) |
| Digital signatures possible | Not for large data |
| Perfect for key exchange | More computational overhead |

**Use Cases:** SSL/TLS (HTTPS), Email encryption (PGP), Digital signatures, Crypto-currencies

---

### **3. Hashing (One-way Encryption)**

**How it works:** One-way function - can't decrypt back to original.

```
Plaintext → [Hash Function] → Fixed-length Hash (can't reverse)
```

**Algorithms:**
- **SHA-256** (Secure Hash Algorithm) - 256-bit output
- **SHA-512** - 512-bit output
- **bcrypt** - Password hashing (with salt + work factor)
- **MD5** - Deprecated (broken, collisions found)

**Example:**
```java
// SHA-256
MessageDigest md = MessageDigest.getInstance("SHA-256");
byte[] hash = md.digest("password123".getBytes());

// bcrypt for passwords
String hashedPassword = BCrypt.hashpw("password123", BCrypt.gensalt(12));
```

**Pros & Cons:**
| Pros | Cons |
|------|------|
| Can't reverse (one-way) | Can't get original back (not encryption actually) |
| Fixed output length | Collision possibility (rare) |
| Fast | Rainbow table attacks (mitigate with salt) |

**Use Cases:** Passwords storage, Data integrity checks, Digital signatures (as part), Blockchain

---

### **Comparison Table:**

| Feature | Symmetric | Asymmetric | Hashing |
|---------|-----------|------------|---------|
| **Keys** | 1 secret key | 2 keys (public + private) | No keys |
| **Reversible** | ✅ Yes | ✅ Yes | ❌ No (one-way) |
| **Speed** | Very fast | Slow | Fast |
| **Key distribution** | Problem | No problem | N/A |
| **Typical key size** | 128-256 bits | 2048-4096 bits | 256-512 bits output |
| **Used for** | Bulk data encryption | Key exchange, signatures | Passwords, integrity |

---

### **Real-world Example - HTTPS:**
```
1. Client → Server: "Hello, I want HTTPS"
2. Server → Client: Sends Public Key (asymmetric)
3. Client generates random Session Key (symmetric)
4. Client encrypts Session Key with Server's Public Key
5. Server decrypts with Private Key → gets Session Key
6. Now both communicate using fast Symmetric encryption (AES)
```

**Interview tip:**  
> "Symmetric fast hai but key sharing problem hai. Asymmetric secure hai but slow. Real world mein **hybrid approach** use hota hai - asymmetric se symmetric key exchange, phir symmetric se actual data encryption. Password store karne ke liye **hashing + salt** use karo (bcrypt recommended)."

## **Q49. What do you know about the Big O notation?**

**Answer:**  
Big O notation **algorithm ki performance** measure karta hai **input size ke respect mein** - specifically **worst-case time complexity** (ya space complexity). Yeh batata hai ki input badhne par algorithm kitna slow hoga.

**Core Concept:**
Big O describes **growth rate** - not exact time (jo machine-dependent hai).

```
n = input size
Time = f(n) ~ O(g(n))  where g(n) is the growth function
```

---

### **Common Big O Complexities (Fastest to Slowest):**

| Big O | Name | Example | n=100 | n=10,000 |
|-------|------|---------|-------|----------|
| **O(1)** | Constant | Array access, hash map get | 1 step | 1 step |
| **O(log n)** | Logarithmic | Binary search | ~7 steps | ~14 steps |
| **O(n)** | Linear | Simple loop, array traversal | 100 steps | 10,000 steps |
| **O(n log n)** | Linearithmic | Merge sort, Quick sort | ~700 steps | ~140,000 steps |
| **O(n²)** | Quadratic | Bubble sort, nested loops | 10,000 steps | 100M steps |
| **O(2ⁿ)** | Exponential | Recursive Fibonacci | 1.2M steps | Astronomical |
| **O(n!)** | Factorial | Traveling salesman (brute force) | 9e157 steps | Impossible |

---

### **Examples with Code:**

**1. O(1) - Constant Time:**
```java
int getFirst(int[] arr) {
    return arr[0];  // Always 1 operation
}

// Hash map lookup
map.get(key);  // O(1) average case
```

**2. O(log n) - Logarithmic Time:**
```java
// Binary search
int binarySearch(int[] sortedArr, int target) {
    int left = 0, right = sortedArr.length - 1;
    while (left <= right) {
        int mid = left + (right - left) / 2;
        if (sortedArr[mid] == target) return mid;
        if (sortedArr[mid] < target) left = mid + 1;
        else right = mid - 1;
    }
    return -1;
}
// Each iteration halves the search space
```

**3. O(n) - Linear Time:**
```java
int sum(int[] arr) {
    int total = 0;
    for (int num : arr) {  // Iterates n times
        total += num;
    }
    return total;
}
```

**4. O(n log n) - Linearithmic:**
```java
// Merge sort, Quick sort, Heap sort
Arrays.sort(arr);  // Java's Dual-Pivot Quick Sort - O(n log n)

// Simple example - n * log n
for (int i = 0; i < n; i++) {        // O(n)
    for (int j = 1; j < n; j *= 2) { // O(log n)
        // operation
    }
}
```

**5. O(n²) - Quadratic Time:**
```java
// Bubble sort / Nested loops
void bubbleSort(int[] arr) {
    for (int i = 0; i < n; i++) {        // O(n)
        for (int j = 0; j < n - 1; j++) { // O(n)
            if (arr[j] > arr[j+1]) {
                swap(arr, j, j+1);
            }
        }
    }
}
```

**6. O(2ⁿ) - Exponential:**
```java
// Recursive Fibonacci (without memoization)
int fib(int n) {
    if (n <= 1) return n;
    return fib(n-1) + fib(n-2);  // 2 calls per recursion
}
// fib(50) would take years to compute!
```

---

### **Space Complexity (Memory usage):**

Big O can also measure **extra memory** needed:

```java
// O(1) space - no extra array
int sum(int[] arr) {
    int total = 0;  // single variable
    return total;
}

// O(n) space - creates new array
int[] copy(int[] arr) {
    int[] newArr = new int[arr.length];  // size = n
    return newArr;
}

// O(n²) space - 2D matrix
int[][] matrix = new int[n][n];
```

---

### **Rules for Calculating Big O:**

1. **Drop constants:**
   - O(2n) → O(n)
   - O(n/2) → O(n)
   - O(100) → O(1)

2. **Drop lower terms:**
   - O(n² + n) → O(n²)
   - O(n + log n) → O(n)

3. **Different inputs = different variables:**
```java
for (int i = 0; i < n; i++) {      // O(n)
    for (int j = 0; j < m; j++) {  // O(m)
        // operation
    }
}
// Total: O(n * m) - NOT O(n²)
```

---

### **Big O in Interviews - Common Questions:**

**Q: What's complexity of HashMap get/put?**
- Average: O(1)
- Worst-case (many collisions): O(n)

**Q: ArrayList vs LinkedList get operation?**
- ArrayList: O(1) (direct index access)
- LinkedList: O(n) (need to traverse)

**Q: Why is binary search O(log n)?**
- Because each step eliminates half the elements
- log₂(1,000,000) ≈ 20 steps only

**Q: How to improve O(n²) to O(n log n)?**
- Use better sorting algorithm (Merge sort instead of Bubble sort)
- Use HashMap for lookups instead of nested loops

---

### **Real-world Example - Finding duplicates:**

```java
// Bad: O(n²)
for (int i = 0; i < n; i++) {
    for (int j = i+1; j < n; j++) {
        if (arr[i] == arr[j]) return true;
    }
}

// Good: O(n) with HashSet
Set<Integer> seen = new HashSet<>();
for (int num : arr) {
    if (seen.contains(num)) return true;
    seen.add(num);
}
```

**Interview tip:**  
> "Big O batata hai ki input size double hone par time kaise change hoga:
> - O(1): Same time
> - O(log n): ~1 extra step
> - O(n): Double time
> - O(n²): 4x time
> 
> Interview mein hamesha worst-case complexity batao unless specifically asked for average case. Space complexity bhi mention karo if relevant."
