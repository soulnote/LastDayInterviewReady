
## **Q1. What is final keyword in Java?**

**Answer:**  
`final` keyword **entity ko modify hone se rokta hai** - variable, method, ya class ke level par.

**Three usages:**

**1. final variable - constant (cannot reassign):**
```java
final int MAX_VALUE = 100;
// MAX_VALUE = 200;  // ❌ Compilation error

// Blank final variable (initialized in constructor)
final int blankFinal;
MyClass() {
    blankFinal = 10;  // Allowed only once
}

// final reference - reference can't change, but object can
final List<String> list = new ArrayList<>();
list.add("Hello");  // ✅ Allowed (object state can change)
// list = new ArrayList<>();  // ❌ Can't reassign reference
```

**2. final method - cannot override:**
```java
class Parent {
    final void display() {
        System.out.println("Parent");
    }
}

class Child extends Parent {
    // void display() { }  // ❌ Cannot override final method
}
```

**3. final class - cannot extend:**
```java
final class ImmutableClass {
    // Class cannot be subclassed
}

// class Child extends ImmutableClass { }  // ❌ Compilation error
```

**Real-world examples:**
```java
// 1. String class is final
public final class String { }  // Can't extend String

// 2. Method parameters as final
public void process(final int value) {
    // value = 10;  // ❌ Can't modify parameter
}

// 3. In anonymous inner classes (Java 7 and below)
final String message = "Hello";
Runnable r = new Runnable() {
    public void run() {
        System.out.println(message);  // Needed final (Java 7)
    }
};
// Java 8+ - effectively final works
```

**Performance benefit:**
```java
// JVM can optimize final methods (inline expansion)
// final method calls can be resolved at compile time
```

**Interview tip:** "Final ka matlab **constant** (variable), **can't override** (method), **can't extend** (class). Final variables ko constructor mein initialize kar sakte ho. Final methods JVM optimize kar sakta hai. String class final hai - security aur immutability ke liye."

## **Q2. What is the finalize() method?**

**Answer:**  
`finalize()` method **Object class** ka method hai - **Garbage Collector object delete karne se pehle call karta hai** (deprecated in Java 9+).

**Basic usage:**
```java
class Resource {
    @Override
    protected void finalize() throws Throwable {
        try {
            // Cleanup resources (close files, connections)
            System.out.println("Cleaning up...");
        } finally {
            super.finalize();  // Always call parent
        }
    }
}
```

**When finalize() is called:**
```java
Resource obj = new Resource();
obj = null;  // Eligible for GC
System.gc();  // Request GC (may call finalize)
// finalize() called BEFORE object is garbage collected
```

**Problems with finalize():**
```java
// 1. Unpredictable timing - GC may never run
// 2. Performance overhead
// 3. Can resurrect object (bad practice)
class BadExample {
    static BadExample instance;
    
    @Override
    protected void finalize() {
        instance = this;  // Object becomes reachable again!
    }
}
```

**Modern replacements (Java 9+):**
```java
// 1. Try-with-resources (AutoCloseable)
try (FileInputStream fis = new FileInputStream("file.txt")) {
    // Auto-closed
}

// 2. Cleaner (Java 9+)
public class Resource implements AutoCloseable {
    private final Cleaner cleaner = Cleaner.create();
    private Cleaner.Cleanable cleanable;
    
    public Resource() {
        cleanable = cleaner.register(this, () -> {
            // Cleanup code
        });
    }
    
    @Override
    public void close() {
        cleanable.clean();
    }
}
```

**Interview tip:** "`finalize()` is **deprecated** (Java 9) and **deprecated for removal** (Java 17+). Never use it. It's unpredictable, slow, and can cause issues. Use `try-with-resources` or `Cleaner` instead."

## **Q3. Difference between throw and throws**

**Answer:**

| Feature | throw | throws |
|---------|-------|--------|
| **Purpose** | Actually throw exception | Declare exception might be thrown |
| **Usage** | Inside method body | Method signature |
| **Followed by** | Exception object (new Exception()) | Exception class names |
| **Multiple exceptions** | Only one at a time | Can declare multiple (comma separated) |
| **Flow** | Immediately exits method | Just declaration |

**throw example:**
```java
public void validateAge(int age) {
    if (age < 18) {
        throw new IllegalArgumentException("Age must be 18+");  // Actual throwing
    }
}
```

**throws example:**
```java
public void readFile(String path) throws IOException, SQLException {
    // Declares that this method might throw these exceptions
    FileInputStream fis = new FileInputStream(path);  // May throw IOException
}
```

**Both together:**
```java
public void process(String data) throws CustomException {
    if (data == null) {
        throw new CustomException("Data is null");  // throw
    }
}
```

**Custom exception with throw:**
```java
class InsufficientFundsException extends Exception {
    public InsufficientFundsException(String message) {
        super(message);
    }
}

public void withdraw(double amount) throws InsufficientFundsException {
    if (amount > balance) {
        throw new InsufficientFundsException("Balance low");
    }
}
```

**Interview tip:** "`throw` = **action** (actually throwing exception). `throws` = **declaration** (method can throw this). Remember: throw + object, throws + class."

## **Q4. Can we re-throw exception in Java?**

**Answer:**  
**Yes** - catch block mein exception ko **dobara throw** kar sakte ho.

**Basic re-throw:**
```java
try {
    // Some code
} catch (IOException e) {
    System.out.println("Logging error: " + e.getMessage());
    throw e;  // Re-throwing same exception
}
```

**Re-throw with different exception (exception chaining):**
```java
try {
    // Database operation
} catch (SQLException e) {
    // Wrap and throw custom exception
    throw new BusinessException("Database operation failed", e);
}
```

**Re-throw with type narrowing (Java 7+):**
```java
public void process() throws IOException, SQLException {
    try {
        // Code that may throw IOException or SQLException
    } catch (Exception e) {
        // Java 7+ - re-throw as actual type, not just Exception
        throw e;  // Preserves original exception type
    }
}
```

**Real-world pattern - Log and rethrow:**
```java
public class ServiceLayer {
    public User getUser(Long id) throws ServiceException {
        try {
            return userRepository.findById(id);
        } catch (DataAccessException e) {
            logger.error("Failed to fetch user: {}", id, e);
            throw new ServiceException("User retrieval failed", e);  // Re-throw as business exception
        }
    }
}
```

**Re-throw after cleanup:**
```java
public void processFile(String path) throws IOException {
    FileInputStream fis = null;
    try {
        fis = new FileInputStream(path);
        // process
    } catch (IOException e) {
        throw e;  // Re-throw after logging
    } finally {
        if (fis != null) {
            try {
                fis.close();
            } catch (IOException e) {
                // Can't throw here - might mask original exception
                logger.error("Error closing file", e);
            }
        }
    }
}
```

**Interview tip:** "Re-throw useful for **logging + propagation**, **exception translation** (wrap in business exception), and **adding context** without losing original stack trace. Always preserve cause chain using constructor with `Throwable cause`."

## **Q5. How can we avoid null pointer exception?**

**Answer:**  
NullPointerException (NPE) se bachne ke liye **defensive programming** karo.

**Strategies:**

**1. Check for null before access (most basic):**
```java
// ❌ Bad
String name = user.getName().toUpperCase();

// ✅ Good
if (user != null && user.getName() != null) {
    String name = user.getName().toUpperCase();
}
```

**2. Use Objects.requireNonNull() (fail fast):**
```java
public void process(User user) {
    Objects.requireNonNull(user, "User cannot be null");
    // Now safe to use user
}

// With default value
String name = Objects.requireNonNullElse(user.getName(), "Unknown");
```

**3. Use Optional (Java 8+):**
```java
// ❌ Bad
String city = user.getAddress().getCity().toLowerCase();

// ✅ Good
String city = Optional.ofNullable(user)
    .map(User::getAddress)
    .map(Address::getCity)
    .map(String::toLowerCase)
    .orElse("Unknown");
```

**4. Return empty collections instead of null:**
```java
// ❌ Bad
public List<String> getNames() {
    if (noData) return null;  // Risk of NPE
}

// ✅ Good
public List<String> getNames() {
    if (noData) return Collections.emptyList();  // Safe
}
```

**5. Use StringUtils (Apache Commons) or similar:**
```java
import org.apache.commons.lang3.StringUtils;

// Null-safe operations
boolean isEmpty = StringUtils.isEmpty(str);
String upper = StringUtils.upperCase(str);  // Returns null if input null
```

**6. Ternary operator for defaults:**
```java
String value = (input != null) ? input : "default";
// Or using Objects.toString()
String value = Objects.toString(input, "default");
```

**7. Annotations for nullability (compile-time checking):**
```java
import javax.annotation.Nullable;
import javax.annotation.Nonnull;

public class UserService {
    public @Nonnull User getUser(@Nonnull Long id) {
        // Compiler warns if null passed
    }
    
    public @Nullable String getOptionalData() {
        // Caller knows to check for null
    }
}
```

**8. Use pattern like Null Object pattern:**
```java
// Instead of null, create a null object
interface Logger {
    void log(String msg);
}

class NullLogger implements Logger {
    @Override
    public void log(String msg) { }  // Do nothing
}

// Use NullLogger instead of null
private Logger logger = new NullLogger();  // Never null
```

**9. Avoid chained calls:**
```java
// ❌ Risky
String city = user.getAddress().getCity().getName();

// ✅ Safe
if (user != null) {
    Address addr = user.getAddress();
    if (addr != null) {
        City city = addr.getCity();
        if (city != null) {
            String name = city.getName();
        }
    }
}

// ✅ Better with Optional
String city = Optional.ofNullable(user)
    .map(User::getAddress)
    .map(Address::getCity)
    .map(City::getName)
    .orElse("Unknown");
```

**Interview tip:** "NPE avoid karne ke liye: **Optional**, **Objects.requireNonNull**, **null checks**, **empty collections instead of null**, **annotations**. Best practice - fail fast with clear message: `Objects.requireNonNull(param, "param cannot be null")`."

## **Q6. Difference between checked and unchecked exceptions**

**Answer:**

| Feature | Checked Exception | Unchecked Exception |
|---------|-------------------|---------------------|
| **Compile-time check** | ✅ Must handle or declare | ❌ Not required |
| **Extends** | `Exception` (but not `RuntimeException`) | `RuntimeException` |
| **Recovery possibility** | Usually recoverable | Usually programming error |
| **Examples** | IOException, SQLException, ClassNotFoundException | NullPointerException, IllegalArgumentException, ArrayIndexOutOfBoundsException |
| **Code pollution** | More (try-catch or throws) | Less |

**Checked exception example:**
```java
// Must handle or declare
public void readFile(String path) throws IOException {  // Must declare
    FileInputStream fis = new FileInputStream(path);  // May throw IOException
    // Or handle with try-catch
}

// Calling code must also handle
try {
    readFile("test.txt");
} catch (IOException e) {
    // Must handle
}
```

**Unchecked exception example:**
```java
// No need to declare or handle
public void divide(int a, int b) {
    if (b == 0) {
        throw new ArithmeticException("Division by zero");  // Unchecked
    }
}

// Calling code doesn't need try-catch
divide(10, 0);  // Compiles fine, fails at runtime
```

**Hierarchy:**
```
Throwable
    ├── Error (unchecked - JVM errors)
    │       ├── OutOfMemoryError
    │       └── StackOverflowError
    │
    └── Exception
            ├── RuntimeException (unchecked)
            │       ├── NullPointerException
            │       ├── IllegalArgumentException
            │       └── ArithmeticException
            │
            └── (Other Exception classes) - checked
                    ├── IOException
                    ├── SQLException
                    └── ClassNotFoundException
```

**When to use which:**
```java
// Checked - caller can recover
public void transferMoney(Account from, Account to, double amount) 
    throws InsufficientFundsException, AccountNotFoundException {
    // Caller can handle (show message, retry, etc.)
}

// Unchecked - programming error
public void setAge(int age) {
    if (age < 0 || age > 150) {
        throw new IllegalArgumentException("Invalid age: " + age);
    }
}
```

**Interview tip:** "Checked exceptions **compile-time enforceable** - for recoverable conditions. Unchecked exceptions **runtime** - for programming errors. Spring often uses unchecked (RuntimeException) to avoid code clutter. Modern thinking: Unchecked for most cases, Checked only when caller can meaningfully recover."

## **Q7. How Do You Handle InterruptedException?**

**Answer:**  
`InterruptedException` tab aati hai jab **thread sleeping/waiting ko interrupt kiya jata hai**. Isko handle karna important hai for graceful shutdown.

**Correct ways to handle:**

**1. Propagate it (if your method can throw it):**
```java
public void doWork() throws InterruptedException {
    Thread.sleep(5000);  // Let caller handle
}
```

**2. Restore interrupt status (if can't throw):**
```java
public void run() {
    try {
        Thread.sleep(10000);
    } catch (InterruptedException e) {
        // Restore interrupted status
        Thread.currentThread().interrupt();  // CRITICAL!
        // Now upper layers know thread was interrupted
    }
}
```

**3. Handle and cleanup (for Runnable):**
```java
@Override
public void run() {
    try {
        while (!Thread.currentThread().isInterrupted()) {
            // Do work
            Thread.sleep(1000);
        }
    } catch (InterruptedException e) {
        // Clean up resources
        cleanup();
        // Restore interrupt status
        Thread.currentThread().interrupt();
    }
}
```

**Common mistakes:**
```java
// ❌ WRONG - Swallowing the interrupt
try {
    Thread.sleep(1000);
} catch (InterruptedException e) {
    // Do nothing - BAD! Lost interrupt signal
}

// ❌ WRONG - Logging only
catch (InterruptedException e) {
    logger.error("Interrupted", e);
    // Still lost the interrupt status
}
```

**Best practice pattern:**
```java
public class WorkerThread implements Runnable {
    private volatile boolean running = true;
    
    public void shutdown() {
        running = false;
        Thread.currentThread().interrupt();
    }
    
    @Override
    public void run() {
        while (running && !Thread.currentThread().isInterrupted()) {
            try {
                // Blocking operation
                Thread.sleep(1000);
            } catch (InterruptedException e) {
                // Restore interrupt status
                Thread.currentThread().interrupt();
                break;  // Exit loop
            }
        }
        
        // Cleanup
        cleanup();
    }
}
```

**In ExecutorService:**
```java
ExecutorService executor = Executors.newSingleThreadExecutor();
Future<?> future = executor.submit(task);

try {
    future.get(5, TimeUnit.SECONDS);
} catch (InterruptedException e) {
    // Restore interrupt
    Thread.currentThread().interrupt();
    // Cancel the task
    future.cancel(true);
}
```

**Interview tip:** "Never swallow InterruptedException. Always **restore interrupt status** using `Thread.currentThread().interrupt()`. This allows upper layers to detect interruption. For Runnable, either propagate or restore + cleanup."

## **Q8. Does Finally always execute in Java?**

**Answer:**  
**Almost always** - but there are **rare exceptions**.

**Normal cases (finally executes):**
```java
// 1. No exception
try {
    // code
} finally {
    System.out.println("Always executes");
}

// 2. Exception occurs
try {
    throw new RuntimeException();
} finally {
    System.out.println("Still executes");
}

// 3. Return in try
try {
    return 1;
} finally {
    System.out.println("Executes before return");  // YES
}

// 4. Break/continue in loop
for(int i=0;i<10;i++) {
    try {
        break;
    } finally {
        System.out.println("Executes");  // YES
    }
}
```

**Scenarios where finally does NOT execute:**

**1. System.exit():**
```java
try {
    System.exit(0);  // JVM shuts down
} finally {
    System.out.println("NEVER executes");  // ❌
}
```

**2. JVM crash:**
```java
try {
    // Native code causing JVM crash
    // Or infinite loop causing stack overflow
} finally {
    // Never reaches here
}
```

**3. Daemon thread termination:**
```java
Thread daemon = new Thread(() -> {
    try {
        // code
    } finally {
        // May not execute if main thread exits
    }
});
daemon.setDaemon(true);
daemon.start();
// Main thread ends - daemon killed, finally may not run
```

**4. Infinite loop before finally:**
```java
try {
    while(true) { }  // Never exits
} finally {
    // Never reaches
}
```

**Important - finally vs return:**
```java
// What does this return?
public int test() {
    try {
        return 1;
    } finally {
        return 2;  // This overrides!
    }
}
// Returns 2 (not 1) - finally's return wins
```

**Interview tip:** "Finally **almost always executes** - except `System.exit()`, JVM crash, daemon thread termination, or infinite loop. Finally block is for cleanup (close files, connections). Don't put `return` in finally - it overrides try/catch returns."

## **Q9. In what scenarios will a finally block not be executed?**

**Answer:** (Detailed version of above)

**Scenarios (remember as "SODI"):**

**1. System.exit():**
```java
try {
    System.exit(0);  // Or Runtime.getRuntime().exit(0)
} finally {
    // ❌ NOT executed
}
```

**2. JVM Crash/Error:**
```java
try {
    // Native method crash
    // Or
    int[] arr = new int[-1];  // NegativeArraySizeException but JVM may crash
} finally {
    // May not execute
}
```

**3. Daemon thread termination:**
```java
Thread t = new Thread(() -> {
    try {
        // work
    } finally {
        // May not execute if main thread exits
    }
});
t.setDaemon(true);
t.start();
// Main ends → JVM exits without waiting for daemon
```

**4. Infinite loop/power loss:**
```java
try {
    while(true) { }  // Never exits
} finally {
    // Never reaches
}
// Also: OS kills process, power failure
```

**5. Thread death (Thread.stop() - deprecated):**
```java
Thread t = new Thread(() -> {
    try {
        while(true) { }
    } finally {
        // May not execute if Thread.stop() called
    }
});
t.start();
t.stop();  // Deprecated - unpredictable finally execution
```

**Demonstration:**
```java
public class FinallyTest {
    public static void main(String[] args) {
        // Case 1: System.exit()
        try {
            System.out.println("In try");
            System.exit(0);
        } finally {
            System.out.println("Finally - WILL NOT print");

        // Case 2: Normal - prints
        try {
            System.out.println("In try 2");
        } finally {
            System.out.println("Finally - WILL print");
        }
    }
}
```

**Interview tip:** "Finally not executed in **SODI**: **S**ystem.exit(), **O**S crash, **D**aemon thread, **I**nfinite loop. For normal flow, finally always executes. Use shutdown hooks for cleanup on System.exit()."

## **Q10. What exactly happens inside the JVM when a NullPointerException is thrown?**

**Answer:**  
Yeh **deep JVM level** question hai - SDE-2 ke liye impressive answer.

**Step-by-step JVM internals:**

**1. Bytecode level:**
```java
// Java code
String s = null;
s.length();

// Bytecode
ALOAD 1      // Load reference from local variable
INVOKEVIRTUAL java/lang/String.length()I  // Try to call method
// JVM checks reference - sees it's null
// Instead of executing, throws exception
```

**2. JVM's null check mechanism:**
```java
// Inside JVM (conceptual C++ code)
if (reference == NULL) {
    EXCEPTION_MARK = true;
    THROW(createNullPointerException());
    return;
}
// else proceed with method invocation
```

**3. What JVM does:**
```java
// 1. Detects null reference during bytecode execution
// 2. Stops normal execution
// 3. Creates NullPointerException object
NullPointerException npe = new NullPointerException();
// 4. Fills stack trace (expensive operation)
npe.fillInStackTrace();
// 5. Sets exception as current pending exception
// 6. Searches for exception handler in current method
// 7. If not found, pops stack frame and continues search
// 8. When found, jumps to exception handler code
// 9. If none found, terminates thread (or JVM if main thread)
```

**Memory impact:**
```java
// Stack trace filling is expensive
public void badMethod() {
    try {
        // code
    } catch (NullPointerException e) {
        // e.printStackTrace();  // VERY expensive - string concat + I/O
    }
}

// Better - create exception only when needed
if (value == null) {
    throw new IllegalArgumentException("Value cannot be null");  // More informative
}
```

**JIT compiler optimization:**
```java
// JIT may optimize null checks in hot code
// Implicit null checks vs explicit
for (int i = 0; i < 1_000_000; i++) {
    // JIT may move null check outside loop
    obj.method();  // Implicit null check
}
```

**Performance consideration:**
```java
// Exception creation is expensive
// Each NullPointerException captures entire stack
// Typical cost: 1-5 microseconds for exception creation
// But if thrown frequently, can be huge

// ❌ Bad - using exception for control flow
try {
    String s = map.get(key);  // May return null
    s.length();
} catch (NullPointerException e) {
    // Handle missing key
}

// ✅ Good - explicit check
String s = map.get(key);
if (s == null) {
    // Handle missing key
}
```

**Interview tip:** "JVM detects null reference during bytecode execution (ALOAD + INVOKEVIRTUAL), creates NPE object, fills stack trace (expensive), searches exception handler, unwinds stack. That's why exceptions shouldn't be used for control flow - stack trace filling is heavy."

## **Q12. Why child methods can't throw broader exceptions?**

**Answer:**  
Yeh **Liskov Substitution Principle (LSP)** ka violation hota hai.

**Rule:**
- Child class method can throw **same** or **subclass** (narrower) exceptions
- Cannot throw **parent** or **broader** exceptions

**Example:**
```java
class Parent {
    void process() throws IOException {
        // Can throw IOException
    }
}

// ✅ Allowed - Same exception
class Child1 extends Parent {
    void process() throws IOException { }
}

// ✅ Allowed - Narrower exception (subclass)
class Child2 extends Parent {
    void process() throws FileNotFoundException { }  // FileNotFoundException extends IOException
}

// ❌ NOT allowed - Broader exception
class Child3 extends Parent {
    void process() throws Exception { }  // Exception is broader than IOException
}
```

**Why? - Liskov Substitution Principle:**
```java
Parent p = new Child3();  // Child3 violates LSP
try {
    p.process();  // Caller expects IOException
} catch (IOException e) {
    // But Child3.process() might throw Exception (broader)
    // This catch block won't catch Exception!
}
```

**For unchecked exceptions (different rule):**
```java
class Parent {
    void process() {  // No throws declaration
        // Can throw unchecked
    }
}

class Child extends Parent {
    // ✅ Allowed - Unchecked exceptions don't need declaration
    void process() throws NullPointerException { }  // Allowed
}
```

**Interface rule:**
```java
interface DataReader {
    String read() throws IOException;
}

// ❌ Not allowed
class BadReader implements DataReader {
    public String read() throws Exception { }  // Broader - not allowed
}

// ✅ Allowed
class GoodReader implements DataReader {
    public String read() throws FileNotFoundException { }  // Narrower - allowed
}
```

**Interview tip:** "LSP ke according, child class **parent ki jagah substitute** ho sakni chahiye. Agar child broader exception throw karega, to caller code (jo parent exception handle kar raha hai) break ho jayega. Rule: Child can throw **same** or **subclass** (narrower) exceptions, never broader."

## **Q13. What happens if an exception occurs inside a thread?**

**Answer:**  
Thread ke andar exception **thread ko terminate** kar deta hai, but **other threads** continue running.

**Default behavior:**
```java
Thread t = new Thread(() -> {
    System.out.println("Thread started");
    throw new RuntimeException("Something went wrong");
    System.out.println("This never prints");  // Not executed
});

t.start();
System.out.println("Main continues");  // Still prints
```

**Output:**
```
Thread started
Main continues
Exception in thread "Thread-0" java.lang.RuntimeException: Something went wrong
```

**Uncaught exception handler (set per thread):**
```java
Thread t = new Thread(() -> {
    throw new RuntimeException("Error in thread");
});

t.setUncaughtExceptionHandler((thread, exception) -> {
    System.err.println("Thread " + thread.getName() + " crashed: " + exception);
    // Cleanup, logging, restart thread
});

t.start();
```

**Default uncaught exception handler (for all threads):**
```java
Thread.setDefaultUncaughtExceptionHandler((thread, exception) -> {
    logger.error("Uncaught exception in thread: {}", thread.getName(), exception);
    // Send alert, log to monitoring
});

// Now all threads use this handler by default
```

**Exception in thread pool:**
```java
ExecutorService executor = Executors.newFixedThreadPool(10);

// Exception in task doesn't kill thread pool
Future<?> future = executor.submit(() -> {
    throw new RuntimeException("Task failed");
});

try {
    future.get();  // Throws ExecutionException
} catch (ExecutionException e) {
    Throwable cause = e.getCause();  // Original exception
    System.out.println("Task failed: " + cause.getMessage());
}

// Thread pool continues - other tasks still run
executor.submit(() -> System.out.println("Still works"));
```

**Exception swallower (what not to do):**
```java
// ❌ BAD - Silent failure
Thread t = new Thread(() -> {
    try {
        riskyOperation();
    } catch (Exception e) {
        // Do nothing - BAD! No one knows thread failed
    }
});

// ✅ GOOD - Log and handle
Thread t = new Thread(() -> {
    try {
        riskyOperation();
    } catch (Exception e) {
        logger.error("Thread failed", e);
        // Notify monitoring, attempt recovery
    }
});
```

**Complete thread exception handling pattern:**
```java
public class RobustThread extends Thread {
    @Override
    public void run() {
        try {
            // Main work
            doWork();
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            logger.info("Thread interrupted, shutting down");
        } catch (Exception e) {
            logger.error("Unexpected error in thread", e);
            // Attempt recovery or restart
            restart();
        } finally {
            cleanup();
        }
    }
    
    private void restart() {
        // Start new thread to replace this one
        new RobustThread().start();
    }
}
```

**Interview tip:** "Thread exception ka default behavior - thread dies, other threads unaffected. Use **UncaughtExceptionHandler** for logging/cleanup. Thread pool mein, exception task fails but thread reused. Never swallow exceptions - always log. For critical threads, implement **restart logic**."
