### **Q2. What is the default method in Java 8?**

**Answer:**  
Default method interface mein **implementation provide** karne ka tareeka hai. Isse **backward compatibility** milti hai.

**Syntax and usage:**
```java
interface Vehicle {
    // Abstract method - must implement
    void start();
    
    // Default method - optional to override
    default void honk() {
        System.out.println("Vehicle honks");
    }
    
    // Static method in interface
    static void service() {
        System.out.println("Service vehicle");
    }
}

class Car implements Vehicle {
    @Override
    public void start() {
        System.out.println("Car starting");
    }
    // honk() optionally override
}

// Usage
Car car = new Car();
car.start();  // Car starting
car.honk();   // Vehicle honks
Vehicle.service();  // Service vehicle
```

**Why needed?**
```java
// Problem: Adding new method to existing interface breaks all implementations
interface OldInterface {
    void existingMethod();
    // void newMethod();  // Would break all existing classes!
}

// Solution: Default method
interface NewInterface {
    void existingMethod();
    
    default void newMethod() {
        // Default implementation - existing classes work fine
    }
}
```

**Diamond Problem Resolution:**
```java
interface A {
    default void m1() { System.out.println("A"); }
}

interface B {
    default void m1() { System.out.println("B"); }
}

class C implements A, B {
    // Must override due to conflict
    @Override
    public void m1() {
        A.super.m1();  // Choose which one
        // Or provide new implementation
    }
}
```

**Interview tip:** "Default methods allow adding methods to interfaces without breaking existing code. Used extensively in Java 8 Collections API (forEach, stream). Multiple inheritance conflict resolve karna padega manually."

### **Q4. What are Types of method reference in Java 8?**

**Answer:** 4 types hote hain:

**1. Static Method Reference - `Class::staticMethod`**
```java
// Lambda
Function<Integer, String> lambda = i -> String.valueOf(i);

// Method reference
Function<Integer, String> ref = String::valueOf;

// Example
List<Integer> numbers = Arrays.asList(1, 2, 3);
numbers.stream()
       .map(String::valueOf)  // Static method reference
       .forEach(System.out::println);
```

**2. Instance Method of Particular Object - `object::instanceMethod`**
```java
// Lambda
List<String> list = Arrays.asList("a", "b");
list.forEach(s -> System.out.println(s));

// Method reference
list.forEach(System.out::println);

// Custom object
class Printer {
    void print(String s) { System.out.println(s); }

Printer printer = new Printer();
list.forEach(printer::print);
```

**3. Instance Method of Arbitrary Object - `Class::instanceMethod`**
```java
// Lambda
List<String> names = Arrays.asList("John", "Jane");
names.sort((a, b) -> a.compareToIgnoreCase(b));

// Method reference
names.sort(String::compareToIgnoreCase);

// Another example
List<Integer> nums = Arrays.asList(1, 2, 3);
nums.stream()
    .map(String::valueOf)  // Not this - static
    .map(String::length);  // This - instance on arbitrary object
```

**4. Constructor Reference - `Class::new`**
```java
// Lambda
Supplier<List<String>> lambda = () -> new ArrayList<>();

// Constructor reference
Supplier<List<String>> ref = ArrayList::new;

// With parameter
Function<String, Integer> func = Integer::new;

// Real example
List<String> strings = Arrays.asList("1", "2", "3");
List<Integer> ints = strings.stream()
    .map(Integer::new)  // Constructor reference
    .collect(Collectors.toList());
```

**Complete reference table:**
| Type | Syntax | Example |
|------|--------|---------|
| Static | `Class::staticMethod` | `Math::max` |
| Instance (specific) | `object::method` | `System.out::println` |
| Instance (arbitrary) | `Class::method` | `String::length` |
| Constructor | `Class::new` | `ArrayList::new` |

**Interview tip:** "4 types yaad rakho: **Static**, **Specific Object**, **Arbitrary Object**, **Constructor**. Arbitrary object wala confusing hota hai - pehla parameter object ban jata hai on which method is called."

### **Q5. What is Metaspace in Java 8?**

**Answer:**  
Metaspace **PermGen ki jagah** aaya Java 8 mein. Yeh **class metadata store** karta hai.

**PermGen (Java 7 and below):**
- Fixed size memory
- Part of heap
- Prone to `OutOfMemoryError: PermGen space`
- Tuned with `-XX:MaxPermSize`

**Metaspace (Java 8+):**
- **Native memory** (not heap)
- **Auto-growing** by default
- No "Metaspace" OOM easily
- Tuned with `-XX:MaxMetaspaceSize`

**What Metaspace stores:**
```java
// Class metadata
// Method metadata
// Constant pool
// Annotations
// Not storing:
// - String pool (moved to heap)
// - Static variables (moved to heap)
```

**Memory difference:**
```
Java 7:
Heap ┌─────────────────────────────┐
     │ Eden │ Survivor │ Old │PermGen│
     └─────────────────────────────┘
     PermGen fixed - often too small

Java 8:
Heap ┌─────────────────────────────┐
     │ Eden │ Survivor │ Old        │
     └─────────────────────────────┘
Native Memory: Metaspace (grows automatically)
```

**Configuration:**
```bash
# Metaspace settings
-XX:MetaspaceSize=128m      # Initial size
-XX:MaxMetaspaceSize=256m   # Max size (no limit by default)
-XX:MinMetaspaceFreeRatio=40 # GC if free < 40%
```

**Why change?**
```java
// Problem: PermGen caused OOM for:
// - Applications with many classes (large frameworks)
// - Dynamic class loading (JSP, reflection)
// - String.intern() abuse

// Solution: Metaspace uses native memory
// - No fixed limit
// - Grows as needed
// - OS manages memory
```

**Interview tip:** "Metaspace = PermGen replacement. Native memory mein hai - auto-growing. No more PermGen OOM. String pool moved to heap. Use `-XX:MaxMetaspaceSize` to limit if needed."

### **Q6. What is an Optional class in Java 8?**

**Answer:**  
`Optional` **container object** hai jo value present ho bhi sakta hai ya nahi. NullPointerException se bachata hai.

**Basic usage:**
```java
// Before - Null checks everywhere
public String getCity(User user) {
    if (user != null) {
        Address addr = user.getAddress();
        if (addr != null) {
            return addr.getCity();
        }
    }
    return "Unknown";
}

// After - Optional
public String getCity(User user) {
    return Optional.ofNullable(user)
        .map(User::getAddress)
        .map(Address::getCity)
        .orElse("Unknown");
}
```

**Creating Optional:**
```java
// Empty optional
Optional<String> empty = Optional.empty();

// Optional with non-null value
Optional<String> present = Optional.of("Hello");  // Null not allowed

// Optional that can be null
Optional<String> nullable = Optional.ofNullable(getValue());  // Can be null
```

**Common methods:**
```java
Optional<String> opt = Optional.ofNullable("Hello");

// Check presence
if (opt.isPresent()) {
    System.out.println(opt.get());
}

// If present, do something
opt.ifPresent(s -> System.out.println(s));

// Return default if empty
String result = opt.orElse("Default");

// Throw exception if empty
String val = opt.orElseThrow(() -> new RuntimeException("No value"));

// Transform value
Optional<Integer> length = opt.map(String::length);

// Filter
Optional<String> filtered = opt.filter(s -> s.length() > 3);
```

**Interview tip:** "Optional is container object for null safety. **Never use Optional.get() without checking isPresent()**. Use orElse/orElseGet. Don't use Optional for fields (not serializable) or method parameters."

### **Q7. How do we avoid NullPointerException in Java 8?**

**Answer:**  
Java 8 mein multiple ways hain NPE avoid karne ke:

**1. Optional Class:**
```java
// Instead of returning null
public Optional<User> findUser(String id) {
    return Optional.ofNullable(database.find(id));
}

// Usage
findUser("123")
    .map(User::getName)
    .ifPresent(System.out::println);
```

**2. Objects.requireNonNull():**
```java
public void process(User user) {
    Objects.requireNonNull(user, "User cannot be null");
    // Now safe to use user
}
```

**3. Optional chaining:**
```java
// Safe navigation
String city = Optional.ofNullable(user)
    .map(User::getAddress)
    .map(Address::getCity)
    .orElse("Unknown");
```

**4. orElse/orElseGet for defaults:**
```java
String name = Optional.ofNullable(getName())
    .orElse("Guest");

String computed = Optional.ofNullable(getName())
    .orElseGet(() -> computeDefault());  // Lazy evaluation
```

**5. Stream API null handling:**
```java
// Skip nulls in streams
List<String> names = list.stream()
    .filter(Objects::nonNull)  // Remove nulls
    .collect(Collectors.toList());
```

**6. Annotations (compile-time):**
```java
import javax.annotation.Nullable;
import javax.annotation.Nonnull;

public void process(@Nonnull String input) {  // Compiler warns if null passed
}
```

**Interview tip:** "Java 8 NPE avoidance: **Optional** for return types, **Objects.requireNonNull** for parameters, **optional chaining** for nested null checks, **orElse/orElseGet** for defaults. Never return null from methods that can return Optional."

### **Q8. What are the different methods in the Optional class that you have used?**

**Answer:** (Most commonly used methods)

**Creating Optional:**
```java
Optional<String> empty = Optional.empty();
Optional<String> of = Optional.of("value");      // Null not allowed
Optional<String> ofNullable = Optional.ofNullable(getValue());  // Null allowed
```

**Checking presence:**
```java
Optional<String> opt = Optional.of("Hello");

boolean isPresent = opt.isPresent();      // true
boolean isEmpty = opt.isEmpty();          // false (Java 11+)
```

**Getting value (with caution):**
```java
String value = opt.get();  // Throws NoSuchElementException if empty - AVOID!
```

**Safe ways to get value:**
```java
// Default value
String result = opt.orElse("Default");

// Lazy default
String result2 = opt.orElseGet(() -> computeValue());

// Throw custom exception
String result3 = opt.orElseThrow(() -> new RuntimeException("Missing"));

// Java 10+ - orElseThrow() with no args
String result4 = opt.orElseThrow();  // Throws NoSuchElementException
```

**Transformation:**
```java
// map - transform if present
Optional<Integer> length = opt.map(String::length);

// flatMap - if transformation returns Optional
Optional<String> upper = opt.flatMap(s -> Optional.of(s.toUpperCase()));
```

**Filtering:**
```java
Optional<String> longString = opt.filter(s -> s.length() > 5);
```

**Side effects:**
```java
// ifPresent - execute if value present
opt.ifPresent(val -> System.out.println(val));

// ifPresentOrElse - Java 9+
opt.ifPresentOrElse(
    val -> System.out.println(val),
    () -> System.out.println("Empty")
);
```

**Real example combining methods:**
```java
public String getEmployeeCity(String empId) {
    return Optional.ofNullable(empId)
        .map(id -> employeeRepo.findById(id))
        .flatMap(Optional::ofNullable)  // Handle null from repo
        .map(Employee::getAddress)
        .map(Address::getCity)
        .filter(city -> !city.isEmpty())
        .orElseThrow(() -> new BusinessException("City not found"));
}
```

**Interview tip:** "Most used: **ofNullable()**, **map()**, **orElse()**, **ifPresent()**. Avoid **get()** without check. Use **orElseThrow()** for required values. **flatMap()** when transformation returns Optional."

### **Q9. When to use map and flatMap?**

**Answer:** (With Optionals and Streams)

**In Optional:**

| Operation | map() | flatMap() |
|-----------|-------|-----------|
| **Returns** | `Optional<U>` | `Optional<U>` |
| **Function returns** | `U` | `Optional<U>` |
| **Result** | Wraps result automatically | Returns as-is |

```java
// map() - function returns value, map wraps it
Optional<String> opt = Optional.of("Hello");
Optional<Integer> length = opt.map(s -> s.length());  // Wraps int in Optional

// flatMap() - function returns Optional, flatMap doesn't wrap again
Optional<String> opt = Optional.of("Hello");
Optional<String> upper = opt.flatMap(s -> Optional.of(s.toUpperCase()));

// Real example
public Optional<Address> getAddress(User user) {
    return Optional.ofNullable(user)
        .flatMap(u -> Optional.ofNullable(u.getAddress()));  // Returns Optional<Address>
}
```

**In Stream:**

| Operation | map() | flatMap() |
|-----------|-------|-----------|
| **Input** | Stream<T> | Stream<T> |
| **Function returns** | R | Stream<R> |
| **Output** | Stream<R> | Stream<R> |
| **Use case** | 1-to-1 transformation | 1-to-many transformation |

```java
// map() - one-to-one
List<String> names = Arrays.asList("john", "jane");
List<String> upper = names.stream()
    .map(String::toUpperCase)  // "john" → "JOHN"
    .collect(Collectors.toList());

// flatMap() - one-to-many
List<List<String>> listOfLists = Arrays.asList(
    Arrays.asList("a", "b"),
    Arrays.asList("c", "d")
);
List<String> flat = listOfLists.stream()
    .flatMap(list -> list.stream())  // Flattens to single stream
    .collect(Collectors.toList());   // ["a", "b", "c", "d"]

// Another example - words to letters
List<String> words = Arrays.asList("Hello", "World");
List<String> letters = words.stream()
    .flatMap(word -> Arrays.stream(word.split("")))  // Each word to multiple letters
    .collect(Collectors.toList());
```

**Decision rule:**
```java
// Use map when: 1 input → 1 output
Function<T, R>   // map

// Use flatMap when: 1 input → multiple outputs OR returns Optional/Stream
Function<T, Stream<R>>   // flatMap
Function<T, Optional<R>> // flatMap
```

**Interview tip:** "map vs flatMap: **map** - one to one, function returns value. **flatMap** - one to many OR function returns container (Optional/Stream). In streams, flatMap flattens nested structures. In optionals, flatMap avoids `Optional<Optional<T>>`."

### **Q10. What is new in date and time changes in Java 8?**

**Answer:**  
Java 8 introduced **java.time package** (JSR-310) - thread-safe, immutable, comprehensive date/time API.

**Before Java 8 problems:**
```java
// Mutable - not thread-safe
Date date = new Date();
date.setYear(2020);  // Mutates!

// Month indexing - 0-based
Date bad = new Date(2020, 0, 1);  // January? Actually year 3920!

// No timezone support
// Poor API design
Calendar cal = Calendar.getInstance();
int month = cal.get(Calendar.MONTH);  // 0-based - confusing
```

**After Java 8 - Main Classes:**

**1. LocalDate (date only):**
```java
LocalDate today = LocalDate.now();
LocalDate birthday = LocalDate.of(1990, Month.JANUARY, 15);
LocalDate parsed = LocalDate.parse("2024-01-15");

// Operations
LocalDate nextWeek = today.plusWeeks(1);
LocalDate prevMonth = today.minusMonths(1);
DayOfWeek day = today.getDayOfWeek();
int year = today.getYear();
```

**2. LocalTime (time only):**
```java
LocalTime now = LocalTime.now();
LocalTime meeting = LocalTime.of(14, 30, 0);
LocalTime parsed = LocalTime.parse("14:30:00");

// Operations
LocalTime later = now.plusHours(2);
LocalTime earlier = now.minusMinutes(30);
int hour = now.getHour();
```

**3. LocalDateTime (date + time):**
```java
LocalDateTime now = LocalDateTime.now();
LocalDateTime datetime = LocalDateTime.of(2024, 1, 15, 14, 30);
LocalDateTime fromParts = LocalDateTime.of(date, time);
```

**4. ZonedDateTime (with timezone):**
```java
ZonedDateTime nowInTokyo = ZonedDateTime.now(ZoneId.of("Asia/Tokyo"));
ZoneId zone = ZoneId.systemDefault();
ZonedDateTime zoned = ZonedDateTime.of(datetime, zone);
```

**5. Period & Duration:**
```java
// Period - date-based
LocalDate start = LocalDate.of(2020, 1, 1);
LocalDate end = LocalDate.of(2024, 1, 1);
Period period = Period.between(start, end);
System.out.println(period.getYears());  // 4

// Duration - time-based
LocalTime startTime = LocalTime.of(9, 0);
LocalTime endTime = LocalTime.of(17, 30);
Duration duration = Duration.between(startTime, endTime);
System.out.println(duration.toHours());  // 8
```

**6. DateTimeFormatter (thread-safe):**
```java
// Predefined formatters
String isoDate = LocalDate.now().format(DateTimeFormatter.ISO_DATE);

// Custom formatter
DateTimeFormatter formatter = DateTimeFormatter.ofPattern("dd/MM/yyyy");
String formatted = LocalDate.now().format(formatter);

// Parse
LocalDate parsed = LocalDate.parse("15/01/2024", formatter);
```

**Key improvements:**
| Feature | Old (Date/Calendar) | New (java.time) |
|---------|--------------------|-----------------|
| **Thread-safety** | Not thread-safe | Immutable, thread-safe |
| **Month indexing** | 0-based | 1-based (Month enum) |
| **Fluent methods** | No | Yes (plusDays, minusWeeks) |
| **Timezone** | Complex | Clean ZoneId API |
| **Duration/Period** | Manual calculation | Built-in |
| **Formatting** | SimpleDateFormat (not thread-safe) | DateTimeFormatter (thread-safe) |

**Interview tip:** "java.time API - **immutable, thread-safe, fluent**. Use LocalDate for dates, LocalTime for times, LocalDateTime for both, ZonedDateTime for timezone. Period for dates, Duration for times. Never use legacy Date/Calendar in new code."

## **Java 11**

### **Q11. What are some new string methods introduced in Java 11?**

**Answer:**  
Java 11 mein **6 new String methods** aaye.

**1. isBlank() - check if string empty or whitespace:**
```java
"".isBlank();           // true
"   ".isBlank();        // true
"Hello".isBlank();      // false
```

**2. lines() - split into stream of lines:**
```java
String multiline = "Line1\nLine2\nLine3";
multiline.lines().forEach(System.out::println);
// Line1
// Line2
// Line3

// Count lines
long count = multiline.lines().count();  // 3
```

**3. strip() / stripLeading() / stripTrailing() - removes whitespace (Unicode-aware):**
```java
String str = "  Hello  ";
str.strip();           // "Hello" (removes both)
str.stripLeading();    // "Hello  " (removes leading)
str.stripTrailing();   // "  Hello" (removes trailing)

// Difference from trim() - strip handles Unicode
String unicode = "\u2000Hello\u2000";
unicode.trim();   // May not remove Unicode spaces
unicode.strip();  // Removes Unicode spaces correctly
```

**4. repeat(int count) - repeat string n times:**
```java
"Hi".repeat(3);        // "HiHiHi"
"-".repeat(10);        // "----------"

// Useful for formatting
String line = "=".repeat(50);
```

**5. formatted() (Object method) - but String has format:**
```java
// Actually part of String's format, but new usage
String template = "Hello %s!";
String result = template.formatted("World");
```

**Complete example:**
```java
public class StringMethodsDemo {
    public static void main(String[] args) {
        // isBlank()
        System.out.println("   ".isBlank());  // true
        
        // lines()
        String text = "Java\nPython\nGo";
        text.lines()
            .filter(l -> l.startsWith("J"))
            .forEach(System.out::println);  // Java
        
        // strip()
        String messy = "\u2000Hello\u2000";
        System.out.println(messy.strip());  // "Hello"
        
        // repeat()
        System.out.println("*".repeat(20));  // 20 stars
        
        // formatted()
        String name = "John";
        System.out.println("Hello %s".formatted(name));
    }
}
```

**Comparison:**
| Method | Java 8 | Java 11 |
|--------|--------|---------|
| Empty/whitespace check | `s.trim().isEmpty()` | `s.isBlank()` |
| Remove whitespace | `trim()` (ASCII only) | `strip()` (Unicode) |
| Split lines | `split("\n")` | `lines()` (Stream) |
| Repeat string | Manual loop | `repeat()` |

**Interview tip:** "Java 11 string methods: **isBlank()** (checks empty/whitespace), **lines()** (stream of lines), **strip()** (Unicode-aware trim), **repeat()** (repeat string). Prefer strip() over trim() for Unicode support."

## **Java 17**

### **Q12. What are sealed classes in Java 17?**

**Answer:**  
Sealed classes **permit** keyword se specify karte hain ki **kaun si classes extend kar sakti hain**. Inheritance control mein aata hai.

**Syntax:**
```java
// Sealed class with permitted subclasses
sealed class Shape permits Circle, Rectangle, Triangle {
    abstract double area();
}

// Subclasses must be final, sealed, or non-sealed
final class Circle extends Shape {
    private double radius;
    Circle(double radius) { this.radius = radius; }
    double area() { return Math.PI * radius * radius; }
}

final class Rectangle extends Shape {
    private double length, width;
    Rectangle(double l, double w) { length = l; width = w; }
    double area() { return length * width; }
}

non-sealed class Triangle extends Shape {
    // non-sealed allows further extension
    double area() { return 0; }
}

// ❌ This would fail - not permitted
// class Square extends Shape { }
```

**Sealed interface:**
```java
sealed interface Vehicle permits Car, Bike, Truck {
    void start();
}

final class Car implements Vehicle {
    public void start() { System.out.println("Car starts"); }
}

final class Bike implements Vehicle {
    public void start() { System.out.println("Bike starts"); }
}
```

**Why sealed classes?**
```java
// Without sealed - anyone can extend
public class Animal { }
class Dog extends Animal { }  // Allowed
class Cat extends Animal { }  // Allowed
class Bird extends Animal { }  // Allowed - no control

// With sealed - controlled extension
sealed class Animal permits Dog, Cat {
    // Only Dog and Cat can extend
}

final class Dog extends Animal { }
final class Cat extends Animal { }
// class Bird extends Animal { }  // ❌ Not permitted
```

**Modifiers for permitted subclasses:**
```java
sealed class Parent permits Child1, Child2, Child3 { }

// Option 1: final - cannot extend further
final class Child1 extends Parent { }

// Option 2: sealed - continues restriction
sealed class Child2 extends Parent permits GrandChild { }
final class GrandChild extends Child2 { }

// Option 3: non-sealed - open for extension
non-sealed class Child3 extends Parent { }
class AnyChild extends Child3 { }  // Allowed
```

**Interview tip:** "Sealed classes = controlled inheritance. Permits clause specifies allowed subclasses. Subclasses must be final, sealed, or non-sealed. Provides exhaustive checking in switch patterns (Java 17+ pattern matching)."

### **Q13. What are Record Classes in Java 17?**

**Answer:**  
Records **transparent data carriers** hain - **immutable data classes** ka concise way.

**Before Records (verbose):**
```java
class Point {
    private final int x;
    private final int y;
    
    public Point(int x, int y) {
        this.x = x;
        this.y = y;
    }
    
    public int x() { return x; }
    public int y() { return y; }
    
    @Override
    public boolean equals(Object o) { ... }
    @Override
    public int hashCode() { ... }
    @Override
    public String toString() { ... }
}
```

**After Records (concise):**
```java
record Point(int x, int y) { }
// Automatically generates:
// - Constructor
// - Accessors (x(), y())
// - equals(), hashCode(), toString()
```

**Complete example:**
```java
// Basic record
record Person(String name, int age) { }

// Usage
Person p1 = new Person("John", 25);
String name = p1.name();  // Accessor (not getter)
```

### **Q14. What restrictions come with using Java records?**

**Answer:**  
Records ke kuch **limitations** hain:

**1. Cannot extend other classes:**
```java
record MyRecord() extends SomeClass { }  // ❌ Not allowed
// Records implicitly extend java.lang.Record
```

**2. Cannot be abstract:**
```java
abstract record MyRecord() { }  // ❌ Not allowed
```

**3. Cannot have instance fields (only state from components):**
```java
record Person(String name) {
    private int age;  // ❌ Not allowed - no extra instance fields
    static int count;  // ✅ Static fields allowed
}
```

**4. Cannot override accessors to change behavior (but can add):**
```java
record Point(int x, int y) {
    // ❌ Cannot change return type
    // public String x() { return String.valueOf(x); }
    
    // ✅ Can add custom methods
    public int sum() { return x + y; }
}
```

**5. No default constructor (compact constructor only):**
```java
record Person(String name) {
    // ✅ Compact constructor
    public Person {
        if (name == null) throw new IllegalArgumentException();
    }
    
    // ❌ Cannot define another constructor with same signature
    // public Person(String name) { this.name = name; }
}
```

**6. Cannot be extended by other classes:**
```java
record Parent(String name) { }
// class Child extends Parent { }  // ❌ Not allowed
```

**7. Cannot have native methods:**
```java
record MyRecord() {
    // native void method();  // ❌ Not allowed
}
```

**8. Cannot be used with certain frameworks:**
```java
// JPA/Hibernate - needs no-arg constructor, proxies
// @Entity record User(String id) { }  // ❌ Won't work

// Spring Data - needs setters for binding
// But works for read-only projections
```

**Workarounds:**
```java
// For frameworks needing no-arg constructor
record Config(String name, int value) {
    // Provide default via factory
    public static Config empty() {
        return new Config("default", 0);
    }
}
```

**Interview tip:** "Records ke limitations: **No inheritance**, **no extra instance fields**, **cannot be abstract**, **no custom accessors** (can't change signature). Perfect for DTOs, not for JPA entities or complex domain objects."

### **Q17. How are sealed classes different from enums?**

**Answer:**

| Feature | Sealed Classes | Enums |
|---------|---------------|-------|
| **Instance count** | Unlimited (controlled) | Fixed set (compile-time) |
| **Instance creation** | Can create new instances | Fixed constants only |
| **State** | Can have different state per instance | Constants are singleton |
| **Methods** | Can have different implementations per subclass | Can have methods per constant |
| **Extensibility** | Permitted subclasses (known set) | No extension possible |
| **Use case** | Known subtypes, but multiple instances | Fixed set of constants |

**Enum example:**
```java
// Fixed set - exactly these 4
enum Day {
    MONDAY, TUESDAY, WEDNESDAY, THURSDAY, FRIDAY, SATURDAY, SUNDAY;
    // Can't create new Day at runtime
}

// Each constant is a singleton
Day d1 = Day.MONDAY;
Day d2 = Day.MONDAY;
System.out.println(d1 == d2);  // true - same instance
```

**Sealed class example:**
```java
// Known subtypes, but multiple instances possible
sealed class Payment permits CreditCard, PayPal, Cash { }

final class CreditCard extends Payment {
    private String cardNumber;
    CreditCard(String cardNumber) { this.cardNumber = cardNumber; }
}

final class PayPal extends Payment {
    private String email;
    PayPal(String email) { this.email = email; }
}

final class Cash extends Payment { }

// Multiple instances of same type
Payment card1 = new CreditCard("1234");
Payment card2 = new CreditCard("5678");  // Different instance
Payment card3 = new CreditCard("1234");  // Another instance
```

**Key difference - State per instance:**
```java
// Enum - no state per instance (singleton)
enum Status {
    PENDING, COMPLETED;
    // Can't have different data per PENDING instance
}

// Sealed - each instance can have different state
sealed class OrderStatus permits Pending, Completed { }

final class Pending extends OrderStatus {
    private LocalDateTime createdAt;
    Pending(LocalDateTime createdAt) { this.createdAt = createdAt; }
}

final class Completed extends OrderStatus {
    private LocalDateTime completedAt;
    Completed(LocalDateTime completedAt) { this.completedAt = completedAt; }
}
```

**When to use which:**
```java
// Use Enum when:
// - Fixed set of constants (like HTTP methods)
// - No additional data per constant needed
// - Singleton behavior desired
enum HttpMethod { GET, POST, PUT, DELETE }

// Use Sealed Class when:
// - Known subtypes but multiple instances
// - Each instance needs different data
// - Complex behavior per type
sealed class Vehicle permits Car, Bike, Truck {
    abstract double calculateFuelEfficiency();
}
final class Car extends Vehicle {
    private double engineSize;
    Car(double engineSize) { this.engineSize = engineSize; }
    double calculateFuelEfficiency() { return 100 / engineSize; }
}
```

**Interview tip:** "Enums = **fixed set of singletons**. Sealed classes = **known set of subtypes**, but can have **multiple instances** with **different state**. Use enum for constants (days, status). Use sealed for hierarchical types where each instance has unique data (payment methods, vehicles)."

### **Q18. How does pattern matching simplify the use of instanceof and switch statements?**

**Answer:**  
Pattern matching **eliminates boilerplate** - combines type checking and casting in one step.

**Before pattern matching (Java 8-16):**
```java
// instanceof + explicit cast
if (obj instanceof String) {
    String s = (String) obj;  // Explicit cast needed
    System.out.println(s.length());
}

// Switch with type checking (verbose)
static String process(Object obj) {
    if (obj instanceof String) {
        String s = (String) obj;
        return "String: " + s;
    } else if (obj instanceof Integer) {
        Integer i = (Integer) obj;
        return "Integer: " + i;
    } else if (obj instanceof List) {
        List<?> list = (List<?>) obj;
        return "List size: " + list.size();
    }
    return "Unknown";
}
```

**With pattern matching (Java 17+):**
```java
// instanceof with pattern variable
if (obj instanceof String s) {
    System.out.println(s.length());  // s is already String type
}

// Switch with pattern matching (Java 21 - preview in 17, 19)
static String process(Object obj) {
    return switch(obj) {
        case String s -> "String: " + s;
        case Integer i -> "Integer: " + i;
        case List<?> list -> "List size: " + list.size();
        case null -> "Null value";
        default -> "Unknown";
    };
}
```

**Advanced pattern matching features:**

**1. Guarded patterns (with when clause):**
```java
// Java 21
static String evaluate(Object obj) {
    return switch(obj) {
        case String s when s.length() > 10 -> "Long string: " + s;
        case String s -> "Short string: " + s;
        case Integer i when i > 0 -> "Positive: " + i;
        case Integer i -> "Non-positive: " + i;
        default -> "Other";
    };
}
```

**2. Record patterns (Java 21):**
```java
record Point(int x, int y) { }
record Circle(Point center, int radius) { }

static void printCircle(Object obj) {
    // Nested pattern matching
    if (obj instanceof Circle(Point(int x, int y), int r)) {
        System.out.println("Circle at (" + x + "," + y + ") radius " + r);
    }
}

// Switch with record patterns
static String describe(Object obj) {
    return switch(obj) {
        case Circle(Point(var x, var y), var r) -> 
            String.format("Circle at (%d,%d) r=%d", x, y, r);
        case Point(var x, var y) -> 
            String.format("Point (%d,%d)", x, y);
        default -> "Unknown shape";
    };
}
```

**3. Array patterns:**
```java
static void processArray(Object obj) {
    if (obj instanceof String[] { String first, String[] rest }) {
        System.out.println("First: " + first);
    }
}
```

**Real-world example - Calculator:**
```java
// Before - Lots of casting
Object evaluate(Object left, String op, Object right) {
    if (left instanceof Integer && right instanceof Integer) {
        int l = (Integer) left;
        int r = (Integer) right;
        switch(op) {
            case "+": return l + r;
            case "-": return l - r;
        }
    } else if (left instanceof Double && right instanceof Double) {
        double l = (Double) left;
        double r = (Double) right;
        // ...
    }
    return null;
}

// After - Clean pattern matching
Object evaluate(Object left, String op, Object right) {
    return switch(left) {
        case Integer l when right instanceof Integer r -> switch(op) {
            case "+" -> l + r;
            case "-" -> l - r;
            default -> throw new IllegalArgumentException();
        };
        case Double l when right instanceof Double r -> switch(op) {
            case "+" -> l + r;
            case "-" -> l - r;
            default -> throw new IllegalArgumentException();
        };
        default -> throw new IllegalArgumentException();
    };
}
```

**Benefits summary:**
| Aspect | Before | After Pattern Matching |
|--------|--------|----------------------|
| **Casting** | Explicit cast needed | Automatic (pattern variable) |
| **Code length** | Longer | Shorter |
| **Type safety** | Same | Same + better |
| **Null handling** | Manual | Switch can handle null |
| **Readability** | Moderate | Excellent |
| **Nested patterns** | Very verbose | Concise |

**Interview tip:** "Pattern matching eliminates **boilerplate casting** - `if (obj instanceof String s)` declares and casts in one step. Switch with patterns (Java 21) enables **type-based dispatch** without manual `if-else` chains. Record patterns allow **deconstruction** of nested structures. Code becomes more readable and less error-prone."

### **Q19. In which scenarios should virtual threads be used?**

**Answer:**  
Virtual threads (Project Loom, Java 21) **lightweight threads** hain jo **blocking I/O operations** ke liye ideal hain.

**What are Virtual Threads:**
```java
// Platform thread (heavy, OS-level)
Thread platform = new Thread(() -> {
    // Blocks OS thread
    Thread.sleep(1000);
});

// Virtual thread (lightweight, JVM-managed)
Thread virtual = Thread.startVirtualThread(() -> {
    // Doesn't block OS thread
    Thread.sleep(1000);
});

// Or using Executors
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    executor.submit(() -> handleRequest());
}
```

**✅ When to use Virtual Threads:**

**1. High-concurrency blocking I/O operations:**
```java

// Each request does database calls, API calls, file I/O

try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 100_000; i++) {
        executor.submit(() -> {
            // Blocking operation - perfect for virtual threads
            String data = httpClient.get("https://api.example.com");
            database.save(data);
        });
    }
}
// Can handle 100,000 concurrent tasks with low memory!
```

**2. Microservice calling multiple downstream services:**
```java
// Scatter-gather pattern
List<Callable<Response>> tasks = List.of(
    () -> callServiceA(),
    () -> callServiceB(),
    () -> callServiceC()
);

try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    List<Future<Response>> futures = tasks.stream()
        .map(executor::submit)
        .toList();
    // Wait for all responses
}
```

**3. Database connection pool replacement:**
```java
// Instead of connection pool (limited by OS threads)
// Virtual threads can have many concurrent DB calls

@GetMapping("/users")
public List<User> getUsers() {
    try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
        List<Future<User>> futures = userIds.stream()
            .map(id -> executor.submit(() -> userRepository.findById(id)))
            .toList();
        
        return futures.stream()
            .map(future -> future.get())
            .collect(Collectors.toList());
    }
}
```

**4. Batch processing with I/O:**
```java
// Processing millions of files
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    Files.list(Paths.get("/data"))
        .forEach(file -> executor.submit(() -> processFile(file)));
}
```

**❌ When NOT to use Virtual Threads:**

**1. CPU-intensive tasks:**
```java
// ❌ Bad for CPU-bound work
// Virtual threads don't make CPU faster
executor.submit(() -> {
    // Heavy calculation
    BigInteger.probablePrime(2048, new Random());
});
// Use platform threads with parallel streams instead
```

**2. Long-running CPU operations:**
```java
// ❌ Image processing, video encoding
// Use platform threads with pool size = CPU cores
```

**3. Thread-local heavy code:**
```java
// ❌ Virtual threads don't work well with ThreadLocal
// (Can use but more expensive)
ThreadLocal<String> context = new ThreadLocal<>();
context.set("value");  // Works but creates many ThreadLocal instances
```

**4. Synchronized blocks (careful):**
```java
// ❌ Can cause pinning
synchronized(lock) {
    // Virtual thread pinned to platform thread
    Thread.sleep(1000);  // Blocks OS thread!
}
// Use ReentrantLock instead
```

**Performance comparison:**
```java
// Platform threads - limited by OS
// 10,000 platform threads = ~1GB memory + slow

// Virtual threads
// 10,000 virtual threads = ~10MB memory + fast
// 1,000,000 virtual threads possible!

// Memory per thread:
// Platform: ~1MB
// Virtual: ~1KB
```

**Real-world example - Web server:**
```java

public class UserController {
    // Reactive (hard to debug, complex)
```

**Interview tip:** "Virtual threads for **I/O-bound, blocking operations** with high concurrency (10,000+). NOT for CPU-bound tasks. Replace reactive programming with simple synchronous code. Memory efficient (1KB vs 1MB per thread). Use `Executors.newVirtualThreadPerTaskExecutor()`."

### **Q20. How do G1 GC and ZGC differ in terms of latency goals and throughput?**

**Answer:**

| Feature | G1 GC | ZGC |
|---------|-------|-----|
| **Goal** | Balanced (latency + throughput) | Ultra-low latency (<1ms) |
| **Pause time** | ~10-200ms (configurable) | <1ms (even for large heaps) |
| **Heap size** | 4GB - 100GB+ | 8MB - 16TB |
| **Throughput** | Higher (90-95%) | Slightly lower (85-90%) |
| **Memory overhead** | 5-10% | 5-15% |
| **Concurrent** | Partially | Mostly concurrent |
| **Default Java** | Java 9+ (default) | Java 15+ (production ready) |

**G1 GC (Garbage First):**
```bash
# Enable G1 GC (default in Java 9+)
-XX:+UseG1GC

# Set pause time goal (soft goal)
-XX:MaxGCPauseMillis=100  # Target 100ms pauses

# Region size
-XX:G1HeapRegionSize=16m

# Young generation size
-XX:G1NewSizePercent=5
-XX:G1MaxNewSizePercent=60
```

**How G1 GC works:**
```
Heap divided into regions (1-32MB each)
┌────┬────┬────┬────┬────┬────┬────┬────┐
│ E  │ E  │ S  │ O  │ H  │ O  │ E  │ F  │
├────┼────┼────┼────┼────┼────┼────┼────┤
│ O  │ F  │ O  │ S  │ O  │ H  │ E  │ O  │
└────┴────┴────┴────┴────┴────┴────┴────┘
E=Eden, S=Survivor, O=Old, H=Humongous, F=Free

- Collects regions with most garbage first ("Garbage First")
- Incremental, concurrent, but has some stop-the-world phases
```

**ZGC (Z Garbage Collector):**
```bash
# Enable ZGC
-XX:+UseZGC

# Set heap size (can be huge)
-Xmx100g

# Concurrent thread count
-XX:ConcGCThreads=4

# Return unused memory to OS
-XX:+ZUncommit
```

**How ZGC works:**
```
- Almost fully concurrent (minimal pauses)
- Uses colored pointers (64-bit)
- Load barriers instead of write barriers
- Pauses never exceed 1ms regardless of heap size

Heap regions (variable size):
- Small: 2MB (objects < 256KB)
- Medium: 32MB (objects 256KB - 4MB)  
- Large: > 4MB (direct mapping)
```

**Latency comparison:**
```java
// Application with 100GB heap

// G1 GC
// Pause times: 50-200ms
// 99.9%ile: ~150ms
// Suitable for most applications

// ZGC
// Pause times: <1ms
// 99.9%ile: <2ms
// Suitable for low-latency trading, gaming
```

**Throughput comparison:**
```java
// G1 GC - Higher throughput
// Application spends ~5% time in GC
// More CPU for application work

// ZGC - Slightly lower throughput  
// Application spends ~10-15% time in GC
// But more consistent performance
```

**When to use which:**

**Use G1 GC when:**
```bash
# Default GC - good for most applications
# - Web servers, microservices
# - Batch processing
# - Moderate heap sizes (4GB-64GB)
# - When throughput matters more than latency
```

**Use ZGC when:**
```bash
# Ultra-low latency required
# - Real-time trading systems
# - Gaming servers
# - Large heaps (100GB+)
# - When pause times must stay under 1ms
```

**Performance example:**
```java
// 100GB heap, 80% live data

// G1 GC
// - 100ms pauses, 4-5 times per minute
// - Application threads: 95% CPU time
// - Good for throughput

// ZGC
// - <1ms pauses, very frequent but negligible
// - Application threads: 85-90% CPU time  
// - 5-10% CPU overhead for low latency
// - Perfect for real-time systems
```

**Choosing GC:**
```
Latency sensitive?
    ├─ Yes, <10ms required → ZGC
    ├─ Yes, <100ms acceptable → G1 GC with tuning
    └─ No, throughput priority → G1 GC (default)

Heap size > 100GB?
    ├─ Yes → ZGC (better large heap performance)
    └─ No → G1 GC is fine

CPU cores?
    ├─ Many cores (16+) → Both work well
    └─ Few cores (<4) → G1 GC (ZGC has higher overhead)
```

**Interview tip:** "G1 GC = **balanced** (default Java 9+), pauses 10-200ms, good throughput. ZGC = **ultra-low latency** (<1ms), slightly lower throughput, great for large heaps. Choose G1 for most apps, ZGC for real-time/large heap. ZGC is concurrent - application runs during GC."

## Functional Interface

### **Q21. What are Functional Interfaces in Java 8?**

**Answer:**  
Functional interface **exactly one abstract method** wala interface hai. Lambda expressions ka target type hota hai.

**Definition:**
```java
@FunctionalInterface  // Optional but recommended
interface MyFunctionalInterface {
    void doSomething();  // Single abstract method
    
    // Can have default methods
    default void log() { System.out.println("Logging"); }
    
    // Can have static methods
    static void utility() { }
}
```

**Built-in functional interfaces:**
```java
// 1. Predicate<T> - test method
Predicate<String> isEmpty = s -> s.isEmpty();
boolean result = isEmpty.test("");  // true

// 2. Function<T,R> - apply method
Function<String, Integer> length = s -> s.length();
int len = length.apply("Hello");  // 5

// 3. Consumer<T> - accept method
Consumer<String> printer = s -> System.out.println(s);
printer.accept("Hello");  // Prints Hello

// 4. Supplier<T> - get method
Supplier<Double> random = () -> Math.random();
double value = random.get();

// 5. Runnable - run method
Runnable task = () -> System.out.println("Running");

// 6. Callable<V> - call method
Callable<Integer> callable = () -> 42;
```

**Using functional interface:**
```java
// Define functional interface
@FunctionalInterface
interface Calculator {
    int calculate(int a, int b);
}

// Use with lambda
Calculator add = (a, b) -> a + b;
Calculator multiply = (a, b) -> a * b;

System.out.println(add.calculate(5, 3));      // 8
System.out.println(multiply.calculate(5, 3)); // 15

// Use with method reference
Calculator subtract = Math::subtractExact;
```

**Why @FunctionalInterface annotation:**
```java
@FunctionalInterface
interface Good {
    void method1();
    // void method2();  // ❌ Compilation error - multiple abstract methods
}

// Without annotation, still functional but compiler doesn't enforce
interface Bad {
    void method1();
    void method2();  // Compiles but not functional
}
```

**Interview tip:** "Functional interface = **single abstract method**. Lambda expressions implement them. Use `@FunctionalInterface` annotation to enforce. Built-in: Predicate, Function, Consumer, Supplier, Runnable, Callable."

### **Q22. Difference between functional interface and abstract class?**

**Answer:**

| Feature | Functional Interface | Abstract Class |
|---------|---------------------|----------------|
| **Abstract methods** | Exactly one | Zero or more |
| **State (fields)** | Only constants (static final) | Can have instance fields |
| **Multiple inheritance** | Class can implement multiple | Class can extend only one |
| **Lambda support** | ✅ Yes (target type) | ❌ No |
| **Constructor** | No | Yes |
| **Default methods** | Yes | No (concrete methods work) |
| **Static methods** | Yes | Yes |
| **Memory** | Lightweight | Heavier (state possible) |

**Code comparison:**
```java
// Functional Interface
@FunctionalInterface
interface Calculator {
    int calculate(int a, int b);  // Single abstract method
    
    default void log() { }  // Default method
    static void help() { }   // Static method
    // No instance fields, no constructor
}

// Abstract Class
abstract class CalculatorBase {
    private String name;  // Can have state
    
    CalculatorBase(String name) {  // Constructor
        this.name = name;
    }
    
    abstract int calculate(int a, int b);  // Abstract method
    
    void log() { }  // Concrete method
}
```

**When to use which:**
```java
// ✅ Use functional interface when:
// - Need behavior only (no state)
// - Want to use lambda expressions
// - Single responsibility

@FunctionalInterface
interface Validator {
    boolean validate(String input);
}
Validator email = s -> s.contains("@");

// ✅ Use abstract class when:
// - Need to share state
// - Need constructors
// - Template method pattern
// - Need common fields

abstract class DataProcessor {
    protected final String source;
    
    DataProcessor(String source) {
        this.source = source;
    }
    
    abstract void process();
    
    void log() { System.out.println("Processing: " + source); }
}
```

**Practical example:**
```java
// Functional interface for strategy pattern
@FunctionalInterface
interface PricingStrategy {
    double calculate(double price);
}

PricingStrategy discount = p -> p * 0.9;
PricingStrategy tax = p -> p * 1.1;

// Abstract class for template pattern
abstract class PaymentProcessor {
    protected final double amount;
    
    PaymentProcessor(double amount) {
        this.amount = amount;
    }
    
    abstract void validate();
    abstract void execute();
    
    final void process() {  // Template method
        validate();
        execute();
        log();
    }
    
    private void log() { }
}
```

**Interview tip:** "Functional interface = **behavior only** (no state), lambda-friendly. Abstract class = **state + behavior**, can have multiple abstract methods. Use functional interface for strategies, predicates. Use abstract class for templates with shared state."

### **Q23. Can Functional Interfaces extend other Interfaces?**

**Answer:**  
**Yes** - functional interface can extend other interfaces, but must maintain **exactly one abstract method**.

**Rules:**
```java
// ✅ Extending interface with no abstract methods
interface Parent1 {
    default void method1() { }
}

@FunctionalInterface
interface Child1 extends Parent1 {
    void method();  // Still one abstract method - OK
}

// ✅ Extending interface with abstract method (same)
interface Parent2 {
    void method();
}

@FunctionalInterface
interface Child2 extends Parent2 {
    // No new abstract methods - OK (inherits one)
}

// ❌ Extending multiple abstract methods
interface Parent3 {
    void method1();
}

interface Parent4 {
    void method2();
}

@FunctionalInterface
interface Child3 extends Parent3, Parent4 {
    // Two abstract methods inherited - NOT functional
}
```

**Valid scenarios:**
```java
// Scenario 1: Inherit single abstract method
interface Runnable {
    void run();
}

@FunctionalInterface
interface MyRunnable extends Runnable {
    // No new abstract methods - OK
}

// Scenario 2: Combine with default methods
interface Base {
    void process();
    
    default void log() {
        System.out.println("Logging");
    }
}

@FunctionalInterface
interface Extended extends Base {
    // Still one abstract method (process)
    // Additional default methods allowed
    default void validate() { }
}

// Scenario 3: Override abstract method (still one)
interface Parent {
    void parentMethod();
}

@FunctionalInterface
interface Child extends Parent {
    @Override
    void parentMethod();  // Overriding same method - OK
}
```

**Invalid scenarios:**
```java
// ❌ Two different abstract methods
interface A {
    void a();
}
interface B {
    void b();
}
@FunctionalInterface
interface C extends A, B { }  // Two abstract methods - Error

// ❌ Adding new abstract method
interface Parent {
    void existing();
}
@FunctionalInterface
interface Child extends Parent {
    void newMethod();  // Two abstract methods - Error
}
```

**Real-world example - Comparator:**
```java
// Comparator is functional interface
@FunctionalInterface
public interface Comparator<T> {
    int compare(T o1, T o2);  // Single abstract method
    
    // Many default and static methods
    default Comparator<T> reversed() { ... }
    static <T> Comparator<T> naturalOrder() { ... }
}

// Can extend Comparator
@FunctionalInterface
interface AdvancedComparator<T> extends Comparator<T> {
    // No new abstract methods - OK
    default Comparator<T> nullFirst() { ... }
}
```

**Interview tip:** "Functional interface can extend other interfaces as long as **only one abstract method** exists in total (inherited + declared). Multiple default/static methods are fine. Useful for extending existing functional interfaces with additional defaults."

### **Q24. Why Functional Interfaces in Java 8 have one Abstract Method?**

**Answer:**  
Lambda expressions ko **target type** chahiye - exactly **one method** to implement.

**Core reason:**
```java
// Lambda expression
(int a, int b) -> a + b

// This lambda could match many interfaces:
// - Calculator: int calculate(int a, int b)
// - BinaryOperator: Integer apply(Integer a, Integer b)
// - BiFunction: Integer apply(Integer a, Integer b)

// Java needs to know WHICH interface
// By having one abstract method, lambda knows what to implement
```

**If multiple abstract methods:**
```java
// What if interface had two methods?
interface BadInterface {
    void method1();
    int method2();
}

// Lambda can't implement both!
BadInterface lambda = () -> ???;  // What to implement?
```

**Historical context:**
```java
// Before Java 8 - Anonymous classes
Runnable r = new Runnable() {
    @Override
    public void run() {
        System.out.println("Run");
    }
};

// Java 8 - Lambda (only works because Runnable has ONE method)
Runnable r = () -> System.out.println("Run");
```

**SAM (Single Abstract Method) concept:**
```java
// SAM interface
@FunctionalInterface
interface SAM {
    void singleMethod();
    // Only one abstract method
}

// Lambda works
SAM sam = () -> System.out.println("Works");
```

**How Java uses functional interfaces:**
```java
// Compiler transforms lambda to implementation
// Source code:
Predicate<String> p = s -> s.isEmpty();

// Compiler generates (roughly):
Predicate<String> p = new Predicate<String>() {
    @Override
    public boolean test(String s) {
        return s.isEmpty();
    }
};
```

**Why not zero methods?**
```java
// Interface with zero methods (marker interface)
interface Marker { }

// Not functional - nothing to implement
// Lambda would have no target method
```

**Why not two+ methods?**
```java
// Interface with two methods
interface TwoMethods {
    void m1();
    void m2();
}

// Lambda can't implement both
TwoMethods t = () -> {};  // Which method?
```

**Exception - Overriding Object methods:**
```java
@FunctionalInterface
interface OK {
    void method();
    
    // Object methods don't count
    String toString();  // Overrides Object - doesn't count
    boolean equals(Object o);  // Doesn't count
}

// Still functional - only one abstract method (method)
OK ok = () -> System.out.println("Works");
```

**Interview tip:** "One abstract method = **target type for lambda**. Lambda provides implementation for that one method. Multiple methods would be ambiguous. Object class methods (toString, equals) don't count. This is called **SAM (Single Abstract Method)** pattern."

```java
public class UserProcessor {

            .filter(isActive.and(isAdult))  // Predicate composition

```

## Lambdas

### **Q26. What is Lambda Expression in Java 8?**

**Answer:**  
Lambda expression **anonymous function** hai - method without name. Functional interface ko implement karta hai.

**Syntax:**
```java
(parameters) -> { body }

// Types of lambda expressions:
() -> System.out.println("No params");
(x) -> x * 2;                    // One param, no braces
(x, y) -> x + y;                 // Two params
(String s) -> s.length();        // Explicit type
() -> { return "Hello"; };       // Block body with return
```

**Basic examples:**
```java
// Without lambda (anonymous class)
Runnable oldWay = new Runnable() {
    @Override
    public void run() {
        System.out.println("Hello");
    }
};

// With lambda
Runnable newWay = () -> System.out.println("Hello");

// Comparator without lambda
Comparator<String> oldComp = new Comparator<String>() {
    @Override
    public int compare(String a, String b) {
        return a.compareTo(b);
    }
};

// With lambda
Comparator<String> newComp = (a, b) -> a.compareTo(b);
```

**Lambda with different inputs:**
```java
// No parameters
Runnable r = () -> System.out.println("Running");

// One parameter (parentheses optional)
Consumer<String> c = s -> System.out.println(s);

// Multiple parameters
BiFunction<Integer, Integer, Integer> add = (a, b) -> a + b;

// Explicit parameter types
BiFunction<String, String, Integer> compare = (String a, String b) -> a.compareTo(b);
```

**Block vs Expression body:**
```java
// Expression body (single expression, no return needed)
Function<Integer, Integer> square = x -> x * x;

// Block body (multiple statements, return needed)
Function<Integer, Integer> factorial = x -> {
    int result = 1;
    for (int i = 1; i <= x; i++) {
        result *= i;
    }
    return result;
};
```

**Variable capture (effectively final):**
```java
// Lambda can access effectively final variables
String prefix = "Hello: ";
Function<String, String> greet = name -> prefix + name;  // Works

// Cannot modify local variables
int count = 0;
Runnable r = () -> {
    // count++;  // ❌ Compilation error - must be effectively final
    System.out.println(count);  // ✅ Can read
};

// Instance variables can be modified
class Example {
    private int count = 0;
    Runnable r = () -> count++;  // ✅ Works (instance variable)
}
```

**Method references as lambda shortcuts:**
```java
// Lambda
Function<String, Integer> lambda = s -> s.length();

// Method reference
Function<String, Integer> ref = String::length;

// Lambda
BiFunction<String, String, String> concat = (a, b) -> a.concat(b);

// Method reference
BiFunction<String, String, String> concatRef = String::concat;
```

**Lambda in streams:**
```java
List<String> names = Arrays.asList("John", "Jane", "Bob");

// Without lambda (verbose)
names.stream()
     .filter(new Predicate<String>() {
         @Override
         public boolean test(String s) {
             return s.startsWith("J");
         }
     })
     .map(new Function<String, String>() {
         @Override
         public String apply(String s) {
             return s.toUpperCase();
         }
     })
     .forEach(new Consumer<String>() {
         @Override
         public void accept(String s) {
             System.out.println(s);
         }
     });

// With lambda (concise)
names.stream()
     .filter(s -> s.startsWith("J"))
     .map(s -> s.toUpperCase())
     .forEach(s -> System.out.println(s));
```

**Interview tip:** "Lambda = anonymous function. Syntax: `(params) -> { body }`. Type inference allows omitting parameter types. Use effectively final variables from outer scope. Method references are shorthand. Lambdas enable functional programming in Java."

### **Q27. Why lambda function is called anonymous? If it returns nothing then what is use of that?**

**Answer:**  
Lambda is **anonymous** because it has **no name** (unlike methods). Even if returns nothing (void), it's useful for **side effects**.

**Why "anonymous"?**

```java
// Regular method - has name
public int add(int a, int b) {  // Method name: "add"
    return a + b;
}

// Lambda - no name
(int a, int b) -> a + b;  // No method name!

// Assign to variable (still anonymous - variable has name, lambda doesn't)
BiFunction<Integer, Integer, Integer> add = (a, b) -> a + b;
// Lambda itself still anonymous - we call via 'add' variable
```

**Void lambda - still useful for side effects:**

```java
// Void lambda examples (Consumer, Runnable)

// 1. Printing (side effect)
Consumer<String> printer = s -> System.out.println(s);
printer.accept("Hello");  // Prints to console - useful!

// 2. Logging
Consumer<String> logger = msg -> logService.info(msg);

// 3. Database update
Consumer<User> updater = user -> userRepository.update(user);

// 4. Sending notification
Consumer<String> notifier = email -> emailService.send(email);

// 5. Event handling
Runnable clickHandler = () -> {
    // Update UI
    // Play sound
    // Log click
};
```

**Real-world void lambda use cases:**

```java
// 1. forEach with side effect
List<String> names = Arrays.asList("John", "Jane");
names.forEach(name -> System.out.println(name));  // Void lambda, but prints!

// 2. Event listeners
button.setOnAction(event -> {
    System.out.println("Button clicked");
    updateCounter();
    saveToDatabase();
});

// 3. Cleanup operations
try (Stream<File> files = Files.list(Paths.get("/tmp"))) {
    files.forEach(file -> {
        if (file.isFile()) {
            file.delete();  // Side effect: delete file
        }
    });
}

// 4. Batch processing with side effects
users.stream()
    .filter(User::isActive)
    .forEach(user -> {
        sendEmail(user);      // Side effect
        updateLastLogin(user); // Side effect
        logActivity(user);     // Side effect
    });
```

**Lambda that does nothing (still valid):**
```java
// Empty void lambda
Runnable doNothing = () -> {};
// Useless? Yes, but valid syntax

// Useful void lambda with condition
Runnable safeTask = () -> {
    if (isReady()) {
        performTask();
    }
};
```

**Comparison: Value-returning vs Void lambda:**

| Type | Return | Use Case | Example |
|------|--------|----------|---------|
| **Value-returning** | Yes | Transformation, calculation | `s -> s.length()` |
| **Void** | No | Side effects, actions | `s -> System.out.println(s)` |

**When void lambdas are useful:**

```java
// 1. Modifying external state
List<String> results = new ArrayList<>();
stream.forEach(item -> results.add(item.toUpperCase()));

// 2. Resource cleanup
try (Connection conn = getConnection()) {
    statements.forEach(stmt -> {
        try (PreparedStatement ps = conn.prepareStatement(stmt)) {
            ps.execute();
        }
    });
}

// 3. Monitoring/metrics
requests.forEach(req -> {
    long start = System.nanoTime();
    process(req);
    metrics.record(System.nanoTime() - start);
});

// 4. Batch operations
users.forEach(user -> {
    if (user.needsUpdate()) {
        updateService.update(user);
        cache.invalidate(user.getId());
    }
});
```

**Interview tip:** "Lambda is **anonymous** because it has no declared name (unlike methods). Void lambdas are useful for **side effects** - printing, logging, database updates, event handling, modifying external state. Not all functions need to return values - actions are equally important in programming."
