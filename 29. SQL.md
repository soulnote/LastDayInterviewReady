# Complete SQL Interview Questions & Answers 

## Topic 1: JOINS 

### INNER JOIN
**Definition:** Returns only matching records from both tables.

```sql
-- Example: Employees with their department names
SELECT e.emp_name, d.dept_name
FROM employees e
INNER JOIN departments d ON e.dept_id = d.dept_id;
```

**Interview Answer:** "INNER JOIN returns only rows where there's a match in both tables. If an employee has no department or a department has no employees, those rows are excluded."

### LEFT JOIN (LEFT OUTER JOIN)
**Definition:** Returns all records from left table, matching records from right table.

```sql
-- Example: All employees, even those without departments
SELECT e.emp_name, d.dept_name
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.dept_id;
```

**Practical Use Case:** "Finding employees who haven't been assigned to any department - the right table columns will show NULL."

### RIGHT JOIN (RIGHT OUTER JOIN)
**Definition:** Returns all records from right table, matching records from left table.

```sql
-- Example: All departments, even empty ones
SELECT e.emp_name, d.dept_name
FROM employees e
RIGHT JOIN departments d ON e.dept_id = d.dept_id;
```

### FULL OUTER JOIN
**Definition:** Returns all records when there's a match in either table.

```sql
-- Example: All employees and all departments
SELECT e.emp_name, d.dept_name
FROM employees e
FULL OUTER JOIN departments d ON e.dept_id = d.dept_id;
```

### SELF JOIN (Highly Important for Hierarchy)

**Definition:** Joining a table with itself using aliases.

```sql
-- Employee-Manager Hierarchy Problem
SELECT 
    e.emp_name AS "Employee",
    m.emp_name AS "Manager"
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.emp_id;

-- Finding employees earning more than their manager
SELECT e.emp_name
FROM employees e
INNER JOIN employees m ON e.manager_id = m.emp_id
WHERE e.salary > m.salary;

-- Finding hierarchy levels (recursive)
WITH RECURSIVE emp_hierarchy AS (
    SELECT emp_id, emp_name, manager_id, 1 AS level
    FROM employees WHERE manager_id IS NULL
    UNION ALL
    SELECT e.emp_id, e.emp_name, e.manager_id, h.level + 1
    FROM employees e
    INNER JOIN emp_hierarchy h ON e.manager_id = h.emp_id
)
SELECT * FROM emp_hierarchy;
```

**Common Interview Question:** "Write a query to find all employees under a specific manager in a multi-level hierarchy"

## Topic 2: Aggregate Functions

### COUNT(*) vs COUNT(column_name)

```sql
-- COUNT(*) - Counts ALL rows including NULLs
SELECT COUNT(*) FROM employees;  -- Returns 100

-- COUNT(column_name) - Counts only NON-NULL values
SELECT COUNT(dept_id) FROM employees;  -- Returns 95 (5 employees have NULL dept_id)

-- COUNT(DISTINCT column_name) - Counts unique non-null values
SELECT COUNT(DISTINCT dept_id) FROM employees;  -- Returns 8
```

**Interview Answer:** "COUNT(*) includes NULL values and counts every row. COUNT(column_name) ignores NULLs. For performance, COUNT(*) is usually faster because it doesn't check for NULLs."

### SUM(), AVG(), MIN(), MAX()

```sql
-- Basic usage
SELECT 
    SUM(salary) AS total_salary,
    AVG(salary) AS avg_salary,
    MIN(salary) AS min_salary,
    MAX(salary) AS max_salary
FROM employees;

-- With condition
SELECT 
    dept_id,
    AVG(CASE WHEN salary > 50000 THEN salary END) AS avg_high_salary
FROM employees
GROUP BY dept_id;
```

**Important Note:** Aggregate functions ignore NULL values automatically.

## Topic 3: GROUP BY & HAVING

### WHERE vs HAVING - THE KEY DIFFERENCE

```sql
-- WHERE filters rows BEFORE grouping
-- HAVING filters groups AFTER grouping

-- Correct understanding:
SELECT 
    dept_id,
    COUNT(*) as emp_count,
    AVG(salary) as avg_salary
FROM employees
WHERE hire_date > '2020-01-01'  -- Filters individual rows first
GROUP BY dept_id
HAVING COUNT(*) > 5  -- Then filters groups
ORDER BY avg_salary DESC;
```

**Interview Answer:** "WHERE filters individual rows before aggregation. HAVING filters groups after aggregation. You cannot use aggregate functions in WHERE, but you can in HAVING."

### Common Mistakes and Solutions

```sql
-- WRONG: Cannot use aggregate in WHERE
SELECT dept_id FROM employees
WHERE COUNT(*) > 5;  -- ERROR!

-- CORRECT: Use HAVING
SELECT dept_id FROM employees
GROUP BY dept_id
HAVING COUNT(*) > 5;

-- WRONG: Cannot use column alias in WHERE (in most SQL dialects)
SELECT salary * 1.1 as bonus FROM employees
WHERE bonus > 5000;  -- ERROR!

-- CORRECT: Repeat expression or use subquery
SELECT salary * 1.1 as bonus FROM employees
WHERE salary * 1.1 > 5000;
```

## Topic 4: Window Functions (Advanced Favorite)

### ROW_NUMBER(), RANK(), DENSE_RANK()

```sql
-- Sample data: Employee salaries by department
SELECT 
    emp_name,
    dept_id,
    salary,
    ROW_NUMBER() OVER(PARTITION BY dept_id ORDER BY salary DESC) as row_num,
    RANK() OVER(PARTITION BY dept_id ORDER BY salary DESC) as rank_num,
    DENSE_RANK() OVER(PARTITION BY dept_id ORDER BY salary DESC) as dense_rank_num
FROM employees;

-- Difference explained:
-- ROW_NUMBER(): Unique number per row (1,2,3,4...)
-- RANK(): Same rank for ties, skips numbers (1,2,2,4...)
-- DENSE_RANK(): Same rank for ties, no skips (1,2,2,3...)
```

### LEAD() and LAG()

```sql
-- Finding salary changes
SELECT 
    emp_name,
    salary,
    LAG(salary) OVER(ORDER BY hire_date) as prev_salary,
    LEAD(salary) OVER(ORDER BY hire_date) as next_salary,
    salary - LAG(salary) OVER(ORDER BY hire_date) as salary_increase
FROM employees;

-- Finding time between events
SELECT 
    emp_name,
    hire_date,
    LAG(hire_date) OVER(ORDER BY hire_date) as prev_hire_date,
    DATEDIFF(day, LAG(hire_date) OVER(ORDER BY hire_date), hire_date) as days_since_last_hire
FROM employees;
```

### Practical Window Function Interview Questions

```sql
-- Q: Find top 3 earners in each department
SELECT * FROM (
    SELECT 
        emp_name,
        dept_id,
        salary,
        ROW_NUMBER() OVER(PARTITION BY dept_id ORDER BY salary DESC) as rn
    FROM employees
) ranked WHERE rn <= 3;

-- Q: Calculate running total of salaries
SELECT 
    emp_name,
    hire_date,
    salary,
    SUM(salary) OVER(ORDER BY hire_date) as running_total
FROM employees;

-- Q: Find percentage of department salary
SELECT 
    emp_name,
    dept_id,
    salary,
    salary * 100.0 / SUM(salary) OVER(PARTITION BY dept_id) as pct_of_dept
FROM employees;
```

## Topic 5: Subqueries vs CTEs

### Subqueries

```sql
-- Scalar subquery (returns single value)
SELECT emp_name, salary
FROM employees
WHERE salary > (SELECT AVG(salary) FROM employees);

-- Correlated subquery (references outer query)
SELECT emp_name, dept_id, salary
FROM employees e1
WHERE salary > (
    SELECT AVG(salary) 
    FROM employees e2 
    WHERE e2.dept_id = e1.dept_id
);

-- EXISTS vs IN
-- EXISTS (faster for large datasets)
SELECT * FROM departments d
WHERE EXISTS (
    SELECT 1 FROM employees e 
    WHERE e.dept_id = d.dept_id
);

-- IN (simpler for small lists)
SELECT * FROM employees
WHERE dept_id IN (1, 2, 3, 4, 5);
```

### CTEs (Common Table Expressions)

```sql
-- Basic CTE
WITH high_earners AS (
    SELECT emp_name, salary, dept_id
    FROM employees
    WHERE salary > 70000
)
SELECT d.dept_name, COUNT(he.emp_name) as high_earner_count
FROM high_earners he
INNER JOIN departments d ON he.dept_id = d.dept_id
GROUP BY d.dept_name;

-- Multiple CTEs
WITH 
dept_stats AS (
    SELECT 
        dept_id,
        AVG(salary) as avg_salary,
        COUNT(*) as emp_count
    FROM employees
    GROUP BY dept_id
),
company_avg AS (
    SELECT AVG(salary) as overall_avg FROM employees
)
SELECT 
    d.dept_name,
    ds.avg_salary,
    ds.emp_count,
    CASE 
        WHEN ds.avg_salary > ca.overall_avg THEN 'Above Average'
        ELSE 'Below Average'
    END as performance
FROM dept_stats ds
CROSS JOIN company_avg ca
INNER JOIN departments d ON ds.dept_id = d.dept_id;

-- Recursive CTE (for hierarchies)
WITH RECURSIVE emp_tree AS (
    -- Anchor member
    SELECT emp_id, emp_name, manager_id, 1 as level
    FROM employees WHERE manager_id IS NULL
    
    UNION ALL
    
    -- Recursive member
    SELECT e.emp_id, e.emp_name, e.manager_id, et.level + 1
    FROM employees e
    INNER JOIN emp_tree et ON e.manager_id = et.emp_id
)
SELECT * FROM emp_tree;
```

**Interview Answer - CTEs vs Subqueries:** "CTEs make complex queries more readable, can be referenced multiple times, and support recursion. Subqueries are better for simple, one-off calculations. Performance-wise, they're often similar, but CTEs can be optimized differently by the query planner."

## Topic 6: Constraints

### Complete Constraint Examples

```sql
-- Creating table with all constraints
CREATE TABLE employees (
    emp_id INT PRIMARY KEY,
    emp_name VARCHAR(100) NOT NULL,
    email VARCHAR(255) UNIQUE,
    age INT CHECK (age >= 18 AND age <= 65),
    salary DECIMAL(10,2) DEFAULT 30000.00,
    dept_id INT,
    status VARCHAR(20) CHECK (status IN ('Active', 'Inactive', 'On Leave')),
    CONSTRAINT fk_department 
        FOREIGN KEY (dept_id) 
        REFERENCES departments(dept_id)
        ON DELETE SET NULL
        ON UPDATE CASCADE
);

-- Adding constraints later
ALTER TABLE employees ADD CONSTRAINT pk_emp PRIMARY KEY (emp_id);
ALTER TABLE employees ADD CONSTRAINT uq_email UNIQUE (email);
ALTER TABLE employees ADD CONSTRAINT chk_age CHECK (age >= 18);
ALTER TABLE employees ALTER COLUMN emp_name SET NOT NULL;
ALTER TABLE employees ALTER COLUMN salary SET DEFAULT 30000;
```

### Interview Questions on Constraints

**Q: Difference between PRIMARY KEY and UNIQUE?**
- PK: Only one per table, cannot be NULL, creates clustered index (usually)
- UNIQUE: Multiple allowed, can have one NULL (in most DBs), creates non-clustered index

**Q: What happens with ON DELETE CASCADE vs ON DELETE SET NULL?**
```sql
-- CASCADE: Deletes child records when parent is deleted
-- SET NULL: Sets foreign key to NULL when parent is deleted
-- RESTRICT/NO ACTION: Prevents deletion if child exists
```

### Date/Time Functions

```sql
-- DATEDIFF variations (syntax varies by DB)
-- MySQL / SQL Server
SELECT 
    DATEDIFF(day, hire_date, GETDATE()) as days_employed,
    DATEDIFF(month, hire_date, GETDATE()) as months_employed;

-- PostgreSQL
SELECT
```

## Topic 8: UNION vs UNION ALL

### Critical Difference for Performance

```sql
-- UNION: Removes duplicates (slower)
SELECT emp_name, dept_id FROM active_employees
UNION
SELECT emp_name, dept_id FROM former_employees;

-- UNION ALL: Keeps all duplicates (faster)
SELECT emp_name, dept_id FROM active_employees
UNION ALL
SELECT emp_name, dept_id FROM former_employees;

-- Performance demonstration
-- UNION does: SELECT DISTINCT on the combined result
-- UNION ALL does: Simple concatenation

-- Practical use case for UNION ALL
SELECT 'Current' as status, emp_name FROM active_employees
UNION ALL
SELECT 'Former' as status, emp_name FROM former_employees;
```

**Interview Answer:** "UNION ALL is faster because it doesn't perform duplicate elimination. Use UNION only when you specifically need to remove duplicates. If you know there are no duplicates or you want to keep them, always use UNION ALL for better performance."

## Topic 9: Indexes

### Clustered vs Non-Clustered Indexes

```sql
-- Clustered Index (only one per table)
-- Physically reorders the table data
CREATE CLUSTERED INDEX idx_emp_id ON employees(emp_id);

-- Non-Clustered Index (multiple allowed)
-- Creates separate structure with pointers
CREATE NONCLUSTERED INDEX idx_emp_name ON employees(emp_name);

-- Composite Index
CREATE INDEX idx_dept_salary ON employees(dept_id, salary);

-- Unique Index
CREATE UNIQUE INDEX idx_unique_email ON employees(email);

-- Covering Index (includes extra columns)
CREATE INDEX idx_covering ON employees(dept_id) INCLUDE (emp_name, salary);
```

### Interview Questions on Indexes

**Q: When do indexes not work?**
```sql
-- Index won't be used for:
SELECT * FROM employees WHERE UPPER(emp_name) = 'JOHN';  -- Function on column
SELECT * FROM employees WHERE emp_name LIKE '%John%';     -- Leading wildcard
SELECT * FROM employees WHERE dept_id != 5;               -- Negation
SELECT * FROM employees WHERE salary > 50000;             -- If low selectivity
```

**Q: Clustered vs Non-Clustered difference?**
- Clustered: Determines physical order, one per table, faster for range queries
- Non-Clustered: Logical order, multiple per table, faster for exact lookups

## Topic 10: Views & Materialized Views

### Regular Views

```sql
-- Creating a view (virtual table)
CREATE VIEW high_salary_employees AS
SELECT 
    e.emp_name,
    e.salary,
    d.dept_name
FROM employees e
INNER JOIN departments d ON e.dept_id = d.dept_id
WHERE e.salary > 70000;

-- Using view
SELECT * FROM high_salary_employees WHERE dept_name = 'IT';

-- Updatable view requirements
CREATE VIEW simple_view AS
SELECT emp_id, emp_name, salary FROM employees;
-- Can update if single table and no aggregates

-- With CHECK OPTION
CREATE VIEW it_employees AS
SELECT * FROM employees WHERE dept_id = 1
WITH CHECK OPTION;  -- Prevents updates that would remove row from view
```

### Materialized Views

```sql
-- PostgreSQL syntax
CREATE MATERIALIZED VIEW dept_salary_stats AS
SELECT 
    dept_id,
    COUNT(*) as emp_count,
    AVG(salary) as avg_salary,
    SUM(salary) as total_salary
FROM employees
GROUP BY dept_id;

-- Refresh materialized view
REFRESH MATERIALIZED VIEW dept_salary_stats;
REFRESH MATERIALIZED VIEW CONCURRENTLY dept_salary_stats;  -- Without locking

-- When to use: For expensive aggregations, data doesn't change frequently
-- When NOT to use: Real-time data needs, frequently updated tables
```

**Interview Answer - View vs Materialized View:** "Regular views are saved queries that always show current data but can be slow. Materialized views store the actual data, are much faster for reads, but need refresh and use storage."

## Topic 11: Stored Procedures & Triggers

### Stored Procedures

```sql
-- Basic procedure with parameters
CREATE PROCEDURE GetEmployeesByDept
    @dept_id INT,
    @min_salary DECIMAL(10,2) = 30000  -- Default parameter
AS
BEGIN
    SELECT emp_name, salary
    FROM employees
    WHERE dept_id = @dept_id 
      AND salary >= @min_salary
    ORDER BY salary DESC;
END;

-- Execute procedure
EXEC GetEmployeesByDept @dept_id = 1, @min_salary = 50000;

-- Procedure with output parameter
CREATE PROCEDURE GetDeptStats
    @dept_id INT,
    @emp_count INT OUTPUT,
    @avg_salary DECIMAL(10,2) OUTPUT
AS
BEGIN
    SELECT 
        @emp_count = COUNT(*),
        @avg_salary = AVG(salary)
    FROM employees
    WHERE dept_id = @dept_id;
END;

-- Procedure with transaction
CREATE PROCEDURE TransferEmployee
    @emp_id INT,
    @new_dept_id INT
AS
BEGIN
    BEGIN TRANSACTION;
    
    UPDATE employees SET dept_id = @new_dept_id
    WHERE emp_id = @emp_id;
    
    INSERT INTO audit_log (emp_id, action, timestamp)
    VALUES (@emp_id, 'DEPT_TRANSFER', GETDATE());
    
    COMMIT TRANSACTION;
END;
```

### Triggers

```sql
-- AFTER Trigger (for auditing)
CREATE TRIGGER trg_employee_audit
ON employees
AFTER INSERT, UPDATE, DELETE
AS
BEGIN
    INSERT INTO audit_table (action, emp_id, changed_by, change_date)
    SELECT 
        CASE 
            WHEN EXISTS(SELECT * FROM inserted) AND EXISTS(SELECT * FROM deleted) 
                THEN 'UPDATE'
            WHEN EXISTS(SELECT * FROM inserted) 
                THEN 'INSERT'
            ELSE 'DELETE'
        END,
        COALESCE(i.emp_id, d.emp_id),
        SYSTEM_USER,
        GETDATE()
    FROM inserted i
    FULL OUTER JOIN deleted d ON i.emp_id = d.emp_id;
END;

-- INSTEAD OF Trigger (for complex views)
CREATE TRIGGER trg_update_employee_view
ON employee_details_view
INSTEAD OF UPDATE
AS
BEGIN
    UPDATE employees
    SET emp_name = i.emp_name,
        salary = i.salary
    FROM employees e
    INNER JOIN inserted i ON e.emp_id = i.emp_id;
END;
```

## Topic 12: ACID Properties

### Complete Explanation with Examples

**Atomicity:** All or nothing
```sql
BEGIN TRANSACTION;
    UPDATE accounts SET balance = balance - 1000 WHERE acc_no = 'A';
    UPDATE accounts SET balance = balance + 1000 WHERE acc_no = 'B';
    -- If either fails, both rollback
COMMIT;
```

**Consistency:** Data follows all rules
```sql
-- Check constraint prevents invalid data
ALTER TABLE accounts ADD CONSTRAINT chk_positive_balance 
CHECK (balance >= 0);
-- Transaction would fail if balance becomes negative
```

**Isolation:** Concurrent transactions don't interfere
```sql
-- Isolation levels
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- Example of dirty read prevention
-- Transaction 1
BEGIN TRANSACTION;
UPDATE employees SET salary = 100000 WHERE emp_id = 1;
-- Not committed yet
-- Transaction 2 (READ COMMITTED or higher won't see this)
SELECT salary FROM employees WHERE emp_id = 1;  -- Sees old value
```

**Durability:** Committed changes persist
```sql
-- After COMMIT, data survives:
-- - Power failure
-- - System crash
-- - Restart
COMMIT;  -- Data written to transaction log and disk
```

## Topic 13: Normalization (1NF, 2NF, 3NF)

### 1NF (First Normal Form)
**Eliminate repeating groups, create separate tables**

```sql
-- UNNORMALIZED (BAD)
CREATE TABLE employee_projects (
    emp_id INT,
    emp_name VARCHAR(100),
    projects VARCHAR(500)  -- "Project1,Project2,Project3"
);

-- 1NF (GOOD) - Atomic values
CREATE TABLE employee_projects (
    emp_id INT,
    project_name VARCHAR(100),
    PRIMARY KEY (emp_id, project_name)
);
```

### 2NF (Second Normal Form)
**Remove partial dependencies (1NF + all columns depend on whole PK)**

```sql
-- 1NF but NOT 2NF
CREATE TABLE order_details (
    order_id INT,
    product_id INT,
    product_name VARCHAR(100),  -- Depends only on product_id, not full PK
    quantity INT,
    PRIMARY KEY (order_id, product_id)
);

-- 2NF Solution - Split into two tables
CREATE TABLE order_items (
    order_id INT,
    product_id INT,
    quantity INT,
    PRIMARY KEY (order_id, product_id)
);

CREATE TABLE products (
    product_id INT PRIMARY KEY,
    product_name VARCHAR(100)
);
```

### 3NF (Third Normal Form)
**Remove transitive dependencies (2NF + no non-key depends on another non-key)**

```sql
-- 2NF but NOT 3NF
CREATE TABLE employees (
    emp_id INT PRIMARY KEY,
    emp_name VARCHAR(100),
    dept_id INT,
    dept_name VARCHAR(100),  -- Depends on dept_id, not emp_id
    dept_location VARCHAR(100)  -- Transitive dependency
);

-- 3NF Solution
CREATE TABLE employees (
    emp_id INT PRIMARY KEY,
    emp_name VARCHAR(100),
    dept_id INT,
    FOREIGN KEY (dept_id) REFERENCES departments(dept_id)
);

CREATE TABLE departments (
    dept_id INT PRIMARY KEY,
    dept_name VARCHAR(100),
    dept_location VARCHAR(100)
);
```

**Interview Answer:** "Normalization reduces redundancy and anomalies. 1NF eliminates repeating groups. 2NF removes partial dependencies. 3NF removes transitive dependencies. But sometimes we denormalize for performance."

## Topic 14: EXISTS vs IN

### Performance and Usage Comparison

```sql
-- IN - Good for small, static lists
SELECT * FROM employees
WHERE dept_id IN (1, 2, 3, 4, 5);

-- IN with subquery (can be slow with large subquery results)
SELECT * FROM employees
WHERE dept_id IN (SELECT dept_id FROM departments WHERE location = 'NYC');
-- SQL evaluates entire subquery first, then compares

-- EXISTS - Better for large subqueries
SELECT * FROM employees e
WHERE EXISTS (
    SELECT 1 FROM departments d 
    WHERE d.dept_id = e.dept_id AND d.location = 'NYC'
);
-- EXISTS stops processing after first match

-- NOT IN vs NOT EXISTS (CRITICAL DIFFERENCE!)
-- NOT IN behaves unexpectedly with NULLs
SELECT * FROM employees
WHERE dept_id NOT IN (SELECT dept_id FROM departments WHERE location = 'NYC');
-- If subquery returns any NULL, the whole query returns NO rows!

-- NOT EXISTS handles NULLs correctly
SELECT * FROM employees e
WHERE NOT EXISTS (
    SELECT 1 FROM departments d 
    WHERE d.dept_id = e.dept_id AND d.location = 'NYC'
);
-- This works correctly even with NULLs
```

**Performance Rule of Thumb:**
- **IN**: When subquery returns small dataset (< 100 rows)
- **EXISTS**: When subquery returns large dataset or checking existence
- **NOT EXISTS**: Always prefer over NOT IN (due to NULL handling)

## Topic 15: Query Optimization

### Understanding EXPLAIN Plan

```sql
-- MySQL / PostgreSQL
EXPLAIN SELECT * FROM employees WHERE dept_id = 1;

-- More detailed
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM employees WHERE dept_id = 1;

-- Look for:
-- - Seq Scan (table scan) - BAD for large tables
-- - Index Scan - GOOD
-- - Nested Loop - BAD for large datasets
-- - Hash Join - GOOD for large datasets
```

### Common Optimization Techniques

```sql
-- 1. Avoid SELECT *
-- BAD
SELECT * FROM employees WHERE dept_id = 1;

-- GOOD
SELECT emp_id, emp_name, salary FROM employees WHERE dept_id = 1;

-- 2. Use EXISTS instead of IN for large subqueries
-- BAD
SELECT * FROM employees 
WHERE dept_id IN (SELECT dept_id FROM departments WHERE budget > 1000000);

-- GOOD
SELECT * FROM employees e
WHERE EXISTS (SELECT 1 FROM departments d WHERE d.dept_id = e.dept_id AND budget > 1000000);

-- 3. Avoid functions on indexed columns
-- BAD (won't use index on hire_date)
SELECT * FROM employees WHERE YEAR(hire_date) = 2023;

-- GOOD
SELECT * FROM employees 
WHERE hire_date >= '2023-01-01' AND hire_date < '2024-01-01';

-- 4. Use UNION ALL instead of UNION when possible
-- BAD (does duplicate elimination unnecessarily)
SELECT emp_id FROM active_employees
UNION
SELECT emp_id FROM terminated_employees;

-- GOOD (if no duplicates or duplicates OK)
SELECT emp_id FROM active_employees
UNION ALL
SELECT emp_id FROM terminated_employees;

-- 5. Optimize JOIN order (put selective tables first)
-- BAD
SELECT * FROM large_table l
INNER JOIN small_table s ON l.id = s.id
WHERE s.status = 'active';

-- GOOD
SELECT * FROM small_table s
INNER JOIN large_table l ON s.id = l.id
WHERE s.status = 'active';

-- 6. Use appropriate JOIN types
-- Replace correlated subquery with JOIN
-- BAD
SELECT e.emp_name, (SELECT d.dept_name FROM departments d WHERE d.dept_id = e.dept_id)
FROM employees e;

-- GOOD
SELECT e.emp_name, d.dept_name
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.dept_id;
```

### Index Optimization Strategies

```sql
-- Create covering indexes
CREATE INDEX idx_covering ON employees(dept_id, status) INCLUDE (emp_name, salary);

-- Composite index order matters (most selective first)
-- GOOD: status has only 3 values, dept_id has 20 values
CREATE INDEX idx_status_dept ON employees(status, dept_id);

-- Remove unused indexes (they slow down writes)
-- Find unused indexes (DB specific query)

-- Use partial indexes for common filters
CREATE INDEX idx_active_employees ON employees(emp_name) 
WHERE status = 'Active';
```

### Common Performance Anti-Patterns to Avoid

```sql
-- 1. Row-by-row processing (use set-based operations)
-- BAD (cursor/loop)
DECLARE @emp_id INT
DECLARE emp_cursor CURSOR FOR SELECT emp_id FROM employees
-- ... loop through

-- GOOD (set-based)
UPDATE employees SET bonus = salary * 0.1 WHERE performance_rating = 'A';

-- 2. N+1 queries problem
-- BAD (in application code)
SELECT * FROM departments;
-- Then for each department: SELECT * FROM employees WHERE dept_id = ?

-- GOOD (single query)
SELECT d.*, e.* FROM departments d
LEFT JOIN employees e ON d.dept_id = e.dept_id;

-- 3. Using DISTINCT unnecessarily
-- BAD (if no duplicates exist)
SELECT DISTINCT emp_id, emp_name FROM employees;

-- GOOD
SELECT emp_id, emp_name FROM employees;

-- 4. OR conditions on different columns
-- BAD
SELECT * FROM employees 
WHERE emp_name = 'John' OR dept_id = 5;

-- GOOD (use UNION if needed)
SELECT * FROM employees WHERE emp_name = 'John'
UNION
SELECT * FROM employees WHERE dept_id = 5;
```

# Top 10 Window Functions Interview Questions & Answers

## Before We Start - Quick Window Functions Refresher

**What makes a function a "Window Function"?**
- Uses `OVER()` clause
- Doesn't collapse rows (unlike GROUP BY)
- Processes a "window" of rows relative to current row

```sql
-- This is NOT a window function (collapses rows)
SELECT dept_id, AVG(salary) FROM employees GROUP BY dept_id;

-- This IS a window function (preserves all rows)
SELECT emp_name, dept_id, salary, 
       AVG(salary) OVER(PARTITION BY dept_id) as dept_avg
FROM employees;
```

## Question 1: ROW_NUMBER() - "Delete Duplicate Records"

**The Problem:** You have a table with duplicate records. Keep only one occurrence per duplicate group.

**Sample Data:**
```sql
-- employees table with duplicates
emp_id | emp_name | dept_id
-------|----------|--------
1      | John     | 10
2      | John     | 10    -- Duplicate!
3      | John     | 10    -- Duplicate!
4      | Sarah    | 20
5      | Sarah    | 20    -- Duplicate!
```

**The Solution:**
```sql
WITH ranked_duplicates AS (
    SELECT *,
           ROW_NUMBER() OVER(PARTITION BY emp_name, dept_id ORDER BY emp_id) as rn
    FROM employees
)
DELETE FROM employees
WHERE (emp_id, emp_name, dept_id) IN (
    SELECT emp_id, emp_name, dept_id 
    FROM ranked_duplicates 
    WHERE rn > 1
);

-- Alternative: Keep only first record
WITH ranked_duplicates AS (
    SELECT *,
           ROW_NUMBER() OVER(PARTITION BY emp_name, dept_id ORDER BY emp_id) as rn
    FROM employees
)
SELECT * FROM ranked_duplicates WHERE rn = 1;
```

**Why ROW_NUMBER()?** It gives a unique number to each row in a partition, perfect for identifying which duplicate to keep.

## Question 2: RANK() vs DENSE_RANK() - "Top 3 Salaries with Ties"

**The Problem:** Find top 3 earners. If two people have same salary, both should be included.

**Sample Data:**
```sql
emp_name | salary
---------|--------
John     | 100000
Sarah    | 90000
Mike     | 90000   -- Tie with Sarah
Lisa     | 85000
Tom      | 80000
```

**The Solution - Showing the Difference:**
```sql
SELECT emp_name,
       salary,
       ROW_NUMBER() OVER(ORDER BY salary DESC) as row_num,
       RANK() OVER(ORDER BY salary DESC) as rank_num,
       DENSE_RANK() OVER(ORDER BY salary DESC) as dense_rank_num
FROM employees;

-- Results:
emp_name | salary | row_num | rank_num | dense_rank_num
---------|--------|---------|----------|----------------
John     | 100000 | 1       | 1        | 1
Sarah    | 90000  | 2       | 2        | 2
Mike     | 90000  | 3       | 2        | 2    -- Same rank as Sarah
Lisa     | 85000  | 4       | 4        | 3    -- Notice rank skips 3
Tom      | 80000  | 5       | 5        | 4

-- To get top 3 earners (including ties):
SELECT emp_name, salary
FROM (
    SELECT emp_name,
           salary,
           DENSE_RANK() OVER(ORDER BY salary DESC) as dr
    FROM employees
) ranked
WHERE dr <= 3;
-- Returns: John, Sarah, Mike, Lisa (4 people because of tie)
```

**Interview Answer:** "RANK() skips numbers after ties (1,2,2,4), while DENSE_RANK() doesn't skip (1,2,2,3). For 'top 3 including ties', use DENSE_RANK(). ROW_NUMBER() never gives ties."

## Question 3: LAG() and LEAD() - "Salary Increase Comparison"

**The Problem:** Compare each employee's salary with previous and next employee (ordered by hire date).

**The Solution:**
```sql
SELECT emp_name,
       hire_date,
       salary,
       LAG(salary, 1, 0) OVER(ORDER BY hire_date) as prev_salary,
       LEAD(salary, 1, 0) OVER(ORDER BY hire_date) as next_salary,
       
       -- Calculate increase from previous
       salary - LAG(salary, 1, salary) OVER(ORDER BY hire_date) as increase_amount,
       
       -- Calculate percentage increase
       ROUND((salary - LAG(salary, 1, salary) OVER(ORDER BY hire_date)) * 100.0 / 
              LAG(salary, 1, salary) OVER(ORDER BY hire_date), 2) as increase_percentage
FROM employees;

-- Find employees whose salary decreased from previous hire
SELECT emp_name, hire_date, salary, prev_salary
FROM (
    SELECT emp_name,
           hire_date,
           salary,
           LAG(salary) OVER(ORDER BY hire_date) as prev_salary
    FROM employees
) comparison
WHERE salary < prev_salary;
```

**Common Variation - Partition by Department:**
```sql
-- Compare salary with previous employee in SAME department
SELECT emp_name,
       dept_id,
       salary,
       LAG(salary) OVER(PARTITION BY dept_id ORDER BY hire_date) as prev_dept_salary
FROM employees;
```

## Question 4: FIRST_VALUE() and LAST_VALUE() - "Department Highest and Lowest Salary"

**The Problem:** Show each employee with their department's highest and lowest salary.

**The Solution:**
```sql
SELECT emp_name,
       dept_id,
       salary,
       FIRST_VALUE(salary) OVER(PARTITION BY dept_id ORDER BY salary DESC) as highest_in_dept,
       FIRST_VALUE(emp_name) OVER(PARTITION BY dept_id ORDER BY salary DESC) as highest_earner,
       LAST_VALUE(salary) OVER(PARTITION BY dept_id ORDER BY salary DESC 
                               ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING) as lowest_in_dept
FROM employees;

SELECT emp_name,
       dept_id,
       salary,

       MIN(salary) OVER(PARTITION BY dept_id) as lowest_in_dept
FROM employees;
```

```sql
-- This WON'T work as expected without frame clause
SELECT emp_name, salary,
       LAST_VALUE(salary) OVER(PARTITION BY dept_id ORDER BY salary) as last_value_wrong
FROM employees;  -- Returns current row's salary!

-- CORRECT way:
SELECT emp_name, salary,
       LAST_VALUE(salary) OVER(PARTITION BY dept_id ORDER BY salary
                               ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING) as last_value_correct
```

## Question 10: ROWS vs RANGE - "Understanding Window Frames"

**The Problem:** Understand the difference between ROWS and RANGE in window frames.

**The Solution with Examples:**
```sql
-- ROWS - Fixed number of rows
SELECT sale_date,
       amount,
       AVG(amount) OVER(ORDER BY sale_date 
                        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW) as rows_avg
FROM daily_sales;
-- Uses exactly 3 rows (2 previous + current), regardless of date gaps

-- RANGE - Based on value difference
SELECT sale_date,
       amount,
       AVG(amount) OVER(ORDER BY sale_date 
                        RANGE BETWEEN INTERVAL '7 days' PRECEDING AND CURRENT ROW) as range_avg
FROM daily_sales;
-- Uses all rows within last 7 days of current date

-- Practical example showing difference
-- Data: Jan 1, Jan 2, Jan 5, Jan 10 (missing Jan 3,4,6,7,8,9)

-- ROWS: Always uses exactly 2 previous rows
-- Jan 1: just Jan 1
-- Jan 2: Jan 1 + Jan 2
-- Jan 5: Jan 2 + Jan 5  (skips Jan 1)
-- Jan 10: Jan 5 + Jan 10

-- RANGE with 7 days:
-- Jan 1: just Jan 1
-- Jan 2: Jan 1 + Jan 2
-- Jan 5: Jan 1,2,5 (all within 7 days of Jan 5)
-- Jan 10: Jan 5,10 (Jan 1,2 are >7 days away)
```

**Default Frame Rules:**
```sql
-- ORDER BY without frame = RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
SUM(salary) OVER(ORDER BY hire_date)  -- Default frame

-- ORDER BY with PARTITION but no frame = same as above
SUM(salary) OVER(PARTITION BY dept_id ORDER BY hire_date)

-- No ORDER BY = entire partition
SUM(salary) OVER(PARTITION BY dept_id)  -- Whole department total
```

## Question 1: Basic CTE - "Simplify Complex Query"

```sql
FROM employees e
CROSS JOIN company_avg ca
INNER JOIN departments d ON e.dept_id = d.dept_id
GROUP BY e.dept_id, d.dept_name, ca.avg_salary
HAVING AVG(e.salary) > ca.avg_salary;
```

## Question 3: Recursive CTE - "Employee Hierarchy (Most Important!)"

**The Problem:** Display complete organizational hierarchy starting from CEO.

**Sample Data:**
```sql
CREATE TABLE employees (
    emp_id INT PRIMARY KEY,
    emp_name VARCHAR(100),
    manager_id INT,
    position VARCHAR(50)
);

INSERT INTO employees VALUES
(1, 'Alice CEO', NULL, 'CEO'),
(2, 'Bob VP', 1, 'VP'),
(3, 'Carol Director', 2, 'Director'),
(4, 'Dave Manager', 3, 'Manager'),
(5, 'Eve Lead', 4, 'Team Lead'),
(6, 'Frank Engineer', 5, 'Engineer'),
(7, 'Grace VP', 1, 'VP'),
(8, 'Henry Director', 7, 'Director');
```

**The Solution - Recursive CTE:**
```sql
WITH RECURSIVE emp_hierarchy AS (
    -- Anchor member: Start with CEO (top level)
    SELECT 
        emp_id,
        emp_name,
        manager_id,
        position,
        1 as level,
        CAST(emp_name AS VARCHAR(500)) as path,
        CAST(emp_id AS VARCHAR(100)) as id_path
    FROM employees
    WHERE manager_id IS NULL
    
    UNION ALL
    
    -- Recursive member: Find direct reports
    SELECT 
        e.emp_id,
        e.emp_name,
        e.manager_id,
        e.position,
        eh.level + 1,
        CAST(eh.path || ' -> ' || e.emp_name AS VARCHAR(500)) as path,
        CAST(eh.id_path || ',' || e.emp_id AS VARCHAR(100)) as id_path
    FROM employees e
    INNER JOIN emp_hierarchy eh ON e.manager_id = eh.emp_id
)
SELECT 
    level,
    REPEAT('  ', level - 1) || emp_name as indented_name,
    position,
    path
FROM emp_hierarchy
ORDER BY id_path;  -- Preserves hierarchy order
```

```
2     |   Bob VP           | VP
3     |     Carol Director | Director
4     |       Dave Manager | Manager
5     |         Eve Lead   | Team Lead
6     |           Frank Engineer | Engineer
2     |   Grace VP         | VP
```

## Question 4: Recursive CTE - "Find All Subordinates"

**The Problem:** Find all employees who report directly or indirectly to a specific manager.

## Question 7: CTE for Pivoting Data - "Department Employee Count"

```sql
    FROM sales s
    INNER JOIN products p ON s.product_id = p.product_id
    GROUP BY p.product_id, p.product_name, p.category_id, DATE_TRUNC('month', s.sale_date)
),
category_averages AS (
    -- Level 2: Calculate category averages
    SELECT 
        category_id,
        sale_month,
        AVG(total_sales) as category_avg_sales
    FROM monthly_product_sales
    GROUP BY category_id, sale_month
),
product_performance AS (
    -- Level 3: Add performance metrics
    SELECT 
        mps.*,
        ca.category_avg_sales,
        CASE 
            WHEN mps.total_sales > ca.category_avg_sales * 2 THEN 'Star'
            WHEN mps.total_sales > ca.category_avg_sales THEN 'Good'
            WHEN mps.total_sales > ca.category_avg_sales * 0.5 THEN 'Average'
            ELSE 'Poor'
        END as performance_category,
        LAG(mps.total_sales) OVER(PARTITION BY mps.product_id ORDER BY mps.sale_month) as prev_month_sales
    FROM monthly_product_sales mps
    INNER JOIN category_averages ca ON mps.category_id = ca.category_id 
                                    AND mps.sale_month = ca.sale_month
),
ranked_products AS (
    -- Level 4: Rank products
    SELECT 
        *,
        ROW_NUMBER() OVER(PARTITION BY sale_month ORDER BY total_sales DESC) as monthly_rank
    FROM product_performance
)
-- Final output
SELECT 
    sale_month,
    product_name,
    total_sales,
    category_avg_sales,
    performance_category,
    monthly_rank,
    CASE 
        WHEN prev_month_sales IS NULL THEN 'New Product'
        WHEN total_sales > prev_month_sales THEN 'Growing'
        ELSE 'Declining'
    END as trend
FROM ranked_products
WHERE monthly_rank <= 3  -- Top 3 products
   OR performance_category = 'Poor'  -- Or underperforming
ORDER BY sale_month DESC, total_sales DESC;
```

## Question 9: CTE with Window Functions - "Employee Performance Ranking"

**The Problem:** Rank employees within department and globally, with running totals.

**The Solution:**
```sql
WITH dept_stats AS (
    -- Department level statistics
    SELECT 
        dept_id,
        AVG(salary) as dept_avg,
        MAX(salary) as dept_max,
        MIN(salary) as dept_min,
        COUNT(*) as dept_size
    FROM employees
    GROUP BY dept_id
),
emp_with_context AS (
    -- Add department context to each employee
    SELECT 
        e.emp_id,
        e.emp_name,
        e.dept_id,
        e.salary,
        e.hire_date,
        ds.dept_avg,
        ds.dept_max,
        ds.dept_min,
        ds.dept_size,
        -- Global ranking
        RANK() OVER(ORDER BY e.salary DESC) as global_rank,
        -- Department ranking
        RANK() OVER(PARTITION BY e.dept_id ORDER BY e.salary DESC) as dept_rank,
        -- Running total by department
        SUM(e.salary) OVER(PARTITION BY e.dept_id ORDER BY e.hire_date) as dept_running_total
    FROM employees e
    INNER JOIN dept_stats ds ON e.dept_id = ds.dept_id
),
performance_tiers AS (
    -- Create performance tiers
    SELECT 
        *,
        CASE 
            WHEN dept_rank <= dept_size * 0.1 THEN 'Top 10%'
            WHEN dept_rank <= dept_size * 0.25 THEN 'Top 25%'
            WHEN dept_rank <= dept_size * 0.5 THEN 'Top 50%'
            WHEN dept_rank <= dept_size * 0.75 THEN 'Bottom 50%'
            ELSE 'Bottom 25%'
        END as performance_tier,
        salary - dept_avg as diff_from_dept_avg
    FROM emp_with_context
)
SELECT 
    emp_name,
    dept_id,
    salary,
    dept_avg,
    diff_from_dept_avg,
    dept_rank,
    global_rank,
    performance_tier,
    dept_running_total
FROM performance_tiers
WHERE global_rank <= 10  -- Top 10 globally
   OR performance_tier = 'Bottom 25%'  -- Or bottom 25% in department
ORDER BY dept_id, dept_rank;
```

## Question 10: Recursive CTE - "Generate Date Series / Calendar"

**The Problem:** Generate all dates between two dates (fill missing dates in reports).

**The Solution:**
```sql
-- Generate all dates in a range
WITH RECURSIVE date_series AS (
    -- Start date
    SELECT '2024-01-01'::DATE as date
    UNION ALL
    -- Add one day recursively
    SELECT date + INTERVAL '1 day'
    FROM date_series
    WHERE date < '2024-12-31'
)
SELECT * FROM date_series;

-- Practical use: Fill missing dates in sales data
WITH RECURSIVE calendar AS (
    SELECT MIN(sale_date) as date, MAX(sale_date) as max_date
    FROM sales
    UNION ALL
    SELECT date + INTERVAL '1 day', max_date
    FROM calendar
    WHERE date < max_date
),
daily_sales AS (
    SELECT 
        sale_date,
        SUM(amount) as total_sales
    FROM sales
    GROUP BY sale_date
)
SELECT 
    c.date,
    COALESCE(ds.total_sales, 0) as sales_amount,
    CASE 
        WHEN ds.total_sales IS NULL THEN 'No Sales'
        WHEN ds.total_sales > AVG(ds.total_sales) OVER(ORDER BY c.date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) THEN 'Above Average'
        ELSE 'Below Average'
    END as performance
FROM calendar c
LEFT JOIN daily_sales ds ON c.date = ds.sale_date
ORDER BY c.date;

-- Generate business days only (skip weekends)
WITH RECURSIVE business_days AS (
    SELECT '2024-01-01'::DATE as date
    UNION ALL
    SELECT date + INTERVAL '1 day'
    FROM business_days
    WHERE date < '2024-12-31'
)
SELECT date
FROM business_days
WHERE EXTRACT(DOW FROM date) NOT IN (0, 6)  -- 0=Sunday, 6=Saturday
ORDER BY date;
```

## Quick Reference Card - When to Use What

### Window Functions
```sql
-- ROW_NUMBER() - Unique numbering
ROW_NUMBER() OVER(PARTITION BY category ORDER BY value)

-- RANK() / DENSE_RANK() - Ranking with ties
RANK() OVER(ORDER BY score DESC)  -- Skips numbers
DENSE_RANK() OVER(ORDER BY score DESC)  -- No skips

-- LAG() / LEAD() - Previous/next row access
LAG(value) OVER(ORDER BY date)
LEAD(value) OVER(ORDER BY date)

-- SUM/AVG as window - Running totals/moving averages
SUM(value) OVER(ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW)

-- NTILE() - Percentiles/quartiles
NTILE(4) OVER(ORDER BY value)  -- Quartiles
```

### CTEs
```sql
-- Simple CTE - Complex calculations
WITH calc AS (SELECT ...)
SELECT * FROM calc

-- Multiple CTEs - Breaking down complex logic
WITH cte1 AS (...), cte2 AS (...)
SELECT * FROM cte1 JOIN cte2

-- Recursive CTE - Hierarchies, trees, graphs
WITH RECURSIVE cte AS (
    SELECT ... WHERE parent IS NULL
    UNION ALL
    SELECT ... JOIN cte ON child.parent = cte.id
)

-- CTE for updates/deletes - Safe batch operations
WITH to_delete AS (SELECT id FROM ... WHERE condition)
DELETE FROM table WHERE id IN (SELECT id FROM to_delete)
```
# ACID Properties - Complete Explanation for Interviews

## Overview
**ACID** is a set of properties that guarantee database transactions are processed reliably. It's a fundamental concept that interviewers LOVE to ask because it separates junior developers from senior ones.

## The 4 Properties

### 1. ATOMICITY (All or Nothing)

**Definition:** A transaction is an indivisible unit - either ALL operations complete successfully, or NONE of them take effect.

**Real-world analogy:** Transferring money from savings to checking. If the power fails mid-transfer, you don't want money debited from savings but never credited to checking.

**Example:**
```sql
-- Atomic transaction - either both updates happen or neither
BEGIN TRANSACTION;
    
    UPDATE accounts 
    SET balance = balance - 1000 
    WHERE account_id = 'SAVINGS_001';
    
    UPDATE accounts 
    SET balance = balance + 1000 
    WHERE account_id = 'CHECKING_001';
    
    -- If either update fails, the entire transaction rolls back
    -- If power fails here, nothing changes
    
COMMIT;  -- Only now are changes permanent
```

**What happens without Atomicity:**
```sql
-- BAD - No transaction wrapper
UPDATE accounts SET balance = balance - 1000 WHERE account_id = 'SAVINGS_001';
-- If system crashes here, money is LOST forever!
UPDATE accounts SET balance = balance + 1000 WHERE account_id = 'CHECKING_001';
```

**Interview Answer:** "Atomicity ensures that a transaction is treated as a single, indivisible unit. If any part fails, the entire transaction is rolled back. This is implemented through write-ahead logging where changes are first recorded in a transaction log before being applied to the actual database."

---

### 2. CONSISTENCY (Data Integrity)

**Definition:** A transaction brings the database from one valid state to another, preserving all defined rules, constraints, and triggers.

**Types of Consistency:**
1. **Declarative Consistency** (Constraints)
2. **Application Consistency** (Business rules)
3. **Referential Integrity** (Foreign keys)

**Example with Constraints:**
```sql
-- Creating constraints that enforce consistency
CREATE TABLE accounts (
    account_id VARCHAR(20) PRIMARY KEY,
    account_holder VARCHAR(100) NOT NULL,
    balance DECIMAL(10,2) NOT NULL,
    account_type VARCHAR(20) CHECK (account_type IN ('Savings', 'Checking', 'Credit')),
    overdraft_limit DECIMAL(10,2),
    
    -- Business rule: Credit cards can have negative balance, others cannot
    CONSTRAINT chk_positive_balance 
        CHECK (
            (account_type = 'Credit' AND balance >= -overdraft_limit) 
            OR 
            (account_type != 'Credit' AND balance >= 0)
        )
);

-- This transaction will FAIL because it violates consistency
BEGIN TRANSACTION;
    UPDATE accounts 
    SET balance = -500 
    WHERE account_id = 'SAVINGS_001';  -- Savings can't go negative!
    -- Database automatically rejects this
COMMIT;  -- Never reaches here
```

**Foreign Key Consistency:**
```sql
-- This maintains referential integrity
CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    order_date DATE,
    FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
);

-- Without consistency, we could have orders for non-existent customers
-- With consistency, this is prevented automatically
```

**Interview Answer:** "Consistency ensures that any transaction brings the database from one valid state to another. All constraints, triggers, and cascades are enforced. It's the 'C' in ACID that separates a database from a simple file system - the database won't allow invalid data even if the application tries to insert it."

---

### 3. ISOLATION (Concurrency Control)

**Definition:** Concurrent transactions execute as if they were running sequentially, without interfering with each other.

**Isolation Levels (from weakest to strongest):**

#### Level 1: READ UNCOMMITTED (Dirty Reads allowed)
```sql
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;

-- Transaction 1 (not yet committed)
BEGIN TRANSACTION;
    UPDATE employees SET salary = 100000 WHERE emp_id = 1;
    -- Still in transaction, not committed
    
-- Transaction 2 (READ UNCOMMITTED can see uncommitted data!)
SELECT salary FROM employees WHERE emp_id = 1;  -- Sees 100000 (dirty read!)
-- If Transaction 1 rolls back, Transaction 2 has wrong data
```

#### Level 2: READ COMMITTED (Default in many DBs)
```sql
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- Transaction 1
BEGIN TRANSACTION;
    UPDATE employees SET salary = 100000 WHERE emp_id = 1;
    -- Not committed yet
    
-- Transaction 2 (can't see uncommitted data)
SELECT salary FROM employees WHERE emp_id = 1;  -- Sees OLD salary
COMMIT;

-- Transaction 1 now commits
COMMIT;

-- Transaction 2 sees new value only on next query
```

#### Level 3: REPEATABLE READ (Prevents non-repeatable reads)
```sql
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- Transaction 1
BEGIN TRANSACTION;
    SELECT salary FROM employees WHERE emp_id = 1;  -- Gets 50000
    
    -- Transaction 2 runs and updates to 60000
    -- UPDATE employees SET salary = 60000 WHERE emp_id = 1;
    -- Transaction 2 commits
    
    SELECT salary FROM employees WHERE emp_id = 1;  -- Still gets 50000!
    -- Same value throughout transaction
COMMIT;
```

#### Level 4: SERIALIZABLE (Highest isolation)
```sql
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- Prevents all anomalies:
-- - Dirty reads
-- - Non-repeatable reads  
-- - Phantom reads (new rows appearing)
-- Essentially runs transactions one after another
```

**Isolation Anomalies Table:**

| Anomaly | Description | Prevented By |
|---------|-------------|---------------|
| Dirty Read | Read uncommitted data | READ COMMITTED+ |
| Non-repeatable Read | Same query returns different data | REPEATABLE READ+ |
| Phantom Read | New rows appear in range | SERIALIZABLE |

**Practical Example - Lost Update Problem:**
```sql
-- Without proper isolation
-- User 1: View balance (5000)
-- User 2: View balance (5000)
-- User 1: Add 1000 (balance becomes 6000)
-- User 2: Add 2000 (balance becomes 7000) - User 1's update LOST!

-- With proper isolation:
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- User 1
BEGIN TRANSACTION;
    SELECT balance INTO @bal FROM accounts WHERE id = 1;  -- Gets 5000
    -- Database locks the row
    
-- User 2's transaction would wait until User 1 completes
```

**Interview Answer:** "Isolation controls how transactions interact. The trade-off is between consistency and performance - higher isolation means more locking and potential deadlocks but better data accuracy. Most production systems use READ COMMITTED as a balance, but financial systems often require SERIALIZABLE."

---

### 4. DURABILITY (Persistence)

**Definition:** Once a transaction is committed, its changes persist permanently, even after system failures, crashes, or power outages.

**How Durability is Achieved:**

```sql
-- When you COMMIT, here's what happens:

BEGIN TRANSACTION;
    UPDATE accounts SET balance = 1000 WHERE id = 1;
    
    -- 1. Changes written to Transaction Log (on disk)
    -- 2. Log entry flushed to disk (fsync)
    -- 3. Acknowledgment received
    
COMMIT;  -- 4. Only now does the transaction complete

-- Even if power fails immediately after COMMIT:
-- - On restart, database reads transaction log
-- - Replays committed transactions
-- - Rollback uncommitted ones
```

**Write-Ahead Logging (WAL):**
```sql
-- Conceptual WAL entries
-- [LSN: 001] BEGIN TRANSACTION T1
-- [LSN: 002] T1: UPDATE accounts SET balance=1000 WHERE id=1 (old=500, new=1000)
-- [LSN: 003] T1: UPDATE accounts SET balance=2000 WHERE id=2 (old=1500, new=2000)
-- [LSN: 004] COMMIT T1

-- After crash recovery:
-- - Reads log from last checkpoint
-- - Applies all committed changes (T1)
-- - Discards uncommitted changes
```

**Durability Mechanisms by Database:**

| Database | Durability Mechanism |
|----------|---------------------|
| PostgreSQL | Write-Ahead Log (WAL) + Checkpoints |
| MySQL/InnoDB | Redo Log + Doublewrite Buffer |
| SQL Server | Transaction Log + Checkpoints |
| MongoDB | Journaling + Checkpoints |

**Performance vs Durability Trade-off:**
```sql
-- Maximum durability (slower but safer)
SET synchronous_commit = ON;  -- PostgreSQL
SET TRANSACTION DURABILITY = FULL;  -- Some DBs

-- Reduced durability for speed (risk losing last few transactions)
SET synchronous_commit = OFF;  -- PostgreSQL
-- Risk: Could lose up to 500ms of transactions if power fails

-- You can also delay durability
BEGIN TRANSACTION;
    -- Do work
COMMIT WITH DELAYED_DURABILITY;  -- SQL Server
```

**Interview Answer:** "Durability guarantees that committed transactions survive system failures. This is implemented through write-ahead logging - changes are written to a transaction log on disk BEFORE the actual data pages. On recovery, the database replays committed transactions from this log and rolls back incomplete ones. Some systems offer a trade-off where you can reduce durability for better performance."

## Complete ACID Example in One Transaction

```sql
-- A banking transaction showing ALL ACID properties

CREATE PROCEDURE TransferMoney(
    @from_account VARCHAR(20),
    @to_account VARCHAR(20),
    @amount DECIMAL(10,2)
)
AS
BEGIN
    SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;  -- ISOLATION
    
    BEGIN TRANSACTION;  -- ATOMICITY start
        
        -- CONSISTENCY check
        IF (SELECT balance FROM accounts WHERE account_id = @from_account) < @amount
        BEGIN
            ROLLBACK;
            RAISERROR('Insufficient funds', 16, 1);
            RETURN;
        END
        
        -- Withdraw
        UPDATE accounts 
        SET balance = balance - @amount 
        WHERE account_id = @from_account;
        
        -- Deposit
        UPDATE accounts 
        SET balance = balance + @amount 
        WHERE account_id = @to_account;
        
        -- Log transaction (for DURABILITY)
        INSERT INTO transaction_log (from_acc, to_acc, amount, timestamp)
        VALUES (@from_account, @to_account, @amount, GETDATE());
        
    COMMIT TRANSACTION;  -- ATOMICITY end, DURABILITY guaranteed
END;
```

## Common Interview Questions & Answers

**Q: Can you have ACID without all four properties?**
A: "Technically no - ACID is a set. However, some NoSQL databases offer BASE (Basically Available, Soft state, Eventual consistency) which sacrifices Consistency for Availability, following the CAP theorem."

**Q: Which is most important?**
A: "It depends on the use case. Banking/finance needs all four equally. Social media might sacrifice Isolation for performance. Logging systems might sacrifice Durability. But traditional RDBMS aim for full ACID compliance."

**Q: How does isolation affect deadlocks?**
A: "Higher isolation levels use more locks, increasing deadlock probability. SERIALIZABLE has the highest deadlock risk but strongest consistency."

**Q: Can you demonstrate a dirty read scenario?**
```sql
-- Session 1 (READ UNCOMMITTED)
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
SELECT * FROM accounts WHERE id = 1;  -- Shows uncommitted data
-- This is dangerous for financial systems!
```

**Q: How do you balance ACID vs Performance?**
A: "Choose appropriate isolation levels per transaction, use optimistic locking when possible, consider READ COMMITTED SNAPSHOT for reporting, and remember that full ACID isn't always necessary - some systems can use BASE model."
